이번 locus 서비스를 개발하면서 사용자들이 작성한 기록들을 효과적으로 찾을 수 있는 검색 기능이 필요하다고 생각했습니다.
처음에는 <span class="text-blue">**`PostgreSQL`**</span>의 기본 쿼리만으로도 충분하다고 생각했지만 실제 요구사항을 구체화하면서 전문 검색 엔진인 **`Elasticsearch`**를 사용하기로 결정을 했고 그 도입 배경과 설계 과정을 한 번 소개해보고자 합니다.

## 🔍 검색 기능의 요구사항
> **무엇을 검색해야 하는가?**

저희는 사용자들이 자신이 작성한 수많은 기록 중에서 특정 키워드를 통해 원하는 내용을 빠르게 찾고 싶어 할 것이라 생각했습니다.
- 특히 기록들이 많아질 수록 이 기능은 필수!

예를 들어
- `카페`라는 키워드로 방문했던 카페 기록들을 찾기
- `성수동`이라는 장소명으로 해당 지역 기록 검색

## ⭐ 빠르고 정확해야 한다
검색 기능에 대한 목표를 세우면서 두 가지 핵심 가치를 설정했습니다.
### **속도**
구글의 연구 자료에 따르면 웹페이지가 3초 이내에 로드되지 않으면 모바일 사용자의 53%가 이탈한다고 합니다.
또한 반환되는 결과는 이상적으로 250ms 이내 늦어도 2초 이내(2-second rule)에는 제공되어야 한다고 합니다.
- 이렇게 사용자가 검색어를 입력하고 기다리는 시간이 길어질수록 서비스에 대한 신뢰도가 떨어질 수 있는 문제가 있을 것이라 생각

### **정확성**
검색 결과의 정확성도 중요하다고 생각습니다.
특히 중요하게 생각한 것은 `Recall(재현율)` 이였습니다.
> 즉, 관련된 기록을 놓치지 않고 모두 보여주는 것이 우선


또한 검색 결과에는 우선순위가 있어야 한다고 생각했습니다.
같은 `카페`라는 키워드라도 아래와 같은 우선순위가 있는 것이죠.
```json
제목 > 태그 > 장소이름 > 본문
```


이는 사용자가 정보를 기록할 때의 의도를 반영한 것입니다. 
**제목**은 기록의 핵심 요약이고 **태그**는 의도적인 분류이며 **장소 이름**은 방문한 곳의 명확한 식별자이며 **본문**은 전체 내용을 담고 있기 때문으로 이런 식으로 우선순위를 설정했습니다.

## 🤔 PostgreSQL vs Elasticsearch
### PostgreSQL FTS를 사용하지 않은 이유
저희는 주 DB로 <span class="text-blue">`PostgreSQL`</span>을 사용하고 있는데요.
<span class="text-blue">`PostgreSQL`</span>도 `Full-Text Search(FTS)` 기능을 제공합니다.
이미 사용하고 있는 데이터베이스에서 검색도 처리할 수 있다면 인프라 관리 측면에서 간단하지 않을까? 라고 생각을 했지만 몇 가지 단점이 있었습니다.


| 항목 | PostgreSQL FTS | Elasticsearch |
| ---- | ---- | ---- |
| 전문 검색 | 제한적 | 강력 |
| 가중치(Boost) | 제한적 | 유연한 설정 가능 |
| 검색 결과 하이라이트 | ❌ | ⭕ |
| 확장성 | 보통 | ⭕ |
| 한글 분석기 | 제한적 | nori 등 다양 |

특히 **필드별 가중치 설정**과 **한글 형태소 분석** 측면에서 <span class="text-red">Elasticsearch</span>가 훨씬 효율적이라고 생각했습니다.

### 지도 기반 조회는 PostgreSQL로
그렇지만 모든 `조회 / 검색` 기능을 **Elasticsearch**로 처리하지는 않기로 했습니다.

**지도 영역 기반 기록 조회** 기능의 경우
- 사용자가 지도에서 보고 있는 영역(Bounding Box) 안의 모든 기록을 조회
- 키워드 검색 없이 순수한 위치 필터링

이런 요구사항은 사실 <u>**"검색" **</u>이 아니라 <u>**"위치 기반 필터링 + 목록 조회"**</u> 에 가깝습니다.
PostgreSQL의 <span class="text-brown">`PostGIS`</span> 확장이 공간 인덱스를 제공하기 때문에 Bounding Box의 조회가 충분히 빠릅니다.

## 👷‍♂️ 아키텍처 설계
### 데이터베이스 역할 분리
```json
PostgreSQL: Source of Truth (정합성과 영속성)
    ↓
   동기화
    ↓
Elasticsearch: Search Index (검색 성능과 유연한 쿼리)
```



| PostgreSQL | Elasticsearch |
| ---- | ---- |
| 모든 데이터의 Source of Truth | 검색 전용 인덱스 |
| 트랜잭션 보장 | 빠른 전문 검색 |
| 데이터 정합성 보장 | 유연한 쿼리 |
| 영구 저장소 역할 | 필드별 가중치 적용 |

### ✅ 데이터 동기화
이렇게 역할을 분리하다보니 이런 고민이 생겼습니다.
> **PostgreSQL과 Elasticsearch의 데이터를 어떻게 동기화할 것인가?**


사용자가 기록을 <u>**생성하거나 수정, 삭제**</u>할 때마다 PostgreSQL뿐만 아니라 Elasticsearch의 인덱스도 함께 업데이트되어야 합니다.

만약 동기화가 제대로 되지 않는다면
- 새로 작성한 기록이 검색 결과에 나타나지 않음
- 수정한 내용이 반영되지 않음
- 삭제한 기록이 여전히 검색

이런 상황이 발생할 수 있고 이는 사용자 경험을 크게 해치고 서비스에 대한 신뢰를 떨어뜨릴 수 있습니다.

## 🔄 동기화 전략 수립
### 동기화 목표 정의
데이터 동기화 방식을 설계하기 전에 명확한 목표를 세우고 진행하고자 했습니다.

1️⃣ **데이터 신뢰성 (Reliability)**
<span class="text-blue">`PostgreSQL`</span>과 `Elasticsearch`의 데이터 상태가 일치해야 합니다. 

2️⃣ **시스템 안정성 (Stability)**
네트워크 장애나 **Elasticsearch** 다운 시에도 서비스는 정상 작동해야 합니다..
- 따라서 ES가 다운되어도 사용자는 기록을 <u>**생성/수정/삭제**</u> 이 가능해야 합니다.

3️⃣** 성능 (Performance)**
동기화 처리가 메인 API 응답 시간에 영향을 주면 안 된다고 생각했습니다. 

### 동기냐 비동기냐
가장 먼저 고민한 것은 **동기화를 동기적으로 처리할 것인가 비동기적으로 처리할 것인가**였습니다.

⚠️ **동기 방식의 문제점**
```typescript
await prisma.record.create(...)      // PostgreSQL에 저장
await elasticsearch.index(...)        // Elasticsearch에 즉시 인덱싱
```


이렇게 처리하면 간단해 보이지만 여러 문제가 있었습니다.
1️⃣ **트랜잭션 경계가 불명확**
> **Elasticsearch** 저장이 실패하면 PostgreSQL을 롤백해야 할까요?

- **Elasticsearch** 장애는 검색 문제일 뿐 비즈니스 데이터 정합성 문제는 아닌데..

2️⃣ **성능 저하**
-  **Elasticsearch** 응답을 기다리는 동안 사용자도 기다려야 합니다.

3️⃣ **장애 전파**
- **Elasticsearch**가 다운되면 기록 생성 자체가 불가능해집니다.


✅ **비동기 방식**
사용자 관점에서 생각해보면 기록을 작성한 직후에 바로 검색해서 확인하는 경우는 많이 없을것이라 생각했습니다. 
대부분 기록 상세 페이지를 보거나 현재 지도내 목록을 확인하죠.
따라서 **Eventually Consistent**를 허용하기로 했습니다!!
- 데이터는 바로 즉시 정확하게 PostgreSQL에 저장
- 검색에 반영되는 것은 몇 초 지연 허용 - Elasticsearch

이처럼 검색 결과가 몇 초 늦게 나오는 것은 치명적이지 않지만 데이터 유실이나 누락은 안되게 처리를 해주어야 합니다.

## 🖐️ 마무리
이렇게 Elasticsearch 도입을 결정하고 비동기 동기화 전략을 간단히 설계해보았습니다. 이후에는 실제로 Elasticsearch의 인덱스를 어떻게 설계했는지 그리고 키워드 검색을 어떻게 구현했는지 간단한 코드와 함께 살펴보고 동기화 과정도 살펴보고자 합니다.
