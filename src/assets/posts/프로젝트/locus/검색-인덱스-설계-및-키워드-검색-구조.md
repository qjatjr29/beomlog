이전 글에서는 Elasticsearch를 도입한 배경과 PostgreSQL과의 역할 분리 그리고 비동기 동기화 전략을 선택한 이유를 다뤘는데요!! 
이번에는 실제로 Elasticsearch 인덱스를 어떻게 설계하고 검색 쿼리를 어떻게 구현했는지 정리해보려고 합니다.

## 🔖 Elasticsearch 인덱스 설계
### 검색 대상 필드 정의
먼저 기록(Record)에서 어떤 정보를 검색 가능하게 만들지 결정해야 해 사용자가 기록을 찾을 때 실제로 기억하는 것들을 생각해봤습니다.
- **제목**: "성수동 카페 투어"
- **태그**: [카페, 브런치, 데이트]
- **장소 이름**: "테라로사 성수점"
- **본문 내용**: 성수동에서 커피를 마시다 니 생각이 났다.

저희는 이 네 가지 필드를 검색 대상으로 설정했고 추가로 검색 필터링을 위해 사용자 ID, 기록 ID, 이미지 포함 여부, 즐겨찾기 여부 등의 메타데이터도 함께 저장하기로 했습니다.

## ↗️ 검색 우선순위 전략
### 필드별 가중치(Boost) 설정
저희는 모든 필드가 동일한 중요도를 가지지는 않는다고 생각했습니다.
같은 `카페`라는 키워드라도 어디에 있느냐에 따라 의미가 다르기 때문입니다.

따라서 저희는 검색 대상에 대해서 아래와 같은 우선순위와 가중치를 설정해주었어요.

| **우선순위** | **필드명** | **가중치** | **설정 이유** |
| ---- | ---- | ---- | ---- |
| **1순위** | **제목 (Title)** | **3.0** | 사용자가 정의한 기록의 **핵심 요약**. 검색어와 일치할 경우 가장 높은 관련성 |
| **2순위** | **태그 (Tag)** | **2.0** | 사용자가 의도적으로 직접 분류한** 메타데이터**.  |
| **3순위** | **장소명 (Place)** | **2.0** | 방문지의 **공식 이름**. 사용자가 특정 장소를 기억할 때 자주 사용하는 정보 |
| **4순위** | **본문 (Content)** | **1.0** | 기록의 상세 내용. 키워드가 포함될 확률은 높으나 **단순 언급**일 가능성을 고려 |

### Multi-Match 쿼리 사용
이렇게 Elasticsearch에서 하나의 키워드를 여러 필드에서 동시에 검색하기 위해서 `multi_match` 쿼리를 사용했습니다.
```typescript
multi_match: {
    query: keyword,
    fields: RECORD_SEARCH_FIELDS,
    type: 'best_fields',
},
```

이를 통해서 "카페"로 검색하면
- title 필드에서 "카페" 검색 → **점수 × 3**
- tags 필드에서 "카페" 검색 → **점수 × 2**
- placeName 필드에서 "카페" 검색 →** 점수 × 2**
- content 필드에서 "카페" 검색 → **점수 × 1**
최종적으로 가장 높은 점수를 받은 기록부터 정렬되어 반환되는 형태입니다.

이 과정에서 **Multi-Match **의 **best_fields** 타입을 선택했는데요! 
> **여러 필드 중 가장 잘 일치하는 필드를 기준으로 점수를 계산**하는 방식입니다.


기록의 제목, 태그, 장소명, 본문은 서로 연관된 정보로 **"성수동 카페 투어" **라는 제목의 기록이라면 태그에도 **카페**가 있고 본문에도 **카페** 이야기가 있을 가능성이 높다고 생각했습니다.
따라서 여러 필드 중 **가장 잘 일치하는 필드를 기준으로 점수를 계산**하기 위해서 `best_fields` 타입을 선택했습니다.
- 제목에 "카페"가 딱 한 번 나오는 기록
- 본문에 "카페"가 3번 나오지만 제목에는 없는 기록
이 두 기록을 비교하면 첫 번째 기록이 더 관련도가 높다고 판단!!

## 🧩 필터링과 검색의 분리
```typescript
bool: {
   must: [
      {
        multi_match: {
          query: keyword,
          fields: RECORD_SEARCH_FIELDS,
          type: 'best_fields',
        },
      },
    ],
    filter: filters,
  }
//...
```

저희가 사용한 검색 쿼리는 크게 두 부분으로 나뉩니다

1️⃣ **Must (검색 조건)**
- 키워드 매칭
- 검색 점수(relevance score)에 영향을 줌
- 점수가 높을수록 상위에 노출

2️⃣ **Filter (필터 조건)**
- 태그 선택, 이미지 포함 여부, 즐겨찾기 여부
- 검색 점수에 영향을 주지 않음
- 단순히 조건에 맞는 것만 걸러냄

이렇게 분리하여 사용자가 원하는 조건을 명확하게 적용할 수 있도록 했습니다.

## 📌 CRUD 작업과 인덱스 관리
### 1️⃣ 기록 생성
새로운 기록 생성 요청이 오면 **Elasticsearch**에도 같은 ID로 문서를 추가합니다.
**PostgreSQL**의 레코드 ID와 **Elasticsearch**의 문서 ID를 동일하게 유지해서 나중에 업데이트나 삭제가 쉽도록 했습니다.

### 2️⃣ 기록 수정
제목이나 내용이 수정되면 Elasticsearch 문서도 업데이트합니다. 만약 Elasticsearch에 해당 문서가 없다면(404 에러) 동기화가 어긋난 상태이므로 별도 처리가 필요합니다.

### 3️⃣ 부분 업데이트 최적화
이때, **모든 필드가 검색과 관련된 것은 **아니였기 때문에 변경된 필드만 부분 업데이트 하는 것이 필요했습니다.
- 즐겨찾기 여부: 검색 결과에는 영향 없지만 필터링에는 필요
- 기록의 연결 수 변경: 검색과는 무관
이런 경우 전체 문서를 다시 인덱싱하는 대신 **변경된 필드만 부분 업데이트**하도록 구현했습니다. 
이를 통해 불필요한 네트워크 트래픽과 Elasticsearch 부하를 줄이고자 했습니다.

### 4️⃣ 기록 삭제
삭제된 기록은 Elasticsearch에서도 제거합니다.

## 🔽 페이지네이션 설계
### Search After 방식
검색 결과를 페이지로 나눌 때 `search_after` 방식을 사용했습니다.
저희는 검색 결과를 무한 스크롤 방식으로 사용자에게 보여주고 있었기에 search After 방식이 더 적합하다고 생각했습니다.

⚠️ **Offset 방식의 문제**
- 10,000번째 페이지를 보려면 10,000개를 건너뛰어야 함 **(Deep Paging 문제)**
- 깊은 페이지일수록 성능이 급격히 저하

👍 **Search After 방식의 장점**
- 마지막으로 본 문서의 정렬 값을 기준으로 다음 문서를 가져옴
- 페이지 깊이와 관계없이 일정한 성능 
- Cursor 기반 페이지네이션 적용
