5장에서는 동일한 데이터의 복사본 여러 개를 다른 노드에 저장하는 복제 개념을 다뤘다.
데이터 셋이 매우 크거나 질의 처리량이 매우 높다면 복제만으로는 부족하고 데이터를 <span class="text-blue">**파티션**</span>으로 쪼갤 필요가 있다.
이 작업을 <span class="text-red">**샤딩**</span>이라 한다.

🎯 **파티셔닝 목적 - 확장성**
각 노드에서 자신의 파티션에 해당하는 질의를 독립적으로 실행할 수 있어 노드를 추가함으로써 **질의 처리량**을 늘릴 수 있다.
- 여러 노드에서 **병렬 실행** 가능 

# 파티셔닝과 복제
보통 **복제와 파티셔닝을** 함께 적용해 각 파티션의 복사본을 여러 노드에 저장한다.
각 레코드는 정확히 한 파티션에 속하더라도 이를 여러 다른 노드에 저장해 `내결함성`을 보장할 수 있다는 의미이다.
# 키-값(Key-Value) 데이터 파티셔닝
**파티셔닝의 목적은 데이터와 질의 부하를 노드 사이에 고르게 분산시키는 것이다.**
파티셔닝이 고르게 이뤄지지 않아 부하가 고르게 분산되지 않고 특정 파티션에만 데이터나 질의가 몰리는 현상을 <span class="text-blue">**쏠림(Skew)**</span>이라고 하며 이렇게 불균형하게 부하가 높은 파티션을 <span class="text-red">**핫스팟(Hot spot)**</span>이라고 부른다.

핫스팟을 피하기 위해서 데이터를 <u>어떤 기준으로 어느 노드에 저장할지 결정</u>해야 한다.

가장 단순한 방법은 레코드를 **할당할 노드를 무작위로 선택**하는 것이다.
- 데이터가 노드들 사이에 매우 고르게 분산되지만 
- 어떤 레코드를 읽으려고 할 때 해당 레코드가 어느 노드에 저장됐는지 알 수 없어 **모든 노드에서 병렬적으로 질의를 실행**해야 하는 단점이 존재.

## 키 범위 기준 파티셔닝
파티셔닝하는 방법 중 하나는 백과사전처럼 각 파티션에 연속된 범위의 키를 할당하는 것
- ex: `A~B`, `C~D` 같은 연속된 범위의 키를 각 파티션에 할당

키 범위 크기는 데이터가 고르게 분포하지 않을 수 있기 때문에 동일할 필요는 없다.
- ex: 알파벳 두 글자마다 할당하면 다른 것들보다 커지는 것이 생길 수 있다.

각 파티션내에서는 키를 **정렬된 순서**로 저장할 수 있어 범위 스캔이 쉬워지고 키를 연쇄된 색인으로 간주해 질의 하나로 관련 레코드 여러 개를 읽어오는데 사용할 수 있다.

⚠️ **키 범위 기준 파티셔닝은 특정한 접근 패턴이 **<span class="text-red">**핫스팟**</span>**을 유발하는 단점이 있다.**
- ex: 키가 '날짜'일 경우 모든 새로운 쓰기 요청이 '오늘 날짜' 파티션 하나로만 집중되어 병목 발생
## 키의 해시값 기준 파티셔닝
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2Fc4beebdb-7d63-4a93-9181-72b276fcbe7a%2Fimage.png?table=block&id=340d67b9-9e80-806a-9e61-c75916f8d4dc&cache=v2" alt="image" width="1258" height="406" loading="lazy" />

<span class="text-blue">**쏠림**</span>과 <span class="text-red">**핫스팟**</span>의 위험 때문에 많은 분산 데이터스토어는 키의 파티션을 정하는데 `해시 함수`를 사용한다.
- 데이터를 입력으로 받아 균일하게 분산되게 한다.
각 파티션에 해시값 범위를 할당하고 해시값이 파티션의 범위에 속하는 모든 키를 그 파티션에 할당하면 된다.
이는 키를 파티션 사이에 균일하게 분산시키는 데 좋다. 
- 파티션 경계는 크기가 동일하도록 나눌 수도 있고 무작위에 가깝게 선택할 수도 있다.
- **일관성 해싱**

⚠️ **키들이 모든 파티션에 흩어져 정렬 순서가 유지되지 않아 **<span class="text-blue">**범위 질의**</span>**가 지원되지 않는다.**
### 카산드라 - 복합 기본키
카산드라는 두 가지 파티셔닝 전략(키 범위 기준, 해시값 기준)의 장점을 사용하기 위해 <span class="text-red">**복합 기본키**</span>라는 방식을 이용한다.
1. 기본키를 여러 컬럼의 조합으로 만든다 (ex: `user_id, timestamp`)
2. **파티셔닝(해시)**: 기본키의 첫 번째 부분(`user_id`)만 해시 함수를 적용해 데이터가 어느 파티션에 저장될지 결정
3. **정렬 (범위 스캔)**: 기본키의 나머지 부분(`timestamp`)은 해시하지 않고 데이터를 정렬하는 연쇄된 인덱스로 사용
따라서 복합 키의 첫 번째 칼럼에 대해서는 값 범위로 검색하는 질의를 사용할 수 없지만 첫 번째 칼럼에 고정된 값을 지정하면 키의 다른 칼럼에 대해서는 범위 스캔을 효율적으로 실행할 수 있다.
### 쏠린 작업 부하와 핫스팟 완화
키를 해실해서 파티션을 정하면 핫스팟을 줄일 수는 있지만 완벽히 제거할 수는 없다.
항상 **동일한 키를 읽고 쓰는 극단적인 상황에서는 모든 요청이 동일한 파티션**으로 쏠리게 된다.

> **상황**
> 수백만 명의 팔로워를 가진 유명인이 SNS에 글을 올리면 수많은 사람들이 동시에 해당 글에 좋아요를 누르고 댓글을 단다.
> 해시 함수를 거쳐봤자 동일한 ID는 동일한 파티션으로 가기 때문에 해당 노드가 터져버림

현대 데이터 시스템은 이를 자동으로 해결해주지 못해 애플리케이션에서 완화를 해주어야 한다.
간단한 해결책: 각 키의 시작이나 끝에 임의의 숫자를 붙이는 것
- ex: 유명인의 게시글 키가 `12345`라면 끝에 `00`~`99` 사이의 난수를 붙여 `12345-00`부터 `12345-99`까지 100개의 키로 분산시켜 저장
- 쓰기 부하는 100개로 쪼개져 분산되지만 나중에 이 데이터를 읽을 때는 100개의 키를 모두 조회해서 애플리케이션이 병합해야 하는 추가 작업이 발생(트레이드 오프)

# 파티셔닝과 보조 색인
지금은 레코드를 `기본키`를 통해서만 파티션을 결정하고 해당 키를 담당하는 파티션으로 읽기 쓰기 요청을 전달했다.
<span class="text-red">**보조 색인(Secondary Index)**</span>가 연관되면 상황은 복잡해진다.
> 📖 **보조 색인**
> - 보통 레코드를 유일하게 식별하는 용도가 아닌 특정한 값이 발생한 항목을 검색하는 수단


보조 색인은 파티션에 깔끔하게 대응되지 않는 문제점이 있다.
- 보조 색인이 있는 데이터베이스를 파티셔닝하는데 쓰이는 두 가지 방법이 있다.
## 문서 기준 보조 색인 파티셔닝
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2Fe3c35963-609e-43fb-8ae3-1ca15066799c%2Fimage.png?table=block&id=340d67b9-9e80-80e7-8363-d9ca7b4dd689&cache=v2" alt="image" width="1240" height="614" loading="lazy" />

각 파티션이 자신들의 문서들만의 보조 색인을 독립적으로 생성
각 파티션은 자신의 보조 색인을 유지하며 그 파티션에 속하는 문서만 담당한다.
따라서 데이터베이스에 문서 추가, 삭제, 갱신등 쓰기 작업을 실행할 때는 쓰려고 하는 문서 ID를 포함하는 파티션만 다루면 된다.
- **지역 색인(local index)**이라고도 한다.

⚠️ **스캐터 / 개더 (scatter/gather)**
특정 데이터를 찾을 때 (ex: 빨간색 자동차를 찾아라) **`모든 파티션`**으로 질의를 보내서(`scatter`) 얻은 결과를 모아야 한다.(`gather`)
- 여러 파티션에 질의를 병렬 실행하더라도 스캐터/개더는 꼬리 지연 시간 증폭이 발생하기 쉽다.
## 용어 기준 파티셔닝
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F693b0414-2c3f-458b-9370-503817b40480%2Fimage.png?table=block&id=340d67b9-9e80-80c1-812b-d2fda5c38ac9&cache=v2" alt="image" width="1258" height="556" loading="lazy" />

각 파티션이 자신만의 보조 색인(지역 색인)을 갖게 하는 대신 **모든 파티션의 데이터를 담당하는 전역 색인(global index)**을 만들 수도 있다.
- 한 노드에만 색인을 저장할 수는 없다. (해당 노드가 병목이 되어 파티셔닝의 목적을 해치기 때문)
모든 파티션에 있는 빨간색 자동차 정보는 색인에서 `color:red` 항목에 저장되지만 색깔 색인은 `a부터 r까지의 글자`로 시작하는 색깔은 `파티션 0`에 `s부터 z까지의 글자`로 시작하는 색깔은 `파티션 1`에 저장되도록 파티셔닝된다.
- 찾고자 하는 용어에 따라 색인의 파티션이 결정되므로 이런 식의 색인을 용어 기준으로 파티셔닝됐다고 한다.
- 여기서는 `color:red` 가 용어의 예시

문서 파티셔닝 색인에 비해 전역 색인이 갖는 이점은<span class="text-red">** 읽기가 효율적**</span>이라는 것이다.
- 모든 파티션에 스캐터/개더를 할 필요없이 원하는 용어를 포함하는 파티션에만 요청을 보내면 된다.
⚠️ 그렇지만 **전역 색인은 쓰기가 느리고 복잡하다는 단점이 있다.**
- 단일 문서를 쓸 때 해당 색인의 여러 파티션에 영향을 줄 수 있기 때문
- 문서 하나를 쓸 때 그 문서에 포함된 용어들이 여러 노드에 걸쳐 있다면 **여러 노드의 색인을 분산 트랜잭션으로 업데이트**. (보통 비동기로 처리)

# 파티션 재균형화
데이터가 많아지거나 트래픽이 늘어나면 장비를 추가해야 하고 장비가 고장 나면 교체해야 합니다. 이때 기존 노드들에 있던 데이터와 요청, 부하를 새로운 노드로 옮기는 과정을 **재균형화(Rebalancing)**라고 합니다.

**재균형화의 최소 요구사항**
1️⃣ 재균형화 후, 부하(데이터, 읽기/쓰기)가 노드 간에 **균등하게 분배**되어야 한다
2️⃣  재균형화가 진행되는 도중에도 데이터베이스는 **계속 읽기/쓰기 요청을 정상적으로 처리**해야 한다
3️⃣ 불필요한 네트워크/디스크 IO를 막기 위해, **꼭 필요한 데이터만 이동**해야 한다.

## 재균형화 전략
### ❌ **해시 모듈러 (해시 값에 모드 N 연산을 실행, ****`hash(key) % N`****)**
키의 해시값 기준으로 파티셔닝을 할 때는 사용 가능한 해시값 범위를 나누고 각 범위를 한 파티션에 할당하는 것이 최선이라고 했다.
🤔 **왜 ****`mod`**** 연산을 사용하지 않을까?**
만약 노드 개수로 나눈 나머지(`% N`)를 사용해 파티션을 배정하면 노드가 10대에서 11대로 늘어나는 순간 거의 모든 키의 위치가 바뀌고 이는 전체 데이터를 대이동시켜야 한다.
- 키가 자주 이동하면 재균형화 비용이 지나치게 커짐

### 파티션 개수 고정
파티션을 노드 대수보다 많이 만들고 각 노드에 여러 파티션을 할당하는 방법
- ex: 10대 노드에 1,000개 파티션, 각 노드마다 약 100개의 파티션 할당
새 노드가 추가되면 새 노드는 파티션이 균일하게 분배될 때 까지 기존 노드에서 파티션 몇개를 뺏어오고 노드가 제거되면 이 과정이 반대로 실행
### 동적 파티셔닝
키 범위 파티셔닝을 사용하는 데이터베이스는 파티션 경계와 개수가 고정되어 있는 것이 불편하기 때문에 파티션을 동적으로 만든다.
파티션 크기가 설정된 값을 넘어서면 파티션을 반으로 쪼개고 각각에 원래 파티션의 절반 정도의 데이터가 포함되도록 한다.
반대로 데이터가 작아지면 인접한 파티션과 합쳐질 수 있다.

동적 파티셔닝은 파티션 개수가 전체 데이터 용량에 맞춰 조정된다는 이점이 있다.
- 데이터 양이 작으면 파티션 개수가 적어도 되어 오버헤드도 작다.
⚠️ **빈 데이터베이스는 파티션 경계를 어디로 정해야 하는지에 관한 사전 정보가 없어 시작할 때는 파티션이 하나다.**
- 데이터 셋이 작을 때는 모든 쓰기 요청이 하나의 노드에서 실행되고 다른 노드들은 유휴 상태에 머물게 된다.
### 노드 비례 파티셔닝
**노드당 할당되는 파티션 개수를 고정한다.**
노드를 새로 추가하면 기존 파티션들중 무작위로 선택해 분할하고 각 분할된 파티션의 절반은 그대로 두고 다른 절반은 새 노드에 할당한다.

# 요청 라우팅
파티션들이 재균형화하고 나면 클라이언트는 **"내가 찾으려는 ****`foo`****라는 키가 지금 어느 IP 주소의 노드에 있는지"** 어떻게 알 수 있을까? 
이 문제는 데이터베이스에 국한되지 않은 더욱 일반적인 문제인 <span class="text-blue">**서비스 탐색(Service Discovery)**</span> 문제의 일종이다.

| **방식** | **설명** |
| ---- | ---- |
| **아무 노드나 접속** | 클라이언트가 **아무 노드에나 접속**하게 하고 그 노드가 자기가 가진 데이터면 처리하고 아니면 올바른 노드로 전달 |
| **라우팅 계층** | 클라이언트의 모든 요청을 **라우팅 계층**으로 먼저 보내고 라우팅 계층에서 각 요청을 처리할 노드를 알아내고 해당 노드로 요청을 전달 |
| **클라이언트가 캐싱** | 클라이언트가 파티셔닝 방법과 파티션이 어떤 노드에 할당됐는지 알게 하여 올바른 노드로 직접 접속 |

모든 경우에 **핵심 문제**는 `라우팅 결정을 내리는 구성요소`(노드 중 하나일 수도, 라우팅 계층일 수도, 클라이언트 일 수 있다)가 **노드에 할당된 파티션의 변경사항을 어떻게 아느냐** 이다.
- 참여하는 모든 곳에서 정보가 일치해야 하므로 다루기 어렵다.
- 그렇지 않다면 요청이 잘못된 노드로 전송되어 제대로 처리되지 않는다.

많은 분산 데이터 시스템은 클러스터 메타데이터를 추적하기 위해 `주키퍼` 같은 별도의 코디네이션 서비스를 사용한다.
각 노드는 주키퍼에 자신을 등록하고 주키퍼는 파티션과 노드 사이의 신뢰성 있는 할당 정보를 관리한다.
파티션에 변경사항이 생기면 주키퍼는 라우팅 계층에 알려서 라우팅 정보를 최신으로 유지할 수 있게 한다.
