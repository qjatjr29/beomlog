<span class="text-red">**복제**</span>란 네트워크로 연결된 여러 장비에 동일한 데이터의 복사본을 유지한다는 의미.

**데이터 복제가 필요한 이유**
1️⃣ **지연 시간 감소**
- 사용자와 지리적으로 가까운 노드에서 데이터를 제공해 지연 시간 감소
2️⃣ **고가용성**
- 한 장비에 장애가 발생하더라도 시스템이 지속해서 동작할 수 있게 해 가용성 증가
3️⃣ **확장성**
- 읽기 요청을 제공하는 장비를 확장하고 분산하여 읽기 처리량을 늘림.

복제중인 데이터가 시간이 지나도 변경되지 않는다면 복제는 쉽다
- 한번에 모든 노드에 데이터를 복사하면 끝.
복제에서 모든 어려움은 <span class="text-red">**데이터의 변경**</span> 처리에 있다.

> 노드 간 변경을 복제하기 위한 세 가지 인기 있는 방법인 `단일 리더`, `다중 리더`, `리더 없는 복제`를 살펴보겠다.


## 리더와 팔로워
<span class="text-blue">**복제 서버**</span>** - 데이터베이스의 복사본을 저장하는 각 노드**
🤔 다중 복제 서버를 사용할 때 **모든 복제 서버에 모든 데이터가 있다는 사실을 어떻게 보장**할 수 있을까??
모든 쓰기는 모든 복제 서버에서 처리되어야 한다. 그렇지 않으면 복제 서버는 동일한 데이터를 유지할 수 없다.

가장 일반적인 해결책은 `리더 기반 복제(leader-based replication)`이다.
복제 서버 중 하나를 <span class="text-blue">리더</span>로 지정
- 클라이언트가 데이터베이스에 쓰기할 때 요청을 리더에게 보내야 한다.
- 리더는 로컬 저장소에 새로운 데이터를 기록
다른 복제 서버는 <span class="text-blue">팔로워</span>라고 한다.
- 리더가 로컬 저장소에 새로운 데이터를 기록할 때마다 데이터 변경을 복제 로그 또는 변경 스트림의 일부로 팔로워에게 전송한다.
- 각 팔로워가 리더로부터 로그를 받으면 리더가 처리한 것과 <u>동일한 순서</u>로 모든 쓰기를 적용해 갱신

**읽기는 리더 또는 임의 팔로워에게 질의할 수 있지만 쓰기는 리더에게만 허용.**

### 동기식, 비동기식 복제
복제 시스템의 중요한 내용중 하나는 복제가 `동기`식으로 발생하는지 `비동기식`으로 발생하는지에 대한 여부이다.
<span class="text-red">**동기식**</span>
- 리더는 팔로워가 쓰기를 수신했는지 확인해줄 때까지 기다리고 확인이 끝나면 사용자에게 성공을 보고한 후 다른 클라이언트에게 해당 쓰기를 보여준다.

| 장점 | 단점 |
| ---- | ---- |
| 팔로워가 리더와 일관성 있게 최신 데이터 복사본을 가지는 것을 보장 | 동기 팔로워가 응답하지 않는다면 쓰기 처리가 불가능 |
| 리더가 작동하지 않아도 데이터는 팔로워에서 계속 사용할 수 있음 | 리더는 모든 쓰기를 `차단(block)`하고 동기 복제 서버가 다시 사용할 수 있을 때까지 기다려야 함. |
|  | 임의 한 노드 장애는 전체 시스템을 멈추게 함 |

보통 <u>**리더 기반 복제**</u>는 **완전히 **<span class="text-blue">**비동기식**</span>으로 구성
- 리더는 메시지를 전송하지만 팔로워의 응답을 기다리지 않는다.
- 팔로워에게 변경 내용을 적용하지만 얼마나 오래 걸릴지 보장할 수 없다.

| 장점 | 단점 |
| ---- | ---- |
| 모든 팔로워가 잘못되더라도 리더가 쓰기 처리를 계속 할 수 있음 | 리더가 잘못되고 복구할 수 없으면 팔로워에 아직 복제되지 않은 모든 쓰기는 유실 |

**반동기식**
- 하나의 팔로워만 동기식으로 유지하고 나머지는 비동기로 설정

### 새로운 팔로워 설정
때때로 복제 서버 수를 늘리거나 장애 노드의 대체를 위해 새로운 팔로워를 설정해야 한다.
- 새로운 팔로워가 리더의 데이터 복제본을 가지고 있는지 어떻게 보장??

데이터베이스를 <u>**잠가서**</u> 일관성 있게 만들 수 있지만 이는 `고가용성` 목표에 부합하지 않는 방식이다.

팔로워 설정은 대개 중단시간 없이 수행할 수 있다.
1. 데이터베이스를 잠그지 않고 리더의 데이터베이스 스냅샷을 일정 시점에 가져온다.
2. 스냅샷을 새로운 팔로워 노드에 복사한다.
3. 팔로워는 리더에 연결해 스냅샷 이후 발생한 모든 데이터 변경을 요청한다.
4. 팔로워가 스냅샷 이후 데이터 변경의 미처리 부분을 모두 처리했을 때 따라잡았다고 말한다.

### 노드 중단 처리
시스템의 모든 노드는 장애로 인해 예기치 않게 중단될 수 있지만 계획된 유지보수로 인해 중단될 수도 있다.
**개별 노드의 장애에도 전체 시스템이 동작하게끔 유지하고 노드 중단의 영향을 최소화**하는 것이 목표이다.
- 리더 기반 복제에서 `고가용성` 달성하기

⚠️ **팔로워 장애 → 따라잡기 복구**
각 팔로워는 리더로부터 수신한 데이터 변경 로그를 로컬 디스크에 보관한다.
팔로워는 보관하고 있는 로그를 바탕으로 장애가 났던 전에 처리한 마지막 트랜잭션을 알아내 리더에 다시 연결하여 팔로워 연결이 끊어진 동안 발생한 데이터 변경을 모두 요청할 수 있다.

**⚠️ 리더 장애 → 장애 복구**
- 리더의 장애를 처리하는 일은 까다롭다.

**자동 장애 복구**는 아래와 같은 단계를 거친다.
1. 리더가 장애인지 확인
  - 대부분의 시스템은 단순히 `타임아웃`을 사용
2. 새로운 리더를 선택
  - 팔로워 중 하나를 새로운 리더로 `선출`
  - 보통 이전 리더의 최신 데이터 변경사항을 가진 복제 서버
3. 새로운 리더 사용을 위한 시스템 재설정
  - 클라이언트는 쓰기 요청을 새로운 리더에게 보내야 한다.
  - 이전 리더가 팔로워가 되고 새로운 리더를 인식할 수 있게끔 해야 한다.

**위 장애 복구 과정에서 발생할 수 있는 문제점**
🚨 **비동기 복제로 인한 쓰기 유실**
- 비동기식 복제 환경에서 이전 리더의 쓰기를 일부 수신하지 못할 수 있다.
- 새 리더가 선출된 후 이전 리더가 클러스터에 다시 추가되었을 때 충돌을 해결하기 위해서 이전 리더의 복제되지 않은 쓰기를 단순 폐기하는 방식이 가장 일반적인 해결책.
- 이는 클라이언트 입장에서는 "완료되었다고 해서 믿었는데 데이터가 사라진" 상황으로 `내구성`에 대한 기대를 저버리게 됌.

🚨  **외부 시스템과의 불일치**
- 위 상황처럼 데이터가 유실될 때 만약 외부 다른 저장소와 연동이 되어 있다면 특히 위험합니다.
- 외부 저장소와의 데이터 불일치

🚨 **스플릿 브레인**
- 특정 결함 시나리오에서 두 노드가 모두 자신이 리더라고 믿어 두 리더가 쓰기를 받으면서 서로 다른 쓰기가 섞여 데이터가 유실되거나 오염된다.

🚨 **적절한 타임아웃 산정의 어려움**
- 리더가 죽었다고 판단하는 `타임아웃`을 **길게** 잡으면 실제로 리더가 죽었을 때 복구까지 너무 오래 걸림
- 타임아웃을 **짧게** 잡으면 잠깐 트래픽이 폭주해서 응답이 지연된 것뿐인데 "리더가 죽었다!"며 불필요한 장애 복구를 시도. 시스템이 이미 부하를 받고 있는데 새 리더 선출등의 장애복구 과정이 상황을 더 악화시킬 수 있다.

### 복제 로그 구현
리더 기반 복제 내부에서 `리더가 팔로워에게 데이터를 어떻게 전달할 지`에 대한 복제 구현 방식은 4가지로 나뉜다.

1️⃣ **구문 기반 복제**
리더가 실행한 SQL 구문(`INSERT`, `UPDATE`, `DELETE`)을 텍스트 그대로 팔로워에게 전달해 팔로워가 똑같은 SQL을 실행하게 하는 방식
`문제점`
- `NOW()`, `RAND()` 같은 비결정적 함수는 리더와 팔로워에서 각기 다른 값을 생성하여 데이터 불일치 발생
- `AUTO_INCREMENT`나 `UPDATE ... WHERE`처럼 기존 데이터에 의존하는 쿼리는 정확히 같은 순서로 실행되지 않으면 결과가 틀어짐.

2️⃣** 쓰기 전 로그 배송**
스토리지 엔진이 디스크에 기록하는 가장 저수준의 물리적 로그(**WAL: Write-Ahead Log**)를 네트워크를 통해 팔로워에게 그대로 전송하는 방식
- 완전히 동일한 로그를 사용해 다른 노드에서 복제 서버를 구축가능
- `PostgreSQL`에서 사용

`문제점`: **복제가 스토리지 엔진과 강하게 결합**
- 디스크 블록의 몇 번째 바이트가 바뀌었는지등의 상세 정보가 담겨 리더와 팔로워의 DB 소프트웨어 버전이 다르면 복제가 깨질 수 있음.
- 이로 인해 팔로워를 먼저 패치하고 리더로 승격시키는 방식의 **무중단 업그레이드가 불가능**하여 운영상 큰 제약이 됨.

3️⃣ **논리적(로우 기반) 로그 복제**
복제용 로그를 스토리지 엔진의 물리적 저장 방식과 **분리**해 **`로우`** 단위의 변경 사항(논리적 로그)을 기록하는 방식

4️⃣ **트리거 기반 복제**
DB 시스템 레벨이 아닌 **애플리케이션 레벨**에서 복제를 제어하고 싶을 때 사용. 
`트리거`나 `스토어드 프로시저`를 사용.
- 애플리케이션 코드를 통해 트리거를 등록
- 애플리케이션 코드는 데이터가 변경되면 자동으로 실행
## 복제 지연 문제
웹 서비스는 대부분 읽기 요청이고 이를 처리하기 위해 하나의 리더와 수많은 팔로워를 두는 **읽기 확장(Read-scaling) 아키텍처**를 사용한다. 
이 접근 방식은 **비동기식 복제에서만 동작**한다. 단일 노드 장애나 네트워크 중단으로 전체 시스템의 쓰기가 불가능해지고 노드가 많아지면 다운될 가능성도 커져 **동기식 설정은 불안정**하다. 

비동기 팔로워에서 데이터를 읽을 때 팔로워가 뒤쳐진다면 지난 정보를 보는 불일치가 발생할 수 있다.
- 모든 쓰기가 팔로워에 반영되지 않은 경우 서로 다른 결과를 얻을 수 있다.
하지만 이런 불일치는 일시적인 상태에 불과하고 데이터베이스에 쓰기를 멈추고 잠시 기다리면 팔로워는 결국 따라잡고 리더와 일치하게 된다. (<span class="text-red">**최종적 일관성**</span>)
이처럼 팔로워가 리더의 최신 데이터를 따라오지 못하는 복제 지연이 있을 때 발생할  있는 3가지 사례와 해결방법을 설명한다.

### 자신이 쓴 내용 읽기
사용자가 데이터를 제출한 직후 데이터를 볼 때 새로운 데이터는 아직 복제 서버에 반영되지 않았을 수 있다.
이는 사용자 입장에서 데이터가 유실된 것처럼 보이기 때문에 불만족스러운 동작이다.
이런 상황에서는 <span class="text-red">**쓰기 후 읽기 일관성(자신의 쓰기 읽기 일관성)**</span>이 필요하다.
- 자신이 제출한 모든 갱신을 볼 수 있음을 보장.
- 다른 사용자에 대해서는 보장하지 않음.

**리더 기반 복제 시스템**에서 **쓰기 후 읽기 일관성**을 구현하는 방법
1. 사용자가 수정한 내용을 읽을 때는 리더에서 읽는다.
2. 애플리케이션이 마지막 수정 시간을 기억해 두었다가 수정 후 1분 동안은 무조건 리더에서 읽게 한다.
3. 클라이언트가 자신의 마지막 쓰기 타임스탬프를 기억하고 그 타임스탬프 이전 상태인 팔로워가 걸리면 최신화될 때까지 대기하거나 다른 팔로워로 라우팅한다.

### 단조 읽기
사용자가 각 다른 복제 서버에서 여러 읽기를 수행할 때 **시간이 거꾸로 흐르는 현상**을 확인할 수 있다.
- 처음 팔로워에서는 남이 쓴 댓글을 보았으나 두 번째는 지연된 팔로워에서 읽어와 댓글이 반환되지 않음

단조 읽기는 강력한 일관성보다는 약하지만 최종적 일관성보다는 강한 보장으로 **한 사용자는 항상 동일한 팔로워에서만 데이터를 읽도록** 한다.

### 일관된 순서로 읽기
A가 먼저 질문을 하고 B가 대답을 했는데 파티셔닝(샤딩)된 분산 데이터베이스 환경에서 각 파티션의 복제 속도가 다르게 동작해 사용자가 데이터베이스에서 읽을 때 B의 대답이 A의 질문보다 먼저 나타나는 현상.
서로 인과성이 있는 쓰기가 동일한 파티션에 기록하게 하거나 인과성을 명시적으로 유지하기 위한 알고리즘을 사용

## 다중 리더 복제
쓰기를 허용하는 노드를 **하나 이상** 두는 것으로 쓰기 처리를 하는 각 노드는 데이터 변경을 다른 모든 노드에 전달해야 한다.
- **각 리더는 동시에 다른 리더의 팔로워 역할도 한다.**

### 사용 사례
1. 여러 데이터 센터 운영: 각 지역 데이터센터마다 리더를 두어 성능 향상
2. 오프라인 작업을 하는 클라이언트: 앱 자체가 로컬 리더 역할
3. 실시간 협업 편집 (ex: 구글 독스): 여러 사용자가 동시에 문서 편집

### 쓰기 충돌 다루기
⚠️ <span class="text-red">**다중 리더 복제는 큰 단점이 있다.**</span>
- **동일한 데이터를 다른 두 개의 리더에서 동시에 변경할 경우 **<span class="text-blue">**쓰기 충돌**</span>이 발생해 이를 반드시 해소해야 한다.

| **방법** | **설명** |
| ---- | ---- |
| **충돌 회피** | 특정 사용자의 요청은 항상 동일한 데이터센터로 라우팅하고 데이터센터 내 리더를 사용해 읽기와 쓰기를 하게끔 보장 |
| **일관된 상태 수렴** | 데이터베이스가 자체적인 규칙을 통해 **모든 복제본이 최종적으로 동일한 값에 도달(수렴)**하도록 하는 방법 |
| **사용자 정의 충돌 해소 로직** | 애플리케이션 코드를 사용해 충돌 해소 로직을 작성 |
| **자동 충돌 해소** | 충돌을 자동으로 안전하게 병합해주는 데이터 구조나 알고리즘 사용 |

### 다중 리더 복제 토폴로지

> - 쓰기를 한 노드에서 다른 노드로 전달하는 통신 경로


| 토폴로지 종류 | 설명 |
| ---- | ---- |
| **원형** | 각 노드가 하나의 노드로부터 쓰기를 받아 다른 하나의 노드로 전달하는 방식 |
| **별 모양** | 중앙에 지정된 루트 노드 하나가 모든 쓰기를 나머지 노드로 전달하는 방식. |
| **전체 연결** | **모든 리더가 자신의 쓰기를 다른 모든 리더에게 직접 전송하는 방식** |

## **리더 없는 복제**
리더의 개념을 버리고 모든 복제 서버가 클라이언트로부터 쓰기 요청을 직접 받을 수 있게 허용하는 접근 방식

### 노드가 다운됐을 때 데이터베이스에 쓰기
세 개의 복제 서버를 가진 데이터베이스가 있고 복제 서버 중 하나를 사용할 수 없다고 가정했을 때, 리더 기반 설정에서 쓰기 처리를 하기 위해서는 장애 복구를 실행해야 한다.
반면 **리더 없는 설정에서는 장애 복구가 필요하지 않다.**

클라이언트가 쓰기를 세 개의 모든 복제 서버에 <span class="text-blue">병렬</span>로 전송한다.
3개의 노드중 2개의 복제 서버가 쓰기를 확인하면 충분하다고 가정하고 2개의 응다을 받은 후에는 쓰기가 성공한 것으로 간주한 후 복제 서버 중 하나가 쓰기를 놓친 것을 무시한다.

**읽기와 쓰기를 위한 **<span class="text-red">**정족수**</span>

| **규칙 ** | **설명** |
| ---- | ---- |
| **`n`** | 복제 서버 수 |
| **`w`** | 쓰기를 성공해야 하는 최소 노드 수 |
| **`r `** | 읽기를 성공해야 하는 최소 노드 수 |
| **`w + r > n`** | 공식 |

`n`개의 복제 서버가 있을 때 모든 쓰기는 `w`개의 노드에서 성공해야 쓰기가 확정되고 모든 읽기는 최소한 `r`개의 노드에 질의해야 한다.
- 최소한 `r`개의 노드 중 하나에서 최신 값을 읽을 수 있기 때문
- 이런 `r`과 `w`를 따르는 읽기와 쓰기를 **정족수 읽기와 쓰기**라고 부른다.
필요한 `w`나 `r`개의 노드보다 **사용 가능한 노드**가 적다면 쓰기나 읽기는 에러를 반환한다.

⚠️ **정족수 일관성의 한계**
**`w + r > n`** 인 경우에도 오래된 값을 반환하는 에지 케이스가 존재한다.
1. 느슨한 정족수를 사용한다면 `w`개의 쓰기는 `r`개의 읽기와 다른 노드에서 수행될 수 있어 `r`개의 노드와 `w`개의 노드가 겹치는 것을 보장하지 않는다.
  - `w + r > n` 을 만족하더라도 읽기 요청이 최신 데이터를 포함하지 않을 수 있음.
2. 두 개의 쓰기가 동시에 발생하면 어떤 쓰기가 먼저 일어났는지 분명하지 않아 정족수만으로는 최신 값을 정하기 어렵다.
  - 동기 쓰기를 합치는 방법으로 해결
3. 쓰기와 읽기가 동시에 발생하면 쓰기는 일부 복제 서버에서만 반영될 수 있고 이 경우 읽기가 예전 값또는 최신 값을 반환하는지 여부가 분명하지 않는다.
  - 같은 것을 읽더라도 어떤 노드에 질의했는지에 따라 결과가 달라짐
4. 쓰기가 일부 복제 서버에서는 성공했지만 다른 복제 서버에서 실패해 전체에서 성공한 서버가 `w` 복제 서버보다 적다면 성공한 복제 서버에서는 롤백하지 않는다. 
  - 클라이언트는 실패 응답을 받았는데도 이후 읽기에서 그 값이 보일 수도 있고 안 보일 수도 있음
5. 새 값을 전달하는 노드에 장애 발생 시 예전 값을 기준으로 데이터가 복구되면 새로운 값을 가진 복제 서버가 `w`보다 낮아져 정족수 조건이 깨진다.

따라서 정족수가 읽기 시 최신 값을 반환하게끔 매개변수 `w`와 `r`로 오래된 값을 읽는 확률을 조정할 수 있지만 <u>절대적으로 보장할 수는 없다.</u>

### **느슨한 정족수와 암시된 핸드오프**
정족수는 <span class="text-red">**내결함성**</span>이 없다.
- 네트워크 중단으로 다수의 데이터베이스 노드와 클라이언트는 쉽게 연결이 끊어질 수 있다.
- 이 상황에서는 응답 가능한 노드가 `w`나 `r`보다 적을 가능성이 있어 클라이언트는 정족수를 충족할 수 없다.
**느슨한 정족수**: 쓰기 가용성을 위해 원래 이 데이터를 저장할 권한이 없는 '이웃 노드'들에게 임시로 데이터를 맡기고 쓰기 성공을 채우는 방식
**암시된 핸드오프**: 네트워크가 복구되면 임시로 데이터를 맡아주던 이웃 노드가 홈 노드에게 일시적으로 수용한 모든 쓰기를 전달
이 방식은 내구성을 높여주지만(데이터가 어딘가에는 저장된다) 암시된 핸드오프가 완료될 때까지는 `r`노드의 읽기가 저장된 데이터를 본다는 보장은 없다.
### 동기 쓰기 감지
리더 없는 복제에서도 동시에 동일한 키에 쓰기를 시도하는 것을 허용합니다.
이로 인해 <u>엄격한 정족수</u>를 사용하더라도 네트워크 지연이나 부분 장애 때문에 노드마다 쓰기 이벤트가 도착하는 순서가 달라져** 충돌이 발생**합니다.

**최종 쓰기 승리 (LWW: Last Write Wins, 동시 쓰기 버리기)**
가장 단순한 충돌 해소 방식은 각 복제본이 가진 예전 값을 버리고 **`가장 최신 값`****으로 덮어쓰는 것**입니다.
- <u>어떤 쓰기가 최신인지 결정할 수 있는 한 모든 쓰기</u>는 **최종적**으로 모든 복제서버에 복사되어 최종적으로 복제본은 동일한 값으로 수렴한다.
쓰기는 임의로 순서를 정할 수 있다.
- 타임스탬프
- UUID (카산드라 사용 시 추천 방식)

**이전 발생 관계 파악하기**
데이터베이스는 값 자체가 아니라 **버전 번호**를 통해 두 작업이 동시에 수행됐는지 여부를 결정
1. 서버는 키가 쓰일 때마다 버전 번호를 1씩 증가한다.
2. 클라이언트는 데이터를 **쓰기 전에 키를 먼저 읽어야 하며** 서버는 최신 데이터와 덮어쓰지 않은 모든 값을 반환한다.
3. 클라이언트가 데이터를 수정해서 쓸 때 이전 읽기의 버전 번호를 포함한다.
4. 서버가 특정 버전 번호를 가진 쓰기를 받으면 해당 버전 이하 모든 값을 덮어쓴다.

**동시에 쓴 값 병합**
이렇게 데이터가 자동으로 삭제되지 않음을 보장하지만 여러 작업이 동시에 발생하면 클라이언트는 동시에 쓴 값을 합쳐 정리를 해야 한다.
- 이런 동시 값을 **형제** 값이라 한다.
클라이언트가 다음번 읽기를 수행할 때 두 개의 형제 값을 모두 반환하여 **애플리케이션 코드가 이를 직접 병합(합집합 등)**하도록 한다.
- 애플리케이션 코드에서 형제 병합은 복잡하고 오류가 발생하기 쉬워 자동으로 병합을 수행할 수 있게 데이터 구조를 설계하려는 노력 몇가지가 있다(ex: CRDT)

**버전 벡터**
단일 노드일 때는 키마다 1개의 버전 번호만 있으면 충분했지만 **다수의 복제 노드가 있는 리더 없는 환경**에서는 그것만으로는 부족하다.
- 다중 복제본 환경에서는 **각 복제본(노드)마다 자체적인 버전 번호를 유지**하고 추적
- `모든 복제본의 버전 번호들을 모아둔 배열`을 **버전 벡터**라고 부른다.
- 데이터베이스는 클라이언트와 데이터를 주고받을 때 이 버전 벡터를 통째로 교환함으로써 여러 노드에 걸쳐 발생한 복잡한 동시 쓰기와 덮어쓰기 관계를 안전하게 식별
