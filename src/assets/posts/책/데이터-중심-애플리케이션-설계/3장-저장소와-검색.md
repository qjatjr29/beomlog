이 장에서는 **데이터베이스가 데이터를 저장하는 방법과 데이터를 요청했을 때 찾을 수 있는 방법**을 설명합니다
## 🔍 색인
색인의 일반적인 개념은 어떤 부가적인 메타데이터를 유지하는 것으로 메타데이터는 원하는 데이터의 위치를 찾는데 도움을 줍니다. 
색인의 유지보수는 쓰기 과정에서 오버헤드가 발생합니다.

저장소 시스템에서 중요한 것은 `트레이드 오프`
- 색인을 잘 선택했다면 읽기 질의 속도가 향상
- 하지만 모든 색인은 쓰기 속도를 떨어뜨림
따라서 개발자나 DB 관리자가 질의 패턴에 대한 지식을 통해 수동으로 색인을 선택해야 합니다.
### 해시 색인
`키-값 데이터 색인`
가장 간단하게 가능한 색인 전략은 키를 데이터 파일의 바이트 오프셋에 매핑해 인메모리 해시 맵을 유지하는 전략입니다.
해당 방식은 key의 값이 자주 갱신되는 상황에 적합
- 즉 쓰기가 많지만 고유 키가 많지 않은 상황 (키당 쓰기 수가 많지만 메모리에 모든 키를 보관 가능)

⚠️ 하지만 파일에 항상 추가만 한다면 디스크 공간이 부족해집니다.
- 특정 크기의 세그먼트로 로그를 나누고 컴팩션을 수행하여 디스크 공간 부족 문제를 해결

**과정**
- 세그먼트가 특정 크기 도달 시 병합을 위한 새 세그먼트 파일 생성
- 세그먼트 병합은 백그라운드 스레드에서 수행되고 이전 세그먼트에 읽기 및 쓰기 처리 수행 가능
- 병합이 끝난 후 새 세그먼트로 전환 → 기존 세그먼트 파일 삭제

💣 **해시 색인의 한계**
- 해시 테이블을 메모리에 저장해야 해 키가 많으면 문제가 된다.
  - 무작위 접근 I/O가 많이 필요
- 범위 질의에 비효율적. 모든 개별 키를 조회

### SS 테이블 & LSM 트리
세그먼트 파일 형식에 **키-값 쌍을 키로 정렬**하는 요구사항을 적용

이처럼 키로 정렬된 형식을 **정렬된 문자열 테이블(Sorted String Table), SS 테이블**이라 합니다.
**방법**
1. 쓰기가 들어오면 `인메모리 균형 트리(balanced tree)`  데이터 구조*(ex: red-black tree) *에 추가. 이 인메모리 트리를 멤테이블(memtable)이라고 합니다.
2. 멤테이블의 크기가 임계값보다 커지면 SS테이블 파일로 디스크에 기록 (트리가 이미 정렬되어 있기 때문에 효율적)
3. 읽기 요청을 처리하려면 먼저 멤테이블에서 키를 찾고 이후 디스크상의 최신 세그먼트부터 찾습니다.
4. 백그라운드에서 지속적으로 병합과 컴팩션 과정을 수행

**LSM 트리 (Log-Structured Merge-Tree)**
- 백그라운드에서 연쇄적으로 SS 테이블을 지속적으로 병합하는 것.
## 🍃 B 트리
**로그 구조화 색인**이 점점 보편화되고 있지만 가장 널리 사용되는 색인 구조는 **B 트리**
B트리는 `4KB 크기의 고정 크기 블록이나 페이지로 나누고 한 번에 하나의 페이지에 읽기 또는 쓰기를 합니다.` 크기의 고정 크기 `블록`이나 `페이지로` 나누고 한 번에 하나의 페이지에 읽기 또는 쓰기를 합니다.
- 각 페이지는 주소나 위치를 이용해 식별 가능
- 이 방식으로 하나의 페이지가 다른 페이지를 참조 가능

페이지는 여러 키와 하위 페이지의 참조를 포함하여 각 하위 페이지는 키가 계속 이어지는 범위를 담당하고 참조 사이의 키는 해당 범위 경계가 어디인지를 나타냅니다.
ex) 키 `251`을 찾기 위해서는 `200~300` 사이의 페이지 참조를 따라 가야하며 좀 더 작은 범위로 나눈 페이지로 이동
최종적으로는 **개별 키(리프 페이지)**를 포함하는 페이지에 도달하고 해당 페이지는 각 키의 값을 포함하거나 값을 찾을 수 있는 페이지의 참조를 포함합니다.

> **B 트리의 키의 값을 갱신하는 과정**

1. 키를 포함하고 있는 리프페이지를 검색 후 페이지의 값을 변경
2. 다음 페이지를 디스크에 다시 기록

> **새로운 키를 추가하는 과정**

1. 새로운 키를 포함하는 범위의 페이지를 찾아 해당 페이지에 키와 값을 추가
2. 새로운 키를 수용한 페이지에 충분한 여유 공간이 없다면 페이지 하나를 반쯤 채워진 페이지 둘로 나누고 상위 페이지가 새로운 키 범위의 하위 부분을 알 수 있도록 갱신

💡 **Tree가 계속 균형을 유지하는 것을 보장**
- `n` 개의 키를 가진 B 트리는 깊이가 항상 `O(log n)`
- 대부분의 DB에서는 깊이가 `3~4` 단계면 충분
### 신뢰할 수 있는 B 트리 만들기
페이지가 많아져 페이지를 나눠야 한다면 분할된 두 페이지를 기록한 후 두 하위 페이지의 참조를 갱신하도록 상위 페이지를 덮어쓰기 해야합니다.
하지만 일부 페이지만 기록한 후 데이터베이스에 문제가 생기면 색인이 훼손되는 상황이 발생합니다.

> 📖** WAL, write-ahead log, redo log**

DB는 고장 상황에서 스스로 복구할 수 있도록 **쓰기전 로그**를 추가해 B 트리를 구현합니다.
- 트리 페이지에 변경된 내용을 적용하기 전 모든 B 트리의 변경 사항을 기록하는 `추가 전용 (append-only)` 파일입니다.
- 이 로그를 바탕으로 데이터베이스가 복구될 때 일관성 있는 상태로 B 트리를 다시 복원합니다.

## ✅ LSM트리와 B 트리 비교
보통 `LSM`트리는 쓰기에서 빠르고 `B트리`는 읽기에서 더 빠르다고 하는데 읽기에서 `LSM`이 더 느린 이유는 각 컴팩션 단계에 있는 여러 데이터 구조와 SS테이블을 확인해야 하기 때문이다. 
### LSM 트리의 장점
**쓰기 처리량**
- 상대적으로 스기 증폭이 더 낮고 트리에서 여러 페이지를 덮어쓰는 것이 아니라 순차적으로 컴팩션된 SS테이블을 쓰기 때문
**압축률**
- B 트리보다 디스크에 더 적은 파일을 생성
- B 트리 저장소 엔진은 파편화로 인해 사용하지 않는 디스크 공간 일부가 남음
- LSM 트리는 페이지 지향적이지 않고 주기적으로 파편화를 없애기 위해 SS 테이블을 다시 기록해 저장소 오버헤드가 더 낮음.
### LSM 트리의 단점
**컴팩션 과정의 영향**
- 컴팩션 과정이 때로는 진행중인 읽기와 쓰기 성능에 영향을 줌
- 비싼 컴팩션 연산이 끝날 때 까지 요청이 대기

**유한한 디스크 대역폭 대비 높은 쓰기 처리량**
- 디스크의 쓰기 대역폭은 유한하지만 백그라운드에서 수행되는 컴팩션 스레드가 이 대역폭을 공유
- 데이터베이스가 커질수록 컴팩션을 위해 더 많은 디스크 대역폭이 필요

**여러 세그먼트에 같은 키의 복사본이 존재할 수 있음**
- B트리가 색인의 한 곳에만 키가 있는 반면 로그 구조화 저장소 엔진은 여러 세그먼트에 같은 키의 다중 복사본이 존재할 수 있습니다. 
- 강력한 트랜잭션 시맨틱(semantic)을 제공하는 데이터베이스에는 B 트리가 훨씬 매력적
## 👀 기타 색인 구조
### 1️⃣ **보조 색인(secondary index)**
**RDB**에서는 `CREATE INDEX`를 이용해 보조 색인을 생성할 수 있는데 주로 조인 질의를 효율적으로 수행하는 역할을 합니다.
보조 색인은 키-값 색인에서 쉽게 생성할 수 있는데 기본키 색인과는 키가 고유하지 않다는 차이가 있습니다.
- 즉 같은 키를 가진 많은 로우(문서, 정점)가 있을 수 있다.

**아래 2가지 방식으로 해결 가능합니다.**
1. 색인의 각 값에 일치하는 로우 식별자 목록을 만들기
2. 로우 식별자를 추가해 각 키를 고유하게 만들기
어느 쪽이든 `보조 색인`으로 B트리 및 로그 구조화 색인을 사용할 수 있습니다.
### **색인 안에 값 저장하기**
색인에서 키는 질의가 검색하는 대상이지만 값은 다음 2가지중 하나에 해당합니다.
- 실제 로우(문서, 정점)
- 다른 곳에 저장된 로우를 가리키는 참조

후자인 경우 로우가 저장된 곳을 **힙 파일(heap file)**이라 하고 순서 없이 데이터를 저장합니다.
- 여러 보조 색인이 존재할 때 데이터 중복을 피할 수 있어 일반적으로 사용.
- 각 색인은 힙 파일에서 위치만 참조하고 실제 데이터는 일정한 곳에 유지

힙 파일 접근 방식은 키를 변경하지 않고 값을 갱신할 때 효율적입니다.
새로운 값이 이전 값보다 많은 공간을 필요로 하지 않으면 레코드를 제자리에 덮어쓸 수 있지만  새로운 값이 많은 공간을 필요로 할 경우 힙에서 충분한 공간이 있는 새로운 곳으로 위치를 이동해야해 복잡해집니다.
이 경우 모든 색인이 레코드의 새로운 힙 위치를 가리키게끔 갱신하거나 이전 힙 위치에 전방향 포인터를 남겨둬야 합니다.

색인에서 힙 파일로 다시 이동하는 일은 읽기 성능에 불이익이 너무 많아서 어떤 상황에서는 색인 안에 바로 색인된 로우를 저장하는 편이 바람직하며 이를 <span class="text-red">**클러스터드 색인(clustered index)**</span>이라 합니다.
- `MySQL의 InnoDB` 저장소 엔진은 테이블의 기본키가 항상 클러스터드 색인
- 보조 색인(힙 파일의 위치가 아닌)은 기본키를 참조
**클러스터드 색인**과 **비클러스터드 색인** 사이의 절충안을 **커버링 색인(covering index)** 이나 **포괄열이 있는 색인(index with included column)**이라 합니다. 
- 색인 안에 테이블의 칼럼 일부를 저장해서 색인만을 사용해 일부 질의에 응답이 가능하다.

클러스터링 색인과 커버링 색인은 읽기 성능을 높일 수 있지만 **추가적인 저장소가 필요하고 쓰기 과정에 오버헤드가 발생**하며 애플리케이션 단에서 **복제로 인한 불일치를 파악**할 수 없기에 데이터베이스는 트랜잭션 보장을 강화하기위해 별도의 노력이 필요합니다.
### **다중 칼럼 색인**
`다중 컬럼`에 동시에 질의를 해야한다면 기존에 설명한 색인으로는 부족하다.

다중 칼럼 색인의 가장 일반적인 유형은 **결합 색인(concatenated index)**
- 하나의 칼럼에 다른 칼럼을 추가하는 방식
- 하나의 키에 여러 필드를 단순히 결합한다.  
- 순서가 정렬

**다차원 색인**은 한 번에 여러 칼럼에 질의하는 좀 더 일반적인 방법입니다.
- `지리 공간 데이터`에서 중요하게 사용
```sql
SELECT * FROM cities WHERE latitude > 120 and latitude < 180 
AND longitude > 30 and longitude < 80;
```

이런 유형의 질의는 표준 B 트리나 LSM 트리 색인으로는 효율적인 응답이 불가능합니다.

💡 **해결하기 위한 방법**
1️⃣ 이차원 위치를 `공간 채움 곡선(space-filling curve)`을 이용해 단일 숫자로 변환하여 일반 B트리 색인을 사용하는 것
2️⃣ `R트리`처럼 `전문 공간 색인(specialized spatial index)`을 사용하는 것. (Ex: PostGIS는 R트리 같은 지리 공간 색인을 구현했다.)
### **전문 검색과 퍼지 색인**
정확한 값이나 정렬된 키의 값의 범위가 아닌 <u>**유사한 키**</u>에 대한 검색은 어떻게??
- 이처럼 **애매모호한(fuzzy)** 질의에는 다른 기술이 필요

<span class="text-red">**루씬**</span>과 같은 전문 검색 엔진은 특정 단어를 검색할 때 다양한 기능을 제공
- 해당 단어의 동의어로 질의를 확장
- 단어의 문법적 활용을 무시하고 동일한 문서에 서로 인접해 나타난 단어 검색
- 언어학적으로 텍스트를 분석해 사용
- 문서나 질의의 오타에 대처하기 위해 특정 편집 거리 내 단어 검색

루씬은 용어 사전을위해 `SS 테이블` 같은 구조를 사용하는데 이 구조는 작은 인메모리 색인이 필요하며 키를 찾는데 필요한 정렬 파일의 오프셋을 질의에 알려주는데 사용합니다.
## 🧑‍💻 트랜잭션 처리나 분석
보통 애플리케이션은** 색인을 사용해 일부 키에 대한 적은 수의 레코드**를 찾고 그리고 레코드는 사용자 입력 기반으로 **추가 및 갱신**됩니다.
이런 애플리케이션은 `대화식`이기에 이런 접근 패턴을 <span class="text-red">**온라인 트랜잭션 처리(online trrasaction processing, OLTP)**</span>이라 합니다.

그러나 데이터베이스를 **데이터 분석**에도 많이 사용하기 시작했는데 이런 데이터 분석은 트랜잭션과 접근 패턴이 매우 다릅니다.
- 많은 수의 레코드를 스캔해 레코드당 일부 칼럼만 읽어 집계 통계(카운트, 합, 평균 등)를 계산
이런 질의는 사용자보다는 비즈니스 분석가가 작성하여 회사 경영진의 의사 결정을 돕는 보고서 작성용으로 사용되며 이런 접근 패턴을<span class="text-red"> </span><span class="text-red">**온라인 분석 처리(online analytic processing, OLAP)**</span>라고 부른다.

🆚 **OLTP와 OLAP의 전형적인 특성비교**

| **특성** | **트랜잭션 처리 시스템(OLTP)** | **분석 시스템(OLAP)** |
| ---- | ---- | ---- |
| 주요 읽기 패턴 | 질의당 적은 수의 레코드, 키 기준으로 가져옴 | 많은 레코드에 대한 집계 |
| 주요 쓰기 패턴 | 임의 접근, 사용자 입력을 낮은 지연 시간으로 기록 | 대규모 불러오기(bulk import, ETL) 또는 이벤트 스트림 |
| 주요 사용처 | 웹 애플리케이션을 통한 최종 사용자/소비자 | 의사결정 지원을 위한 내부 분석가 |
| 데이터 표현 | 데이터의 최신 상태(현재 시점) | 시간이 지나며 일어난 이벤트 이력 |
| 데이터셋 크기 | 기가바이트에서 테라바이트 | 테라바이트에서 페타바이트 |

처음에는 트랜잭션 처리와 분석 질의를 위해 `동일한 데이터베이스`를 사용했습니다.
최근에 들어서는 OLTP 시스템을 분석 목적으로 사용하지 않고 개별 데이터베이스에서 분석을 수행하는 경향을 보였으며 이러한 개별 데이터베이스를 **데이터 웨어하우스(data warehouse)**라고 불렀습니다.
### 데이터 웨어하우징
<span class="text-red">**OLTP**</span> 시스템은 **높은 가용성과 낮은 지연 시간**의 트랜잭션 처리를 기대합니다.
- 즉석 분석 질의(ad hoc analytic query)를 실행하는 것을 꺼려합니다.
분석 질의가 데이터 셋의 많은 부분을 스캔하여 동시에 실행되는 트랜잭션 성능을 저하시킬 수 있습니다.

반대로 **데이터 웨어하우스**는 분석가들이 **OLTP** 작업에 영향을 주지 않으면서 질의할 수 있는 데이터베이스로 **OLTP** 데이터베이스에서 추출하여 변환하고 적재한 것입니다.
- **`ETL`**** (Extract-Transform-Load)**

### OLTP 데이터베이스와 데이터 웨어하우스의 차이점
SQL은 일반적으로 분석 질의에 적합하기에 데이터웨어 하우스는 일반적인 관계형 모델을 사용합니다.
- SQL 질의를 생성하고 결과를 시각화하고 분석가가 `드릴 다운(drill-down), 슬라이싱(slicing), 다이싱(dicing)`같은 작업을 통해 데이터를 탐색할 수 있게 해주는 여러 그래픽 데이터 분석 도구가 존재
**OLTP**나 **데이터 웨어하우스** 둘 다 SQL 질의 인터페이스를 지원하지만 각각 서로 매우 다른 질의 패턴에 맞게 최적화되었기 때문에 시스템의 내부는 완전히 다릅니다.
그렇기에 다수의 데이터베이스 벤더는 트랜잭션 처리와 분석 작업부하를 둘 다 지원하기보단 하나에 특화되도록 지원하는데 중점을 둡니다.

## 칼럼 지향 저장소
테이블에 엄청난 수의 로우와 페타바이트 데이터가 있다면 효율적인 저장 및 질의는 매우 어려운 문제가 됩니다.
테이블은 칼럼이 **100**개 이상인 경우가 많지만 일반적인** 데이터 웨어하우스 질의**는 한 번에 **4~5**개 칼럼에만 접근합니다.

🤔 **이런 질의를 어떻게 효율적으로 실행할 수 있을까요?!?**
대부분의 **OLTP** 데이터베이스 저장소는 **로우 지향**방식으로 데이터를 배치합니다.
- 테이블에서 한 로우의 모든 값은 서로 인접하게 저장

> <span class="text-blue">**칼럼 지향 저장소**</span>의 기본 개념은 간단합니다.
> 모든 값을 하나의 로우에 함께 저장하는게 아니라 각 칼럼별로 모든 값을 함께 저장.


각 칼럼을 개별 파일에 저장하면 질의에 사용되는 칼럼만 읽고 구분 분석을 하면 됩니다.

### 칼럼 압축
압축을 통해 데이터를 압축하면 디스크 처리량을 줄일 수 있는데** 칼럼 지향 저장소**는 대게 압축에 적합합니다.
- 칼럼의 데이터에 따라 다양한 압축 기법을 사용가능
그 중 한 가지 기법이 데이터 웨어하우스에서 특히 효과적인 **비트맵 부호화(bitmap encoding)**입니다.

칼럼에서 고유 값의 수는 로우 수에 비해 적습니다. 
- 위 `product_sk`만 봐도 18개의 데이터가 있지만 고유한 값은 6가지(29, 30, 31, 68, 69, 74) 뿐
`n`개의 고유 값을 가진 칼럼을 가져와 `n`개의 개별 비트맵으로 변환할 수 있습니다.  
위 그림과 같이 고유 값 하나가 하나의 비트맵이고 각 로우는 한 비트를 가져 로우가 해당 값을 가질 경우 비트는 `1`이고 아니면 `0`입니다.
여기서 `n`이 클수록 대부분의 비트맵은 `0`이 더 많아지는데 이를 희소(sparse)하다 말합니다. 이 경우 위 그림의 하단처럼 **런 렝스 부호화(run-length encoding)**할 수 있습니다.
- 비트맵 색인은 데이터 웨어하우스에서 일반적으로 사용되는 질의종류에 매우 적합하다.

### 칼럼 저장소의 순서 정렬
**칼럼 저장소에서 로우가 저장되는 순서가 반드시 중요하지는 않습니다.**
- 삽입된 순서로 저장하는 방식이 가장 쉬움
- 새로운 로우를 삽입하는 작업은 각 칼럼 파일에 덧붙여 추가하는 것을 의미하기 때문
하지만 이전의 **SS 테이블**에서 했던 것처럼 순서를 도입해 이를 색인 메커니즘으로 사용할 수 있습니다.
각 칼럼을 독립적으로 정렬할 수는 없습니다. 
- 그렇게 되면 칼럼의 어떤 항목이 동일한 로우에 속하는지 알 수 없기 때문
**한 칼럼의 ****`k`****번째 항목이 다른 칼럼의 ****`k`****번째 항목과 같은 로우에 속하기 때문에 로우를 재구성할 수 있습니다.**

**칼럼별로 저장되었을지라도 데이터는 한번에 전체 로우를 정렬해야 합니다.**
데이터베이스 관리자는 테이블에서 정렬해야 하는 칼럼을 선택할 수 있으며 첫 번째 칼럼에서 같은 값을 가진 로우들의 정렬 순서를 두 번째 칼럼에서 정할 수 있습니다.
- **그룹화**와 **필터링** 질의에 도움

**정렬된 순서**의 또 다른 장점으로는 `칼럼 압축`이 있는데 기본 정렬 칼럼에 고유 값을 많이 포함하지 않는다면 정렬한 후 기본 정렬 칼럼은 연속해서 같은 값이 연속해서 길게 반복됩니다.
간단한 런 렝스 부호화는 수십억 개의 로우를 가진 테이블도 수 킬로바이트로 칼럼을 압축할 수 있습니다.
이런 압축은 첫 번째 정렬키가 가장 강력하고 두 번째, 세 번째로 내려갈수록 뒤섞여 있기에 반복된 값이 그리 길지 않아 정렬 우선순위가 낮은 컬럼은 거의 임의 순서로 나타나 잘 압축되지 않습니다. 
하지만 초반 정렬된 몇개의 칼럼을 압축하는 것은 전체적으로 이득!
### 다양한 순서 정렬
다양한 질의는 서로 다른 정렬 순서의 도움을 받으니 **같은 데이터를 ****`다양한 방식`****으로 정렬**해 저장한다면?
- 질의를 처리할 때 가장 적합한 버전을 사용하면 높은 성능을 기대
하나의 장비가 고장나도 데이터를 잃지 않기 위해서는 동일한 데이터를 여러 장비에 복제하는 작업이 필요합니다.
### 칼럼 지향 저장소에 쓰기
데이터 웨어하우스에서 이런 최적화는 합리적입니다.
- 대부분의 작업은 분석가가 수행하는 대량의 **읽기 전용 질의**이기 때문
칼럼 지향 저장소 뿐만아니라 압축, 정렬 모두 읽기 질의를 빠르게 하는 방법이지만 쓰기를 어렵게 하는 단점이 존재

**B 트리**와 같은 제자리 갱신(update-in-place)접근 방식은 압축된 칼럼에선 불가능하지만 **LSM트리**를 이용해 쓰기 문제를 해결합니다. 
**모든 쓰기는 먼저 인메모리 저장소로 이동해 정렬된 구조에 추가하고 디스크에 쓸 준비를 한 후 그 다음 충분한 쓰기가 모이면 디스크의 칼럼 파일에 병합하고 대량으로 새로운 파일에 기록합니다.**
