> 책 전반에 걸쳐 사용하는 전문 용어와 접근 방식 소개

- 신뢰성(reliability)
- 확장성(scalability)
- 유지보수성(maitainability)
이와 같은 단어의 실제 의미와 이 같은 목표를 달성하기 위해 어떻게 해야하는지 알아본다.

많은 애플리케이션은 계산 중심과 다르게 <u>**데이터 중심**</u>적이다.
- 데이터의 양, 데이터의 복잡도, 데이터의 변화 속도가 중요하다.

## 🤔 데이터 시스템에 대한 생각
일반적으로 데이터베이스. 큐, 캐시 등을 매우 다른 범주에 속하는 도구로 생각한다.
- 표면적으로 비슷하더라도(얼마 동안 데이터 저장) 매우 다른 접근 패턴을 갖고 서로 다른 성능 특성이 있어 구현 방식이 매우 다르다.

### 이 모든 것을 왜 데이터 시스템이라는 포괄적 용어로 묶어야 할까?
1️⃣  **데이터 저장과 처리를 위한 여러 도구들이 최근에 만들어지며 경계가 모호하다**
- 각 도구들이 다양한 사용 사례(use case)에 최적화됐기 때문에 전통적인 분류에 들어맞지 않음

2️⃣ **애플리케이션이 단일 도구로는 더 이상 데이터 처리와 저장 모두를 만족시킬 수 없는 광범위한 요구사항을 가지고 있다.**
- 작업은 단일 도구에서 효율적으로 **수행할 수 있는 태스크**로 나누고 다양한 도구들은 **애플리케이션 코드를 이용해 서로 연결**

데이터 시스템이나 서비스를 설계할 때 까다로운 문제가 많이 생긴다.
- 내부적으로 문제가 있어도 데이터를 정확하고 완전하게 유지하려면?
- 시스템의 일부 성능이 저하되더라도 클라이언트에게 일관되게 좋은 성능을 어떻게 제공?

대부분의 소프트웨어 시스템에서 중요하게 아래 3가지 관심사에 중점을 두자.

| **신뢰성 (Reliability)** | 하드웨어나 소프트웨어 결함, 인적 오류 같은 문제에 직면하더라도 **시스템은 지속적으로 ****`올바르게`**** 동작해야 한다. (원하는 성능 수준에서 정확한 기능을 수행)** |
| ---- | ---- |
| **확장성 (Scalability)** | 시스템의 데이터 양, 트래픽 양, 복잡도가 **증가**하면서 이를 처리할 수 있는 방법이 있어야 한다. |
| **유지보수성 (Maintainability)** | 모든 사용자가 시스템 상에서 **생산적으로 작업**할 수 있게 해야 한다. |

## 🤝 신뢰성
소프트웨어의 경우 신뢰성의 일반적인 기대치는 다음과 같다.
- 애플리케이션은 사용자가 기대한 기능을 수행
- 사용자가 범한 실수나 예상치 못한 소프트웨어 사용법을 허용 가능
- 시스템 성능은 예상된 부하와 데이터 양에서 필수적인 사용 사례를 충분히 만족
- 시스템은 허가되지 않은 접근과 오남용을 방지

### 결함(fault), 장애(failure)
결함을 예측하고 대처할 수 있는 시스템을 `내결함성(fault-tolerant)` 또는 `탄력성(resilient)`을 지녔다고 한다.

일반적으로 결함은 <u>사양에서 벗어난 시스템의 한 구성 요소</u>로 정의되지만 장애는 <u>사용자에게 필요한 서비스를 제공하지 못하고 시스템 전체가 멈춘 경우</u>이다.

결함 확률을 0으로 줄이는 것은 불가능하기 때문에 결함으로 인해 장애가 발생하지 않도로 설계하는 것이 중요하다.
- `카오스 엔지니어링`처럼 고의적으로 결함을 일으켜 시스템을 지속적으로 탐지, 개선하는 방법도 내결함성을 높일 수 있는 접근 방식이다.

### 하드웨어 결함
시스템 장애의 원인을 생각했을 때 하드웨어 결함이 바로 떠오른다.
- 하드디스크 고장
- 정전
- 등등

이런 하드웨어 결함으로 인한 시스템 장애를 대응하는 방법
- 각 하드웨어 구성 요소에 중복을 추가하는 방법 (디스크 - RAID 구성, 서버 - 이중 전원 디바이스 등)

하지만 데이터 양과 애플리케이션 계산 요구가 늘어나며 애플리케이션이 많은 장비를 사용하게 되었고 하드웨어 결함율도 증가했다.
- 따라서 소프트웨어 내결함성 기술을 사용하거나 하드웨어 중복성을 추가

### 소프트웨어 오류
소프트웨어 오류(시스템 내 체계적 오류)는 예상하기 더 어렵고 노드간 상관관계 때문에 시스템 오류를 더 많이 유발하는 경향이 있다.
- 잘못된 특정 입력이 있을 때 모든 애플리케이션 서버 인스턴스가 죽는 버그
- 일부 프로세스가 공유 자원(CPU, 메모리, 디스크, 대역폭)을 과도하게 사용
- 시스템 속도가 느려져 반응이 없거나 잘못된 응답을 반환하는 서비스
- 한 구성 요소의 작은 결함이 다른 구성 요소의 결함을 야기하며 더 많은 결함이 발생하는 연쇄 장애

이런 시스템 장애를 어떻게 대비???
- 시스템 내 상호작용에 대해 고민
- 테스트
- 프로세스 격리
- 지속적인 모니터링

### 인적 오류
소프트웨어 시스템을 설계하고 구축하고 운영하는 건 결국 사람이다.

이런 사람의 실수나 오류에도 시스템을 어떻게 신뢰성있게 만들까??
1. 오류의 가능성을 최소화하는 방향으로 시스템 설계
  - 잘 설계된 추상화 / API / 관리 인터페이스
1. 사람이 많이 실수하는 장소에서 장애가 발생할 수 있는 부분을 분리
2. 단위 테스트부터 전체 시스템 통합 테스트와 수동 테스트까지 철저히 테스트
3. 인적 오류를 빠르고 쉽게 복구할 수 있도록 한다.
  - ex) 롤백 또는 롤아웃
1. 명확한 모니터링 대책

## 🚀 확장성
> 증가한 부하에 대처하는 시스템 능력

- 시스템이 특정 방식으로 커지면 이에 대처하기 위한 선택은 무엇인가?
- 추가 부하를 다루기 위해 계산 자원을 어떻게 투입할까?

### 부하 기술하기
시스템의 현재 부하를 간결하게 기술해야 한다.
- 부하는 부하 매개변수라고 부르는 숫자들로 나타낼 수 있다.

🤖 **트위터 예시**
트위터의 두 동작은 다음과 같다.

| 트윗 작성 | 사용자는 팔로워에게 새로운 메시지를 게시 |
| ---- | ---- |
| 홈 타임라인 | 사용자는 팔로우한 사람이 작성한 트윗을 볼 수 있다. |

이 두 가지 동작을 구현하는 방법은 크게 2가지이다.
1. 트윗 작성은 간단히 삽입하고 타임라인 조회시 필요한 테이블들을 모두 조인하여 조회
2. 트윗 작성시 사용자를 팔로우하는 사람을 모두 찾아 타임라인에 삽입, 타임라인 읽기 요청은 미리 계산해서 저렴

⚠️ 1번 방식은 타임라인 질의 부하를 버텨내기 힘들고 2번 방식의 경우에는 팔로워가 많은 사람의 경우 문제가 발생

💡 따라서 트위터는 사용자의 트윗은 계속해서 사람들이 작성할 때 홈 타인라인에 펼쳐지지만 팔로워 수가 매우 많은 소수 사용자(ex: 유명인)는 팬 아웃에서 제외하는 방식을 사용
- 유명인의 트윗은 별도로 가져와 읽는 시점에 사용자의 홈 타임라인에 합쳐진다.

### 성능 기술하기
시스템 부하를 기술하면 부하가 발생할 때 어떤 일이 일어났는지 조사할 수 있다.

> 
> 부하 매개변수를 증가시키고 시스템 자원은 변경하지 않고 유지했을 때 시스템 성능은 어떻게 영향?
> 부하 매개변수를 증가시켰을 때 성능이 변하지 않고 유지되길 원한다면 자원을 얼마나 늘려야 할까?


위 두가지 질문을 위해서는 성능 수치가 필요하다.

> 📖 **지연시간과 응답시간**
> 응답시간: 클라이언트 관점. 요청을 처리하는 실제 시간외에도 네트워크 지연과 큐 지연도 포함
> 지연시간: 요청이 처리되길 기다리는 시간으로 서비스를 기다리며 휴지상태인 시간


서비스의 응답 시간을 살피는 것은 **산술 평균**보다는 <span class="text-red">**백분위**</span>를 사용하는 편이 더 좋다.
- p50: 가운데 값(`중앙값`, 절반은 이거보다 빠르고 절반은 느림).
- p95: 상위 5% 지점, `가장 느린 5% 중 제일 빠른 값`
- p99: 상위 1% 지점, `가장 느린 1% 중 제일 빠른 값`

<span class="text-blue">**꼬리 지연 시간(tail latency)**</span>으로 알려진 상위 백분위 응답시간은 서비스의 사용자 경험에 직접 영향을 주기 때문에 중요하다.
- 백엔드 호출이 많아질 수록 꼬리가 증폭되어 사용자의 요청의 상당수가 느려지는 현상
- ex) 요청을 처리하기 위해 여러번 백엔드 호출이 필요한 상황에서 단 하나의 느린 백엔드 요청이 전체 최종 사용자 요청을 느리게 한다.

### 부하 대응 접근 방식
**비공유 아키텍처 (shared-nothing)**
- 다수의 장비에 부하를 분산하는 아키텍처

현실적으로 좋은 아키텍처는 실용적인 접근 방식의 조합이 중요하다.
- ex) 적절한 사양의 장비 몇 대가 다량의 낮은 사양 가상 장비보다 훨씬 간단하고 저렴

시스템의 아키텍처는 해당 시스템을 사용하는 애플리케이션에 특화되어 있고 <span class="text-red">범용적이고 모든 상황에 맞는 확장 아키텍처는 없다.</span>
- 주요 동작이 무엇이고 잘 하지 않는 동작이 무엇인지에 대한 가정을 바탕으로 구축

## 🔧 유지보수성
- 버그 수정, 시스템 운영 유지, 장애조사, 새로운 기능 추가 등

유지보수를 위해 주의를 기울ㅇ야 할 소프트웨어 설계 원칙은 다음 3가지이다.

| 설계 원칙 | 설명 |
| ---- | ---- |
| 운용성 | 운영팀이 시스템을 원활하게 운영할 수 있게 쉽게 만들어라 |
| 단순성 | 시스템에서 복잡도를 최대한 제거해 새 엔지니어가 시스템을 이해하기 쉽게 만들어라 |
| 발전성 | 엔지니어가 이후에 시스템을 쉽게 변경할 수 있게 해라.  |


### 운용성 - 운영의 편리함 만들기
동일하게 반복되는 태스크를 쉽게 수행하도록 해서 운영팀이 더 높은 가치의 활동에 노력할 수 있도록 한다.
- `모니터링`을 통해 런타임 동작과 시스템 내부에 대한 가시성 제공
- `자동화`와 통합을 위한 자원을 제공하기 위한 도구이용
- 개별 장비 의존성 회피. 시스템 전체에 영향을 주지 않고 계속해서 운영 가능해야 함.
- 좋은 `문서`와 이해하기 쉬운 운영 모델

### 단순성 - 복잡도 관리
프로젝트가 커짐에 따라 시스템은 복잡하고 이해하기 어려워진다.
- 복잡도는 같은 시스템에서 작업해야 하는 모든 사람의 진행을 느리게 하고 나아가 유지보수 비용이 증가한다.

우발적 복잡도를 제거하기 위한 좋은 도구는 <span class="text-red">**추상화**</span>이다.
- 많은 세부 구현을 숨길 수 있음
- 재사용 가능

> 📖 **우발적 복잡도**
> 소프트웨어 개발에서 해결하려는 문제 본질과 무관하게, 구현 방식, 개발자 실수, 도구 선택, 혹은 제약 사항 등 외부 요인으로 인해 발생하는 불필요한 복잡성을 의미
> - 단순한 기능을 위해 지나치게 복잡한 프레임워크나 아키텍처를 도입하는 경우.
> - 데이터베이스 스키마를 잘못 설계하여 데이터 조회가 비효율적으로 변하는 경우.
> - 가독성이 낮고 유지보수가 어려운 스파게티 코드.
> - 문제를 해결하는 데 필요하지 않은 과도한 설정 파일이나 보일러플레이트 코드


### 발전성 - 변화를 쉽게 만들기
시스템의 요구사항은 끊임없이 변화한다.

데이터 시스템 변경을 쉽게 하고 변화된 요구사항에 시스템을 맞추는 방법은 시스템의 간단함과 추상화와 밀접한 관계가 있다.

## 🤔 1장을 읽고나서.. 
### 🔍 **“소프트웨어에는 환경에 대한 일종의 가정이 있다”?**
`신뢰성` 에 대한 부분을 읽다 아래 글에 대해서 이해가 잘 안됐어서 다시 정리를 해보려고 합니다.
> 소프트웨어에는 환경에 대한 일종의 가정이 있다는 사실을 알 수 있다.
> 이 가정은 대개 사실이지만 어떤 이유로 최종적으로는 사실이 아니게 된다.


**가정 : 설계 / 코드가 믿고 있는 전제 조건**
- ex) 동시 접속하는 유저는 1만명을 넘지 않는다.
- ex) 네트워크 타임아웃은 최대 5초면 충분하다.

이런 것들은 평소 상황에서는 사실이라 시스템이 문제없이 잘 동작.
버그가 발생하더라도 이런 가정을 깨는 상황이 아니라면 드러나지 않는다.

⚠️ **문제 상황 - **`동시 접속 1만 명`만 고려한 큐 사이즈, 스레드 풀, 커넥션 풀 설정이 되어 있는 시스템
- 평소 3천 명 접속 → 아무 문제 없음.
- 이벤트로 인해 3만 명이 몰림 → 큐가 꽉 차고 타임아웃, 메모리 부족, 스레드 고갈 등으로 시스템이 죽음.
- **가정이 더 이상 사실이 드러남.** 

따라서 위에 나온 문구는 아래와 같은 것을 의미하는 것으로 이해했습니다.
> 
> 1. **소프트웨어 버그는 특정 조건(가정이 깨지는 순간)이 오기 전까지 잠복**해 있다.
> 2. 그 조건이 오게되면 `원래 설계가 전제로 삼고 있던 환경(트래픽, 데이터 분포, 네트워크 등)`이 더 이상 맞지 않게 되고 그때서야 문제가 드러난다.


즉, **가정은 특정 기간 동안은 유효하지만 시스템, 환경이 변화하면서 언젠가는 깨진다.**
- 트래픽 패턴, 데이터 양과 분포, 인프라, 운영 방식등이 계속 변화하는 상황에서 처음 설계했을 때의 가정(그땐 맞았을..)이 영원할 수 없다!!

따라서 우리는 **우리 시스템과 환경에 대한 가정이 무엇인지 계속 생각하면서 테스트와 모니터링, 장애 격리, 재시도 등의 방법을 통해서 가정이 깨졌을 때에도 실패로 이루어지지 않도록 해야한다!!**

### 😯 산술 평균과 백분위
지난 프로젝트들을 진행하면서 서비스의 응답시간에 대한 지표를 계산할 때 평균과 백분위(p50, p95, p99)를 측정을 해보면서 시각화를 했었는데 평균에 대한 생각이 이 장을 읽고나서 조금 달라졌다.

```plain text
p50: 50ms (중앙 값)
p95: 250ms (5% 사용자) 
p99: 2,000ms (1% 사용자)
→ 평균 100ms = "시스템 양호" 오판
```

위와 같은 상황에서 평균으로만 보게된다면 꼬리 지연시간 같은 것을 알 수 없고 실제 사용자의 체감과는 다른 무의미한 지표가 될 수도 있다는 것을 이해했다.

또, 백분위 중에서도 `p99`와 같은 수치에 대해서 어떻게 활용할지에 대한 궁금증이 있었는데 이 부분도 해소가 된 것 같다.
