이 책의 목적은 애플리케이션과 시스템을 **신뢰할 수 있고 확장 가능하며 유지보수하기 쉽게 만드는 방법**을 탐구하는 것이다.
이번 장에서의 목적은 현재보다 나은, 즉 견고하고 정확하며 발전 가능한 애플리케이션을 설계하는 방법을 발견하는 것이다.
# 데이터 통합
복잡한 애플리케이션에서는 데이터를 <u>여러 가지 다른 방법</u>으로 사용한다.
데이터를 사용하는 모든 다른 상황에 적합한 소프트웨어가 있을 가능성은 낮으므로 **원하는 애플리케이션 기능을 제공하기 위해서는 반드시 여러 다른 소프트웨어를 함께 사용해야 한다.**
## 파생 데이터에 특화된 도구의 결합
데이터를 다루는 범위는 매우 넓다.
어떤 사람에게는 의미 없는 기능이 다른 사람에게는 핵심 요구사항일 수 있다.
**데이터 통합의 필요성은 조직 전체 데이터플로를 고려할 때야 명확해진다.**
### 데이터플로에 대한 추론
여러 저장소에 데이터가 복제될 때 데이터가 어디서 처음 쓰이고 어떻게 흘러가는지 분명히 할 필요가 있다.
파생 데이터 시스템은 이벤트 로그를 기반으로 갱신하면 결정적이고 멱등성을 지녀 결함에서 복구하기가 상당히 쉬워진다.
### 파생 데이터 대 분산 트랜잭션
서로 다른 데이터 시스템 간 일관성을 유지하는 2가지 접근법

| **구분** | **쓰기 순서 결정 ** | **정확히 한 번 실행 보장 방식** | **타이밍 보장 ** | **단점 및 평가** |
| ---- | ---- | ---- | ---- | ---- |
| **분산 트랜잭션** | 상호 배타성을 위한 **잠금** 사용 | **원자적 커밋 (Atomic commit)** 사용 | **선형성 제공**(자신이 쓴 내용 읽기 등의 유용한 보장을 내포함) | **내결함성**과 **성능**이 나빠 유용성이 크게 제한됨. |
| **파생 데이터 시스템**(CDC, 이벤트 소싱) | **로그** 사용 | **결정적 재시도(Deterministic retry)**와 **멱등성(Idempotence)** 기반 | **비동기**로 갱신됨 | 이종 데이터 시스템을 통합하는 장래성 있는 접근법 |

### 전체 순서화의 제약
규모가 커지고 복잡한 작업부하가 발생함에 따라 이벤트 로그의 순서 전체를 보장하는데 한계가 드러난다.
- 트래픽이 많아 **파티셔닝**하면 파티션 간의 순서를 보장할 수 없다.
- **다중 데이터센터**나 **마이크로서비스** 환경에서는 서로 다른 노드에서 발생한 이벤트 간의 순서를 명확히 정의하기 어렵다

### 인과성 획득을 위한 이벤트 순서화
순서가 없어도 괜찮은 데이터가 있지만 `인과성`이 중요한 데이터도 있다. 
예시: 사회 연결망 서비스에서 관계를 맺은 두 사용자가 관계를 끊은 경우
- 친구 상태를 저장하는 곳과 메시지를 저장하는 곳이 다른 시스템에서의 친구 끊기 이벤트와 메시지 보내기 이벤트 사이에는 의존성이 없다.
- 인과성을 획득하지 못한다면 새 메시지에 알림을 보내는 서비스가 메시지 보내기 이벤트를 친구 끊기 이벤트 전에 보내 이전 친구에게 잘못된 알림을 보낼 가능성이 있다.

#### ✔️ 해결 방법

| **방법** | **설명** |
| ---- | ---- |
| **논리적 타임 스탬프** | 코디네이션 없이 전체 순서화를 지원하여 도움이 될지 모르지만 수신자가 잘못된 순서로 전달된 이벤트를 처리해야 하고 추가로 메타데이터를 전달해야 한다. |
| **읽기 이벤트를 식별자로 함께 로깅** | 사용자가 본 시스템 상태를 기록하는 이벤트를 로깅하고 해당 이벤트에 고유 식별자를 부여해 이후 이벤트는 인과적 의존성을 기록하기 위한 이벤트 식별자를 참조 |
| **충돌 해소 알고리즘** | 예상치 못한 순서로 전송된 이벤트를 처리하는데 도움을 준다. 상태를 유지하는 데 유용하지만 활동에 외부 부수효과(사용자에게 알림을 보내는 작업 등)가 있다면 도움이 되지 않는다. |

## 일괄 처리와 스트림 처리
데이터 통합의 목표는 데이터를 올바른 형태와 장소에 두는 것이며 **일괄(Batch)** 처리와 **스트림(Stream)** 처리가 이를 수행하는 도구다.
일괄 처리와 스트림의 처리의 출력은 파생 데이터 셋이다.
- 데이터 셋: 검색 색인이나 구체화 뷰, 사용자에게 보여주기 위한 추천, 집계 지표 등
일괄 처리와 스트림 처리는 여러 공통 원리가 있다.
하지만 일괄 처리와 스트림 처리의 근본적인 주요 차이점은 스트림 처리는 끝이 없는 데이터 셋 상에서 운영되는 반면 일괄 처리는 유한한 크기의 입력을 사용한다는 것이다.
### 파생 상태 유지
**일괄 처리**는 결정적이고 출력이 입력에만 의존하여 명시적 출력 외에는 **다른 부수 효과가 없는 순수 함수**를 장려하며 입력을 불변으로 간주하고 출력은 추가 전용으로만 사용한다.
**스트림 처리**도 유사하지만 연산자를 확장해 상태를 관리할 수 있고 내결함성을 지니게 한다.
입력과 출력을 잘 정의한 결정적 함수의 원리는 내결함성에 도움이 될 뿐 아니라 조직 내의 데이터플로 추론을 단순화한다.

이론상으로 파생 데이터 시세틈은 관계형 데이터베이스가 색인할 테이블에 기록하는 트랜잭션 내에서 보조 색인을 동기식으로 갱신하는 것처럼 동기식으로 운영할 수 있다.
하지만 **비동기 방식**을 사용하면 이벤트 로그 기반 시스템을 훨씬 견고하게 만든다.
- 특정 시스템의 장애가 전체로 번지는 것을 막을 수 있어 견고
### 애플리케이션 발전을 위한 데이터 재처리
파생 데이터를 유지할 때 일괄 처리와 스트림 처리는 모두 유용하다.
- 스트림 처리를 이용하면 입력의 변화를 빠르게 파생 뷰에 반영 가능
- 일괄 처리를 사용하면 누적된 상당한 양의 과거 데이터를 재처리해 기존 데이터셋을 반영한 새 파생뷰를 만들 수 있다.
기존 데이터를 **재처리**하는 것은 시스템을 유지보수하기 위한 좋은 메커니즘으로 새로운 기능 추가와 변경된 요구사항에 대응할 수 있다.

> 파생 뷰를 사용하면 **점진적 발전**이 가능하다.

이전 스키마와 새 스키마를 함께 유지해 같은 데이터를 기반으로 두 개의 독립적인 파생 뷰를 만들 수 있다.
점진적으로 새 뷰를 접근하는 사용자의 비율을 늘려가서 기존 뷰를 결국에는 내릴 수 있게 된다.
- **장점: 처리의 모든 단계에서 뭔가 잘못됐을 때 쉽게 이전으로 되돌릴 수 있다.**
### 람다 아키텍처
일괄 처리를 과거 데이터를 재처리하는 데 사용하고 스트림 처리를 최근 갱신 데이터를 처리하는 데 사용한다고 할 때 어떻게 두 방식을 조합해 사용할 수 있을까?
- <span class="text-blue">**람다 아키텍처**</span>
:::callout 💡** 람다 아키텍처 핵심 아이디어**
:::callout 입력 데이터를 불변 이벤트로서 증가하기만 하는 데이터 셋에 추가하는 방식으로 기록해야 한다.
:::callout - `이벤트 소싱`과 유사
:::callout - 이 이벤트에서 읽기에 **최적화된 뷰**를 파생한다.
람다 아키텍처는 두 개의 다른 시스템을 병행해서 운용하기를 제안한다.
- 스트림 처리자는 이벤트를 소비해 근사 갱신을 뷰에 빠르게 반영
- 일괄 처리자가 같은 이벤트 집합을 소비해 정확한 버전의 파생 뷰에 반영

:::callout ⚠️ **람다 아키텍처의 실질적인 문제**
:::callout - 일괄 처리와 스트림 처리 양쪽 프레임워크에서 같은 로직을 유지해야 하는 데는 상당한 노력이 필요하다.
:::callout - 조인이나 세션화 같은 좀 더 복잡한 연산을 사용해 뷰를 만든다든지 출력이 시계열이 아니면 상당히 어려워진다.
### 일괄 처리와 스트림 처리의 통합
최근에는 람다 아키텍처의 단점을 빼고 장점을 취할 수 있게 하는 **같은 시스템에서 일괄 처리 연산과 스트림 연산을 모두 구현하는 작업**이 진행되고 있다.
- 과거 이벤트 재생 능력, 정확히 한 번(Exactly-once) 처리 보장, 그리고 처리 시간이 아닌 **이벤트 시간 기준의 윈도우 처리 기능**이 필수
# 데이터베이스 언번들링
유닉스와 관계형 데이터베이스는 정보 관리 문제를 다른 철학으로 접근했다.
**유닉스**는 논리적이지만 꽤 저수준인 하드웨어 추상화를 프로그래머에게 제공하는 것을 목적으로 뒀지만 **관계형 데이터베이스**는 디스크 상의 자료 구조, 동시성, 장애 복구 등의 복잡성을 감추는 고수준 추상화를 애플리케이션 프로그래머에게 제공하려 했다.
## 데이터 저장소 기술 구성하기
**데이터베이스에 내장된 기능과 일괄 처리와 스트림 처리로 구축하는 파생 데이터 시스템 사이**에는 유사점이 있다.
### 색인 생성하기
DB에서 `CREATE INDEX` 를 통해 새 색인을 생성하는 과정은 새 팔로워 복제본을 구축하는 과정과 스트림 시스템에서 `CDC`의 예비 과정과도 상당히 유사하다.
- 테이블의 일관된 스냅숏을 사용해 스캔하고 색인할 필드 값을 모두 골라 정렬하고 색인에 기록
- 이후 일관된 스냅숏을 만든 후 실행된 쓰기의 백로그를 처리
- 색인 생성을 완료하면 데이터베이스는 트랜잭션이 테이블에 쓸 때마다 꾸준히 색인에 반영
### 모든 것의 메타데이터베이스
**전체 조직의 데이터플로가 거대한 데이터베이스처럼 보이기 시작한다.**
1️⃣ **연합 데이터베이스 (federated database)**
- 많은 하단 저장소 엔진과 처리 메서드를 통합해 질의하는 인터페이스를 제공
- 다른 여러 저장소를 읽기 전용으로 질의하는 문제를 해결
- 여러 시스템에 걸친 쓰기를 동기화하기에는 적합하지 않음

2️⃣ **언번들링 데이터베이스: 쓰기를 통합**
- DB의 색인 유지 기능을 다른 기술에 걸친 쓰기를 동기화할 수 있는 방식
### 언번들링이 동작하게 만들기
쓰기를 동기화하는 전통적인 접근법은 **분산 트랜잭션**이다.
하지만 데이터가 다른 기술 사이의 경계를 오간다면 **멱등성을 기반으로 쓰기를 수행하는 **<u>**비동기 이벤트 로그**</u>**를 사용하는 편이 훨씬 더 강력하고 현실적인 접근법**이라고 본다.

로그 기반 통합의 큰 장점은 다양한 구성 요소간 <span class="text-red">**느슨한 결합(loose coupling)**</span>이다.
- 전체 시스템이 개별 구성 요소의 장애나 성능 저하가 생겨도 잘 견디게 할 수 있다.
- 구성요소와 서비스를 다른 팀에서 각자 개발하고 개선하고 독립적으로 유지보수가 가능하다.
### 언번들링 대 통합 시스템
언번들링이 실제로 미래에 사용될 방법이라고 가정해도 현재 형태의 데이터베이스를 대체하지는 못 할 것이다. 데이터베이스는 여전히 스트림 처리자의 상태를 유지하기 위해 필요하고 일괄 처리와 스트림 처리자의 출력에 대한 질의를 처리하기 위해서도 필요하다.
여러 다른 인프라에서 수행하는 복잡성도 문제가 될 수 있다. 
- 각 소프트웨어마다 학습 곡선과 설정 문제, 운영상에서만 나타나는 특성이 있다.
따라서 동적 부분을 가능하면 적게 배포해야 유리하다.

물론, 단일 통합 소프트웨어 제품은 설계된 용도에 맞춰 사용하면 애플리케이션 코드로 연결한 여러 도구로 구성된 시스템보다 뛰어나고 예측 가능한 성능을 낼 수 있다.
- 필요하지도 않은 확장성을 고려해 시스템을 구축하는 노력은 낭비일 뿐더러 유연하지 못한 설계에 갇힐 수 있다.

🎯 **언번들링의 목표는 단일 소프트웨어로 가능한 것보다 더 넓은 범위의 작업부하에 대해 좋은 성능을 달성하기 위함이다.**
## 데이터플로 주변 애플리케이션 설계
애플리케이션 코드로 특화된 저장소와 처리 시스템을 조립하는 언번들링 데이터베이스 접근법은 데이터베이스 <span class="text-red">**인사이드 아웃 접근법**</span>이라고도 한다.
:::callout ℹ️ **현대 데이터 시스템**
:::callout - 현대 데이터 시스템은 내결함성과 확장성이 있어야 하고 지속성 있게 데이터를 저장해야 한다.
:::callout - 또한 데이터 시스템은 시간이 흐름에 따라 **다른 그룹의 사람들이 개발한 기술과도 통합이 가능해야 할 뿐 아니라 이미 존재하는 라이브러리와 서비스를 재사용 가능**해야 한다.
### 파생 함수로서의 애플리케이션 코드
DB의 보조 색인용 파생 함수는 일반적인 요구사항이라 많은 데이터베이스에 핵심 기능으로 내장되어 있다.
좀 더 복잡한 기능을 지원하려면 특화된 튜닝이 종종 필요하다.

파생 데이터셋을 생성하는 함수가 보조 색인 생성 함수와 같은 비슷비슷한 표준 함수가 아니라면 사용자 정의 코드를 써서 애플리케이션에 특화된 측면을 다뤄야 한다.
- 트리거, 스토어드 프로시저, 사용자 정의 함수
### 애플리케이션 코드와 상태의 분리
오늘날 대부분의 웹 애플리케이션이 **무상태** 서비스로 배포된다.
그러나 상태는 어딘가에 꼭 있어야 하는데 일반적으로 데이터베이스가 된다.
- **상태 유지(Stateful, 데이터베이스)**와 **무상태(Stateless, 애플리케이션 서버)**애플리케이션 로직을 분리한다.
### 데이터플로: 상태 변경과 애플리케이션 코드 간 상호작용
데이터베이스를 애플리케이션이 직접 조작하는 수동적 변수로 취급하는 대신(주기적 폴링) 상태와 상태 변경, 그리고 상태를 처리하는 코드 간의 상호작용과 협동에 관해 좀 더 생각해 볼 수 있다.
- 데이터베이스에서 데이터 변경으로 트리거가 발생하거나 색인된 테이블에 변경사항을 반영하기 위해 보조 색인을 갱신할 때 비슷한 일이 발생한다.

> ⚠️  **파생 데이터를 유지하는 것이 전통적인 메시징 시스템의 설계 목적인 비동기 작업 실행과는 같지 않다는 것을 명심해야 한다.**
> - 파생 데이터를 유지할 때 상태 변경 순서가 중요할 때가 있지만 메시지 브로커가 이런 속성을 가지고 있지 않다.
> - 내결함성은 파생 데이터의 핵심이다. 메시지 전달과 파생 상태 갱신 양쪽 모두 반드시 신뢰성이 있어야 한다.

### 스트림 처리자와 서비스
최신 유행하는 애플리케이션 개발 스타일은 동기 네트워크(ex: REST API)요청을 통해 **통신하는 서비스**의 집합으로 나누는 것이다.
- **장점: 느슨한 연결을 통한 조직적 확장성**

**스트림 연산자로 데이터플로 시스템을 구성하는 것은 마이크로서비스 접근법**과 유사한 특징이 상당히 많지만 **기반이 되는 통신 메커니즘은 매우 다르다.**
- `마이크로서비스`: 동기식 요청-응답 상호작용
- `스트림 연산자로 구성된 시스템`: 단방향 비동기식 메시지 스트림
## 파생 상태 관찰하기
데이터플로 시스템은 파생 데이터셋을 생성하고 최신 상태로 유지하는 과정인 **쓰기 경로**와 파생 데이터셋을 질의하는 **읽기 경로**가 있다.

읽기 경로와 쓰기 경로를 종합하면 데이터를 수집하는 지점에서 데이터를 소비하는 지점까지 데이터의 모든 여정을 포함한다.
- 쓰기 경로: 데이터의 여정 중 미리 계산된 부분이다.
- 읽기 경로: 데이터의 여정 중에서 누군가 요청했을 때만 발생하는 부분이다.
### 구체화 뷰와 캐싱
`전문 검색 색인`은 좋은 예시이다.
- 쓰기 경로는 색인을 갱신하고 읽기 경로는 색인을 사용해 키워드를 찾는다.
**읽기와 쓰기 모두 어떤 작업이 필요하다.**
- 쓰기는 문서에 출현한 모든 용어의 색인 항목을 갱신해야 한다.
- 읽기는 질의에 포함된 각 단어를 검색한 다음, 질의에 포함된 모든 단어를 포함하는 문서를 찾거나 각 단어의 동의어 중 어떤 것이든 포함하는 문서를 찾기 위해 불 논리를 적용해야 한다.

**색인이 존재하지 않는다면** 검색 질의는 모든 문서를 스캔해야 한다.
- 색인이 없으면 쓰기 경로의 작업량이 줄지만 읽기 경로의 작업이 상당히 늘어난다.

반대로 모든 가능한 질의의 검색 결과를 미리 계산해놓는다면?
- 읽기 경로에서 처리하는 작업량이 줄어든다.
- 하지만 쓰기 경로에 많은 비용이 발생한다.

:::callout ℹ️ **공통 질의 캐시**
:::callout - 고정된 가장 공통적인 질의 집합의 검색 결과를 미리 계산해 두는 방법
:::callout - 색인까지 가지 않고 빠르게 처리 가능하다.
### 오프라인 대응 가능한 상태 저장 클라이언트
<span class="text-red">**오프라인 우선 애플리케이션**</span>은 인터넷 연결 요구 없이 같은 장치의 로컬 데이터베이스를 이용해 가능한 많은 일을 하고 네트워크 연결이 가능할 때 **백그라운드에서 원격 서버와 동기화한다.**
모바일 장치는 때때로 느리거나 인터넷 연결이 불안정하기 때문에 사용자 인터페이스가 동기식 네트워크 요청을 기다리지 않아도 된다면 그리고 앱이 대부분 오프라인으로 작동한다면 사용자에게 상당한 이점이다.
**무상태 클라이언트가 항상 중아 서버와 통신한다는 가정에서 벗어나 최종 사용자 장치에서 상태를 유지하는 쪽으로 나아가면 새로운 기회가 있는 세상이 열린다.**
- 장치 상 상태를 서버 상 상태의 캐시로 생각할 수 있다. 
### 상태 변경을 클라이언트에게 푸시하기
웹 브라우저에서 웹 페이지가 로드된 후 서버에서 데이터가 변경된다면 페이지가 새로 로드될 때까지 브라우저는 해당 변경 사항을 알 수 없다.
클라이언트가 변경사항을 폴링할 때까지 기다리지 말고 `WebSocket`이나 `서버 전송 이벤트(Server Sent Events)`를 통해 서버의 상태 변경을 클라이언트로 **push** 한다. 
- 서버의 로컬에 저장된 상태에 변경이 발생했을 때 서버에서 주도적으로 최종 사용자에게 알려줄 수 있어 클라이언트 측 상태의 신선도가 떨어지는 상황을 줄일 수 있다.
### 종단 간 이벤트 스트림
`엘름(Elm)` 언어와 `리액트(React)`, `플럭스(Flux)`, `리덕스(Redux)` 같은 상태 저장 클라이언트와 사용자 인터페이스 개발용 최신 도구는 이미 내부적으로 사용자 입력을 표현하는 **이벤트 스트림이나 서버 응답 스트림을 구독하는 방식을 사용해 클라이언트 측 상태를 관리**한다.

서버가 상태 변경 이벤트를 클라이언트 측 이벤트 파이프라인으로 푸시하게끔 프로그래밍 모델을 확장하는 것 또한 매우 자연스럽다.
- 따라서 **상태 변경은 종단 간(end-to-end) 쓰기 경로를 따라 흐를 수 있다.**
- 즉 상태 변경이 트리거된 한 장치의 상호작용으로 이벤트 로그를 거쳐 여러 파생 데이터 시스템과 스트림 처리자를 통해 다른 장치의 상태를 보고 있는 사람의 인터페이스까지 이어진다.
### 읽기도 이벤트다
저장소에 `쓰기`는 **이벤트 로그**를 거치지만 `읽기`는 질의 대상 데이터가 **저장된 노드로 직접 보내는 일시적 네트워크 요청**이다.
이는 합리적이지만 유일하게 가능한 설계는 아니다.
읽기 요청을 이벤트 스트림으로 표현하고 읽기 이벤트와 쓰기 이벤트 모두를 스트림 처리자를 통해 보내는 방법도 가능하다.
- 스트림 처리자는 읽기 결과를 출력 스트림으로 방출해 읽기 이벤트에 응답한다.

> 쓰기와 읽기 모두 이벤트로 표현하고 이벤트 처리를 위해 같은 스트림 연산자로 라우팅하는 것은 사실상 **읽기 질의 스트림과 데이터베이스 사이의 스트림 테이블 조인 수행과 동일**하다.
> - 읽기 이벤트는 해당 데이터를 가지고 있는 데이터베이스 파티션으로 보내야 한다.
> - 이는 일괄처리나 스트림 처리자가 조인할 때 입력을 같은 키를 기준으로 함께 파티셔닝해야 하는 방식과 거의 같다.


또한 읽기 이벤트 로그를 기록하면 잠재적으로** 인과적 의존성과 시스템 전체의 데이터 출처를 추적할 수 있다는 이점**이 있다.
- 읽기 이벤트 로그를 기록하면 사용자가 특정 결정을 내리기 전에 봤던 것을 재구성할 수 있다.
### 다중 파티션 데이터 처리
단일 파티션 조회에서는 스트림 처리가 오버헤드일 수 있지만 여러 파티션의 데이터 통합이 필요한 복잡한 질의를 분산 실행할 때는 유리하다.

**스톰의 분산 RPC 기능이 이런 사용 패턴을 지원한다.**
- 트위터에서 특정 URL을 본 사람 수를 계산하는 데 사용 가능하다.
- 이 URL을 트윗한 모든 사람의 팔로워 집합의 합집합을 구하는 작업으로 트위터 사용자 집합이 파티션돼 있다면 이 계산은 많은 파티션의 결과를 결합해야 한다.
# 정확성을 목표로
`무상태` 서비스에서 문제가 발생했을 때 재시작을 하면 정상으로 돌아오지만 데이터베이스와 같은 `상태 유지` 시스템은 데이터를 영구적으로 저장하게 설계되었기 때문에 무언가 잘못되면 그 효과가 영원히 지속될 수 있어 신중하게 생각할 필요가 있다.
애플리케이션을 구축하기 위해서 `ACID 트랜잭션`을 선택했었는데 이는 생각보다 약하다.
- **완화된 격리 수준**을 사용할 때 생기는 한계
이번 절에서 데이터플로 아키텍처의 맥락에서 정확성에 관한 몇 가지 방법을 제안한다.
## 데이터베이스에 관한 종단간 논증
애플리케이션이 **직렬성(Serializable)** 트랜잭션 같은 비교적 강력한 안전성 속성을 지원하는 데이터 시스템을 사용한다해도** 데이터 손실이나 손상이 없다고 보장할 수는 없다.**
애플리케이션 자체 버그로 인해 잘못된 데이터를 쓰거나 지운다면 직렬성 트랜잭션도 이런 문제를 해결해줄 수 없다.
### 연산자의 정확히 한 번 실행
장애가 발생해 메시지를 재시도했을 때 첫 번째 시도에 성공했을 때에도 실제로 성공했는지 확인할 수 없어 두 번째 메시지도 처리된다.
⚠️ **두 번 처리되는 것은 데이터 손상의 한 형태이다!!**

:::callout 한 번이든 여러 번 실행하든 관계 없이 같은 효과가 나타나는 것을 보장한다.
### 중복 억제
스트림 처리 외에도 많은 곳에서 동일한 중복 제거 패턴이 발생하는데 <span class="text-red">**TCP**</span>는 패킷의 일련 번호를 사용해 수신자에게 올바른 순서로 패킷을 전달하고 네트워크 상에서 패킷을 잃어버렸는지 중복됐는지 확인한다.
#### 👀 데이터베이스 클라이언트가 트랜잭션 `COMMIT`을 보낸 직후 네트워크가 끊겼다고 가정
해당 연결에 타임아웃이 발생했으나 데이터베이스 서버로부터 응답을 받지 못했다면 해당 트랜잭션이 커밋을 성공했는지 실패했는지 알 수 없다.
클라이언트는 데이터베이스에 재연결해 트랜잭션을 재시도할 수 있지만 이는 `TCP 중복 억제 범위`를 벗어나 버린다.
또한 사용자가 에러 메시지를 받고 수동으로 재시도를 하게되면 웹 서버 관점에서 재시도가 별도의 요청이고 데이터베이스 관점에서 재시도는 분리된 트랜잭션이다.
### 연산 식별자
여러 네트워크 통신 홉을 통과하는 연산을 멱등적으로 만들기 위해 데이터베이스가 제공하는 트랜잭션 메커니즘에 의존하는 것은 충분하지 않다. 요청의 **종단 간 흐름**을 생각할 필요가 있다.
예를 들어 **클라이언트 단에서 연산의 고유 식별자(UUID 등)를 만들어 종단 간으로 전달해야 한다.**
- 요청을 두 번 제출한다면 두 요청은 같은 연산 ID를 갖기 때문에 특정 ID에 대해 딱 한 번만 연산을 실행할 수 있다.
### 종단 간 논증(end-to-end argument)
중복 트랜잭션 억제 시나리오는 <span class="text-blue">**종단 간 논증**</span>이라 부르는 일반적인 원리의 한 예다.
> **“문제 기능은 통신 시스템의 종단점에 있는 애플리케이션 지식과 도움이 있어야만 완벽하고 정확하게 구현 가능하다.” (Saltzer, Reed, Clark - 1984)**

TCP의 중복 억제나 이더넷의 체크섬 같은 저수준의 신뢰성 기능은 유용하지만 그것만으로 종단 간 정확성을 보장하기에는 충분하지 않다. 
### 종단 간 사고를 데이터 시스템에 적용하기
**애플리케이션 자체가 중복 억제와 같은 종단 간 대책을 갖출 필요가 있다.**
<span class="text-blue">**트랜잭션**</span>은 **광범위한 문제(동시 쓰기, 제약 조건 위반, 충돌, 네트워크 끊김, 디스크 고장)**를 `커밋`과 `어보트` 두가지로 단순화해주는 훌륭한 **추상화이다**. 
하지만 트랜잭션은 비용이 많이든다. (특히 이종 저장소 기술을 함께 사용할 때 더 그렇다)
- 비용이 커 분산 트랜잭션 사용을 거부하면 애플리케이션 코드 안에서 내결함성 메커니즘을 재구현해야 한다.
따라서 대규모 분산 환경에서는 좋은 성능을 유지하면서도 애플리케이션에 특화된 종단 간 정확성을 제공하는 **내결함성 추상화**를 고민해야 한다.
## 제약 조건 강제하기
계좌 잔고가 마이너스가 되지 않게 하거나 비행기 좌석이 중복 예약되지 않게 하는 등의 유일성 조건은 어떻게 강제할 수 있을까?
### 유일성 제약 조건은 합의가 필요하다
분산 환경에서 `유일성`을 보장하려면 합의가 필요하다.
- 동시 요청 중 어떤 것을 승인하고 어떤 것을 거절할지 결정해야 하기 때문이다.
합의를 달성하는 가장 일반적인 방법은 단일 노드를 리더로 만들어 리더 노드에 요청을 몰아 해결하지만 다중 마스터 환경에서는 충돌이 발생해 유일성이 깨질 수 있다.
### 로그 기반 메시징의 유일성
로그는 모든 소비자가 동일한 순서로 메시지를 보도록 보장하는 <span class="text-red">**전체 순서 브로드캐스트**</span>를 제공한다.
#### 👀 예시: 같은 사용자명을 쓰고자 하는 여러 사용자의 경우
1. 사용자명 요청은 사용자명의 해시값을 기준으로 **특정 파티션에 추가**
2. 스트림 처리자가 로그에서 요청을 순차적으로 읽고 로컬 데이터베이스를 이용해 사용 가능한 이름이면 성공 메시지를 이미 사용중이라면 거부 메시지를 출력 스트림에 방출한다.
3. 클라이언트는 출력 스트림을 보며 보낸 요청에 해당하는 성공 또는 거부 메시지가 오기를 기다린다.
### 다중 파티션 요청 처리
데이터베이스의 전통적인 접근법에서는 트랜잭션을 실행할 때 모든 파티션에 걸쳐 원자적 커밋이 필요하다.
- 이러면 파티션 중 하나라도 쓰는 다른 모든 트랜잭션에 대해 전체 순서를 정해야 한다.
- 파티션에 걸친 코디네이션이 있기 때문에 더 이상 파티션을 독립적으로 처리하지 못하고 처리량에 대한 고민이 생긴다.

하지만 <span class="text-red">**파티셔닝된 로그**</span>를 사용하면 동등한 정확성을 달성할 수 있다.
#### 👀 예시: A 계좌에서 B 계좌로 돈을 이체할 때 A와 B가 서로 다른 파티션에 있는 경우
1️⃣ 클라이언트가 고유한 `요청 ID`를 붙여 **단일 로그 파티션에 이체 요청을 추가**
2️⃣ 스트림 처리자는 요청 로그를 읽어 A 계좌(출금), B계좌(입금) 파티션으로 각각 지시 메시지를 보냄
3️⃣  이 과정에서 장애가 나 후속 처리자가 재시작되더라도 `요청 ID`를 통해 각 계좌 파티션에 중복없이 처리
## 적시성과 무결성
데이터 일관성이라는 용어는 두 가지 개념을 합쳤다고 생각하는데 이는 분리해서 생각해 볼 가치가 있다.

| **구분** | **개념 / 특징** | **위반시** |
| ---- | ---- | ---- |
| **적시성(Timeliness)** | 사용자가 **항상 최신 상태의 데이터**를 관찰하는 것(ex: 선형성, 자신이 쓴 내용 읽기) | **결과적 일관성**: 잠시 기다렸다가 재시도하면 해결되는 일시적 문제 |
| **무결성(Integrity)** | 데이터의** 손실이나 모순이 없는** 상태 | **영구적 불일치**: 위반되면 재시도해도 고쳐지지 않는다. |

> 💡 **대부분의 애플리케이션에서 무결성이 적시성보다 훨씬 중요하다.**

### 데이터플로 시스템의 정확성
`ACID 트랜잭션`은 대개 적시성과 무결성을 모두 제공한다.
반면 **`이벤트 기반 데이터플로 시스템`****은 이 둘을 분리해서 처리한다.**
비동기로 처리해 명시적으로 메시지 도착을 기다리도록 구축하지 않는다면 적시성은 보장되지 않는다. 그러나 무결성이 스트림 시스템의 핵심이다.
- **정확히 한 번 처리와 멱등성을 통해 무결성을 보장한다.**
### 느슨하게 해석되는 제약 조건
많은 비즈니스 맥락에서 제약조건을 일시적으로 위반하고 나중에 사과해 바로잡는 것은 실제로 수용 가능한 방법이다.
- 초과 예약된 비행기 좌석 ➡️ 바우처 지급 및 다음 항공편 제공
- 재고보다 많이 팔린 물건 ➡️ 환불 및 할인 쿠폰 제공
### 코디네이션 회피 데이터 시스템
1️⃣ 데이터플로 시스템은 원자적 커밋, 선형성, 동기 코디네이션 없이도 파생 데이터에 대한 무결성을 보장할 수 있다.
2️⃣ 많은 애플리케이션은 느슨한 제약 조건을 사용해도 실제로 괜찮다. (나중에 복구 가능)
이 2가지를 통해 알 수 있는 것은 데이터플로 시스템은 코디네이션 없이도 많은 애플리케이션용 데이터 관리 서비스를 제공할 수 있을 뿐 아니라 무결성을 강력하게 보장해준다는 것이다.
이런 **코디네이션 회피 데이터 시스템**은 성능이 더 좋고 더 나은 내결함성을 가진다.
## 믿어라. 하지만 확인하라.
지금까지 정확성, 무결성, 내결함성에 대해 설명할 때 어떤 것을 잘못될 테지만 다른 것은 그렇지 않을 것이라 가정했다.
디스크 기록이나 CPU 연산이 항상 정확할 것이라 가정한다. 하지만 우주 방사선이나 하드웨어 결함에 의한 메모리 비트 플립 같은 문제가 발생할 수 있다.
### 소프트웨어 버그가 발생해도 무결성 유지하기
하드웨어 뿐 아니라 소프트웨어 버그의 위험성은 항상 존재한다.
`MySQL`이나 `PostgreSQL` 같은 좋은 평가를 받는 데이터베이스도 제약 조건이나 격리 수준에서 문제가 발생하고 한다.
애플리케이션이 데이터베이스를 정확하게 사용하지 않으면 데이터 무결성을 보장할 수 없다.
### 약속을 맹목적으로 믿지 마라
데이터가 안전할 것이라고 모든 것이 잘 동작한다고 맹목적으로 믿지 마라. 
S3나 HDFS 같은 대규모 저장소 시스템은 디스크를 완전히 믿지 않고 백그라운드 프로세스가 지속적으로 파일을 다시 읽어 다른 복제본과 비교하는 작업을 수행한다. 백업 데이터 역시 실제로 복원 테스트를 해보지 않으면 안전을 보장할 수 없다.
:::callout ℹ️ **감사(Auditing)**
:::callout - 데이터 무결성을 체크하는 작업
### 감사 기능 설계
트랜잭션 로그를 보더라도 다양한 테이블 내에서 발생한 삽입, 갱신, 삭제만으로 왜 이런 변경이 일어났는지 추적하기 어렵다.
반면** 이벤트 소싱은 사용자 입력을 불변 이벤트로 기록하여 감사가 용이하다. **
데이터의 출처가 명확해지고 해시를 통해 데이터 손상을 확인하고 언제든 로직을 재시도하여 같은 결과가 나오는지 확인할 수 있다.
### 다시 종단 간 논증
데이터 무결성 검증 방법은 `종단 간 방식`이 최선이다.
종단 간 전체 파생 데이터 파이프라인이 정확한지 확인이 가능하다면 파이프라인을 따르는 디스크와 네트워크, 서비스와 알고리즘이 이 확인에 암묵적으로 포함된다.
# 옳은 일 하기
> **모든 시스템은 목적을 가지고 구축된다.**

**시스템을 구축하는 엔지니어는 결과를 신중하게 생각해서 의식적으로 어떤 세상에 살고 싶은지 결정할 책임이 있다.**
소프트웨어 엔지니어는 오로지 기술에만 집중하고 그 영향을 무시하는 것이 아니라 소프트웨어 엔지니어도 **윤리적 책임**을 져야 한다. 
## 예측 분석
예측 분석은 `빅데이터` 판촉에 중요한 부분이다.
- ex) 날씨 예측이나 질병 확산 예측 분석에 데이터를 사용하는 경우

날씨를 예측하는 것과 채무자의 채무 불이행 가능성 또는 기결수의 재범 여부 등을 예측하는 것은 전혀 다른 문제이다.
:::callout ⚠️ **알고리즘 감옥 (Algorithmic prison)**
:::callout 알고리즘에 의한 의사 결정이 보편화되면서 사실이든 아니든 특정 알고리즘으로 위험성이 있다고 판단된 사람은 많은 `아니오` 결정으로 고통받는다.
:::callout - 시스템 적으로 사회의 주요 측면에서 배제되는 일은 개인 자유의 커다란 제약사항이다.
:::callout - 자동화 시스템은 시스템적으로나 임의로 유죄 증명 없이 그리고 변론할 기회도 거의 없이 사람을 사회 참여에서 배제시킬 수 있다.
### 편견과 차별
> **알고리즘이 내린 결정이 사람이 내린 결정보다 반드시 좋거나 나쁜 것은 아니다. **

알고리즘에 투입된 입력에 시스템적 `편견`이 있다면 시스템은 그 편견을 학습하고 그 편견을 증폭해서 출력을 내보낼 가능성이 높다.
예측 분석 시스템은 오직 과거로부터 추정하기 때문에 과거가 차별적이라면 예측 분석 시스템은 차별을 성문화한다.
### 책임과 의무
의사결정 자동화는 아직 책임과 의무에 대한 문제가 해결되지 않았다.
사람이 실수를 하면 책임을 질 수 있고 영향을 받은 사람은 이의를 제기할 수 있지만 **알고리즘이 실수하면 누가 책임을 지어야 할까?**
머신러닝 알고리즘은 투명하지 않아 차별 여부등을 입증하기 어렵고 특정 집단 통계가 개별적으로는 잘못될 수 있다.
따라서 의사결정에 데이터를 최우선으로 하는 맹목적 믿음은 절대적으로 위험하다.
### 피드백 루프

## 사생활과 추적
데이터를 사용해 사람에 대한 결정을 자동화하는 문제 외에도 `데이터 수집` 그 자체로 윤리적 문제가 있다.
- 데이터를 수집하는 조직과 수집당하는 사람들 사이의 관계는 무엇일까??

사용자가 시스템이 특정 방식으로 데이터를 저장하고 처리하길 원하기 때문에 시스템이 사용자가 입력한 데이터만 저장한다면 시스템과 사용자 관계는 서비스를 제공하는 주체와 고객이 된다.
💣 그러나 사용자의 활동을 추적하고 사용자가 하는 행동의 부수 효과로서 로그를 남긴다면 이 관계가 불명확하다.
- 서비스는 더 이상 사용자가 원하는 것만 하지 않고 사용자의 이익과 충돌이 발생할 수 있는 자신의 관심사를 수집한다.

:::callout 행동 데이터 추적(사용자 추적)은 주로 개인을 위한 것이 아닌 서비스에 돈을 대는 광고주의 필요로 제공한다.
:::callout 이 관계를 **감시** 라는 단어가 적절히 설명한다고 생각한다.
### 감시
`데이터`란 단어를 `감시`로 바꾸면 사람들은 자발적이고 열정적으로 전체 감시 세계에 자신을 던지고 있다.
### 동의와 선택의 자유
사용자가 사용자 행동을 추적하는 서비스 사용을 선택하고 서비스 약관과 개인 정보 보호 정책에 동의했기 때문에 데이터 수집에 동의했다고 주장할 수 있다.
**사용자는 사용자가 제공하는 데이터가 무엇이고 어떻게 유지하며 처리하는지에 대한 지식이 거의 없다.**
- 사용자가 사용자의 데이터가 어떤 일을 하는지 알지 못한 채 받은 동의는 의미가 없다.

> **서비스와 사용자 간의 관계는 매우 비대칭적이고 일방적이다.**
> - 데이터는 일방적인 과정을 통해 사용자로부터 추출하며 사용자가 데이터를 얼마나 제공하고 대가로 받는 서비스가 무엇인지 협상할 수 있는 대화나 선택 사항이 없다.
> - 약관은 사용자가 아닌 서비스가 설정한다.

**감시**에 동의하지 않는 사람이 선택할 수 있는 대안은 서비스를 사용하지 않는 것뿐이다. 사실상 서비스를 사용하도록 **강제**한다.
### 사생활과 데이터 사용
`사생활`은 누구에게 어떤 것을 보여줄 것인지, 어떤 것을 공개로 할 것인지, 어떤 것을 비밀로 할 것인지 고를수 있는 자유를 가진다는 의미이다. 사생활 보호 권리는 결정권이다.
기업은 데이터 수집이 실제로 얼마나 침입적인지에 대한 질문은 피하는 대신 사용자 인식을 관리하는 데 집중한다.
- 심지어 사용자 인식도 관리가 매우 형편없다.

**서비스는 사용자 데이터에 제한 없이 접근할 수 있고 데이터를 개인 정보 보호 정책에서 허용된 방식으로 사용하는 것은 자유다. **서비스 자체에는 데이터를 내부적으로 처리하고 분석할 권리를 제한 없이 허용한다.
이런 서비스에서 종종 사용자에게 명백히 보이는 것보다 훨씬 지나치게 데이터를 사용할 때가 있다.
**인터넷 서비스는 의미 있는 동의를 받지 않고도 막대한 양의 민감한 정보를 축적하기 쉽게 만들었다.**
또한 인터넷 서비스는 사용자가 자신의 사생활 데이터에 무슨 일이 생기는지 이해하지도 못한 채로 이런 데이터를 거대한 규모로 사용하기도 쉽게 만들었다.
### 자산과 권력으로서의 데이터
사용자가 서비스와 상호작용을 하면서 나오는** 행동 데이터는 서비스의 핵심 자산**이다.
데이터가 범죄자나 해외 정보 기관에게 침해당할 수도 있고 내부자가 데이터를 유출하거나 비양심적 경영자의 손에 떨어지거나 억압적인 정부의 손에 넘어갈 수도 있다.
오늘날 기술 기업들이 축적한 데이터와 지식은 공공의 감시를 벗어나 기업에게 많은 권력을 부여한다.

### 법률과 자기 규제
데이터 보호법으로 개인의 권리를 보호할 수 있다지만 오늘날 인터넷 맥락에 효과적인지는 의심스럽다.
이 규칙은 직접적으로 `빅데이터` 철학을 거스른다.
**빅데이터**는 최대로 데이터를 수집하고 다른 데이터셋과 수집한 데이터를 결합해 새로운 통찰을 얻기 위해 실험하고 탐구한다. 탐구는 예측하지 못한 목적으로 데이터를 사용한다는 의미이다.

소프트웨어에 의존하는 사람들의 신뢰를 확립하고 유지하기 위해서는 데이터 수집과 처리를 수행할 때 자기 규제가 필요하다.
기업과 소프트웨어 개발자는 사용자 데이터가 어떻게 사용되는지 감추지 않고 최종 사용자에게 교육할 책임이 있다.
