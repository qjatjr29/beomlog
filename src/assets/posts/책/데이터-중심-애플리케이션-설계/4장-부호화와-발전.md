애플리케이션은 필연적으로 시간이 지남에 따라 변한다.
새로운 제품을 출시하거나 사용자 요구사항을 이해하게 되거나 비즈니스 환경이 변함에 따라 애플리케이션 기능은 추가되거나 변경된다.
대부분의 경우 애플리케이션 기능을 변경하기 위해 저장하는 데이터도 변경해야 한다.

데이터 타입이나 스키마가 변경될 때 애플리케이션의 코드에 대한 변경이 종종 발생하지만 대규모 애플리케이션에서 코드 변경은 대개 즉시 반영할 수 없다.
이는 **예전 버전의 코드와 새로운 버전의 코드, 이전의 데이터 타입과 새로운 데이터 타입이 모든 시스템에 동시에 공존**할 수 있다는 의미이다.
- 시스템이 계속 원활하게 실행되게 하려면 양방향으로 호환성을 유지해야 한다.

> **하위 호환성**
> - 새로운 코드는 예전 코드가 기록한 데이터를 읽을 수 있어야 한다.


> **상위 호환성**
> - 예전 코드는 새로운 코드가 기록한 데이터를 읽을 수 있어야 한다.

## **데이터 부호화 형식**
프로그램은 보통 두 가지 형태로 표현된 데이터를 사용해 동작한다.
1. **인메모리:** 구조체, 객체, 해시테이블 등 CPU가 효율적으로 접근하고 조작할 수 있게(포인터 이용) 최적화된 상태.
2. **바이트열:** 데이터를 파일에 쓰거나 네트워크를 통해 전송하기 위한 상태. 
  - 인메모리 표현에서 바이트열로의 전환을 **부호화**라고 하며 그 반대를 **복호화**라고 한다.

### **JSON과 XML, 이진 변형**
> 텍스트 기반 포맷 (JSON, XML, CSV)


**장점**
- 사람이 읽기 편하고 널리 표준화되어 있다. 
- 데이터 교환 형식으로 사용하기 좋다.

**단점**
- XML은 지나치게 복잡하고 장황하다.
- 수의 부호화에 많은 애매함이 있다. 

**Binary Encoding (이진 부호화)**
작은 데이터셋의 경우에는 부호화 형식 선택으로 얻는 이득이 무시할 정도지만 테라바이트 정도가 되면 데이터 타입의 선택이 큰 영향을 미친다.
JSON은 XML보다 덜 장황하지만 이진 형식과 비교하면 둘 다 훨씬 많은 공간을 사용한다.
MessagePack, BSON 등을 사용하면 JSON을 이진으로 압축할 수 있다.
- JSON을 이진 부호화 한다고 해도 객체의 필드 이름을 포함해야 한다.

### **스리프트와 프로토콜 버퍼**
`아파치 스리프트(Apache Thrift)`와 `프로토콜 버퍼(Protocol Buffers)`는 같은 원리를 기반으로 한 이진 부호화 라이브러리이다.

스리프트로 데이터를 부호화하려면 <u>**스리프트 인터페이스 정의 언어(interface definition language, IDL)**</u>로 스키마를 기술해야 한다.
```bash
# IDL
struct Person {
	1:required string userName, 
	2:optional i64 favoriteNumber, 
	3:optional list<string> interests 
}
```


**프로토콜 버퍼**로 정의한 스키마는 스리프트 스키마와 매우 비슷하다.
```bash
message Person {
	required string user_name = 1; 
	optional int64 favorite_number = 2; 
	repeated string interests = 3; 
}
```


스리프트와 프로토콜 버퍼는 이런 스키마 정의를 통해 코드를 생성하는 도구가 있고 생성된 코드를 호출해 스카미의 레코드를 부호화, 복호화할 수 있다.

💡 **특징**
필드 이름이 없다.
대신 부호화된 데이터는 숫자 같은 <span class="text-red">**필드 태그(field tag)**</span>를 포함한다.
- 숫자는 스키마 정의에 나타난 숫자.
- 필드 태그는 어떤 필드를 다루는지 알려주는 방법

👀 **호환성(하위, 상위) 유지 방식**
- 시간에 지남에 따라 스키마는 변한다. (**schema evolution, 스키마 발전)**

| **호환성 타입** | 방법 |
| ---- | ---- |
| **상위 호환성** | 예전 코드가 최신 데이터를 읽을 때 자신이 모르는 필드 태그 번호가 나오면 해당 데이터의 타입(길이)만큼 바이트를 건너뛰고 무시한다 |
| **하위 호환성** | 삭제할 수 있는 필드는 `optional` 뿐이며 최초 배포 이후에는 새로운 `required` 필드를 추가할 수 없다. 추가되는 모든 필드는 `optional`로 하거나 기본값을 가져야 한다. |

### **아브로 (Apache Avro)**
- **프로토콜 버퍼와 스리프트와 다른 또 하나의 이진 부호화 형식.**
- 부호화할 데이터 구조를 지정하기 위해 스키마를 이용 (두 가지)
- 사람이 편집할 수 있는 **아브로 IDL**
- 기계가 더 쉽게 읽을 수 있는 **JSON** 기반 언어

**아브로 IDL로 작성한 예제 스키마**
```bash
record Person {
	string userName;
	union { null, long } favoriteNumber = null;
	array<string> interests;
}
```


예제 스키마와 동일한 **JSON 표현**
```json
{
  "type": "record",
  "name": "Person",
  "fields": [
    {"name": "userName", "type": "string"},
    {"name": "favoriteNumber", "type": ["null","long"], "default": null},
    {"name": "interests","type": {"type": "array","items": "string"}}
  ]
}
```


**특징**
- 인코딩된 이진 데이터 안에 **필드 이름도 없고 필드 태그 번호도 없다.**
- 모든 부호화중 길이가 가장 짧다.

바이트열을 살펴보면 필드나 데이터타입을 식별하기 위한 정보가 없다.
- 부호화는 단순히 연결된 값으로 구성

아브로를 이용해 이진 데이터를 파싱하기 위해서는 스키마에 나타난 순서대로 필드를 살펴보고 스키마를 이용해 각 필드의 데이터타입을 <u>**미리**</u> 파악해야 한다.
> **데이터를 읽는 코드가 데이터를 기록한 코드와 정확히 같은 스키마**를 사용하는 경우에만 이진 데이터를 올바르게** 복호화**할 수 있음을 의미.


**쓰기 스키마와 읽기 스키마**

| **쓰기 스키마** | 애플리케이션이 인메모리 데이터를 파일, 데이터베이스, 또는 네트워크로 내보내기 위해 **부호화할 때 기준이 되는 스키마** |
| ---- | ---- |
| **읽기 스키마** | 애플리케이션이 파일, 데이터베이스, 또는 네트워크로부터 수신한 바이트열 데이터를 다시 인메모리 객체로 **복호화할 때 기대하는 스키마** |


아브로의 핵심 아이디어는 **쓰기 스키마와 읽기 스키마가 동일하지 않아도 되며 단지 호환 가능**하면 된다는 것.
데이터를 복호화(읽기)할 때 아브로는 쓰기 스키마와 읽기 스키마를 함께 살펴본 후 <u>쓰기 스키마에서 읽기 스키마로 데이터를 변환</u>해 차이를 해소한다.
- ex) 쓰기, 읽기 스키마는 필드 순서가 달라도 스키마 해석에서 이름으로 필드를 일치시키기 때문에 문제 없다.
- ex) 데이터를 읽는 코드가 읽기 스키마에는 없고 쓰기 스키마에 존재하는 필드를 만나면 해당 필드는 무시
- ex) 데이터를 읽는 코드가 기대하는 어떤 필드가 쓰기 스키마에 포함되어 있지 않는 경우 읽기 스키마에 선언된 기본값으로 채움.

👀 **호환성 유지**
아브로에서 **상위 호환성**은 새로운 버전의 쓰기 스키마와 예전 버전의 읽기 스키마를 가질 수 있음을 의미
**하위 호환성**은 새로운 버전의 읽기 스키마와 예전 버전의 쓰기 스키마를 가질 수 있음을 의미

| 호환성 타입 | 스키마 조합 | 호환성 달성을 위한 필수 제약 (규칙) |
| ---- | ---- | ---- |
| **상위 호환성** | 새로운 버전의 쓰기 스키마 **+** 예전 버전의 읽기 스키마 | **기본값(default)이 있는 필드만 삭제 가능**(최신 스키마에서 필수 필드를 삭제해 버리면 과거 코드는 그 필드를 읽으려다 에러가 발생함) |
| **하위 호환성** | 새로운 버전의 읽기 스키마 **+** 예전 버전의 쓰기 스키마 | **기본값(default)이 있는 필드만 추가 가능**(최신 코드가 새 필드를 원할 때 과거 데이터에는 값이 없으므로 기본값으로 채워야 함) |

> **동적 생성 스키마**
> 프로토콜 버퍼와 스리프트에 비해 아브로 방식은 **스키마에 태그 번호가 포함되지 않은 장점**이 존재.

*이는 아브로가 동적 생성 스키마에 더 친숙하다는 점을 보여준다.*

각 데이터베이스 테이블에 맞게 레코드 스키마를 생성하고 각 칼럼은 해당 레코드의 필드가 된다.
- 데이터베이스의 칼럼 이름은 아브로의 필드 이름에 매핑

데이터베이스 스키마가 변경되면 갱신된 데이터베이스 스키마로부터 새로운 아브로 스키마를 생성하고 새로운 아브로 스키마로 데이터를 내보낸다.
- 데이터를 내보내는 과정은 스키마 변경에 신경 쓸 필요가 없다.
- 스리프트나 프로토콜 버퍼를 이런 용도로 사용시 필드 태그를 수동으로 할당해주어야 한다.
## **데이터플로 모드**
하나의 프로세스에서 다른 프로세스로 데이터를 전달하는 추상적인 개념
- 데이터베이스를 통한 방법
- 서비스 호출을 통한 방법
- 비동기 메시지 전달을 통한 방법
### **데이터베이스를 통한 데이터플로**
데이터베이스에 기록하는 프로세스는 데이터를 부호화하고 읽는 프로세스는 복호화한다
- 새로운 버전의 애플리케이션이 기록한 데이터를 예전 버전의 애플리케이션이 갱신할 때 해당 데이터가 유실되지 않도록 주의해야 한다.

### **서비스를 통한 데이터플로: REST와 RPC**
네트워크를 통해 통신해야하는 프로세스가 있을때 해당 통신을 배치하는 몇가지 방법이 있다
가장 일반적인 방법으로는 클라이언트와 서버의 두 역할로 배치해 서버가 네트워크를 통해 API를 공개하고 클라이언트는 이 API로 요청을 만들어 서버에 연결한다.

**웹서비스**
- 서비스와 통신하기 위한 기본 프로토콜로 HTTP를 사용할 때 이를 웹 서비스라고 한다
- 웹서비스에는 대중적인 두가지 방식 REST와 SOAP이 있다

↗️ **REST**
HTTP를 원칙을 토대로 한 설계 원칙으로 간단한 데이터 타입을 강조하며 URL을 사용해 리소스를 식별하고 캐시 제어, 인증, 콘텐츠 유형 협상에 HTTP 기능을 사용

**SOAP**
네트워크 API 요청을 위한 XML 기반 프로토콜
- HTTP 상에서 일반적으로 사용되지만 HTTP와 독립적이며 대부분의 HTTP 기능을 사용하지 않는다.

**원격 프로시저 호출(RPC, Remote Procedure Call) 문제**
RPC(Remote Procedure Call)는 원격 네트워크 서비스 요청을** 같은 프로세스 안에서 특정 프로그래밍 언어의 함수나 메서드를 호출하는 것**과 동일하게 사용 가능하게 해준다.

⚠️ **RPC가 처음에는 편리한 것 같지만 RPC 접근 방식은 근본적으로 결함이 있다**

| 문제점 | 설명 |
| ---- | ---- |
| 실패 원인이 외부에 있음 | 로컬 함수는 내 프로세스 안에서 실행되지만 네트워크 요청은 서버 장애나 네트워크 문제로 실패할 수 있음 |
| 타임아웃의 모호성 | 응답이 오지 않으면 요청이 아예 도착하지 않았는지 이미 처리되었지만 응답만 유실된 것인지 알 수 없음 |
| 재시도의 위험 | 같은 요청을 다시 보내면 실제 작업이 중복 수행될 수 있으므로 멱등성이나 중복 제거 설계가 필요함 |
| 성능과 지연의 불안정성 | 로컬 함수 호출보다 훨씬 느리고 지연 시간도 상황에 따라 크게 달라질 수 있음 |
| 포인터 전달 불가 | 네트워크를 넘길 때는 객체 참조를 직접 넘길 수 없고 데이터를 바이트로 직렬화해서 보내야 함 |
| 언어 간 타입 차이 | 클라이언트와 서버가 다른 언어로 작성될 수 있으므로 RPC 프레임워크가 데이터 타입 변환 문제까지 처리해야 함 |

**RPC의 현재 방향** 
이러한 문제에도 RPC는 사라지지 않고 지금까지 언급한 모든 부호화 위에 다양한 RPC 프레임워크가 개발되었다.
- `gRPC`는 프로토콜 버퍼를 이용한 RPC 구현
### **메시지 전달 데이터플로**
`REST`와 `RPC`는 하나의 프로세스가 네트워크를 통해 다른 프로세스로 요청을 전송하고 가능한 빠른 응답을 기대하는 방식.
`데이터베이스`는 하나의 프로세스가 부호화한 데이터를 기록하고 다른 프로세스가 언젠가 해당 데이터를 다시 읽는 방식

<span class="text-red">**비동기 메시지 전달 시스템**</span>은 클라이언트 요청을 낮은 지연 시간으로 다른 프로세스에 전달한다는 점에서 `RPC`와 비슷하다.
메시지를 직접 네트워크 연결로 전송하지 않고 임시로 메시지를 저장하는 **메시지 브로커**나 **메시지 지향 미들웨어라는 중간 단계**를 거쳐 전송한다는 점은 `데이터베이스`와 유사하다.

메시지 전달 통신은 일반적으로 단방향이라는 점이 `RPC`와 다르다.

**메시지 브로커 (RabbitMQ, Apache Kafka 등)**
프로세스 하나가 메시지를 이름이 지정된 **큐**나 **토픽**으로 전송하고 브로커는 해당 큐나 토픽 하나 이상의 **소비자** 또는 **구독자**에게 메시지를 전달한다.
- 동일한 토픽에 여러 생산자와 소비자가 있을 수 있다.
메시지 브로커는 특정 데이터 모델을 강요하지 않고 메시지는 일부 메타데이터를 가진 바이트열로 모든 부호화 형식을 사용할 수 있다.

**분산 액터 프레임워크**
액터 모델은 단일 프로세스 안에서 동시성을 위한 프로그래밍 모델이다.
- 스레드(경쟁 조건, Locking, deadlock 와 연관된 문제들)를 직접 처리하는 대신 로직이 액터에 캡슐화된다.
- 로직을 액터라는 단위로 캡슐화하고 비동기 메시지로만 소통
