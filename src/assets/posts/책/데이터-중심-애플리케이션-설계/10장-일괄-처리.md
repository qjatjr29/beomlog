현대 데이터 시스템에서 데이터 처리 방식은 시스템에 요청한 후 시스템으로부터 결과를 반환받는 방식이다.
이런 시스템에서는 <span class="text-red">**응답 시간 **</span>단축에 노력을 많이 기울인다.

이 `요청-응답` 방식이 매우 흔해져 당연한 것으로 여기기 쉽지만 시스템을 구축하는 유일한 방식은 아니다!!
1️⃣ **서비스(온라인 시스템)**
- 서비스는 클라이언트로 요청이나 지시가 올 때까지 기다린다.
- 요청이 들어오면 서비스는 가능한 빨리 요청을 처리해 응답을 되돌려 보내고자 한다.
- `응답시간`은 서비스 성능을 측정할 때 중요한 지표이다.

2️⃣ **일괄 처리 시스템(오프라인 시스템)**
- 매우 큰 입력 데이터를 받아 데이터를 처리하는 작업을 수행하고 결과 데이터를 생산한다.
- 이는 수 분에서 때로 수 일이 걸려 대부분 사용자가 작업이 끝날 때까지 대기하지 않는다.
- 입력 데이터중 특정 크기만큼 처리할 때 걸리는 시간인 `처리량`이 주요 성능 지표이다.

3️⃣ **스트림 처리 시스템(준실시간 시스템)**
- 온라인과 오프라인/일괄 처리 사이 어딘가에 있는 준실시간 처리
- 요청에 대해 응답하지 않으며 입력 데이터를 소비하고 출력 데이터를 생산한다.
- 하지만 스트림 처리는 입력 이벤트가 발생한 직후 바로 동작한다.(일괄 처리 작업은 정해진 크기의 입력 데이터를 대상으로 함)

> 💡 <span class="text-red">**일괄 처리**</span>는 신뢰할 수 있고 확장 가능하며 유지보수하기 쉬운 애플리케이션을 구축하는데 매우 중요한 구성요소이다.

## 맵리듀스와 분산 파일 시스템
<span class="text-red">**맵리듀스**</span>는 수천 대의 장비로 분산해 실행이 가능하다.
- 단일 맵리듀스 작업은 하나 이상의 입력을 받아 하나 이상의 출력을 만들어 낸다.
- 맵리듀스 작업은 입력을 수정하지 않아 출력을 생산하는 것 외에 다른 부수 효과가 없다.
- 출력 파일은 순차적으로 한 번만 쓰여지고 파일에 이미 쓰여진 부분은 고치지 않는다.

맵리듀스 작업은 분산 파일 시스템상의 파일을 입력과 출력으로 사용한다..
- HDFS(하둡 맵리듀스 구현에서의 파일시스템, Hadoop Distributed File System)

### HDFS
HDFS는 <u>**비공유 방식**</u>으로 특별한 하드웨어가 필요 없이 일반적인 데이터센터 네트워크에 연결된 컴퓨터면 충분하다.
HDFS는 각 장비에서 실행되는 데몬 프로세스로 구성된다. 
데몬 프로세스는 다른 노드가 해당 장비에 저장된 파일에 접근 가능하게끔 네트워크 서비스를 제공하고 **네임노드**라고 부르는 중앙 서버는 특정 파일 블록이 어떤 장비에 저장됐는지 추적한다.

## 맵리듀스 작업 실행하기
맵리듀스는 HDFS 같은 분산 파일 시스템 위에서 대용량 데이터셋을 처리하는 코드를 작성하는 프로그래밍 프레임워크이다.
### 맵리듀스의 데이터 처리 패턴
1. 입력 파일을 읽어 레코드로 쪼갠다.
2. 각 입력 레코드마다 `매퍼` 함수를 호출하여 키와 값을 **추출**한다.
3. 키를 기준으로 키-값 쌍을 모두 **정렬**한다.
4. 정렬된 키-값 쌍 전체를 대상으로 `리듀스` 함수를 호출한다.
### 매퍼와 리듀서
맵리듀스 작업을 생성하려면 <span class="text-red">**매퍼**</span>와 <span class="text-red">**리듀서**</span>라는 두 가지 콜백함수를 구현해야 한다.
> **매퍼(Mapper)**
> - 모든 입력 레코드마다 한 번씩만 호출된다.
> - 매퍼는 입력 레코드로부터 **키와 값을 추출하는 작업**이다.
> - 각 레코드를 독립적으로 처리한다.


> **리듀서(Reducer)**
> - 매퍼가 생상한 키-값 쌍을 받아 같은 키를 가진 레코드를 모으고 해당 값의 집합을 반복해 리듀서 함수를 호출한다.
> - 리듀서는 **출력 레코드를 생**산한다.
> - 출력 레코드 예시: 동일한 URL이 출현한 횟수

### 맵 리듀스의 분산 실행
맵리듀스는 <u>**병렬로 수행하는 코드를 직접 작성하지 않고도 여러 장비에서 동시에 처리가 가능**</u>하다.
매퍼와 리듀서는 한 번에 하나의 레코드만 처리하고 입력이 어디서 오는지 출력이 어디로 가는지 신경쓰지 않는다.
- 맵리듀스 프레임워크가 장비 간 데이터가 이동하는 복잡한 부분을 처리하기 때문

대부분의 경우 맵 태스크에서 실행되는 애플리케이션 코드는 작업이 할당된 장비에 아직 존재하지 않기 때문에 맵리듀스 프레임워크가 작업을 수행하기에 적절한 코드를 복사한다.
복사가 끝난 후 장비에서 매퍼 태스크가 시작되고 입력파일을 읽기 시작하면 입력 파일에서 한 번에 레코드 하나씩 읽어 매퍼 콜백 함수로 전달한다.

**리듀서 연산도 파티셔닝이 된다**.
- 맵 태스크 수는 입력 파일의 블록 수로 결정되지만 리듀스 태스크 수는 사용자가 설정한다.
맵리듀스 프레임워크는 같은 키를 가진 모든 키-값 쌍을 같은 리듀서에서 처리하는 것을 보장한다.
- 키의 해시 값을 사용해 키-값 쌍이 어느 리듀스 태스크에서 수행될지 결정한다.
### 맵리듀스 워크플로
맵리듀스 작업을 연결해 워크플로로 구성하는 방식은 일반적이다.
- 맵리듀스 작업 하나의 출력을 다른 맵리듀스 작업의 입력으로 사용하는 방식
일괄 처리 작업의 출력은 작업이 성공적으로 끝났을 때만 유효하기 때문에 워크플로 상에서 해당 작업의 입력을 생성하는 선행 작업이 끝나야만 다음 작업을 시작할 수 있다.
- 하둡 맵리듀스 작업 간 수행 의종성을 관리하기 위해 다양한 스케줄러가 개발됐다.
## 리듀스 사이드 조인과 그룹화
### 정렬 병합 조인
매퍼 출력을 키로 정렬된 후에 리듀서가 조인의 양측의 정렬된 레코드 목록을 병합한다.
### 같은 곳으로 연관된 데이터 가져오기
병합 정렬 조인 중 매퍼와 정렬 프로세스는 조인 연산을 할 때 필요한 모든 데이터를 한 곳으로 모은다.
- 같은 키를 가진 키-값 쌍은 모두 같은 리듀서 호출한다.
- 맵리듀스는 데이터 모으는 연산의 물리적 네트워크 통신과 데이터를 처리하는 애플리케이션 로직을 분리한다.
- 맵리듀스는 부분적으로 실패가 발생하더라도 애플리케이션 로직에 영향이 가지 않게 실패한 태스크는 확실하게 재시도한다.
### 그룹화
맵리듀스로 그룹화 연산을 구현하는 가장 간단한 방법은 매퍼가 키-값 쌍을 생성할 때 그룹화할 대상을 키로 하는 것이다.
- 파티션 및 정렬 프로세스가 같은 키를 가진 모든 레코드를 같은 리듀서로 모은다.
### 쏠림 다루기
유명 인사 한 사람에 대한 활동을 리듀서 한 개에 모으면 상당한 **쏠림 현상**이 생긴다.(`핫스팟`)
- 한 리듀서가 다른 리듀서보다 많은 레코드를 처리해야 한다는 의미
⚠️ 모든 래퍼와 리듀서가 완전히 끝나야지만 맵리듀스 작업이 끝나기 때문에 **가장 느린 리듀서가 작업을 완료할 때까지** 후속 작업들은 시작하지 못한 채 기다려야 한다.

핫스팟을 완화할 몇 가지 알고리즘
- Pig의 쏠린 조인(skewed join)
- Crunch의 공유 조인(shared join)
- Hive의 맵 사이드 조인
핫 키로 레코드를 그룹화하고 집계하는 작업은 2단계로 수행된다.
1. 레코드를 임의의 리듀서로 보내 핫 키 레코드의 일부를 그룹화하고 키별로 집계해 간소화한 값을 출력
2. 첫 단계 모든 리듀서에서 나온 값을 키별로 모두 결합해 하나의 값으로 만든다.
## 일괄 처리 워크플로의 출력
> 일괄 처리는 어디에 적합할까 🤔

트랜잭션 처리도 아니고 분석도 아니다. 일괄처리는 입력 데이터셋 대부분을 스캔하는 것이 일반적이라 분석에 더 가깝다. 
그러나 분석 목적으로 사용하는 SQL 질의와 다르고 출력은 보고서가 아닌 다른 형태의 구조이다.
### 검색 색인 구축
맵리듀스는 `구글`에서 검색 엔진에 사용할 색인을 구축하기 위해 처음 사용됐었다.
- 더 이상 구글이 색인 구축 목적으로 사용하지 않음.

**정해진 문서 집합을 대상**으로 **전문 검색이 필요하다면 일괄 처리가 색인을 구축하는 데 매우 효율적**이다.
매퍼는 필요에 따라 문서 집합을 파티셔닝하고 각 리듀서가 해당 파티션에 대한 색인을 구축한다.
- 색인 파일은 분산 파일 시스템에 저장된다.
- 문서 기준으로 파티셔닝해 색인을 구축하는 과정은 병렬화가 잘된다.
### 일괄 처리의 출력으로 키-값을 저장
검색 색인 뿐 아니라 일괄 처리 워크플로 출력의 다른 예로 머신러닝 시스템을 구축하거나 추천 시스템을 구축할 수 있다.
이런 일괄 처리 작업의 출력은 일종의 데이터베이스가 된다.
- ex) 사용자에게 추천할 친구를 가져오기 위해 사용자 ID로 질의하는 데이터베이스 
일괄 처리 작업 내부에 새로운 데이터베이스를 구축해 분산 파일 시스템의 작업 출력 디렉터리에 저장.
## 맵리듀스를 넘어
일괄 처리 방법의 대안을 살펴보자.
### 중간 상태 구체화
모든 맵리듀스 작업은 다른 작업과 모두 독립적으로 한 작업의 출력은 다른 작업의 입력으로만 사용된다.
이 경우에 분산 파일 시스템 상에 있는 파일들은 단순히 한 작업에서 다른 작업으로 데이터를 옮기는 수단, **중간 상태**다.
- 복잡한 워크플로는 많은 맵리듀스 작업이 구성되는데 여기에는 많은 중간 상태가 존재한다.
> **구체화**
> - 중간 상태를 파일로 기록하는 과정


중간 상태를 완전히 구체화하는 맵리듀스 접근법은 단점이 있다.
1. 맵리듀스 작업은 입력을 생성하는 모든 선행 작업이 완료됐을 때만 시작 가능하다.
  - 모든 선행 작업 태스크가 종료될 때가지 기다리면 워크플로 전체 수행 시간은 느려진다.
2. 매퍼는 중복되기도 한다.
  - 이 매퍼들은 리듀서에서 막 기록된 동일한 파일을 읽어 다음 단계인 파티셔닝과 정렬단계를 준비한다.
3. 분산 파일 시스템에서 중간 상태를 저장하는 것은 여러 장비에 복제됐다는 것으로 이런 임시데이터에게 과잉조치이다.
### 데이터플로 엔진
맵리듀스에 있는 문제를 해결하기 위해서 분산 일괄 처리 연산을 수행하는 엔진이 몇가지 개발됐다.
- `스파크`, `테즈`, `플링크` 
이 엔진들에는 공통점 하나가 있다.
> **전체 워크플로를 독립된 하위 작업으로 나누지 않고 작업 하나로서 다룬다.**


이 엔진들은 여러 처리 단계를 통해 데이터 흐름을 명시적으로 모델링하기 때문에 이 시스템을 데이터 플로 엔진이라 부른다.
- 맵리듀스처럼 단일 스레드에서 사용자 정의 함수를 반복 호출해 한번에 레코드 한 개씩 처리
- 입력을 파티셔닝해 병렬화한다.
- 한 함수의 출력을 다른 함수의 입력으로 사용하기 위해 네트워크를 통해 복사
### 내결함성
분산 파일 시스템에서 중간 상태를 모두 구체화하면 `내구성`에 이점이 생긴다.
태스크가 실패하더라도 다른 장비에서 태스크를 재실행할 수 있고 파일 시스템으로부터 동일한 입력을 다시 읽을 수 있다.

스파크와 플링크, 테즈는 중간 상태를 쓰지 않기 때문에 내결함성 확보를 위해 다른 접근법을 사용한다.
- 유효한 데이터로부터 계산을 다시 해 복구
