시스템이 신뢰성을 가지기 위해서는 여러 결함을 처리하여 전체 시스템의 치명적인 장애로 이어지는 것을 막아야 한다(내결함성을 갖춘 시스템)
- 구현하기 위해서는 할 일이 많다

수십년 동안 <span class="text-red">**트랜잭션**</span>은 이런 문제를 단순화하는 메커니즘으로 사용해왔다.
> 💡 **트랜잭션**
> - 애플리케이션에서 몇 개의 읽기와 쓰기를 하나의 논리적 단위로 묶는 방법

트랜잭션은 전체가 성공하거나 실패한다.(`커밋` or `롤백`)
- 트랜잭션 실패시 애플리케이션에서 안전하게 재시도
- 오류처리를 하기 훨씬 단순해진다.
- 부분 실패를 걱정할 필요없다.
# 애매모호한 트랜잭션의 개념
트랜잭션은 이점과 한계가 있다.
이 트레이드오프를 이해하기 위해 정상적인 운영상황과 다양한 극단적인(그러나 현실적인) 환경에서 트랜잭션이 제공하는 보장의 세부사항을 살펴보자.
## ACID 의미
트랜잭션이 제공하는 **안정성 보장**은 흔히 `원자성(Atomicity)`, `일관성(Consistency)`, `격리성(Isolation)`, `지속성(Durability)`를 의미하는 <span class="text-red">**ACID**</span>로 잘 알려져 있다.
**데이터베이스마다 ACID 구현이 제각각이다.**
- 격리성의 의미 주변에는 모호함이 많이 있다.

### 1️⃣ **원자성**
원자성은 여러 변경을 적용할 때 어떤 변경은 효과가 있고 어떤 것은 그렇지 않은 문제를 단순하게 만들어준다.
- 트랜잭션이 실패했다면 애플리케이션에서 해당 트랜잭션이 어떤 것도 변경하지 않았음을 알 수 있어 안전하게 재시도가 가능
**원자성은 동시성과 관련이 없다.**
- 이 문제는 `격리성`과 관련

### 2️⃣ **일관성**
일관성은 항상 진실이어야 하는, 데이터에 관한 <u>**어떤 선언(불변식)**</u>이 있다는 것이다.
일관성을 유지하도록 트랜잭션을 올바르게 정의하는 것은 애플리케이션의 책임이다.
- `원자성, 격리성, 지속성(데이터베이스의 속성)`과 달리 **일관성은 애플리케이션 속성**

### 3️⃣ **격리성**
클라이언트들이 동일한 데이터베이스 레코드에 접근하면 <span class="text-red">**동시성**</span> 문제가 발생한다.
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F0bdd530c-740b-4af7-bbf9-06c01449c414%2Fimage.png?table=block&id=348d67b9-9e80-80cf-bd9b-eff2bec84773&cache=v2" alt="image" width="1216" height="394" loading="lazy" />

**격리성은 동시에 실행되는 트랜잭션은 서로 격리된다는 것을 의미한다.**
- 트랜잭션은 다른 트랜잭션을 방해할 수 없다.

**직렬성**: 각 트랜잭션이 전체 데이터베이스에서 실행되는 유일한 트랜잭션인 것처럼 동작할 수 있다는 의미
- 데이터베이스는 실제로는 여러 트랜잭션이 동시에 실행됐더라도 커밋됐을 때 결과가 트랜잭션이 순차적으로 실행됐을 때의 결과와 동일하도록 보장한다.

### 4️⃣ **지속성**
지속성은 트랜잭션이 성공적으로 커밋됐다면 하드웨어 결함이 발생하거나 데이터베이스가 죽더라도 **트랜잭션에서 기록한 모든 데이터는 손실되지 않는다는 보장**이다.
단일 노드 데이터베이스에서 지속성은 비휘발성 저장소에 기록이 되었다는 뜻이다.
- 저장된 데이터 구조가 오염되었을 때 복구할 수 있게 해주는 `쓰기 전 로그(WAL)`나 비슷한 방법을 동반

복제 기능이 있는 데이터베이스에서 지속성은 데이터가 성공적으로 다른 노드 몇개에 복사가 됐다는 것을 의미
- 쓰기나 복제가 완료될 때까지 기다렸다가 트랜잭션이 성공적으로 커밋됐다고 보고
## 단일 객체 연산과 다중 객체 연산
### 다중 객체 연산
다중 객체 트랜잭션은 데이터의 여러 조각이 동기화된 상태로 유지되어야 할 때 필요하다.

👀 예시: 사용자의 읽지 않은 메시지 개수를 보여주기 (이메일 애플리케이션)
```sql
 SELECT COUNT(*) FROM emails WHERE recipient_id = 2 AND unread_flag = true
```

이때, 이메일이 너무 많아 질의가 느려지면 읽지 않은 메시지 개수를 별개의 필드로 저장하도록 할 수 있다.(일종의 비정규화)
- 새로운 메시지가 올 때마다 읽지 않은 메시지 개수도 증가
- 메시지를 읽을 때마다 읽지 않은 메시지 개수 감소
⚠️ <span class="text-red">**이상현상 (dirty read)**</span>
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F2f6919af-7064-469c-bc31-8ff72f05dfec%2Fimage.png?table=block&id=348d67b9-9e80-8010-bc60-f550e1a47901&cache=v2" alt="image" width="1230" height="518" loading="lazy" />

- `격리성 위반`
- 메일함에는 읽지 않은 메시지가 있지만 메시지 개수는 증가되지 않아서 개수가 0개로 표시

⚠️ <span class="text-red">**이상현상 **</span>
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F8f83e6a4-e854-454e-9352-474346736e9a%2Fimage.png?table=block&id=348d67b9-9e80-8042-b429-e4d1478b48c9&cache=v2" alt="image" width="1222" height="362" loading="lazy" />

- `원자성 위반`
- 트랜잭션 실행 도중 어느 시점에서 오류가 발생 시 메임함의 내용과 읽지 않은 메시지 개수가 동기화되지 않을 수 있다.
- 원자적 트랜잭션에서는 개수 갱신에 실패하면 이전에 실행한 쓰기(이메일 삽입)을 취소(롤백)해 일관성이 깨진 상태가 되지 않게 한다.

다중 객체 트랜잭션은 어떤 읽기 연산과 쓰기 연산이 동일한 트랜잭션에 속하는지 알아낼 수단이 필요하다.
- `BEGIN TRANSACTION ~ COMMIT` 사이의 모든 것은 같은 트랜잭션에 속하는 것으로 여겨진다.
### 단일 객체 쓰기
단일 객체 연산은 여러 클라이언트에서 동시에 같은 객체에 쓰려고 할 때 갱신 손실을 방지하므로 유용하다.
- 하지만 일반적으로 쓰이는 의미의 트랜잭션이 아니다.
**트랜잭션은 보통 다중 객체에 대한 다중 연산을 하나의 실행단위로 묶는 메커니즘으로 이해된다.**
### 다중 객체 트랜잭션의 필요성
단일 객체 삽입, 갱신, 삭제만으로 충분한 사례도 있지만 많은 다른 경우에는 여러 개의 다른 객체에 실행되는 쓰기 작업은 코디네이션되어야 한다.
1. 관계형 데이터 모델에서는 한 테이블이 다른 테이블의 로우를 참조(외래키)
  - 서로 참조하는 레코드를 삽입할 때 참조 키는 알맞아야 하며 최신 정보를 반영해야 한다.
2. 문서 데이터베이스는 비정규화를 하기도 한다.
  - 비정규화된 정보를 갱신할 때는 한 번에 여러 문서를 갱신해야 한다.
  - 트랜잭션은 비정규화된 데이터가 동기화가 깨지는 것을 방지하는데 유용
3. 보조 색인이 있는 데이터베이스
  - 트랜잭션 관점에서 색인은 서로 다른 데이터베이스 객체이다.
  - 트랜잭션 격리성이 없다면 어떤 색인에서는 레코드가 보이지만 다른 색인은 아직 갱신되지 않아 레코드가 보이지 않을 수 있다.

트랜잭션이 없더라도 구현할 수 있지만 `원자성`이 없다면 오류처리가 훨씬 복잡해지고 `격리성`이 없다면 동시성 문제가 발생할 수 있다.
# 완화된 격리 수준
동시성 문제(경쟁 조건)은 트랜잭션이 다른 트랜잭션에서 동시에 변경한 데이터를 읽거나 두 트랜잭션이 <span class="text-red">동시에 같은 데이터를 변경</span>하려고 할 때만 나타난다.
데이터베이스는 **트랜잭션 격리**를 제공함으로 애플리케이션 개발자들에게 동시성 문제를 감추려 했다.
- 이론상 격리성은 동시성이 없는 것처럼 행동할 수 있음
- `Serializable` 격리는 데이터베이스가 여러 트랜잭션들이 순차적으로 실행되는 것과 동일한 결과가 나오도록 보장함

⚠️ 하지만 현실에서는 격리가 그리 간단치 않다.
- `Serializable` 격리는 성능 비용이 발생
**따라서 어떤 동시성 이슈로부터는 보호해주지만 모든 이슈로부터 보호해주지는 않는 완화된 격리 수준을 사용하는 시스템들이 흔하다.**
🚀 **신뢰성 있고 올바르게 동작**하는 애플리케이션을 위해 존재하는 **동시성 문제의 종류를 잘 이해하고 방지**하는 방법을 배울 필요가 있다.
## Read Committed - 커밋 후 읽기
가장 기본적인 수준의 트랜잭션 격리는 <span class="text-blue">`read committed(커밋 후 읽기)`</span>이다.
이 격리 수준은 2가지를 보장해준다.
1️⃣ 데이터베이스에서 **읽을 때 커밋된 데이터**만 보게된다 (`no dirty read`)
2️⃣ 데이터베이스에 **쓸 때 커밋된 데이터**만 덮어쓰게 된다. (`no dirty write`)
### 더티 읽기 방지
> 📖 **dirty read**
> 트랜잭션이 데이터베이스에 데이터를 썼지만 아직 커밋이나 어보트가 되지 않은 상황에서 다른 트랜잭션에서 커밋되지 않은 데이터를 볼 수 있는 경우

<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F07b0b3a3-e104-4022-98d2-77f3ab4ab7e3%2Fimage.png?table=block&id=348d67b9-9e80-80d5-9a54-c626c02b01c5&cache=v2" alt="image" width="1268" height="402" loading="lazy" />

<span class="text-blue">**`read committed`**</span>** 격리 수준에서의 트랜잭션은 ****`dirty read`****를 막아야 한다.**
- 트랜잭션이 쓴 내용은 커밋된 후에야 다른 트랜잭션에 보인다는 의미.
### 더티 쓰기 방지
> 📖 **dirty write**
> 두 트랜잭션이 데이터베이스에 있는 동일한 객체를 동시에 갱신하려는 상황
> - 먼저 쓴 내용이 아직 커밋되지 않은 트랜잭션에서 쓴 내용이고 나중에 실행된 쓰기 작업이 커밋되지 않은 값을 덮어쓰는 경우

<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F83f6d4e6-c276-4067-b6ad-787111fd2731%2Fimage.png?table=block&id=348d67b9-9e80-8018-835b-d492a433ff9b&cache=v2" alt="image" width="1282" height="602" loading="lazy" />

<span class="text-blue">**`read committed`**</span>** 격리 수준에서의 트랜잭션은 ****`dirty write`****를 막아야 한다.**
- 보통 먼저 쓴 트랜잭션이 커밋되거나 어보트될 때까지 두 번째 쓰기를 지연시키는 방법을 사용
### Read Committed 한계
⚠️ 그러나 <span class="text-blue">**`read committed`**</span>** **은 두 번의 카운터 증가 사이에 발생하는 경쟁 조건은 막지 못한다.
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F229bc7b1-5c0a-410b-895d-60d5a7bd4b7d%2Fimage.png?table=block&id=348d67b9-9e80-80ad-b44a-e22cad0e6cb8&cache=v2" alt="image" width="1288" height="416" loading="lazy" />

이 경우 첫 번째 트랜잭션이 커밋된 후 두 번째 쓰기가 일어났으므로 `dirty write`이 아니다.
- 이후 <u>**갱신 손실 방지**</u>에서 카운터 증가를 아전하게 만드는 방법을 다룸
### Read Committed 구현
<span class="text-blue">**`read committed`**</span>** **은 많이 사용되는 격리 수준이다.(PostgreSQL, Oracle 11g 등)

로우 수준 잠금을 사용해 `dirty write` 방지
- 오직 한 트랜잭션만 주어진 객체에 대한 잠금을 보유
- 다른 트랜잭션에서 동일한 객체에 쓰기를 원한다면 잠금을 가지고 있는 트랜잭션이 커밋되거나 어보트된 후에야 잠금을 얻어 진행가능

`dirty read` 방지
- 데이터베이스는 과거에 커밋된 값과 현재 쓰기 잠금을 가지고 있는 트랜잭션에서 쓴 새로운 값을 기억
- 해당 트랜잭션이 실행되는 동안 그 객체를 읽는 다른 트랜잭션들은 이전의 값을 읽음.

앨리스는 은행에 `1,000` 달러를 저축했고 두 계좌에 `500` 달러씩 나눈 상황
그녀의 계좌중 하나에서 다른 계좌로 `100` 달러를 전송하는 트랜잭션을 실행.
트랜잭션이 처리되는 동안 계좌 잔고를 보게되면 한 계좌는 입금이 되기전 상태(`500`달러)를 보고 다른 계좌는 출금이 된 후 상태(`500→400`)를 볼 수도 있다.
- 계좌 총액이 `900` 달러만 있는 것처럼 보임.
이런 이상 현상을 <span class="text-red">**non-repeatable read(비반복 읽기)**</span> 나 <span class="text-red">**read skew(읽기 스큐)**</span>라고 한다.
- <span class="text-blue">`read committed`</span> 격리 수준에서는 발생할 수 있다.
### 스냅샷 격리
💡 **스냅샷 격리는** 이런 문제의 가장 흔한 해결책이다.
각 트랜잭션은 데이터베이스의 **일관된 스냅샷**으로부터 읽는다.
즉, 트랜잭션은 시작할 때 데이터베이스에 커밋된 상태였던 모든 데이터를 본다.
- 데이터가 나중에 다른 트랜잭션에 의해 바껴도 각 트랜잭션은 특정한 시점의 과거 데이터를 본다.

스냅샷 격리의 핵심 원리는 <u>**읽는 쪽에서 쓰는 쪽을 차단하지 않고 쓰는 쪽에서 읽는 쪽을 차단하지 않는다는 것**</u>이다.
데이터베이스는 객체마다 커밋된 버전 여러 개를 유지할 수 있어야 한다.
- 진행중인 여러 트랜잭션에서 서로 다른 시점의 데이터베이스 상태를 봐야할 수도 있기 때문
- <span class="text-red">**MVCC (다중 버전 동시성 제어. multi-version concurrency control)**</span>

위 그림에서는 PostgreSQL에서의 MVCC 기반 스냅샷 격리를 구현한 방법을 나타낸다.
트랜잭션이 시작하면 계속 증가하는 고유한 `트랜잭션 ID(txid)`를 할당받아 트랜잭션이 데이터베이스에 데이터를 쓸 때마다 쓰기를 실행한 **트랜잭션의 ID**가 함께 붙게된다.

👀 **일관된 스냅**샷**을 보는 가시성 규칙**
데이터베이스는 가시성 규칙을 정의하여 일관된 스냅샷을 애플리케이션에게 제공할 수 있다.
1️⃣ 데이터베이스는 각 트랜잭션을 시작할 때 그 시점에 진행중인 모든 트랜잭션의 목록을 만든다.
- **해당 트랜잭션들이 생성한 데이터는 모두 무시한다.**
- 아직 완료되지 않은 작업이므로 결과가 바뀔 수 있음 (Dirty Read 방지)
2️⃣ 어보트된 트랜잭션이 쓴 데이터는 모두 무시
- 롤백된 트랜잭션이 생성한 데이터는 존재하지 않는 것으로 간주한다.
3️⃣ 트랜잭션 ID가 더 큰(현재 트랜잭션이 시작한 후에 시작한) 트랜잭션이 쓴 데이터는 그 트랜잭션의 커밋 여부에 관계없이 모두 무시된다.
- 내 스냅샷 이후 생긴 변화는 모두 무시
4️⃣ 그 밖의 모든 데이터는 애플리케이션의 질의로 볼 수 있다.
- 이미 커밋되었고 현재 트랜잭션 시작 이전에 완료된 데이터
### 색인과 스냅샷 격리
<span class="text-blue">**다중 버전 데이터베이스**</span>에서는 하나의 `row`가 아래처럼 존재
```plain text
id=1 → version A (과거)
     → version B (현재)
     → version C (더 최신)
```

즉, 같은 데이터가 여러버전으로 존재

> 🤔 **그렇다면 인덱스는 어디를 가리켜야 하나?**

**방식1. 인덱스는 모든 버전을 가리킨다.**
```plain text
Index → (A, B, C 전부 가리킴)
```

이후 질의를 할 때 현재 트랜잭션을 기준으로 볼 수 없는 버전은 걸러냄
- 인덱스 탐색 → 여러 버전 가져옴
- 각 버전에 대해서 "이거 내 스냅샷에서 보이는 데이터인가?" 판단
- 보이는 것만 결과로 사용

**방식 2. Copy-on-Write (append-only B-tree)**
- CouchDB, LMDB에서 사용
데이터 수정 발생시 해당 페이지를 **복사해서 수정**
- 부모 노드도 복사해서 새 페이지 가리킴
- 결국 root까지 새로 만들어짐
```plain text
Root1 → 과거 스냅샷
Root2 → 최신 스냅샷
Root3 → 더 최신
```

각 root 자체가 완전한 시점의 DB 상태 (스냅샷 자체가 분리)
## 갱신 손실 방지
동시에 실행되는 쓰기 트랜잭션 사이에 발생할 수 있는 충돌이 몇 가지 더 존재한다.
가장 널리 알려진 것은 <span class="text-red">**갱신 손실(lost update)**</span> 문제이다.
> ⚠️ **갱신 손실**
> **데이터베이스에서 값을 읽고 변경한 후 변경된 값을 다시 쓸 때 발생할 수 있다.**
> 두 트랜잭션이 이 작업을 동시에 할 경우 두 번째 쓰기 작업이 첫 번째 변경을 포함하지 않아 변경중 하나가 손실될 수 있다.

### 원자적 쓰기 연산
여러 데이터베이스에서는 `원자적 갱신 연산`을 제공한다.
- 이 연산은 애플리케이션 코드에서 `read-modify-write` 주기를 구현할 필요를 없애준다.

```sql
UPDATE counters SET value = value + 1 WHERE key = 'foo';
```

이런 명령은 대부분의 관계형 데이터베이스에서 <u>**동시성 안전**</u>하다.
- **읽기 + 수정 + 쓰기가 DB 내부에서 모두 한 번에 처리**

👀 **동시 실행 상황 가정**
```sql
T1: UPDATE 실행 → row lock 획득
T2: UPDATE 실행 → lock 때문에 대기

T1: value = 10 → 11로 업데이트 → commit
T2: 이제 실행 → value = 11 → 12로 업데이트
```

원자적 연산은 `row`에 `exclusive lock`을 걸어 구현한다.
- 다른 트랜잭션은 갱신이 적용될 때까지 읽지 못함 (접근 못함)
- 순서대로 처리

⚠️ <span class="text-blue">**객체 관계형 매핑 프레임워크(ORM)**</span>을 사용하면 데이터베이스가 제공하는 **원자적 연산을 사용하는 **대신 불안전한 `read-modify-write` 를 실행하는 코드를 작성하기 쉽다.
```java
@Transactional
public void increase() {
    Counter c = repository.findById("foo");
    c.setValue(c.getValue() + 1);
    repository.save(c);
}
```

👀** 이 상황에서 동시에 두 요청이 들어오면??**
```plain text
T1 시작 (@Transactional)
T2 시작 (@Transactional)

T1: SELECT → 10
T2: SELECT → 10   

T1: UPDATE → 11
T1: commit

T2: UPDATE → 11
T2: commit
```

**결과가 ****`12`****가 아니라 ****`11`**** !!**
두 개의 트랜잭션이 정상적으로 동작하고 에러도 없고 로그도 이상이 없는데 결과가 틀림
### Explicit Locking (명시적인 잠금)
데이터베이스에 내장된 원자적 연산이 필요한 기능을 제공하지 않을 때 갱신 손실을 막는 다른 선택지는 갱신할 객체를 명시적으로 잠그는 것이다.
- 애플리케이션이 `read-modify-write` 를 수행할 수 있고 다른 트랜잭션이 동시에 같은 객체를 읽으려고 하면 첫 번째 처리가 완료될 때까지 기다리도록 강제
## 쓰기 스큐와 팬텀
### 쓰기 스큐
두 트랜잭션이 두 개의 다른 객체를 갱신하지만 트랜잭션이 동시에 실행됐기 때문에 이상 동작이 나타날 수 있다
- 서로 다른 데이터를 수정한다 (같은 row가 아님)
- 따라서 **lost update나 dirty write는 발생하지 않는다**
- 하지만 비즈니스 규칙은 깨질 수 있다
- 즉, 데이터 자체는 일관되지만 전체 조건은 깨지는 상황

### 예시: 온콜 의사 시스템
> ✔️ 항상 최소 1명의 의사가 당직 상태여야 한다

초기 상태
```plain text
Alice: on-call
Bob: on-call
```

동시 실행 상황
```plain text
T1 (Alice):
- 현재 on-call 수 조회 → 2
- 문제 없다고 판단
- Alice off-call로 변경

T2 (Bob):
- 현재 on-call 수 조회 → 2
- 문제 없다고 판단
- Bob off-call로 변경
```

👀** 결과**
```plain text
Alice: off
Bob: off
```

> ❌ 아무도 `on-call` 아님 → 규칙 위반


🤔** 왜 이런 일이 발생??**
- 두 트랜잭션 모두 **snapshot isolation** 하에서 실행
- 동일한 스냅샷을 보고 판단
- 서로의 변경은 보지 못함
### 팬텀 (Phantom)
> 📖 **팬텀**
> 어떤 트랜잭션에서 실행한 쓰기가 다른 트랜잭션의 검색 질의 결과를 바꾸는 효과


> 🔗 **Write Skew와 Phantom의 관계**
> Write Skew는 Phantom으로 인해 발생할 수 있다


`예시`
- <u>조건을 만족하는 데이터가 없다</u> 를 확인
- 이후 해당 조건을 만족하는 데이터를 INSERT
⚠️ 조회 시점에는 `row`가 없어서 락을 걸 수 없음

### 충돌 구체화
팬텀 문제에서의 문제는 락을 걸 대상이 없다는 것으로 락을 걸 수 있는 대상을 만들어 처리하는 것이다.
👀 **예시: 회의실 예약**
별도의 테이블 생성
```plain text
[ room_id + time_slot ]

Room 123 - 12:00~12:15
Room 123 - 12:15~12:30
...
```

- 예약 시 해당 `time slot row`를 `SELECT ... FOR UPDATE`
- 락 획득 > 기존 예약 확인 > 예약 생성
이런 방식은 설계가 복잡하고 실수하기 쉽고 비즈니스 로직에 동시성 제어 메커니즘이 적용되는 문제등이 있다. 따라서, Serializable 격리 수준을 더 선호된다.
# Serializability (직렬성)
**Serializable 격리는 보통 가장 강력한 격리 수준이라 여겨진다.**
여러 트랜잭션이 병렬로 실행되더라도 **최종 결과는 동시성 없이 한 번에 하나씩 순차적으로 실행**될 때와 같도록 보장한다.
- 데이터베이스가 발생할 수 있는 모든 경쟁 조건을 막아준다.
## 실제적인 직렬 실행
동시성 문제를 피하는 가장 간단한 방법 = 동시성을 완전히 제거
- 한 번에 트랜잭션 하나씩만 직렬로 **단일 스레드**에서 실행

### Stored Procedure 방식
**상호작용식 트랜잭션**
- 애플리케이션이 질의를 실행하고 결과를 읽어 질의 결과에 따라 다른 질의를 실행
- 애플리케이션 코드와 데이터베이스 사이에서 질의와 결과를 주고 받는다.
- 이는 네트워크 통신에 많은 시간을 소비하고 데이터베이스에서 동시성을 허용하지 않고 한 번에 트랜잭션 하나씩만 처리한다면 처리량이 매우 좋지 않다.

**Stored Procedure 방식**
트랜잭션 코드 전체를 DB 내부에서 실행
- 네트워크 왕복 제거 
- 빠른 실행이 가능

⚠️ **단점**
- 데이터베이스 벤더마다 스토이드 프로시저용 언어가 있는데 매우 조잡
- 데이터베이스에서 실행되는 코드는 관리하기 어려움
- 잘못된 코드가 있다면 애플리케이션 서버에 미치는 것보다 데이터베이스에 훨씬 더 곤란한 상황을 만들 수 있다.
### Partitioning
모든 트랜잭션을 순차적으로 처리하면 동시성 제어는 훨씬 간단해지지만 **데이터베이스의 트랜잭션 처리량이 단일 장비에 있는 단일 CPU 코어의 속도로 제한된다.**
💡 여러 CPU 코어와 여러 노드로 확장하기 위해 데이터를 <span class="text-blue">**파티셔닝**</span>할 수도 있다.
- 각 파티션은 독립적으로 직렬 실행

⚠️ **문제**
- 여러 파티션에 접근해야 하는 트랜잭션이 있다면 데이터베이스가 해당 트랜잭션이 접근하는 모든 파티션에 걸쳐 코디네이션을 해야 한다.
- 트랜잭션이 단일 파티션에서 실행될 수 있는 지 여부는 애플리케이션에서 사용되는 데이터 구조에 크게 의존한다.
## 🙋2단계 잠금(2PL)
<span class="text-red">**2PL**</span>에서 쓰기 트랜잭션은 다른 쓰기 트랜잭션뿐만 아닌 **읽기 트랜잭션도 진행하지 못하게 막고** 그 역도 성립한다.
### 2단계 잠금 구현
<span class="text-blue">**MySQL(InnoDB)**</span>에서 2PL은 **Serializable 격리 수준**을 구현하는데 사용된다.
**읽는 쪽과 쓰는 쪽을 막는 것은 데이터베이스의 각 객체에 잠금을 사용해 구현한다.**
- `공유 모드(shared mode)`와 `독점 모드(exclusive mode)`로 사용될 수 있다.

1️⃣ **공유 모드**
트랜잭션이 객체를 **읽기** 원한다면 공유 모드로 잠금을 획득해야 한다.
- 동시에 여러 트랜잭션도 동시에 `공유 모드`로 잠금을 획득하는 것은 허용
- 하지만 그 객체에 이미 `독점 모드`로 잠금을 획득한 트랜잭션이 있다면 기다려야 한다.
2️⃣ **독점 모드**
트랜잭션이 객체에 **쓰기**를 원한다면 독점 모드로 잠금을 획득해야 한다.
- 다른 어떤 트랜잭션도 동시에 잠금을 획득할 수 없어 잠금이 존재한다면 대기해야 한다.

트랜잭션이 객체를 **읽다가 쓰기를 실행할 때는 공유 잠금을 독점 잠금으로 업그레이드 해야 한다.**
- 업그레이드는 독점 잠금을 획득할 때와 똑같이 동작한다.
트랜잭션이 잠금을 획득한 후에는 **트랜잭션이 종료될 때까지 잠금을 가지고 있어야 한다.**

⚠️ **교착 상태**
두 트랜잭션이 서로가 가진 락을 기다리는 상태
```plain text
T1: A 락 획득 → B 기다림
T2: B 락 획득 → A 기다림
```

**데이터베이스는 교착 상태를 자동으로 감지하고 트랜잭션 중 하나를 어보트 시켜 다른 트랜잭션들이 진행할 수 있게 한다.**
### 2PL의 성능 문제
2단계 잠금을 사용하면 완화된 격리 수준을 쓸 때보다 `트랜잭션 처리량`과 `질의 응답 시간`이 크게 나빠진다.
1️⃣ **동시성 저하**
두 트랜잭션이 경쟁 조건을 유발하는 일을 하고자 한다면 한 트랜잭션이 다른 트랜잭션이 완료될 때까지 기다려야 한다.
2️⃣ **대기 시간 증가**
한 트랜잭션이 다른 트랜잭션이 끝나기를 기다릴 때 얼마나 오래 기다려야 하는지에 대한 제한이 없다.
3️⃣** 교착 상태 (deadlock)**
`read committed` 격리 수준에서도 교착 상태가 발생할 수 있지만 2PL 직렬성 격리에서는 훨씬 더 자주 발생한다.
트랜잭션이 교착 상태 때문에 어보트되어 재시도하면 작업을 전부 다시 해야 한다.
## 직렬성 스냅샷 격리(SSI)
> **데이터베이스의 동시성 제어 방법**
> - 1️⃣ 성능이 좋지 않거나 확장이 잘 되지 않는 제어 방법: <span class="text-red">**Serializability**</span>
> - 2️⃣ 성능은 좋지만 다양한 `경쟁 조건(갱신 손실, 쓰기 스큐, 팬텀 등)`에 취약한 <span class="text-red">**완화된 격리 수준**</span>

🤔 **Serializable** 격리 수준과 좋은 성능은 공존할 수 없을까??
❗아니!! **직렬성 스냅샷 격리(Serializable Snapshot Isolation, SSI) **라는 알고리즘이 유망하다!!
- **완전한 직렬성을 제공하지만 스냅샷 격리에 비해 약간의 성능 손해만 있을 뿐이다.**
- 상당히 최근에 등장.

**SSI는 단일 노드 데이터베이스와 분산 데이터베이스 모두에서 사용된다.**
- 🐘  `PostgreSQL`은 9.1 버전 부터 `serializable` 격리 수준을 제공한다.
### 비관적 동시성 제어 대 낙관적 동시성 제어
**Serializable 실행**은 어떤 면에서 보면 극단적으로 <span class="text-red">**비관적**</span>이다.
- **2단계 잠금**은 **비관적** **동시성 제어 **메커니즘이다.
- 각 트랜잭션이 실행되는 동안 전체 데이터베이스 또는 데이터베이스 파티션 하나에 독점 잠금을 획득하는 것과 같다.

반대로 **직렬성 스냅샷 격리는 **<span class="text-red">**낙관적**</span>** 동시성 제어 기법이다.**
- 트랜잭션이 커밋될 때 데이터베이스는 격리가 위반되었는지 확인하고 그렇다면 어보트된다.
- 직렬로 실행된 트랜잭션만 커밋이 허용

**SSI**는 **스냅샷 격리**를 기반으로 하여 트랜잭션에서 실행되는 모든 읽기는 데이터베이스의 일관된 스냅샷을 보게 된다.
- 이것이 이전의 낙관적 동시성 제어 기법과 크게 다른 점이다.
- SSI는 스냅샷 격리 위에 쓰기 작업 사이의 직렬성 충돌을 감지하여 어보트시킬 트랜잭션을 결정하는 알고리즘을 추가한다.
### 뒤처진 전제에 기반한 결정
스냅샷 격리에서 트랜잭션이 커밋되는 시점에 원래 질의의 결과가 최신이 아닐 수 있다. (도중에 데이터가 변경됐을 수도 있기 때문)
즉, 트랜잭션은 어떤 **전제**(트랜잭션을 시작할 때 참이었던 사실) 를 기반으로 어떤 동작을 한다.
- 커밋을 하고자 할 때 원래 데이터가 변경되어 그 전제가 더 이상 참이 아닐 수도 있다.

🤔 데이터베이스가 어떻게 질의 결과가 바뀌었는지 알 수 있을까?

- 과거의 읽기에 영향을 미치는 쓰기 감지하기
### 오래된 MVCC 읽기 감지하기
**스냅샷 격리**는 보통 `MVCC`를 기반으로 구현이 된다.

- 이 과정에서 트랜잭션은 어떤 변경을 `무시`

위 그림에서 `트랜잭션 43`이 커밋하기 원할 때는 `트랜잭션 42`가 이미 커밋된 상태이다.
- 일관된 스냅샷에서 무시했던 쓰기가 커밋할 때는 영향이 있고 `트랜잭션 43`의 **전제가 더 이상 참이 아니라는 의미**이다.

👀 **SSI**는 이러한 상황을 감지하기 위해 트랜잭션이 읽는 동안 어떤 변경을 무시했는지를 추적한다. 
그리고 트랜잭션이 커밋하려는 시점에 그 무시된 변경이 실제로 커밋되었는지를 확인한다. 
- 커밋된 것이 있다면 트랜잭션은 롤백 (유효한 전제가 아니기 때문)
### 과거의 읽기에 영향을 미치는 쓰기 감지하기
또 고려해야 하는 상황은 **데이터를 읽은 후 다른 트랜잭션에서 그 데이터를 변경하는 상황**이다.
트랜잭션이 데이터베이스에 쓸 때 영향받는 데이터를 최근에 읽은 트랜잭션이 있는지 확인해야 한다.
- 읽는 쪽에서 커밋될 때까지 차단하지 않고 트랜잭션이 읽은 데이터가 더 이상 최신이 아니라고 트랜잭션에게 알려준다.
커밋 시점에 직렬성이 깨지는 트랜잭션을 어보트한다.
### 직렬성 스냅샷 격리의 성능
SSI는 스냅샷 격리의 특성을 유지하기 때문에 읽기와 쓰기가 서로를 막지 않는다는 장점이 있다. 
즉, 읽기 작업은 락 없이 일관된 스냅숏을 기반으로 수행될 수 있어 이는 읽기 중심의 작업에서 매우 유리하다.
