10장에서 살펴본 일괄처리는 입력을 유한한 크기로 한정한다는 중요한 가정이 있다.
그래서 일괄처리는 입력을 읽는 작업이 끝나는 시점을 알 수 있다.
**실제로는 많은 데이터가 시간이 지나면서 점진적으로 도착하기 때문에 유한한 크기로 한정되지 않는다.**

<span class="text-red">**일간 일관 처리**</span>**의 문제점은 입력의 변화가 하루가 지나야 반영된다는 것인데 사용자가 느끼기에는 너무 느리다.**
이런 지체를 줄이려면 좀 더 자주 처리를 실행해야 한다.
- ex) 매초 데이터 처리, 이벤트가 발생할 때마다 처리 
- <span class="text-red">**스트림 처리**</span>의 기본 개념
이벤트 스트림은 일괄 처리 데이터와는 반대로 한정되지 않고 점진적으로 처리된다.
# 이벤트 스트림 전송
일괄 처리 환경에서 작업은 입출력이 `파일`이다.
🤔 스트림 처리에서는 어떻게 할까??
- 대개 첫 번째 단계로 파일을 분석해 레코드의 연속으로 바꾸는 처리를한다 
- 스트림 처리 문맥에서 레코드는 보통 <span class="text-red">**이벤트**</span>라고 하지만 특정 시점에 일어난 사건에 대한 세부 사항을 포함하는 작고 독립된 불변 객체라는 점에서 동일하다.

일괄 처리에서 파일은 한 번 기록하면 여러 작업에서 읽을 수 있다. <u>**스트리밍도 생산자가 이벤트를 한 번 만들면 해당 이벤트를 복수의 소비자 또는 수신자가 처리할 수 있다.**</u>
스트림 시스템에서는 `토픽`이나 `스트림`으로 관련 이벤트를 묶는다.
#### 폴링 방식
생산자는 만든 모든 이벤트를 데이터스토어에 기록하고 각 소비자는 주기적으로 데이터스토어를 폴링해 마지막으로 처리한 이벤트 이후에 새로 발생한 이벤트가 있는지 확인하여 처리한다.
⚠️ **지연 시간이 낮으면서 지속해서 처리하는 방식을 지향할 때 데이터스토어를 용도에 맞게 설계하지 않았다면 폴링 방식은 비용이 크다.**
폴링이 잦을 수록 새로운 이벤트를 반환하는 요청 비율이 낮아지기 때문에 폴링을 수행하는 오버헤드가 커진다.
오히려 **새로운 이벤트가 나타날 때마다 소비자에게 알리는 편이 더 낫다.**
## 메시징 시스템
:::callout ℹ️ <span class="text-red">**메시징 시스템**</span>
:::callout 새로운 이벤트에 대해 소비자에게 알려주려고 쓰는 일방적인 방법
:::callout 생산자는 이벤트를 포함한 메시지를 전송하고 이 메시지는 소비자에게 전달된다.
**메시징 시스템**은 **다수의 생산자 노드가 같은 토픽으로 메시지를 전송**할 수 있고 **다수의 소비자 노드가 토픽하나에서 메시지를 받아** 갈 수 있다.
#### 발행-구독(Publish-Subscribe)모델에서 여러 시스템들은 다양한 접근법을 사용
> 아래 두 가지 질문이 이 시스템을 구별하는 데 도움이 된다.

🔍 **생산자가 소비자가 메시지를 처리하는 속도보다 빠르게 메시지를 전송한다면 어떻게 될까?**
시스템은 많은 트래픽을 처리하기 위해 크게 3가지 방법을 사용한다.

| **방법** | **설명** | **동작 방식 및 예시** |
| ---- | ---- | ---- |
| **버리기(drop)** | 넘치는 메시지를 버림 | 시스템 보호를 위해 초과분 삭제 |
| **버퍼링(Buffer)** | 큐(Queue)에 메시지를 쌓아둠 | 큐가 커질 때 메모리가 터질지, 디스크에 쓸지 고민 필요 |
| **백프레셔(Backpressure)** | 생산자의 송신을 차단 | 유닉스 파이프, TCP |

:::callout ⚠️ 버퍼링(Buffering) 선택 시 주의사항
:::callout 큐에 메시지를 쌓는 방식을 택했다면 큐가 메모리 용량을 초과할 때 시스템이 어떻게 동작하는지 반드시 이해해야 한다.  디스크에 메시지를 기록한다면 이 디스크 접근이 메시징 시스템의 전체 성능에 어떤 영향을 미치는지 파악해야 한다.

🔍 **노드가 죽거나 일시적으로 오프라인이 된다면 어떻게 될까?**
> 메시지의 **내구성(Durability)과** **성능(Performance)** 사이의 트레이드오프를 결정


| **상황** | **설명** | 예시 |
| ---- | ---- | ---- |
| **내구성이 필요한 경우** | DB처럼 디스크에 기록하거나 복제가 필요하고 이는 성능 저하(비용)이 수반. | 이벤트 횟수를 카운팅하는 시스템 (메시지 하나 유실 = 잘못된 카운트 결과) |
| **유실을 허용하는 경우** | 내구성을 포기하는 대신 훨씬 높은 처리량과 낮은 지연시간을 얻음 | 주기적으로 전송되는 센서 데이터나 지표 (다음 주기에 최신 값이 오므로 1~2개 누락은 괜찮음. 단, 대량 유실 시 지표 왜곡 주의) |

### 생산자에서 소비자로 메시지를 직접 전달하기
**많은 메시지 시스템은 중간 노드를 통하지 않고 **<span class="text-blue">**생산자와 소비자를 네트워크로 직접 통신**</span>**한다.**
- **UDP 멀티캐스트:** 주식 시장 피드처럼 짧은 지연 시간이 중요한 금융업계에서 사용. 
  - 패킷 유실 복구는 애플리케이션 단에서 직접 구현해야 함.
- **ZeroMQ / nanomsg:** TCP나 IP 멀티캐스트 위에서 발행-구독 패턴을 구현하는 **브로커가 없는 라이브러리**.
- **StatsD / Brubeck:** 네트워크 모니터링이나 메트릭 수집을 위해 신뢰성은 낮지만 빠른 UDP를 사용.
- 소비자가 네트워크에 서비스를 노출하면** 생산자는 직접 HTTP나 RPC 요청을 직접 보냄**

⚠️ **직접 메시징 시스템**은 설계 상황에서는 잘 동작하지만 **일반적으로 **<u>**메시지가 유실될 수 있는 가능성**</u>**을 고려해 애플리케이션 코드를 작성해야 한다.**
- 시스템이 항상 온라인 상태여야 한다. 
- 소비자가 오프라인일 때 전송된 메시지는 유실될 수 있다.
- 생산자가 재시도를 위해 실패한 메시지를 버퍼에 들고 있다가 생산자마저 죽어버리면 데이터는 영영 사라진다.
### 메시지 브로커 (Message Brokers)
:::callout ℹ️<span class="text-red">** 메시지 브로커 (메시지 큐)**</span>
:::callout - **직접 메시징 시스템의 대안으로 널리 사용되는 방법**
:::callout - 메시지 스트림을 처리하는데 최적화된 데이터베이스의 일종
메시지 브로커는 서버로 구동되고 생산자와 소비자는 서버의 클라이트로 접속한다.
생산자는 브로커로 메시지를 전송하고 소비자는 브로커에서 메시지를 읽어 전송받는다.
#### 지속성 문제
브로커에 데이터가 모이기 때문에 클라이언트의 상태 변경(접속, 접속 해제, 장애)에 쉽게 대처할 수 있다.
지속성 문제가 생산자와 소비자에서 브로커로 옮겨갔기 때문이다.
#### 비동기 처리
큐 대기를 하면 소비자는 일반적으로 `비동기`로 동작한다.
생산자는 브로커가 메시지를 버퍼에 넣었는지만 확인하고 소비자가 메시지를 처리하기까지 기다리지 않는다.
### 메시지 브로커와 데이터베이스 비교

| **구분** | **일반적인 데이터베이스 (RDBMS 등)** | **메시지 브로커** |
| ---- | ---- | ---- |
| **데이터 수명** | 명시적으로 삭제할 때까지 영구 보관 | 대부분 소비자가 성공적으로 처리하면 **자동으로 메시지 삭제** |
| **적정 데이터량** | 대용량 저장에 최적화 | 큐를 짧게 유지하는 것을 권장. 큐가 길어지면 성능 저하 발생 |
| **데이터 검색** | 인덱스를 통한 다양한 조건 검색 지원 | 특정 패턴이나 토픽의 부분 구독만 지원 |
| **데이터 확인 방식** | 클라이언트가 쿼리를 날린 시점의 스냅샷 반환 | 데이터가 변경(새 메시지 생성)되면 클라이언트에게 **알림** |

### 복수 소비자
> **여러 소비자가 같은 토픽의 메시지를 읽을 때** 사용하는 두 가지 핵심 패턴


| **패턴** | **동작 방식** | **목적** |
| ---- | ---- | ---- |
| **로드 밸런싱 (Load Balancing)** | 각 메시지는 **단 하나의 소비자에게만** 전달됨 | 메시지 처리 비용이 비쌀 때 소비자를 늘려 작업을 **병렬 분산 처리** |
| **팬아웃 (Fan-out)** | 각 메시지는 **모든 소비자에게** 전달됨 | 여러 독립적인 소비자가 동일한 메시지를 목적에 맞게 **동시 사용** |

<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2Fedeedc37-7238-4d26-a668-b9c478382379%2Fimage.png?table=block&id=36bd67b9-9e80-80f9-aea9-c384f90aa7e0&cache=v2" alt="image" width="1296" height="650" loading="lazy" />

이 두 패턴은 함께 사용 가능하다.
- ex) 두 개의 소비자 그룹에서 하나의 토픽을 구독하고 각 그룹은 모든 메시지를 받지만 그룹 내에서는 각 메시지를 하나의 노드만 받게하는 방식
### 확인 응답과 재전송
**소비자**가 메시지를 받긴 했지만 처리 도중 죽어버리면 메시지가 유실될 수 있다.
이를 막기 위해 <span class="text-red">**확인 응답(Acknowledgment, Ack)**</span>을 사용
- 클라이언트는 메시지 처리가 끝났을 때 브로커가 메시지를 큐에서 제거할 수 있도록 브로커에게 명시적으로 알려야 한다.
브로커가 확인 응답을 받기 전에 클라이언트로의 연결이 닫히거나 타임아웃되면 브로커는 메시지가 처리되지 않았다고 가정하고 다른 소비자에게 다시 전송한다.
:::callout ⚠️ 로드밸런싱과 결합할 때 재전송 행위는 메시지 순서에 영향을 미침
:::callout - 네트워크 지연이나 소비자 장애로 인해 Ack를 받지 못하면 브로커는 다른 소비자에게 메시지를 재전송
:::callout - 이 과정에서 메시지의 처리 순서가 달라질 수 있다.
## 파티셔닝된 로그
브로커는 확인 응답을 받으면 브로커에서 메시지를 삭제하기 때문에 이미 받은 메시지는 복구할 수 없다.
데이터베이스의 지속성 있는 저장 방법과 메시징 시스템의 지연 시간이 짧은 알림 기능을 조합할 수 없을까??
- 이것이 <span class="text-red">**로그 기반 메시지 브로커**</span>의 기본 아이디어이다.

### 로그를 사용한 메시지 저장소
생산자가 보낸 메시지는 로그 끝에 추가하고 소비자는 로그를 순차적으로 읽어 메시지를 받는다.
#### 파티셔닝
디스크를 하나 쓸 때보다 처리량을 높이기 위해 확장하는 방법으로 **로그를 **<span class="text-red">**파티셔닝**</span>하는 방법이 있다.
- 다른 장비에서 파티션을 서비스할 수 있다.
- 각 파티션은 다른 파티션과 독립적으로 읽고 쓰기가 가능한 분리된 로그가 된다.
#### 토픽과 오프셋
`토픽`은 같은 형식의 메시지를 전달하는 파티션들의 그룹으로 정의한다.
파티션 내의 각 메시지는 단조 증가하는 일련번호인 `오프셋(Offset)`을 가지며 파티션 내부에서는 순서를 보장한다.
- 다른 파티션 간 메시지의 순서는 보장하지 않는다.
### 로그 방식과 전통적인 메시징 방식의 비교
로그 기반 접근법은 팬 아웃 메시징 방식을 제공한다. 소비자가 서로 영향 없이 독립적으로 로그를 읽을 수 있고 메시지를 읽어도 로그에서 삭제되지 않기 때문이다.
클라이언트는 할당된 **파티션의 메시지를 모두** 소비한다.(개별 메시지가 아닌 **파티션 단위**로 소비자를 할당)
- 소비자는 단일 스레드로 파티션에서 순차적으로 메시지를 읽는다.
- 토픽 하나를 소비하는 소비자 노드 수는 해당 **토픽의 로그 파티션 수로 제한**된다.
- **Head-of-line blocking**: 특정 메시지 처리가 느리면 파티션 내 후속 메시지 처리가 지연된다.
### 소비자 오프셋
파티션 하나를 순서대로 처리하면 메시지를 어디까지 처리했는지 알기 쉽다.
단지 주기적으로 `소비자 오프셋`을 기록하면 된다.
소비자에 장애가 발생하면 소비자 그룹 내 다른 소비자가 해당 파티션을 넘겨받아 마지막으로 기록된 오프셋부터 처리하기 시작한다.
### 디스크 공간 사용
로그를 계속 추가한다면 결국 디스크 공간을 전부 사용하게 된다.
디스크 공간을 **재사용**하기 위해 실제로는 **로그를 여러 조각으로 나누고 오래된 조각을 삭제하거나 보관 저장소로 이동**한다.
메시지 보관 기간과 관계없이 모든 메시지를 디스크에 기록하기 때문에 로그 처리량은 일정하다.
### 소비자가 생산자를 따라갈 수 없을 때
소비자 처리 속도가 느려 메시지가 생산되는 속도를 따라잡지 못하면 소비자가 너무 뒤처져 소비자 오프셋이 이미 삭제한 조각을 가리킬 수도 있다.
- 소비자가 뒤처져 필요한 메시지가 디스크에 보유한 메시지보다 오래되면 필요한 메시지는 읽을 수 없다.
- 브로커는 버퍼 크기를 넘는 오래된 메시지를 버린다. (메시지 일부를 잃어버릴 가능성이 있다)
소비자가 뒤처지는 경우 경고할 수 있고 사람이 소비자 처리가 느린 문제를 고쳐 메시지를 읽기 전에 따라 잡을 시간을 충분히 벌 수 있다.
어떤 소비자가 너무 뒤처져서 메시지를 읽기 시작해도 해당 소비지만 영향을 받고 다른 소비자들의 서비스를 망치지는 않는다.
### 오래된 메시지 재생
로그 기반 메시지 브로커는 읽기 전용 연산으로 로그를 변화시키지 않는다.
- 유일한 부수 효과는 소비자 오프셋 이동이다.
소비자의 오프셋을 조정하면 전날 분량의 메시지를 재처리같은 작업을 할 수 있다.
# 데이터베이스와 스트림
## 시스템 동기화 유지하기
대부분의 애플리케이션이 요구사항을 만족하기 위해 `몇 가지 다른 기술의 조합`이 필요하다.
- ex) OLTP **데이터베이스**, 응답 속도를 높이기 위한 **캐시**, 검색 질의를 다루기 위한 **전문 색인**, 분석용 **데이터** **웨어하우스**
관련이 있거나 동일한 데이터가 여러 다른 장소에 나타나기 때문에 <span class="text-red">**서로 동기화가 필수**</span>다.
데이터베이스의 아이템 하나를 갱신하면 캐시와 색인과 데이터 웨어하우스도 갱신해야 한다.

#### 이중 기록(dual write)
주기적으로 데이터베이스 전체를 덤프하는 작업이 너무 느리다면 대안으로 사용하는 방법으로 <span class="text-blue">**이중 기록**</span>이 있다.
이중 기록을 사용하면 데이터가 변할 때마다 애플리케이션 코드에서 명시적으로 각 시스템에 기록한다.
⚠️ **그러나 이중 기록에는 몇 가지 심각한 문제가 있다.**
1️⃣ **경쟁 조건(race condition)에 의한 데이터 불일치**

## 변경 데이터 캡처 (Change Data Capture, CDC)
:::callout ℹ️ **변경 데이터 캡처 (Change Data Capture, CDC)**

:::callout - 데이터가 기록되자마자 변경 내용을 스트림으로 제공할 수 있으면 유용하다.

### 변경 데이터 캡처의 구현
**변경 데이터 캡처는 레코드 시스템에 발생하는 모든 변경 사항을 파생 데이터 시스템에 반영하는 것을 보장해주는 메커니즘이다.**
:::callout 💡 **파생 데이터 시스템**
:::callout - 검색 색인이나 데이터 웨어하우스에 저장되는 데이터는 레코드 시스템에 저장된 데이터의 또 다른 뷰일 뿐이다.
변경 데이터 캡처는 변경 사항을 캡처할 **데이터베이스 하나를 ****`리더`**로 하고 **나머지를 ****`팔로워`**로 한다.
**로그 기반 메시지 브로커**는 원본 데이터베이스에서 변경 이벤트를 전송하기에 적합하다.
- 메시지 순서를 유지하기 때문이다.

✅ **변경 데이터 캡처는 ****`비동기`**** 방식으로 동작한다.**
레코드 데이터베이스 시스템은 변경 사항을 커밋하기 전에 변경 사항이 소비자에게 적용될 때까지 기다리지 않는다.
- 복제 지연의 모든 문제가 발생하는 단점이 있다.

### 초기 스냅숏
데이터베이스에서 발생한 모든 변경 로그가 있다면 로그를 재현해 데이터베이스의 전체 상태를 재구축할 수 있다.
⚠️ 그러나 대부분 **모든 변경 사항을 영구적으로 보관하는 일은 디스크 공간이 너무 많이 필요하고 모든 로그를 재생하는 작업도 너무 오래 걸린다.**
- 그래서 로그를 적당히 잘라야 한다.
전문 색인을 새로 구축할 때 전체 데이터베이스 복사본이 필요하다.
전체 로그 히스토리가 없다면 전체 DB의 일관된 스냅숏을 사용하고 그 스냅숏이 생성된 시점의 로그 오프셋(Offset)부터 변경 스트림을 이어받아 적용하는 방식을 사용한다.
### 로그 컴팩션
로그 히스토리의 양을 제한한다면 새로운 파생 시스템을 추가할 때마다 스냅숏을 만들어야 한다.
하지만 <span class="text-blue">**로그 컴팩션**</span>이라는 대안이 있다.
- 주기적으로 같은 키의 로그 레코드를 찾아 중복을 제거하고 각 키에 대해 가장 최근에 갱신된 내용만 유지
- 컴팩션과 병합과정은 백그라운드로 실행
> CDC 시스템에서 모든 변경에 기본키가 포함되게 하고 키의 모든 갱신이 해당 키의 이전 값을 교체한다면 **특정 키에 대해 최신 쓰기만 유지하면 충분**하다.

이를 활용하면 별도의 DB 스냅숏 없이 컴팩션된 로그를 처음부터 읽어 들이는 것만으로도 전체 데이터베이스의 콘텐츠의 복사본을 얻을 수 있다.
## 이벤트 소싱 (Event Sourcing)
:::callout ℹ️ **이벤트 소싱(Event Sourcing)**
:::callout - `도메인 주도 설계(domain-driven design, DDD)` 커뮤니티에서 개발한 방법
:::callout - 변경 데이터 캡처와 유사하게 **애플리케이션 상태 변화를 모두 변경 이벤트 로그로 저장**한다.
변경 데이터 캡처와 가장 큰 차이점은 아이디어를 적용하는 추상화 레벨이 다르다는 점이다.
#### ✅ **변경 데이터 캡처 (CDC)**
변경 데이터 캡처에서 애플리케이션은 데이터베이스를 변경 가능한 방식으로 사용해 레코드를 자유롭게 갱신하고 삭제한다.
- 변경 로그는 데이터베이스에서 **저수준**으로 추출한다.
- 변경 로그는 데이터베이스에서 추출한 쓰기 순서가 실제로 데이터를 기록한 순서와 일치하고 경쟁 조건이 나타나지 않게 보장한다.
- 데이터베이스에 기록하는 애플리케이션은 `CDC`가 실행 중인지 알 필요가 없다.

#### 💡 이벤트 소싱
애플리케이션 로직은 이벤트 로그에 기록된 <span class="text-red">**불변 이벤트**</span>를 기반으로 명시적으로 구축한다.
- 이벤트 저장은 단지 추가만 가능하고 갱신이나 삭제는 권장하지 않거나 금지한다.

> **이벤트 소싱**은 데이터 모델링에 쓸 수 있는 강력한 기법이다.

애플리케이션 관점에서 사용자의 행동을 **불변 이벤트로 기록하는 방식**은 변경 가능한 데이터베이스 상에서 사용자의 행동에 따른 효과를 기록하는 방식보다 훨씬 유의미하다.
이벤트 소싱은 어떤 상황이 발생한 후 상황 파악이 쉽기 때문에 디버깅에 도움이 되고 애플리케이션 버그를 방지하여 애플리케이션을 지속해서 개선하기가 매우 유리하다.
### 이벤트 로그에서 현재 상태 파생하기
사용자는 시스템의 현재 상태를 보기 원하지 수정 히스토리를 원하지 않는다. 
따라서 **이벤트 소싱 애플리케이션은 기록한 데이터를 표현한 이벤트 로그를 가져와 **<span class="text-blue">**사용자에게 보여줄 상태**</span>**로 변환해야 한다.**
### 명령과 이벤트
이벤트 소싱 철학은 **이벤트**와 **명령**을 구분하는 데 주의한다.
1. 사용자 요청이 처음 도착했을 때 이 요청은 명령이고 애플리케이션은 먼저 명령이 실행 가능한지 확인해야 한다.
2. 무결성이 검증되고 명령이 승인되면 명령은 **지속성 있는 불변 이벤트**가 된다.
## 상태와 스트림 그리고 불변성
> 불변성의 원리가 이벤트 소싱과 변경 데이터 캡처를 매우 강력하게 만든다.

상태가 변할 때마다 해당 <span class="text-red">**상태**</span>는 **시간이 흐름에 따라 변한 이벤트의 마지막 결과**다.
- 상태가 어떻게 바뀌었든 항상 이런 변화를 일으킨 일련의 이벤트가 있다.
- 모든 **변경 로그**는 시간이 지남에 따라 바뀌는 상태를 나타낸다.
### 불변 이벤트의 장점
추가만 하는 불변 이벤트 로그를 사용했다면 **문제 상황의 진단과 복구가 훨씬 쉽다.**
또한 불변 이벤트는 현재 상태보다 훨씬 많은 정보를 포함한다.
`ex) 장바구니 넣었다가 제거`: 고객이 특정 항목을 구매하려 했다가 하지 않았다는 것을 알 수 있는 유용한 정보
### 동일한 이벤트 로그로 여러 가지 뷰 만들기
불변 이벤트 로그에서 **가변 상태를 분리하면 동일한 이벤트 로그로 다른 여러 읽기 전용 뷰**를 만들 수 있다.
#### 명령과 질의 책임의 분리(command query responsibility segregation, CQRS)
데이터를 쓰는 형식과 읽는 형식을 분리해 다양한 읽기 뷰를 허용한다면 상당한 유연성을 얻을 수 있다.
### 동시성 제어
:::callout ⚠️ **이벤트 소싱과 변경 데이터 캡처의 큰 단점**
:::callout 이벤트 로그의 소비가 대개 비동기로 이뤄진다는 점
그래서 **사용자가 로그에 이벤트를 기록하고 이어 로그에서 파생된 뷰를 읽어도 기록한 이벤트가 아직 읽기 뷰에 반영되지 않았을 가능성이 있다.**
#### 동기식 처리
읽기 뷰의 갱신과 로그에 이벤트를 추가하는 작업을 `동기식`으로 수행하는 방법이다.
이 방법을 쓰려면 트랜잭션에서 <u>**여러 쓰기를 원자적 단위로 결합**</u>해야 해 이벤트 로그와 읽기 뷰를 같은 저장 시스템에 담아야 한다.
- 다른 시스템에 있다면 <span class="text-red">**분산 트랜잭션**</span>이 필요

#### 이벤트 로그
반면 이벤트 로그로 현재 상태를 만들면 동시성 제어 측면이 단순해진다.
이벤트 소싱을 사용하면 사용자 동작에 대한 설명을 자체적으로 포함하는 이벤트를 설계할 수 있다.
그러면 **사용자 동작은 한 장소에서 한 번 쓰기만 필요**하다. 즉 이벤트를 로그에 추가만 하면 되어 원자적으로 만들기 쉽다.
### 불변성의 한계
상대적으로 작은 데이터 셋에서 빈번히 갱신과 삭제를 하는 작업 부하는 불변 히스토리가 감당하기 힘들 정도로 커지거나 파편화 문제가 발생할 수도 있다.
또한 컴팩션과 가비지 컬렉션의 성능 문제가 견고한 운영을 하는데 큰 골칫거리가 되기도 한다.

성능적인 이유 외에도 데이터가 모두 불변성임에도 관리상의 이유로 데이터를 삭제할 필요가 있는 상황이 있을 수 있다.
- ex) 사생활 침해 규제로 인해 사용자가 계정을 폐쇄한 이후 사용자의 개인 정보를 지울 필요가 있음
이런 상황에서는 이전 데이터를 삭제해야 한다는 또 다른 이벤트를 로그에 추가한다고 해결되지 않는다.
실제로 원하는 것은 히스토리를 새로 쓰고 문제가 되는 데이터를 처음부터 기록하지 않았단 것처럼 하는 것이다.
# 스트림 처리
스트림을 처리하는 방법은 크게 3가지 선택지가 있다.
1. 데이터를 DB, 캐시, 검색 색인에 기록하고 다른 클라이언트가 데이터를 질의한다.
2. 이메일, 푸시 알림, 실시간 대시보드 등을 통해 사용자에게 직접 이벤트를 보낸다.
3. **하나 이상의 입력 스트림을 처리해 하나 이상의 출력 스트림을 생산한다.**
나머지 부분에서는 스트림을 처리해 다른 파생 스트림을 생산하는 3번 선택지에 대해 설명한다.
## 스트림 처리의 사용
스트림 처리는 특정 상황이 발생하면 경고를 해주는 모니터링 목적으로 오랜 기간 사용돼 왔다.
그러나 시간이 흘러 다른 용도로 스트림 처리를 사용하는 사용자들이 나타나기 시작했다.
### 복잡한 이벤트 처리
**복잡한 이벤트 처리(complex event processing, CEP)**는 1990년대에 이벤트 스트림 분석용으로 개발된 방법이다.
- 특정 이벤트 패턴을 찾기 위한 방법. 
- 문자열에서 정규표현식을 찾는 것과 유사하게 스트림에서 특정 이벤트 패턴을 찾는 규칙을 규정할 수 있다.
이 시스템에서 질의와 데이터의 관계는 일반적인 DB와 비교했을 때 반대다.
**CEP는 질의(query)가 저장되고 입력 스트림으로 들어오는 이벤트는 질의를 지나면서 패턴이 일치하면 복잡한 이벤트를 방출한다.**
### 스트림 분석
스트림 처리를 사용하는 다른 영역으로 **스트림 분석**이 있다.
CEP와 스트림 분석 사이의 경계는 불분명하지만 일반적으로 **분석은 연속한 특정 이벤트 패턴을 찾는 것보다 대량의 이벤트를 집계하고 통계적 지표를 뽑는 것을 더 우선한다.**
이런 통계는 고정된 시간 간격 기준으로 계산하는데 이 **집계 시간 간격**을 <span class="text-red">**윈도우(window)**</span>라 한다.
### 구체화 뷰 유지하기
데이터베이스 변경에 대한 스트림은 캐시, 검색 색인, 데이터 웨어하우스 등의 파생 데이터 시스템이 최신 내용을 따라잡게 하는 데 쓸 수 있다.
이벤트 소싱에서 애플리케이션 상태는 이벤트 로그를 적용함으로써 유지된다. 여기서 애플리케이션 상태는 일종의 구체화 뷰이다.
스트림 분석과 달리 특정 시간 윈도우가 아니라 구체화 뷰를 만드려면 임의의 시간 범위에 발생한 **모든** 이벤트가 필요하다.  
### 스트림 상에서 검색하기
복수 이벤트로 구성된 패턴을 찾는 CEP외에도 전문 검색 질의와 같은 복잡한 기준을 기반으로 개별 이벤트를 검색해야 하는 경우도 있다.
스트림 검색은 질의를 먼저 저장하고 CEP와 같이 문서는 질의를 지나가면서 실행된다.
### 메시지 전달과 RPC
이전 챕터(메시지 전달 데이트플로)에서 메시지 전달 시스템을 RPC 대안으로 사용할 수 있다고 설명했다.
즉 액터 모델 등에서 쓰이는 서비스간 통신 메커니즘으로 사용할 수 있다.
이런 시스템은 메시지와 이벤트에 기반을 두지만 일반적으로 이것들을 스트림 처리자로 생각하지는 않는다.

| **구분** | **액터 모델 (Actor Frameworks)** | **스트림 처리 (Stream Processing)** |
| ---- | ---- | ---- |
| **주요 목적** | **동시성 관리** 및 통신 모듈 분산 실행 | **데이터 관리** 및 변환 |
| **통신의 성격** | 주로 단기적이며 **1:1 통신** | 지속성이 있으며 **1:N (다중 구독)** 가능 |
| **데이터 흐름** | 임의적 (**순환 요청/응답 패턴** 허용) | 정해진 입력에서 출력으로 흐르는 **비순환적 파이프라인** |

유사 RPC 시스템과 스트림 처리 사이에 겹치는 영역이 있다.
✔️ **Apache Storm의 분산 RPC:** 사용자 질의를 이벤트 스트림과 섞어서 여러 노드에 분산 처리한 뒤 결과를 모아서 반환
✔️ **액터 프레임워크로 스트림 처리:** 액터 프레임워크를 이용해 스트림을 처리할 수는 있지만 대부분 장애 상황에서 메시지 전송을 보장하지 않기 때문에 내결함성을 보장하려면 추가적인 재시도 로직을 구현해야 합니다.
## 시간에 관한 추론
스트림 처리자는 종종 시간을 다뤄야 할 때가 있다(분석 목적으로 사용할 때). 이때 주로 시간 윈도우를 자주 사용한다.
많은 스트림 처리 프레임워크는 윈도우 시간을 결정 할 때 처리하는 장비의 시세틈 시계를 이용한다.
이 접근 법은 간단하다는 장점이 있지만 처리가 눈에 띌 정도로 지연되면 즉, 이벤트가 실제로 발생한 시간보다 처리 시간이 많이 늦어지면 문제가 생긴다.
### 이벤트 시간 대 처리 시간
처리가 지연되는 데는 많은 이유가 있고 메시지가 지연되면 메시지 순서를 예측하지 못할 수도 있다.
예를 들어 사용자가 첫 번째 웹 요청(웹 서버 A에서 처리)을 보낸 후 두 번째 웹 요청(웹 서버 B에서 처리)을 보냈다고 가정.
1. 웹 서버 A와 B는 각각이 처리한 요청을 담은 이벤트를 방출
2. 그러나 웹 서버 A의 이벤트가 메시지 브로커에 도달하기 전에 웹 서버 B의 이벤트가 먼저 도착
3. 스트림 처리자는 실제로 이벤트가 발생한 순서가 반대임에도 B 이벤트를 먼저 처리하고 A 이벤트를 처리
**스트림 처리 알고리즘은 타이밍과 순서 문제를 처리하게끔 명확히 작성할 필요가 있다.**
### 준비 여부 인식
**이벤트 시간 기준으로 윈도우를 정의할 때 발생하는 문제**는 특정 윈도우에서 모든 이벤트가 도착했다거나 아직도 이벤트가 계속 들어오고 있는지를 확신할 수 없다는 점이다.
- 일부 이벤트는 네트워크 지연으로 뒤늦게 도착할 수 있는 가능성이 있다.
이런 지연 이벤트를 처리할 방법이 필요하다.

| 방법 | 설명 |
| ---- | ---- |
| 지연 이벤트 무시하고 누락 비율 모니터링 | 정상적인 환경에서 이런 이벤트는 적은 비율을 차지하기 때문에 지표로 추적해 많은 양의 데이터가 누락되는 경우 경고를 보낸다. |
| 지연 이벤트가 포함된 수정된 윈도우 값을 다시 발행하기 |  |

### 어쨌든 어떤 시계를 사용할 것인가?
오프라인 상태의 모바일 앱처럼 며칠 뒤에 서버로 전송되는 이벤트가 있을 수 있다. 스트림 소비자는 이 이벤트를 지연된 이벤트로 본다.
잘못된 장치 시계를 조정하는 한 가지 방법은 세 가지 타임스탬프를 로그로 남기는 것이다.
1. 이벤트가 발생한 시간(장치 시계)
2. 이벤트를 서버로 보낸 시간(장치 시계)
3. 서버에서 이벤트를 받은 시간(서버 시계)
두 번째와 세 번째의 타임스탬프 차이를 구하면 장치 시계와 서버 시계간 오프셋을 추정할 수 있다.
계산한 오프셋을 이벤트 타임스탬프에 적용해 이벤트가 실제로 발생한 시간을 추정할 수 있다.
### 윈도우 유형
윈도우는 이벤트 수를 세거나 윈도우 내 평균값을 구하는 등 집계를 할 때 사용한다.

| **윈도우 유형** | **크기(길이)** | **윈도우 간 겹침(Overlap)** | **동작 방식 및 특징** |
| ---- | ---- | ---- | ---- |
| **텀블링 윈도우(Tumbling)** | 고정됨 | **겹치지 않음** | 각 이벤트는 **정확히 단 하나의 윈도우**에만 속함.  |
| **호핑 윈도우(Hopping)** | 고정됨 | **겹침 허용** | 윈도우 크기는 고정되어 있지만 특정 이동 간격(Hop)마다 윈도우가 생성되어 여러 인접한 윈도우를 모아 구현 |
| **슬라이딩 윈도우(Sliding)** | 유동적(이벤트 기준) | **겹침 허용** | 각 시간 간격 사이에 발생한 모든 이벤트를 포함. |
| **세션 윈도우(Session)** | 고정되지 않음(사용자 기준) | 없음 (세션별 독립) | 사용자가 짧은 시간 동안 발생시킨 모든 이벤트를 그룹화해서 세션 윈도우 정의. |

## 스트림 조인
스트림 상에서 새로운 이벤트가 언제든 나타날 수 있다는 사실은 **스트림 상에서 수행하는 조인**을 일괄 처리 작업에서 수행하는 조인보다 훨씬 어렵게 만든다.
### 스트림 스트림 조인(stream-stream join, 윈도우 조인)
> 두 개의 이벤트 스트림을 특정 시간 윈도우(Window) 내에서 결합하는 방식

#### 🤖 웹사이트에 검색 기능이 있고 검색된 URL의 최신 경향을 파악하고 싶다고 가정.
- 누군가 검색 질의를 타이핑할 때마다 질의와 반환된 결과가 있는 이벤트 로깅
- 누군가 그 검색 결과에서 하나를 클릭할 때마다 클릭을 기록하는 다른 이벤트 로깅
검색 결과에서 각 URL당 클릭율을 계산하려면 같은 세션 ID를 가져서 서로 연관되는 **검색 활동 이벤트**와 **클릭 활동 이벤트**를 함께 모아야 한다.

#### ⚠️ **문제 상황**
1. 사용자가 검색만 하고 클릭은 하지 않고 이탈할 수 있다.
2. 검색과 클릭 사이의 시간 간격이 가변적이다. (몇 초 일수도 있지만 검색 후 잊어버리고 늦게 클릭)
3. 네트워크 지연으로 인해 '클릭 이벤트'가 '검색 이벤트'보다 서버에 먼저 도착할 수 있다.

#### 🚨 주의 사항
클릭 이벤트에 검색에 대한 내용을 추가하면 조인이 필요하지 않지 않나?!
이렇게 하면 사용자가 클릭한 검색 결과만 알려주지 검색은 했지만 클릭하지 않은 경우를 알려주지 않는다.
**검색 품질을 측정하기 위해서는 정확한 클릭율이 필요하기 때문에 검색 이벤트와 클릭 이벤트가 모두 필요하다.**

#### 🧑‍💻 구현
이런 유형의 조인을 구현하려면 스트림 처리자가 <span class="text-brown">**상태(state)**</span>를 유지해야 한다.
- ex) 지난 시간에 발생한 모든 이벤트를 세션 ID로 색인

1. 검색, 클릭 이벤트가 발생할 때마다 해당 색인에 추가하고 스트림 처리자는 같은 세션 ID로 이미 도착한 다른 이벤트가 있는지 확인한다.
2. 이벤트가 매칭되면 검색한 결과를 <u>클릭했다고 말해주는 이벤트</u>를 방출한다.
3. 설정한 윈도우 시간이 되도록 클릭 이벤트 매칭이 없이 만료되면 <u>클릭되지 않았다고 말해주는 이벤트</u>를 방출한다.
### 스트림 테이블 조인(stream-table join, 스트림 강화)
> 흘러가는 이벤트 스트림에 데이터베이스(테이블)의 정적인 정보를 덧붙여(강화) 데이터를 만드는 방식

#### 🤖 사용자 활동 이벤트 분석
`사용자 활동 이벤트 스트림(사용자 ID만 존재)`에 `사용자 프로필 테이블(나이, 지역 등)`을 조인하기
- 이 과정을 데이터베이스의 정보로 활동 이벤트를** 강화**한다고 한다.

#### 🧩 방법
1️⃣ 이벤트가 들어올 때마다 매번 원격 DB를 질의하여 프로필 정보를 추가한다.
원격 질의는 느리고 데이터베이스에 과부하를 줄 위험이 있다.

2️⃣ **스트림 처리자 내부에 데이터베이스 사본을 적재**
- 네트워크 왕복 없이 로컬에서 질의가 가능
- 스트림 처리는 오랜 기간 수행하기 때문에 데이터베이스의 내용이 변할 가능성이 높아 로컬 복사본을 최신 상태로 유지해야 한다.
💡 **`변경 데이터 캡처`**를 사용하면 해결 가능하다.
- 스트림 처리자는 활동 이벤트를 스트림으로 받는 것처럼 사용자 프로필 데이터베이스의 변경 로그를 구독
- 프로필을 생성하거나 수정하면 스트림 처리자는 로컬 복사본을 갱신한다.
이렇게 활동 이벤트와 프로필 갱신이라는 두 개의 스트림을 조인할 수 있다.
### 테이블 테이블 조인(table-table join, 구체화 뷰 유지)
> 두 개의 스트림(테이블의 변경 로그)을 조인하여 특정 쿼리의 결과를 미리 계산해 두는 캐시(구체화 뷰)를 실시간으로 유지하는 방식

#### 🤖 트위터 타임라인 
사용자가 자신의 홈 타임라인을 볼 때 **사용자가 팔로우한 사람 모두를 순회하여 최근 트윗을 찾아 병합**하는 데는 너무 많은 비용이 든다.
대신 트윗이 전송될 때 기록되는 일종의 사용자별 `‘받은 편지함’` 같은 타임라인 <span class="text-blue">**캐시**</span>를 사용한다.
#### 🧑‍💻 구현
**트윗 이벤트 스트림(전송, 삭제)과 팔로우 관계 이벤트 스트림(팔로우, 언팔로우)이 필요하다.**
- A가 새 트윗 작성 → A의 팔로위들 타임라인 캐시에 트윗 추가
- 사용자가 트윗을 삭제하면 모든 사용자의 타임라인 캐시에서 해당 트윗 삭제
- 사용자 A가 B를 팔로우하기 시작하면 B의 최근 트윗을 A의 타임라인 캐시에 추가
- 사용자 A가 B 팔로우를 취소했을 때 사용자 B의 트윗을 사용자 A의 타임라인에서 제거
### 조인의 시간 의존성
설명한 3가지 조인 유형은 공통점이 많다.
> 세 가지 유형 모두 **스트림 처리자가 하나의 조인 입력을 기반으로 한 특정 상태를 유지**해야 하고** 다른 조인 입력에서 온 메시지에 그 상태를 질의**한다.

**상태를 유지하는 이벤트의 순서는 매우 중요하다.**(먼저 팔로우한 다음 언팔로우했는지 그 반대인지..)
파티셔닝된 로그에서 **단일 파티션 내 이벤트 순서는 보장**되지만 **다른 스트림이나 다른 파티션 사이에서 순서를 보장하는 일반적인 방법은 없다.**

#### 🤔 비슷한 시각에 다른 스트림에서 발생한 이벤트는 어떤 순서로 처리될까?
복수개의 스트림에 걸친 이벤트 순서가 결정되지 않으면 조인도 비결정적이다.
- 동일한 입력으로 같은 작업을 재수행하더라도 같은 결과를 얻지는 못한다는 의미이다.
이 문제를 데이터 웨어하우스에서는 **천천히 변하는 차원(slowly changing dimension, SCD)**이라 한다.
🚀 **조인되는 레코드의 특정 버전을 가리키는 데 유일한 식별자를 사용해 해결한다.**
## 내결함성
스트림 처리는 끝이 없어 일괄 처리방식 처럼 처음부터 다시 실행하는 것이 불가능
### 마이크로 일괄 처리와 체크포인트
#### **마이크로 일괄 처리 (Microbatching)**
> 스트림을 1초 등의 작은 블록 단위로 쪼개어 각 블록을 일괄 처리 같이 다루는 방법

일괄 처리 크기가 **작을수록** 스케줄링과 코디네이션 비용이 커지고 **클수록** 스트림 처리의 결과를 보기까지 지연시간이 길어진다.
#### **체크포인트 (Checkpointing)**
> 주기적으로 상태의 롤링 체크포인트를 생성하고 지속성 있는 저장소에 저장

장애 시 마지막 체크포인트부터 다시 시작

두 방식 모두 `정확히 한 번`을 보장하지만 이메일 전송이나 DB 쓰기 등 출력이 스트림 처리자를 떠나자마자 실패한 일괄 처리 출력을 더 이상 지울 수 없다.
이 경우 실패한 태스크를 재시작하면 외부 부수 효과가 두 번 발생한다.
마이크로 일괄 처리와 체크포인트 접근법만으로는 이 문제를 방지하기에 충분하지 않다.
### 원자적 커밋 재검토
장애가 발생했을 때 정확히 한 번 처리되는 것처럼 보일려면 **처리가 성공했을 때만** 모든 출력과 이벤트 처리의 부수 효과가 발생하게 해야 한다.
이런 효과는 원자적으로 모두 일어나거나 또는 모두 일어나지 않아야 하지만 서로 동기화가 깨지면 안된다.
### 멱등성
결국 목표는 처리 효과가 두번 일어나는 일 없이 재처리하기 위해 실패한 태스크의 부분 출력을 버리는 것이다.
분산 트랜잭션도 한 가지 방법이지만 다른 방법으로 <span class="text-red">**멱등성**</span>에 의존하는 방법이 있다.
:::callout ℹ️ **멱등 연산**
:::callout 여러 번 수행하더라도 오직 한 번 수행한 것과 같은 효과를 내는 연산

💡 연산 자체가 멱등적이지 않아도(ex: 카운터를 증가 시키는 것) 약간의 **여분 메타데이터로 연산을 멱등적으로 만들 수 있다.**
예를 들어 카프카로부터 메시지를 소비할 때 **모든 메시지는 영속적이고 단조 증가하는 오프셋**이 있다.
외부 데이터베이스에 값을 기록할 때 마지막으로 그 값을 기록하라고 트리거한 메시지의 오프셋을 함께 포함하여 이미 갱신이 적용됐는지 확인할 수 있어 반복해서 같은 갱신이 수행되는 것을 막을 수 있다.
### 실패 후에 상태 재구축하기
윈도우 집계나 조인을 위해 사용하던 상태가 필요한 스트림 처리는 **실패 후에도 해당 상태가 복구됨을 보장**해야 한다.
#### 원격 데이터 저장소에 상태를 유지하고 복제하기
개별 메시지는 질의하는 것은 느리다.
#### 스트림 처리자의 로컬에 상태를 유지하고 주기적으로 복제하기
스트림 처리자가 실패한 작업을 복구할 때 새 태스크는 복제된 상태를 읽어 데이터 손실 없이 처리를 재개할 수 있다.
#### 윈도우가 작은 경우 입력 이벤트를 재생
어떤 경우에는 상태 복제가 필요 없을 수도 있다. 입력 스트림을 사용해 재구축할 수 있기 때문이다.
상당히 작은 크기의 윈도우를 집계해 만든 상태라면 그냥 해당 윈도우에 해당하는 입력 이벤트를 재생해도 충분히 빠르다.
