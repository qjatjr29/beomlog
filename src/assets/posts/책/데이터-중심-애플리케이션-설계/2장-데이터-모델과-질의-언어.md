데이터 모델은 소프트웨어 개발에서 제일 중요한 부분일 것이다.
- **소프트웨어가 어떻게 작성됐는지** 뿐만 아니라 **해결하려는 문제를 어떻게 생각해야 하는지**에 대해서도 영향을 미치기 때문

### <span class="text-red">추상화</span>
대부분의 애플리케이션은 하나의 데이터 모델을 다른 데이터 모델 위에 계층을 둬서 만든다.
- 각 계층의 핵심 문제는 다음 하위 계층 관점에서 데이터 모델을 표현하는 방법이다.

`예시`
**1단계 (애플리케이션 개발자)**: 현실(사람, 상품 등)을 객체(Object)나 데이터 구조로 표현
**2단계 (데이터 구조 저장)**: 객체를 JSON, XML, 관계형 테이블, 그래프 등의 범용 데이터 모델로 추상화하여 표현
**3단계 (데이터베이스 엔지니어)**: 데이터 모델을 메모리나 디스크의 바이트 배열로 어떻게 표현할지 결정
**4단계 (하드웨어 엔지니어)**: 바이트를 전기 신호, 빛의 파동, 자기장 등으로 표현하는 방법을 찾아 처리

각 계층은 명확한 데이터 모델을 제공해 하위 계층의 복잡성을 숨긴다.
- 각 계층은 자신만의 데이터 모델을 제공

다양한 유형의 데이터 모델이 있고 각 데이터 모델은 사용 방법에 대한 가정을 나타낸다.
- `사용자는 이런 식으로 데이터를 읽고 쓸 것이다`
- 어떤 가정에서는 데이터 변환이 자연스럽지만 가정이 맞지 않으면 느려지고 복잡해진다.

🚀 **애플리케이션에 적합한 데이터 모델을 선택하는 작업이 중요하다.**

## 🆚 관계형 모델과 문서 모델
오늘날 가장 잘 알려진 데이터 모델은 <span class="text-red">관계형 모델</span>을 기반으로한 <span class="text-red">**SQL**</span>이다.
- 데이터는 관계로 구성되고 각 관계는 순서 없는 튜플 모음

관계형 데이터베이스의 근원은 **비즈니스 데이터 처리**에 있다.
- 트랜잭션 처리
- 일괄 처리

### NoSQL의 탄생
NoSQL은 대규모 데이터셋과 높은 쓰기 처리량을 감당할 수 있는 확장성을 제공하며 관계형 모델에서 지원하지 않는 특수 질의 동작, 관계형 스키마의 제약에서 벗어나 유연하고 동적인 데이터 모델을 원했던 사람들의 불만을 해소

애플리케이션은 요구사항이 각각 다르기 때문에 **관계형 데이터베이스가 다양한을 가진 비관계형 데이터스토어와 함께 사용**될 것이다.
- **다중 저장소 지속성**

## ⚠️ 객체 관계형 불일치
현대의 애플리케이션 개발은 대부분 객체지향 언어를 사용하지만 데이터를 관계형 테이블에 저장하려면 애플리케이션 코드와 데이터 베이스 모델 객체 사이에 전환을 위한 계층이 필요하다.

**ORM(Object-Relational Mapping)** 프레임워크(ActiveRecord, Hibernate 등)는 보일러플레이트 코드의 양을 줄여주지만 두 모델 간의 차이를 완전히 숨길 수는 없다.

### 예시 - 링크드인 프로필 (이력서)
사용자 프로필 데이터(이력서)를 생각해보자.
- `user_id`, `first_name`, `last_name`은 사용자당 한 번 나타남 (1:1)
- `positions`(경력), `education`(학력), `contact_info`(연락처)는 **1:N 관계**

### **관계형 모델에서의 접근**
**1. 전통적인 SQL 방식**
- `users`, `positions`, `education`, `contact_info` 테이블을 분리
- 외래 키(Foreign Key)로 `user_id` 참조
- 프로필 전체를 가져오려면 **여러 쿼리** 또는 **복잡한 다중 조인(Multi-way Join)** 필요

**2. 최신 SQL 표준 지원**
- 일부 RDBMS는 `JSON/XML` 타입을 지원하여 단일 행에 다중 값 저장 가능

### **문서 모델의 접근(JSON)**
```json
{
  "user_id": 251,
  "first_name": "Bill",
  "last_name": "Gates",
  "region_id": "us:91",
  "industry_id": 131,
  "positions": [
    {"job_title": "Co-chair", "organization": "Gates Foundation"},
    {"job_title": "Co-founder", "organization": "Microsoft"}
  ],
  "education": [...],
  "contact_info": {...}
}
```

- **지역성(Locality)**: 관련 정보가 한 곳에 모여 있어 **한 번의 쿼리**로 전체 프로필 조회 가능
- **트리 구조 명시**: 1:N 관계가 의미상 데이터 트리 구조와 같음

### 다대일과 다대다 관계
🤔 프로필 예시에서 `region_id`와 `industry_id`를 **ID로 저장**하는 이유
- ID를 사용하여 의미 있는 한 곳에 저장하고 그것을 참조하는 모든 것은 ID를 이용
- 텍스트를 직접 저장한다면 그것을 사용하는 모든 레코드에서 중복해서 저장하게 된다.

> ⚠️ **문제점**
> **중복된 데이터를 정규화하기 위해서는 N:1 관계가 필요한데 N:1 관계는 문서 모델에 적합하지 않다.**
> - `관계형 DB`: 조인이 쉬움
> - `문서형 DB`: 1:N 트리 구조에서는 조인이 필요하지 않지만 조인 지원이 약하거나 없음.


초기에는 조인 없는 문서 모델이 적합해 보이지만 기능이 추가되면서 데이터가 점점 더 **상호 연결**되는 경향이 있습니다.

`예시 - organization(회사)와 school_name(공부한 곳)` 
- 단순 문자열(`"Microsoft"`)에서 → 엔터티(로고, 웹페이지, 뉴스 피드 포함)로 변경
- 여러 프로필이 같은 조직을 참조 → **N:1 관계** 발생

`예시 - 추천서`
- 사용자 A가 사용자 B를 추천
- 추천인의 정보가 B의 프로필에 표시되며 A의 이름과 사진 포함
- A가 사진을 변경하면 모든 추천글에 반영되어야 함 → **N:M 관계(Many-to-Many)** 발생

## 🤔 **문서 데이터베이스는 역사를 반복하고 있는가?**
오늘날 문서형 DB(NoSQL)가 겪고 있는 N:1, N:M 관계의 어려움은 사실 1970년대 초기 데이터베이스 시스템이 겪었던 문제와 매우 유사하다.

### **계층 모델 (Hierarchical Model)과 네트워크 모델 (Network Model)**
**계층 모델 (1970년대 IBM IMS 등)**
- 현재의 JSON 문서 모델과 비슷하게 **모든 데이터를 트리(Tree) 구조로 중첩**하여 저장. 
- 1:N 관계에는 탁월했지만 N:M 관계 처리가 어려워 데이터를 중복하거나 수동으로 개발자가 참조를 수동으로 해결

**네트워크 모델 (CODASYL)**
- 계층 모델의 트리구조에서는 모든 레코드는 정확히 하나의 부모만 있었다면 네트워크 모델에서는 레코드가 다중 부모를 가질 수 있다.
- 다대다 관계에서는 다양한 다른 경로가 같은 레코드로 이어질 수 있어 다양한 접근 경로를 계속 추적을 해줘야해 질의와 갱신을 위한 코드가 복잡하고 유연하지 못한 문제가 발생.

### **관계형 모델**
관계형 모델은 개발자가 복잡한 `접근 경로`를 생각할 필요 없이 `쿼리 옵티마이저(Query Optimizer)`가 어떤 인덱스를 탈지 어떤 순서로 조인할지, 실행할지 자동으로 결정

## ✅ 관계형 데이터베이스와 문서 데이터베이스
문서 데이터 모델을 선호하는 주요 이유
- 스키마 유연성
- 저장소 지역성

관계형 모델의 강점
- 조인
- 다대다, 다대일 관계 지원

### 문서 모델에서의 스키마 유연성 
문서 데이터베이스와 관계형 데이터베이스에서 지원하는 JSON은 문서의 데이터에 스키마를 강요하지 않는다.
문서 데이터베이스는 읽기 쓰기마로 **쓰기(Write) 시점에 스키마를 강제하지 않고 읽기(Read) 시점에 애플리케이션 코드가 데이터 구조를 해석한다**.** (동적 타입 확인과 유사).**

애플리케이션 데이터 타입이 변경될 때 유리하다.
- 새로운 필드를 가진 새로운 문서를 작성하고 애플리케이션 단에서 예전 문서를 읽은 경우를 처리
- RDBMS처럼 `ALTER TABLE`로 인한 서비스 중단이나 마이그레이션 문제가 없다.

> 💡 **쓰기 스키마**
> - 스키마는 명시적이고 데이터베이스는 쓰여진 모든 데이터가 스키마를 따르고 있음을 보장


> 💡 **읽기 스키마**
> - 데이터 구조는 암묵적이고 데이터를 읽을 때만 해석


### 질의를 위한 데이터 지역성
문서 전체가 하나의 연속된 문자열으로 저장되므로 **한 번에 해당 문서의 많은 데이터를 조회할 때** 디스크 탐색이 줄어들어 성능이 향상
- 문서의 작은 일부만 수정해도 전체를 다시 써야 하므로 문서를 작게 유지하는 것이 좋다.

## 🔍 데이터를 위한 질의 언어
관계형 모델이 등장했을 때 데이터를 질의하는 새로운 방법도 나타남.
- 선언형 질의 (SQL)
- 명령형 코드 (IMS, 코다실)

많이 사용하는 프로그래밍 언어는 **명령형 언어**이다.
- 특정 순서로 특정 연산을 수행하게끔 컴퓨터에게 지시
- <span class="text-red">**`How`**</span>

SQL이나 관계 대수 같은 선언형 질의 언어에서는 목표를 달성하기 위한 <span class="text-red">**방법**</span>이 아니라 <span class="text-red">**알고자 하는 데이터의 패턴과 조건(결과)**</span>만 명시한다.
- 코드가 훨씬 짧고 읽기 쉽다.
- **데이터베이스의 자동 최적화: **`어떻게` 가져올지는 DB의 `쿼리 옵티마이저`가 결정.
  - 데이터베이스 엔진의 상세 구현이 숨겨져 있어 질의를 변경하지 않아도 데이터베이스 시스템 성능을 향상시킬 수 있음.
- 병렬 실행에 적합
  - 실행 순서에 의존하지 않아 다중 코어나 다중 장비에서 작업을 병렬로 쪼개어 처리하기 쉬움

### 맵리듀스 질의
`맵리듀스` 는 많은 컴퓨터에서 대량의 데이터를 처리하기 위한 프로그래밍 모델
- 몽고 DB, 카우치DB를 포함한 일부 NoSQL 데이터 저장소는  읽기 전용 쿼리를 위해 제한된 형태의 맵리듀스를 지원

맵리듀스는 선언형(SQL)도 아니고 명령형도 아닌 **그 중간 지점**이다.
분산 처리, 병렬 실행 등은 은 처리 프레임워크가 담당하지만 개발자는 `map`과 `reduce` 함수의 로직을 직접 작성해야 하기 때문
- **`map`**** (collect):** 문서를 순회하며 특정 키(Key)와 값(Value)을 추출해 방출(emit)
- **`reduce`**** (fold):** 같은 키를 가진 값들을 모아서 하나로 집계

## 〽️ 그래프형 데이터 모델
데이터에서 다대다 관계가 매우 일반적이라면 어떻게 해야 할까?
관계형 모델은 단순한 다대다 관계를 다룰 수 있지만 데이터간 연결이 복잡해지면 **그래프**로 데이터를 모델링하기 시작하는 편이 더 자연스럽다.

그래프는 두 유형의 객체로 이뤄진다.
**정점(vertex)**: 노드나 엔티티
**간선(edge)**: 관계나 호

많은 유형의 데이터를 그래프로 모델링할 수 있다.
- 소셜 그래프
- 웹 그래프
- 도로나 철도 네트워크

### 속성 그래프 (Property Graph)
`Neo4j`, `Titan`
**정점과 간선으로** 구성되어 있으며 아래와 같은 특징이 있다.
- 어떤 정점이든 다른 정점과 간선으로 연결될 수 있음. (스키마 제약 없음)
- 각 정점과 간선은 내부에 `key-value` 쌍의 **속성**을 가질 수 있음.
- 정점이 주어지면 정점의 유입과 유출 간선을 효율적으로 찾아 그래프를 순회할 수 있음.

이렇게 그래프는 데이터 모델링을 위한 많은 유연성을 제공

### 사이퍼 질의 언어 (Cypher)
**사이퍼**는 속성 그래프를 위한 `선언형` 질의 언어
- Neo4j 데이터베이스용
- `() -[:MATCH] → ()` 같은 화살표 문법을 통해서 시각적으로 노드간의 패턴을 매칭

질의를 작성할 때 수행할 내용에 대해서 자세히 지정할 필요없이 복잡한 경로 탐색 쿼리를 단 몇 줄의 코드로 작성하면 내부 실행 전략은 쿼리 옵티마이저가 가장 효율적이라 예측한 전략을 자동으로 선택

### 트리플 저장소(Triple-Store)
**트리플 저장소 모델**은 속성 그래프 모델과 거의 유사하지만 모든 정보를 (`주어 - Subject`, `서술어 - Predicate`, `목적어 - Object`)라는 세 부분 구문 형식(three-part statements)으로 저장한다.
ex) `(짐, 좋아하다, 바나나`) : 짐은 주어, 좋아하다는 서술어(동사), 바나나는 목적어

**주어**: 그래프의 **정점**과 동등
**목적어**는 아래 두 가지 중 하나
- 문자열이나 숫자와 같은 원시 데이터타입의 값 (루시, 나이, 33)
  - 서술어, 목적어가 주어의 속성의 키, 값과 동등
- 그래프의 다른 정점 - (루시, 결혼하다, 알랭)
  - 서술어는 두 정점을 잇는 간선

### 스파클(SPARQL)
RDF 데이터 모델을 사용한 **트리플 저장소 질의 언어**

### 추상화 계층
이 챕터의 앞 부분에서는 추상화 계층에 대한 이야기가 나왔는데 이 책에서 이 `추상화 계층`을 중요하게 생각하는 이유는 **"우리가 어느 계층의 문제를 풀고 있는지 명확히 인지하기 위해서" **인 것 같다고 생각했다.
가끔 개발을 하다 보면 비즈니스 로직에서 풀어야 할 문제를 데이터베이스의 복잡한 쿼리로 억지로 해결하려다 시스템 전체가 느려지거나 반대로 DB가 쉽게 처리할 수 있는 관계형 데이터를 애플리케이션 메모리로 전부 끌고 와서 느리게 조인(Join)하는 실수를 할 수 있다..!!

> 💡 따라서 <u>**"지금 다루고 있는 데이터 모델이 어떤 추상화의 결과물이며 어떤 가정을 하고 만들어졌는지 이해해야만 아키텍처의 함정에 빠지지 않는다"**</u> 는 것을 계속해서 인지하고 있어야겠다.


### RDBMS 쿼리 옵티마이저와 실행계획
관계형 DB가 강력한 이유는 `어떻게 가져올지`를 개발자가 아닌 <span class="text-blue">**쿼리 옵티마이저(Query Optimizer)**</span>가 자동으로 결정해 주기 때문이다!!
따라서 실제 쿼리 옵티마이저는 어떻게 결정을 하는지 알아보고자 했다. - ***Join할 때 최적화된 접근 경로 선택***

🏃 **쿼리 옵티마이저의 동작 원리**
1. **실행 순서(Join Order) 결정**
  - 결과 수가 가장 적을 것으로 예상되는 테이블을 항상 먼저 읽는다 (Driving Table)
2. **조인 알고리즘 선택**
  1. **소량 데이터:** `Nested Loop Join` + B-Tree 인덱스 활용
  1. **대량 데이터:** `Hash Join` + 인덱스 무시하고 Full Scan (메모리 해시 테이블 활용)
  1. **정렬된 데이터:** `Sort-Merge Join` (이미 정렬되어 있거나 인덱스로 정렬이 쉬울 때)
