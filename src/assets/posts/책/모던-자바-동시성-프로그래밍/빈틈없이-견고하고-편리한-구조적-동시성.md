:::callout 🎯 **챕터 목표**
:::callout - **프로젝트 룸(Project Loom)**의 가장 중요한 혁신 중 하나인 <span class="text-red">**StructuredTaskScope**</span>를 다룬다.
:::callout - <span class="text-red">**비구조적 동시성**</span>에 존재하는 문제를 어떻게 해결하는지 살펴보고 다양한 **조인(join) 정책을 익히고** **예외를 우아하게 처리하며 견고한 동시성 애플리케이션을 구축**하는 방법을 배운다.
:::callout - 기본 사용법부터 커스텀 조이너(joiner)와 중첩 스코프(nested scope)까지 폭넓게 다룬다.
수천 개의 가벼운 가상 스레드를 거의 오버헤드 없이 생성할 수 있게되고 블로킹 호출은 걱정되지 않지만 해결해야할 중요한 문제가 남아 있다.
#### ⚠️ <span class="text-red">**여러 스레드에 나뉘어 분산된 태스크들 사이 관계와 의존성을 어떻게 관리할 것인가**</span>
하나의 부모 태스크가 여러 개의 서브태스크에 의존하고 각각의 서브태스크가 서로 다른 스레드에서 실행한다고 가정
- 부모 태스크가 예상치 못하게 종료되었더라도 서브태스크들은 결과가 의미가 없음에도 독립적으로 실행을 계속한다. 
- 부모 태스크가 모든 서브태스크가 완료된 뒤에만 끝난다면 서브태스크 중 하나라도 실패할 경우 전체 작업이 취소되어야 한다. (고아 스레드가 발생할 수 있음)

**기존의 동시성 모델**(ex: `ExecutorService`, `Future`)는 태스크의 위계 구조를 지원하지 않아 취소 신호를 전파하거나 오류를 구조적으로 처리하기 어렵다.
- 기존의 동시성 프로그래밍은 단편화되고 오류가 발생하기 쉽다.
:::callout ℹ️ **구조적 동시성 (structured concurrency)**
:::callout - 관련된 **여러 태스크를 하나의 단일 작업 단위**로 취급할 수 있게 하여 이런 문제를 해결하는 패러다임
:::callout - 서브태스크의 생애주기를 부모 태스크에 종속시켜 동시 실행 구조를 명확하고 관리하기 쉽게 한다.

## 비구조적 동시성의 문제
`ExecutorService` 와 `Future` 같은 추상화를 사용해 동시성을 관리하는 것은 태스크를 동시에 실행할 수는 있지만 태스크 간의 조율이 필요할 때 어려움을 겪는다.

```java
public ProductInfo fetchProductInfo(Long productId) {
    log("Fetching product & reviews for id: " + productId);

    try (var executorService = Executors.newVirtualThreadPerTaskExecutor()) {

        Future<Product> productTask = executorService.submit(() -> fetchProduct(productId));
        Future<List<Review>> reviewsTask = executorService.submit(() -> fetchReviews(productId));

        Product product = productTask.get();

        List<Review> reviews = reviewsTask.get();
        return new ProductInfo(product, reviews);

    } catch (ExecutionException | InterruptedException e) {
        Throwable cause = e.getCause() != null ? e.getCause() : e;
        if (e instanceof InterruptedException) {
            Thread.currentThread().interrupt();
        }

        throw new ProductServiceException("Fetch failed for id: " + productId, cause);
    }
}

void main() {
    ProductService productService = new ProductService();
    long testProductId = 1L;
		try {
        ProductInfo productInfo = productService.fetchProductInfo(testProductId);
    } catch (ProductServiceException e) {
        //...
    }
}
```

위 코드는 <span class="text-red">상품 상세 조회</span>와 <span class="text-red">리뷰 조회</span> 이 두 가지 동시 실행 태스크로 나눠서 동작하는 것처럼 보이지만 두 작업을 실행하는 메서드가 각각 독립적으로 동시에 실행되어 각 메서드는 성공하거나 실패할 수 있다.
⚠️ 만약 상품 데이터베이스에 접근할 수 없는 상황이라면 `fetchProduct()` **태스크는 예외를 던져 메인 스레드를 멈추게 하지만 ****`fetchReviews()`**** 태스크는 이를 알지 못하고** <span class="text-red">**자신의 스레드에서 계속 작업을 실행하게 되어 스레드 누수가 발생하여 잠재적으로 불필요하게 자원을 사용하고 서버에 부하를 유발한다.**</span>

:::callout ⚠️ **비구조적 동시성의 핵심 문제**
:::callout - 태스크들을 서로 독립적인 존재로 취급하고 그 **사이의 관계나 의존성을 무시**한다.
:::callout - 오류, 자원 누수, 성능 병목 현상이 쉽게 발생한다.
:::callout - 관측 도구를 사용하더라도 태스크와 서브태스크간의 관계를 파악하기 어렵다.
:::callout - 태스크나 스레드가 예측 불가능한 순서로 실행될 수 있어 특정 시점에 프로그램이 어떤 상태에 있는지 파악하기 어렵다.
## 든든한 구조적 동시성
<span class="text-red">**구조적 동시성**</span>**은 프로그램의 논리적 구조를 반영하여 동시에 실행되는 태스크들을 체계적으로 조직화한다.**
### 💡 **핵심 원칙**
어떤 태스크가 여러 개의 동시 실행 서브태스크를 생성한다면 그 서브태스크들은 모두 부모 태스크 코드의 동일한 지점으로 돌아와야 한다.
<u>**부모 태스크는 감독자의 역할을 한다 **</u>
- 서브태스크 모니터링
- 완료될 때까지 대기
- 문제 발생시 개입
### 👋 구조적 동시성 이점

| **이점** | **설명** |
| ---- | ---- |
| 관련 태스크 집합 전체에 걸친 **통합적 오류 처리와 취소** | 하나의 서브태스크가 오류를 만나거나 취소되면 같은 위계에 있는 태스크들은 자동으로 종료되며 오류는 상위로 전달된다. |
| 태스크의 부모-자식 생애주기를 엄격하게 보장해 **스레드 누수를 방지** | 태스크는 자신의 유효범위를 벗어나 존재할 수 없으며 부모 태스크는 모든 서브태스크를 정리한 뒤에만 종료될 수 있어 시스템 동작에 대한 예측 가능성이 높아진다. |
| 태스크간 **의존 관계를 명확하고 직관적으로 표현(관측성)** | 문제가 발생했을 때 근본 원인 분석과 해결 조치를 더 빠르게 수행할 수 있다. |
| **선언적인 프로그래밍 스타일** | 개발자는 스레드 관리의 복잡한 세부 사항보다 각 태스크의 비즈니스 로직에 집중할 수 있다. 이해와 수정이 쉬운 구조를 만들어 코드의 가독성과 유지보수성을 높이고 개발 시간과 비용을 줄여준다. |
| **구조적 동시성과 가상 스레드** | 구조적 동시성은 가상 스레드 위에서 동작하여 최소한의 오버헤드로도 굉장히 많은 수의 동시 실행 태스크를 생성할 수 있다. 구조적 동시성은 이런 가상 스레드들을 조율하는 핵심 프레임워크를 제공하며 대규모 환경에서도 안정하고 효율적인 실행을 보장한다. |

## 구조적 동시성 API 이해
구조적 동시성 API 중심에는 `java.util.concurrent` 패키지의 `StructuredTaskScope`가 있다.
### StructuredTaskScope
`StructuredTaskScope` 인터페이스를 사용하여 관련된 태스크들의 그룹을 관리할 수 있다.
- 같은 그룹에 속한 태스크들의 생애주기가 함께 묶여 동작하도록 보장

`StructuredTaskScope` 는 제네릭 타입 `T`와 `R`을 파라미터로 갖는 `sealed` 인터페이스이다.
- `T`: 스코프안에서 실행되는 서브태스크의 결과 타입
- `R`: 이 스코프의 결과 타입
```java
public sealed interface StructuredTaskScope<T, R> extends AutoCloseable
```


#### 주요 메서드 확인

| 메서드 | 설명 | 특징 |
| ---- | ---- | ---- |
| `<U extends T> Subtask<U> fork(Callable<? extends U> task)` | 스코프 내 새로운 서브태스크를 생성하고 실행 | 태스크는 별도의 스레드에서 동시에 실행되며 실행 중인 서브태스크를 나타내는 `SubTask` 객체를 반환한다. |
| `<U extends T> Subtask<U> fork(Runnable task)` | 스코프 내 새로운 서브태스크를 생성하고 실행 | 반환값이 없는 `Runnable` 태스크이기 때문에 결과를 반환하지 않고 성공적으로 완료되면 `Subtask.get()`은 `null`을 반환한다. |
| `join()` | 모든 서브태스크가 완료되거나 스코프가 취소될 때까지 기다린 후 결과를 반환 | 호출한 스레드를 블로킹해 모든 서브태스크가 끝날 때까지 기다린다. |
| `close()` | 스코프를 닫으며 모든 서브태스크가 완료되었거나 취소되었음을 보장한다. |  |

#### StructuredTaskScope를 사용한 fetchProductInfo 메서드 구현
```java
public ProductInfo fetchProductInfo(Long productId) {
    log("Fetching product & reviews for id: " + productId);

    try (var scope = StructuredTaskScope.open()) {
        StructuredTaskScope.Subtask<Product> productTask = 
            scope.fork(() -> fetchProduct(productId));
        StructuredTaskScope.Subtask<List<Review>> reviewsTask = 
            scope.fork(() -> fetchReviews(productId));
        scope.join();
        
        return new ProductInfo(productTask.get(), reviewsTask.get()); 
    } catch (InterruptedException | StructuredTaskScope.FailedException e) {
        Thread.currentThread().interrupt();
        throw new ProductServiceException("Fetch failed for id: " + productId, cause);
    }
}
```

`scope.join()`을 호출하여 2개의 서브태스크가 모두 완료되거나 둘 중 하나가 실패하거나 스코프 자체가 취소될 때까지 블로킹하며 결과를 기다린다.
2개의 서브태스크가 모두 성공하면 `join()` 메서드는 정상적으로 반환하여 결과를 반환하는 get()을 통해 인스턴스를 생성하고 반환한다.
> ⚠️ `fetchProduct()` 서브태스크가 실패하여 예외를 던진다면??!
> - **StructuredTaskScope**는 즉시 동시 실행중인 `fetchReviews` 서브태스크를 취소하고 부모 스레드는 예외를 전달받아 에러 처리를 한다.


#### 🚀 **구조적 동시성은 비구조적 동시성 방식의 문제를 해결해준다.**

| **장점** | **설명** |
| ---- | ---- |
| **불필요한 자원 낭비 방지** | 서브태스크 중 하나가 실패하면 나머지는 자동 취소된다. |
| **불완전한 데이터 문제 발생 방지** | 부모 태스크가 취소되면 스코프가 닫혀 서브태스크들도 자동으로 종료된다. |

### 스코프와 서브태스크: 관계와 생애주기
구조적 동시성은 **StructuredTaskScope**를 부모 컨테이너로 사용하여 자식** 서브태스크들**을 관리하며 **서브태스크**들의 생애주기와 실행을 책임진다.
- **서브태스크**는 **스코프안에서 가상 스레드로 실행되는 동시 작업 단위**를 의미.
> **StructuredTaskScope**는 동시 작업 수행을 위해 명확히 경계 지어진 컨텍스트를 정의하고 서브태스크 전체의 생애주기를 관리하여 서브태스크의 질서 있는 실행과 종료를 보장한다.


#### 서브태스크 포크하기
`fork()` 메서드를 사용해 동시 작업을 시작한다.
- `fork(Callable<? extends U> task)`: 값을 반환하도록 설계된 서브태스크를 만들 때 사용
- `fork(Runnable task)`: 값을 반환하지 않는 서브태스크를 만들 때 사용

#### 포크 과정
1. 비동기 태스크를 표현하는 **Subtask** 객체 생성
2. 내부 **Joiner** 정책 객체가 `onFork()` 메서드를 통해 호출
3. 정책에 위배되지 않는다면 새로운 가상 스레드가 시작되어 태스크가 실행

#### 서브태스크 완료
서브태스크가 작업을 끝내면 태스크를 실행하던 스레드는 **Joiner**의 `onComplete()` 메서드를 호출하여 작업 종료를 알린다.
스코프 소유자는 `join()`을 사용해 서브태스크 정책이 만족될 때까지 기다리고 하나의 서브태스크가 실패하면 스코프는 실행중인 다른 서브태스크들을 모두 취소한다.
- 실패하면 서브태크스에서 발생한 예외를 `FailedException`으로 감싸 던진다.
스코프가 닫힐 때 모든 서브태스크 스레드가 종료될 때까지 기다린 후 부모 스레드가 진행되도록 보장한다.
- 자원 누수 방지 및 동시 작업의 구조 유지 보장
### Joiner를 통한 조인 정책
> **Joiner**는 `join()` 메서드가 <u>어떤 조건을 만족할 때 완료되는지</u>, <u>어떤 결과를 반환할지</u>를 정의해 **서브태스크의 생애주기를 관리할 수 있는 메커니즘**이다.

### 공통적인 조인 정책들
**Joiner**는 여러 정책이 적용되어 있는** 정적 팩토리 메서드**를 제공한다.

| **정적 팩토리 메서드** | **설명** |
| ---- | ---- |
| `Joiner.awaitAllSuccessfulOrThrow()` | **기본 정책(전부 성공 아니면 무효). **완료된 태스크 수를 추적하여 하나라도 실패하면 스코프를 취소하고 모든 서브태스크가 성공할 때만 `join()`이 성공적으로 완료된다. |
| `Joiner.anySuccessfulResultOrThrow()` | **우승자를 선택하는 정책 구현(선착순)**. 여러 동시 실행 서브태스크 사이에서 가장 먼저 성공적으로 완료한 태스크가 있다면 스코프를 취소한다.(나머지 서브태스크들 즉시 취소) |
| `Joiner.allSuccessfulOrThrow()` | **전부 성공 아니면 무효. **기본 정책과 달리 모든 서브태스크가 성공했을 때 성공 결과를 Stream 형태로 제공하여 쉽게 처리할 수 있다. |
| `Joiner.awaitAll()` | 모든 서브태스크가 성공하든 실패하든 완료될 때까지 기다린다. 모든 태스크 결과를 알아야 하고 각각 개별적으로 처리해야할 때 유용. 가용성과 안정성이 최우선인 서버 애플리케이션에 적합(장애 격리) |
| `Joiner.allUntil(...)` | 유연한 정책을 지원하는 메서드. 사용자 정의 중단 조건 정의할 수 있다. |

### **StructuredTaskScope의 예외처리**
구조적 동시성에서 <span class="text-red">예외처리</span>는 사용중인 **Joiner** 정책에 따른 패턴을 따른다.
:::callout ℹ️ **StructuredTaskScope.FailedException**
:::callout - 예외가 던져지면 `join()` 메서드는 던져진 예외를 값으로 갖는 `StructuredTaskScope.FailedException`**으로 감싸서 던진다.**
:::callout - 여러 **Joiner** 구현체에서 이 예외는 실패한 서브태크스에서 발생한 예외를 나타낸다.
:::callout - ex) `allSuccessfulOrThrow()`나 `awaitAllSuccessfulOrThrow()`에서는 가장 먼저 실패한 서브태스크의 예외가 전달

#### 1️⃣ 기본적인 예외처리
가장 흔한 패턴은 구조적 동시성 코드를 `try-catch`로 감싸 **`FailedException`** 예외를 잡아 처리하는 방식이다. 이를 통해** 실제 원인을 확인하고 어떻게 처리할지 결정**할 수 있다.

#### 2️⃣ 패턴 매칭을 활용한 에러처리
여러 종류의 예외에 대해 각각 다른 복구 전략을 적용해야 할 때 **패턴 매칭을 사용하면 실패 유형에 따라 맞춤형으로 해결**할 수 있다.
👀 **전자상거래 주문 처리 시스템 예시 - 고객이 주문**
- 결제 처리, 재고 확인, 배송비 계산 같은 여러 작업이 성공해야 한다.
이 과정에서 각 작업은 서로 다른 방식으로 실패할 수 있고 고객에게 정확한 안내를 제공하여 문제를 이해하도록 하고 싶다.
💣 전통적인 예외 처리 방식에서는 **포괄적인 예외를 하나 잡아 애매한 메시지**를 반환할 수밖에 없다.
💡 **구조적 동시성과 패턴 매칭을 활용하면 사용자에게 구체적이고 의미있는 응답을 제공할 수 있다.**
```java
public class OrderProcessingService {

    private static OrderResult handleOrderProcessingError(Throwable cause) {
        return switch (cause) { 
            case PaymentDeclinedException pde -> new OrderResult(...);
            case InsufficientInventoryException iie -> new OrderResult(...);
            case ShippingNotAvailableException snae -> new OrderResult(...);
            case NetworkException ne -> new OrderResult(...);
            case SecurityException se -> new OrderResult(...);
            default -> new OrderResult(...);
        };
    }
    public OrderResult processOrder(String customerId,
                                    String productId, double amount) {

        try (var scope = open(Joiner.<String>allSuccessfulOrThrow())) { 

            var paymentTask = scope.fork(() ->
                    processPayment(customerId, amount));
            var inventoryTask = scope.fork(() ->
                    checkAndReserveInventory(productId));
            var shippingTask = scope.fork(() ->
                    calculateShipping(customerId, productId));

            var results = scope.join()
                    .map(Subtask::get)
                    .toList();

            String orderId = generateOrderId();
            return new OrderResult(orderId, "CONFIRMED",
                    "Order confirmed successfully", true);

        } catch (FailedException e) {
            Throwable cause = e.getCause(); 
            return handleOrderProcessingError(cause);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("Operation interrupted", e);
        }
    }
 }
```

`FailedException`에 감싸져 있는 원래 예외를 추출해 실패 유형을 결정하고 패턴 매칭을 통해 각 비즈니스 예외 타입을 사용자 에러 응답과 연결하여 반환한다.

#### 3️⃣ 전략적 예외 전파
때로는 예외를 상위 수준으로 전파하고 더 넓은 컨텍스트에서 에러를 처리하는 것이 가장 좋은 전략일 때도 있다.
- 다양한 수준에서 서로 다른 책임이 연관되는 계층형 애플리케이션을 만들 때 적합하다.
```java
public class ExceptionPropagationExample {
    // 상위 수준에서 중앙 집중적으로 처리할 수 있도록 예외를 전파
    public List<String> fetchCriticalData(List<String> sources)
            throws StructuredTaskScope.FailedException, InterruptedException {

        try (var scope = open(StructuredTaskScope.Joiner.
                <String>allSuccessfulOrThrow())) {

            var tasks = sources.stream()
                    .map(source -> scope.fork(()
                            -> fetchFromSource(source)))
                    .toList();

            // 실패해도 FailedException을 잡아 처리하지 않고 상위로 전파
            var results = scope.join();
            return results.map(StructuredTaskScope.Subtask::get)
                    .toList();
        }
        // catch 블록 없이 FailedException을 전파
    }

    // 중앙 집중적으로 예외를 처리하는 상위 수준 메서드
    public void processDataWithCentralizedHandling() {
        try {
            var sources = List.of("source1", "source2");
            var data = fetchCriticalData(sources);
            log("Successfully fetched data: " + data);
        } catch (StructuredTaskScope.FailedException e) {  // 예외처리
            log("Critical data fetch failed: " + e.getCause().getMessage());
            handleCriticalSystemFailure(e);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            log("Operation was interrupted");
        }
    }
    private String fetchFromSource(String source) { ... }
    private void handleCriticalSystemFailure(FailedException e) { ... }
}
```

> 하위 레벨 메서드가 실패를 적절히 처리할 충분한 컨텍스트를 가지고 있지 않을 경우 예외를 상위로 전파하라. 예외를 올려보내 회복 전략을 판단할 수 있는 컴포넌트가 예외를 처리하도록 한다.


#### 4️⃣ 서브태크스 안에서 예외 처리
특정 예외가 전체 작업 실패 대신 기본 결과를 반환해야 하는 경우 서브태스크 자체에서 예외를 처리하는 편이 더 낫다.
- 일부 실패가 하더라도 작업 전체 수준에서는 성공 처리될 수 있다.
```java
public class SubtaskExceptionHandling {

    public List<ServiceResponse> gatherOptionalData(List<String> services)
            throws InterruptedException {
        try (var scope
                 = open(Joiner.<ServiceResponse>allSuccessfulOrThrow())) {
            var tasks = services.stream()
                    .map(service -> scope.fork(() ->
                            fetchWithDefaults(service))) 
                    .toList();

            var results = scope.join();
            return results.map(Subtask::get)
                    .toList();

        } catch (FailedException e) { 
            // 서브태스크에서 처리하고 있으니 자주 발생하면 안되는 예외
            log("Unexpected failure: " + e.getCause().getMessage());
            throw new RuntimeException("System error", e);
        }
    }

    // 서브태스크안에서 예외처리 후 기본값 반환
    private ServiceResponse fetchWithDefaults(String service) {
        try {
            String data = fetchServiceData(service);
            return new ServiceResponse(service, data, true);
        } catch (IOException e) {
            log("Network error for " + service + ": " + e.getMessage());
            return new ServiceResponse(service, "Default data", false);
        } catch (TimeoutException e) {
            log("Timeout for " + service + ": " + e.getMessage());
            return new ServiceResponse(service, "Cached data", false);
        } catch (Exception e) {
            log("Unexpected error for " + service + ": " +  e.getMessage());
            return new ServiceResponse(service, "Error", false);
        }
    }

    private String fetchServiceData(String service)
            throws IOException, TimeoutException, InterruptedException {
        Thread.sleep(Duration.ofMillis(100));
        return "Data from " + service;
    }
}
```

> 부분적인 성공이 허용되는 선택적 작업에서는 우아한 실패 처리를 위해 서브태스크 예외를 서브태스크 내부에서 처리하라.


#### 5️⃣ 조이너별 예외 처리 차이
**Joiner** 선택은 예외 처리 방식에 중요한 영향을 미친다.

🆚 **allSuccessfulOrThrow()와 awaitAll() 예외 처리 방식 차이**
```java
public class JoinerExceptionHandlerDemo {
    private void demonstrateAllSuccessfulOrThrow() {
        try (var scope = open(StructuredTaskScope.Joiner.
                <String>allSuccessfulOrThrow())) {

            scope.fork(() -> successfulTask("Task1"));
            scope.fork(() -> failingTask("Task2"));
            scope.fork(() -> successfulTask("Task3"));

            var results = scope.join(); // ③
            log("All tasks completed successfully");

        } catch (StructuredTaskScope.FailedException e) {
            log("Failed due to: " + e.getCause().getMessage());
            log("Remaining tasks were cancelled");
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
    // ...
}
```

**`allSuccessfulOrThrow()`**** 방식의 특징**
- 모든 서브태스크가 <span class="text-red">**전부 성공 아니면 전부 실패**</span>하는 스코프를 생성
- 서브태스크에서 예외를 던지면 <span class="text-blue">**실패 즉시 중단**</span> 동작 시작
- **Task2**가 실패하면 **Task3**도 실패하면서 `join()` 호출이 예외를 던진다.

```java
public class JoinerExceptionHandlerDemo {
    private void demonstrateAwaitAll() {
        try (var scope = open(StructuredTaskScope.Joiner.
                <Void>awaitAll())) {

            scope.fork(() -> successfulTask("Task1"));
            scope.fork(() -> failingTask("Task2"));
            scope.fork(() -> successfulTask("Task3"));

            scope.join();

            log("All tasks were allowed to complete");
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
    // ...
}
```

**`awaitAll()`**** 방식의 특징**
- 개별 태스크가 <span class="text-red">**실패하더라도 모든 태스크가 완료될 때까지 기다린다**</span>.
- Task2가 예외를 던지지만 다른 태스크가 완료될 때까지 실행되는 것을 막지 않는다.
- `awaitAll()`이 적용된 스코프의 `join()` 메서드는 `FailedException`을 던지지 않는다.

#### 6️⃣ 일반적인 예외
`StructuredTaskScope.open()` 메서드에 <span class="text-brown">**null**</span>을 인자로 전달하면 `NullPointerException`이 발생한다.
이런 일관된 동작은 프로그래밍 오류를 초기에 발견하도록 돕고 API의 예측 가능성을 높여준다.

### 스코프 구성
**StructuredTaskScope**의 기본 구성은 많은 상황에서 충분히 잘 동작하지만 실제 애플리케이션에서는 동시 실행되는 태스크에 대해 더 세밀한 제어가 필요할 때가 있다.
구성을 사용하면 스레드 생성 방식, 모니터링 설정, 타임아웃 정책 등을 요구사항에 맞게 커스터마이징할 수 있다.
#### 구성 이해하기
기본적인 `open()`, `open(Joiner)` 메서드로 **StructuredTaskScope**를 생성하면 기본 구성이 적용된다.
- **ThreadFactory**: 이름 없는 가상 스레드 생성
- **모니터링용 이름**: 없음 (익명 스코프)
- **타임아웃**: 없음 (작업 완료 또는 취소될 때까지 동작)

**Configuration API**는 구성 함수를 인자로 받는** 오버로딩된** `open()` 메서드를 제공하는데 이를 통해 더 정교한 제어를 할 수 있다.
```java
static <T, R> StructuredTaskScope<T, R> 
        open(Joiner<? super T, ? extends R> joiner,
            Function<Configuration, Configuration> configFunction) {
    return StructuredTaskScopeImpl.open(joiner, configFunction);
}
```


**Configuration은 sealed 인터페이스로 3가지 설정 메서드를 가지고 있다.**
- 각 메서드는 기존 객체를 수정하지 않고 변경사항을 반영한 **새 Configuration 객체**를 반환한다**.**
```java
sealed interface Configuration {
    Configuration withThreadFactory(ThreadFactory threadFactory);
    Configuration withName(String name);
    Configuration withTimeout(Duration timeout);
}
```


#### 이름 있는 스레드
가장 보편적인 커스터마이징 방법은 **디버깅이나 모니터링을 더 쉽게 할 수 있도록 **<span class="text-blue">**이름있는 스레드**</span>를 생성하는 것이다.
```java
public class NamedThreadExample {

    public void processUserRequests(List<String> userIds) {
        ThreadFactory factory = Thread.ofVirtual()
                .name("user-processor-", 0)
                .factory();

        try (var scope = open(Joiner.<String>allSuccessfulOrThrow(),
                cf -> cf.withThreadFactory(factory))) {

            var tasks = userIds.stream()
                    .map(userId -> scope.fork(() -> processUser(userId)))
                    .toList();

            var results = scope.join()
                    .map(Subtask::get)
                    .toList();

            System.out.println("Processed users: " + results);

        } catch (FailedException | InterruptedException e) {
            System.out.println("Processing failed: " + e.getMessage());
        }
    }

    private String processUser(String userId)
            throws InterruptedException {
        System.out.println("Processing user " + userId +
                " on thread: " + Thread.currentThread().getName());
        Thread.sleep(Duration.ofMillis(100));
        return "User " + userId + " processed";
    }

    void main() {
        processUserRequests(List.of("user1", "user2", "user3"));
    }
}
---------------------출력 결과 -----------------------
Processing user user2 on thread: user-processor-1
Processing user user3 on thread: user-processor-2
Processing user user1 on thread: user-processor-0
```


#### 타임아웃 설정
타임아웃을 설정하여 애플리케이션이 무기한으로 멈춰있는 상황을 방지할 수 있고 외부 서비스가 응답하지 않을 때 빠르게 실패 처리를 할 수 있다.
```java
public class TimeoutExample {
    public List<String> fetchDataWithTimeout(List<String> sources)
            throws TimeoutException,
            FailedException,
            InterruptedException {
        Duration timeout = Duration.ofSeconds(5); // ①

        try (var scope = open(Joiner.<String>allSuccessfulOrThrow(),
                cf -> cf.withTimeout(timeout))) {  // ②
            var tasks = sources.stream() 
                    .map(source -> scope.fork(() -> fetchFromSource(source)))
                    .toList();

            // join()이 완료되기 전 타임아웃이 발생하면 TimeoutException이 던져진다.
            return scope.join() // ③
                    .map(Subtask::get)
                    .toList();
        }
    }

    private String fetchFromSource(String source) throws InterruptedException {
        int delay = switch (source) {
            case "fast_source" -> 1000;
            case "slow_source" -> 3000;
            case "very_slow_source" -> 8000; // ④
            default -> 2000;
        };

        Thread.sleep(Duration.ofMillis(delay));
        return "Data from " + source;
    }

    void main() {
        var sources = List.of("fast_source", "slow_source", "very_slow_source");

        try {
            var results = fetchDataWithTimeout(sources);
            System.out.println("Success: " + results);
        } catch (TimeoutException e) { // ⑤
            System.out.println("Operation timed out after 5 seconds");
            System.out.println("Some sources were too slow to respond");
        } catch (FailedException e) {
            System.out.println("Task failed: " + e.getCause().getMessage());
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            System.out.println("Operation interrupted");
        }
    }
}
```

1. 전체 작업에 대해 5초 타임아웃을 설정
2. 타임아웃이 설정된 **Configuration**을 사용해 스코프를 생성한다
  - 타임아웃은 스코프가 시작되는 시점부터 측정된다.
3. 5초안에 완료되지 못하는 서브태스크가 하나라도 있다면 `join()` 메서드는 `TimeoutException`을 던진다.
4. 8초가 걸리는 작업에 의해 전체 작업에 설정되어 있던 5초 타임아웃도 발생한다.
5. 타임아웃 관련 예외 처리

:::callout ℹ️** StructuredTaskScope.Joiner<T, R> 인터페이스**
:::callout - **커스텀 조이너**는 해당 인터페이스를 구현하여 만들 수 있다.
:::callout - 서브태스크 완료 이벤트를 처리하고<u> 스코프를 언제 종료할지 어떤 결과를 반환할지</u>를 결정하는 메서드를 제공한다.
모든 조이너는 **StructuredTaskScope.Joiner<T, R> 인터페이스**를 구현한다.
- `T`: 서브태스크가 생성하는 결과 타입
- `R`: 최종 결과 타입

#### 🤖 **StructuredTaskScope.Joiner<T, R> 인터페이스의 3가지 핵심 메서드**

| **메서드** | **설명** |
| ---- | ---- |
| `onFork(Subtask<? extends T> subtask` | 서브태스크가 fork 될 때 스레드가 생성되어 실행되기 전에 호출된다. |
| `onComplete(Subtask<? extends T> subtask)` | 서브태스크가 종료될 때 호출되며 서브태스크가 완료될 때 호출된다. 결과를 처리하고 계속할지 / 조기 종료할지 결정할 수 있다. |
| `result()` | 스코프가 최종 결과를 생성할 준비가 됐을 때 호출된다. |

#### 성공과 예외 모두 수집
성공을 하던지 실패를 하던지 모든 서브태스크의 결과와 예외를 수집하는 시나리오를 가정.
- 성공이나 실패에 대한 정보를 필요로 하는 일괄 처리에 적합하다.
```java
public class CollectingJoiner<T> implements StructuredTaskScope.Joiner<T, CollectingJoiner.Result<T>> {
    private final Queue<T> results = new ConcurrentLinkedQueue<>();
    private final Queue<Throwable> exceptions = new ConcurrentLinkedQueue<>();

    @Override
    public Result<T> result() {
        return new Result<>(
                results.stream().toList(),
                exceptions.stream().toList());
    }
    @Override
    public boolean onComplete(StructuredTaskScope.Subtask<? extends T> subtask) {
        switch (subtask.state()) {
            case SUCCESS -> results.add(subtask.get());  // ①
            case FAILED -> exceptions.add(subtask.exception());   // ②
            case UNAVAILABLE -> {    // ③
                // 태스크가 취소되고 실패로 처리
                exceptions.add(new RuntimeException("Task was cancelled"));
            }
        }
        return false;  // ④
    }

    public record Result<T>(List<T> successes, List<Throwable> failures) {
        public boolean hasFailures() {
            return !failures.isEmpty();
        }
        public int totalTasks() {
            return successes.size() + failures.size();
        }
    }
}
```

1️⃣ 서브태스크가 성공으로 종료되면 나중에 조회할 수 있도록 큐에 결과를 저장
2️⃣ 서브태스크가 실패하면 전체 작업을 즉시 실패로 처리하는 대신 예외 정보를 수집
3️⃣ UNAVAILABLE 상태의 서브태스크는 종료되기 전 취소된 것으로 실패로 간주하고 적절한 예외 생성
4️⃣ **false를 반환하여 모든 서브태스크가 종료될 때까지 계속 실행되게 한다.**

#### 정족수(quorum) 기반 완료
분산 시스템에서 사용하는 방식에서 영감을 받아 **일정 수의 서브태스크가 성공하면 완료되는 정족수** 기반 **조이너**를 구현해보자.
- 어떤 작업을 진행하기 전 정해진 수의 승인이 필요할 때 유용
```java
public class QuorumJoiner<T> implements StructuredTaskScope.Joiner<T, Boolean> {
    private final int requiredSuccesses;
    private final AtomicInteger successCount = new AtomicInteger(0);  // ①
    private final AtomicInteger totalCount = new AtomicInteger(0);    // ②
    private volatile boolean quorumReached = false;   // ③

    public QuorumJoiner(int requiredSuccesses) {
        this.requiredSuccesses = requiredSuccesses;
    }

    @Override
    public Boolean result() { return quorumReached; // ④ }

    @Override
    public boolean onFork(StructuredTaskScope.Subtask<? extends T> subtask) {
        totalCount.incrementAndGet();   // ⑤
        return false;  // 모든 태스크가 진행되도록 false 반환 // ⑥
    }

    @Override
    public boolean onComplete(StructuredTaskScope.Subtask<? extends T> subtask) {
        if (subtask.state() == StructuredTaskScope.Subtask.State.SUCCESS) {
            int currentSuccess = successCount.incrementAndGet(); // ⑦

            if (currentSuccess >= requiredSuccesses) {   // ⑧
                quorumReached = true;
                return true;  // ⑨
            }
        }
        return false;   // ⑩
    }

    public int getSuccessCount() {
        return successCount.get();
    }

    public int getTotalCount() {
        return totalCount.get();
    }
}
```

1️⃣ 스레드 안정성이 있는 **AtomicInteger**를 이용해 성공한 서브태스크 수 추적
2️⃣ 몇 개의 서브태스크가 fork 되었는지 추적해 전체 진행률 파악
3️⃣ 정족수가 충족되었는지 여부를 나타내는 `volatile` 플래그
- 스레드 사이에 즉시 상태가 공유가 되도록 한다.
4️⃣ 스코프 종료 시점에 정족수 달성 여부 반환
5️⃣ 서브태스크가 포크될 때마다 총 개수 증가
6️⃣ 포크할 때는 항상 **false**를 반환해 모든 서브태스크가 실행되게 한다.
7️⃣ 서브태스크가 성공하면 성공 카운터 증가 및 계산
8️⃣ 성공한 태스크의 수가 정족수에 도달했는지 검사
9️⃣ 정족수 도달시 플래그 **true**로 설정하고 남은 태스크 취소하여 스코프를 조기에 종료
🔟 아직 조건을 만족시키지 못했다면 **false**를 반환하여 다른 태스크가 계속 실행되도록 한다.

#### 적응형 완료(adaptive completion)
실행 중에 관찰한 결과에 따라 동적으로 완료 여부를 결정하고 싶을 때가 있다.
- **ex) 실패가 너무 많아질 경우 조기에 완료되는 방식의 조이너(적응형 완료)**
```java
public class AdaptiveJoiner<T> implements StructuredTaskScope.Joiner<T, CollectingJoiner.Result<T>> {
    private final int minSampleSize;
    private final double maxFailureRate;
    private final Queue<T> successes = new ConcurrentLinkedQueue<>();
    private final Queue<Throwable> failures = new ConcurrentLinkedQueue<>();

    public AdaptiveJoiner(double maxFailureRate, int minSampleSize) {
        this.maxFailureRate = maxFailureRate;
        this.minSampleSize = minSampleSize;
    }

    @Override
    public CollectingJoiner.Result<T> result() {
        return new CollectingJoiner.Result<>(
                successes.stream().toList(),
                failures.stream().toList()
        );
    }

    @Override
    public boolean onComplete(StructuredTaskScope.Subtask<? extends T> subtask) {
        switch (subtask.state()) {
            case SUCCESS -> successes.add(subtask.get());       // ①
            case FAILED -> failures.add(subtask.exception());   // ①
            case UNAVAILABLE -> failures.add(
                    new RuntimeException("Task cancelled"));
        }

        int total = successes.size() + failures.size();

        // 최소한의 샘플 크기 이상일 때만 실패율 체크
        if (total >= minSampleSize) {          // ②
            double failureRate = (double) failures.size() / total;
            return failureRate > maxFailureRate;        // ③
        }

        return false;  // ④ 중단하지 않고 계속 처리
    }
}
```

1️⃣ 성공과 실패 모두 수집
2️⃣ **의미 있는 결정을 내리는데 필요한 최소한의 데이터를 수집한 후 실패율 평가**
3️⃣ **실패율이 한계치를 넘어갈 경우 작업을 중단하여 자원 낭비 방지**
4️⃣ 수용할만한 수준이라면 중단하지 않고 계속 데이터를 처리

#### 요청 제한 조이너
서브태스크의 실행 속도를 조절해 외부 서비스에 과도한 요청을 보내는 것을 막아야 할때도 있다.
- **ex) Semaphore를 사용해 요청을 제한하는 조이너**
```java
public class RateLimitedJoiner<T>
        implements StructuredTaskScope.Joiner<T, List<T>> {
    private final Semaphore semaphore;  // ①
    private final Queue<T> results = new ConcurrentLinkedQueue<>();
    private final Queue<Throwable> failures = new ConcurrentLinkedQueue<>();

    public RateLimitedJoiner(int maxConcurrentTasks) {
        this.semaphore = new Semaphore(maxConcurrentTasks); // ②
    }

    @Override
    public boolean onFork(StructuredTaskScope.Subtask<? extends T> subtask) {
        try {
            semaphore.acquire();       // ③
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        return false;    // ④
    }

    @Override
    public boolean onComplete(StructuredTaskScope.Subtask<? extends T> subtask) {
        switch (subtask.state()) {
            case SUCCESS -> results.add(subtask.get());    // ⑤
            case FAILED -> failures.add(subtask.exception());   // ⑥
            case UNAVAILABLE -> failures.add(new RuntimeException("Task cancelled"));
        }

        semaphore.release();    // ⑦
        return false;        // ⑧
    }

    @Override
    public List<T> result() {
        return results.stream().toList();
    }
}
```

1️⃣ **Semaphore를 사용해 동시에 실행될 수 있는 태스크 개수 제어**
2️⃣ **동시 실행 허용 최대 태스크 수로 Semaphore를 생성**
3️⃣ 태스크 실행 시작전 세마포어로부터 출입증 획득
- 없다면 다른 태스크가 반환할 때까지 블로킹
4️⃣ 서브태스크가 생성되어 실행되도록 false 반환
5️⃣ 성공적으로 태스크가 끝나면 결과 수집
6️⃣ 실패한 경우에도 실패 정보 수집
7️⃣ 태스크 완료 여부와 상관없이 출입증 반환 - 다른 대기중인 태스크가 사용할 수 있도록
8️⃣ 조기 종료하지 않고 모든 태스크가 완전히 실행되도록 한다.

#### 조건부 조이너
`onFork` 메서드는 런타임 상태에 따라 조건부 실행을 구현할 때도 사용할 수 있다.
```java
public class ConditionalJoiner<T> implements StructuredTaskScope.Joiner<T, List<T>> {

    private final Supplier<Boolean> shouldContinue;          // ①
    private final Queue<T> results = new ConcurrentLinkedQueue<>();
    private final Queue<Throwable> failures = new ConcurrentLinkedQueue<>();

    public ConditionalJoiner(Supplier<Boolean> shouldContinue) {
        this.shouldContinue = shouldContinue;        // ②
    }

    @Override
    public boolean onFork(StructuredTaskScope.Subtask<? extends T> subtask) {
        if (!shouldContinue.get()) {       // ③
            System.out.println("Condition failed, stopping new tasks");
            return true;  // 새로운 태스크가 실행되지 않도록 스코프를 취소
        }
        return false;     // 조건이 만족되면 태스크 실행 허용
    }

    @Override
    public boolean onComplete(StructuredTaskScope.Subtask<? extends T> subtask) {
        switch (subtask.state()) {
            case SUCCESS -> results.add(subtask.get());         //  ④
            case FAILED -> failures.add(subtask.exception());    // ④
            case UNAVAILABLE -> failures.add(new RuntimeException("Task cancelled"));
        }

        return false;  // 실행중인 태스크 계속 처리  ⑤
    }

    @Override
    public List<T> result() {
        return results.stream().toList();
    }
    public List<Throwable> getFailures() {
        return failures.stream().toList();
    }
}
```

1️⃣ 계속 실행할지 여부를 결정하는 `Supplier<Boolean>` 형태의 조건을 받아 들인다.
2️⃣ 태스크 포크시 평가할 수 있는 **조건을 저장**
3️⃣ 새로운 태스크가 시작될 때마다 조건을 검사
- 조건이 만족되지 않는다면 새로운 태스크가 시작되지 않도록 스코프 중단
- 조건이 만족되면 태스크가 정상적으로 진행되도록 허용
4️⃣ 성공한 결과, 실패한 결과 수집
5️⃣ 이미 실행중인 태스크는 완료까지 진행하도록 한다.

### 메모리 일관성 효과
**StructuredTaskScope**는 <u>**스레드 간 공유 데이터의 가시성**</u>을 쉽게 파악할 수 있도록 명확하게 정의된 **메모리 일관성 보장 수단**을 가지고 있다.
1. **StructuredTaskScope**를 소유한 스레드에서 서브태스크를 포크하기 전 실행된 동작은 포크된 서브태스크가 수행하는 모든 작업보다 항상 먼저 실행된다.
2. 그리고 서브태스크가 수행하는 동작은 그 서브태스크의 결과를 가져오는 동작보다 먼저 실행된다.
> 이런 명확한 <span class="text-red">**선후 관계**</span>를 통해 스코프를 소유한 스레드가 `fork()` 호출 전에 **수행한 데이터 변경을 포크된 서브태스크에서 볼 수 있도록 보장**하고 **서브태스크에서 수행한 데이터 변경을 서브태스크 결과를 조회할 때 볼 수 있도록 보장**한다.


선후 관계 보장은 <span class="text-red">**소유자 스레드 ↔ 서브태스크**</span> 이렇게 양방향으로 동작한다.
- 서브 태스크가 공유상태를 수정하면 `join()` 완료 후 그 변경사항은 소유자 스레드에서 볼 수 있다.
