:::callout 🎯 **챕터 목표**
:::callout - 동시성의 내부 동작을 깊이 있게 다룬다.
:::callout - **스레드 풀**, **Executor 프레임워크**, **가상 스레드를 구동하는 ForkJoinPool을 분석**한다.
:::callout - 특히 `Continuation`을 사용해 가상 스레드 구현을 처음부터 직접 만들어본다.
:::callout - `NanoThread`를 만들어보며 가상 스레드가 내부적으로 어떻게 동작하는지 깊이 있게 이해할 수 있다.

가상 스레드의 구현을 이해하기 위해서는 2가지 개념을 알아야 한다.
- `ForkJoinPool`
- `Continuation`
`ForkJoinPool`은 가상 스레드를 스케줄링하는 스케줄러이고 `컨티뉴에이션(continuation)`은 가상 스레드가 실행을 멈췄다가 나중에 멈춘 지점에서 이어서 다시 실행할 수 있게 하는 핵심 메커니즘이다.

## 스레드 풀
> `Executor`는 **스레드 풀의 구현체**이다.


스레드 풀은 애플리케이션이 시작될 때 미리 생성되어 계속 실행중인 스레드들의 집합이다.
스레드 풀을 만드는 전략은 종류에 따라 다를 수 있지만 일정 수의 스레드를 미리 만들어 준비해 두고 계속 유지하며 사용한다는 핵심 아이디어는 동일하다.
- 태스크를 담아둘 큐가 있고 태스크가 큐에 들어오면 스레드 풀은 사용 가능한 스레드가 생기는 즉시 큐에서 태스크를 꺼내 실행한다.
- 하나의 태스크 처리가 끝나면 다음 태스크를 꺼내 실행하고 태스크가 큐에 있는 한 이를 반복한다.
### 스레드 풀이 필요한 이유
스레드가 필요할 때마다 스레드를 반복해서 생성하게 되면 문제가 발생한다.
- 계속해서 스레드를 만들다 보면 운영체제에서 허용하는 최대 스레드 수에 도달하여 애플리케이션이 다운될 수 있다.
따라서 실행 중인 애플리케이션에서 생성할 수 있는 스레드 개수에 대한 통제가 필요하다.

💡 **스레드 풀은 이 통제를 가능하게 해주며 너무 많은 동시 작업이 한꺼번에 자원을 사용하는 것을 방지하는 요청 제한 역할도 한다.**
- 적절한 스레드 개수를 미리 정하고 애플리케이션 시작 시점에 한도를 설정
- 많은 수의 동시 요청이 들어올 때 자원이 고갈되는 일을 막을 수 있다.
- 새로운 스레드를 매번 생성할 필요 없이 이미 생성된 스레드를 재사용해 즉시 태스크를 실행할 수 있어 응답 속도가 빨라진다.
- 스레드의 생애주기를 알아서 관리하여 개발자가 직접 스레드를 관리할 때 생길 수 있는 복잡성과 스레드 누수 같은 오류를 줄여준다.

### 단순한 스레드 풀 만들기
간단한 스레드 풀을 만들어보자 - `SimpleThreadPool` 클래스
먼저 내부 클래스인 **Worker** 클래스를 살펴보자.
```java
class Worker extends Thread {
    public Worker(ThreadGroup threadGroup, String name) {
       super(threadGroup, name);
    }

    @Override
    public void run() {
        while (running) {
            try {
                Runnable task = queue.take();
                task.run();
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }
}
```

**Worker 클래스는 Thread 클래스를 상속받는 독립적인 스레드로 공유된 큐에서 태스크를 계속 꺼내와 실행한다.**

**SimpleThreadPool**을 살펴보자.
```java
public class SimpleThreadPool implements AutoCloseable {

    private final BlockingQueue<Runnable> queue;
    private final ThreadGroup threadGroup;
    private volatile boolean running = true; // ①

    public SimpleThreadPool(int poolSize, int queueSize) {
        Worker[] threads = new Worker[poolSize];
        this.queue = new LinkedBlockingDeque<>(queueSize); // ②
        this.threadGroup = new ThreadGroup("SimpleThreadPool");

        for (int i = 0; i < poolSize; i++) {
            threads[i] = new Worker(threadGroup, "Worker-" + i);
            threads[i].start(); // ③
        }
    }

    public void submit(Runnable task) {
        try {
            queue.put(task); // ④
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }

    public void shutdown() {
        this.running = false;
        threadGroup.interrupt(); // ⑤
    }

    @Override
    public void close() {
        while (!queue.isEmpty()) { // ⑥
            try {
                Thread.sleep(100);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                return; // exit gracefully
            }
        }
        shutdown();
    }

    class Worker extends Thread {
        public Worker(ThreadGroup threadGroup, String name) {
            super(threadGroup, name);
        }

        @Override
        public void run() {
            while (running) {
                try {
                    Runnable task = queue.take();
                    task.run();
                } catch (InterruptedException e) {
                    // 풀은 인터럽트를 사용해 셧다운하지 않으므로
                    // 인터럽트 발생 시에는 그냥 현재 스레드를 인터럽트한다.
                    Thread.currentThread().interrupt();
                }
            }
        }
    }
}
```


#### 스레드 풀 3가지 필드
1️⃣ **running 플래그**
- <u>running 플래그</u>는 워커의 생애주기를 제어하는 역할을 한다.

2️⃣ **queue**
- 큐에 태스크가 있고 풀의 상태가 실행중이라면 풀에 있는 스레드는 큐에서 태스크 꺼내기를 시도한다.
- 큐에 태스크가 없다면 스레드는 새로운 태스크가 들어올 때까지 기다린다.
- <u>블로킹 큐</u>를 사용해 태스크가 큐 안에 없다면 스레드는 큐에 태스크가 들어올 때까지 기다린다.

3️⃣ **worker**
- 모든 worker 스레드는 그룹 지어져 ThreadGroup에 속해 스레드 관리가 더 쉬워진다.
- 그룹에 인터럽트 신호를 보내면 그룹에 속한 모든 스레드가 종료되어 그룹 내 모든 스레드를 종료하기 위해 개별 스레드마다 직접적으로 인터럽트를 보낼 필요가 없다.

#### 구현 세부 내용
1. 모든 스레드가 **running 플래그**의 변화를 인지할 수 있도록 `volatile` 키워드를 사용
  - 워커가 셧다운 신호를 받지 못하는 일이 발생하지 않게 보장
2. `LinkedBlockingDeque`는 **스레드 안정성이 보장**되는 방식으로 태스크의 제출과 조회를 처리할 수 있다.
  - 큐 크기에 한도를 정해 태스크 누적에 의해 메모리가 고갈되는 문제를 예방할 수 있다.
3. **워커**는 생성 후 바로 시작되어 태스크가 제출되자마자 가져와 실행할 수 있다.
4. `put()` 메서드는 큐가 태스크로 가득 차 있으면 **블로킹**한다. 
  - 풀에 과부하가 걸리지 않게 막아주는 `backpressure` 역할
5. 스레드 그룹에 인터럽트 신호를 보내면 그룹 내 모든 스레드에 동시에 인터럽트 신호가 전달되어 안전하게 셧다운 처리가 진행된다.
6. 안전한 셧다운은 처리 중인 태스크가 완료될 때까지 기다려 풀을 폐쇄하더라도 작업 일부가 유실되지 않도록 보장한다.
### 이그제큐터 프레임워크
이그제큐터 프레임워크에는 공통 인터페이스인 `ExecutorService`를 구현하는 다양한 구현체가 있다.
팩토리 클래스인 `Executors`를 사용해 원하는 구현체를 골라 사용할 수 있다.

> `Executors` 클래스의 여러 팩토리 메서드는 각자의 목적에 맞게 설정된 인자를 `ThreadPoolExecutor` 생성자에 전달해 `ThreadPoolExecutor` 인스턴스를 생성하고 반환한다.

```java
// 고정 크기를 가진 10개의 스레드 풀이 필요한 경우
ExecutorService service = Executors.newFixedThreadPool(10);
```


#### ThreadPoolExecutor 
```java
public ThreadPoolExecutor(int corePoolSize,
                          int maximumPoolSize,
                          long keepAliveTime,
                          TimeUnit unit,
                          BlockingQueue<Runnable> workQueue,
                          ThreadFactory threadFactory,
                          RejectedExecutionHandler handler) {
        //...
}
```


| **생성자에 사용되는 파라미터** | **설명** |
| ---- | ---- |
| **corePoolSize** | 스레드가 아무 태스크도 처리하지 않고 대기 중일 때도 유지되는 스레드의 최소 개수 |
| **maximumPoolSize** | 풀에 허용되는 스레드의 최대 개수 |
| **keepAliveTime** | 스레드 수가 `corePoolSize`보다 많을 때 초과된 유휴 스레드가 새로운 태스크를 기다리며 살아있는 최대 시간. 시간이 지나면 초과 스레드는 종료된다. |
| **unit** | `keepAliveTime`의 시간 단위(예: 나노초, 밀리초, 초 등의 단위) |
| **workQueue** | 태스크가 실행되기 전에 보관되는 큐 |
| **threadFactory** | Executor가 새로운 스레드를 생성할 때 사용하는 팩토리 |
| **handler** | 스레드 수와 큐 용량이 모두 한계에 도달해 실행이 막힌 상황을 처리하는 핸들러 |

### 스레드 풀을 설정 시 고려해야 하는 점
스레드 풀을 설정할 때는 **두 가지 중요한 크기**를 고려해야 한다.
- **코어 풀 크기(corePoolSize)**
- **최대 풀 크기**
#### 1️⃣ 코어 풀 크기
스레드가 아무 일도 하지 않고 대기 중이라도 항상 풀안에 유지되는 최소한의 스레드 개수
스레드를 **새로 생성하는 데 드는 비용이 크기 때문에** 이런 코어 스레드 집합을 유지하는 것이 매우 중요하다.
- 너무 많은 스레드를 만들어 대기 상태에 있는 스레드가 많다면 CPU 사이클과 메모리 같은 자원이 불필요하게 낭비되기 때문에 적절하게 최소 스레드 수를 설정하는 것이 중요하다.
#### 2️⃣ 최대 풀 크기
스레드 풀이 가질 수 있는 활성 스레드의 상한선을 정의한다.
이를 통해 스레드 풀은 태스크가 급격히 늘어날 때 더 많은 스레드를 만들어 부하를 흡수할 수 있다.

코어 풀 크기와 최대 풀 크기를 최적으로 설정하는 것은 성능 튜닝의 핵심으로 애플리케이션이 실행되는 <span class="text-blue">작업 부하</span>의 특성이나 <span class="text-blue">하드웨어</span> 환경에 따라 크게 달라진다.

| **상황** | **설정** | **이유** |
| ---- | ---- | ---- |
| **CPU 집중적인 작업이 많은 상황** | 스레드 수는 가능한** CPU 코어 수와 맞추는 것**이 좋다 | 스레드를 과도하게 늘릴 시 컨텍스트 전환과 동기화 오버헤드가 증가하여 오히려 성능이 떨어질 수 있다. |
| **I/O 집중적인 작업이 많은 상황(데이터 베이스 호출)** | **CPU 코어 수보다 많은** 스레드를 두는 것이 좋다. | 입출력을 기다리는 동안 다른 태스크를 처리할 수 있기 때문이다. |

### 이그제큐터 프레임워크에서 JDK가 제공하는 다양한 스레드 풀 타입
#### ✅ FixedThreadPool
**스레드의 개수가 고정되어 있는 풀**, 모든 스레드가 바쁜 상태일 때 새로 제출된 태스크는 큐에 쌓여 기다리고 태스크 처리를 마친 스레드가 큐에 있는 태스크를 가져와 실행한다.
- **활성 스레드의 개수를 일정하게 유지하여 작업 부하의 규모를 예측할 수 있을 때 유용하다.**
- 동시에 일정한 수의 스레드가 작업을 수행해야 하는 애플리케이션에 적합
- ex: 서버 애플리케이션처럼 예측 가능한 수준의 동시성을 유지해야 하는 경우
- ex: 자원이나 데이터베이스 연결 수가 제한된 환경에서 고정된 스레드 풀을 사용하여 시스템 과부하를 막을 수 있다.

#### ✅ CachedThreadPool
이전에 생성된 스레드가 유휴 상태이면 재사용하고 새 스레드가 필요하면 동적으로 생성한다.
**스레드 풀이 태스크의 양에 따라 자동으로 확장 또는 축소될 수 있어 작업 부하의 크기가 자주 변할 때 유용하다.**
- **짧은 시간동안 수행되는 태스크를 굉장히 많이 처리해야 하거나 태스크가 불규칙하게 몰리는 상황에 적합하다.**
- 하지만 **실행 시간이 긴 태스크를 처리해야 하는 상황**에서는 너무 많은 스레드가 동시에 생성되어 시스템 자원이 과도하게 사용될 수 있기 때문에 적합하지 않다.

#### ✅ SingleThreadExecutor
**태스크를 단 하나의 스레드만 사용해서 순차적으로 실행하는 것을 보장한다.**
- 태스크가 동시에 실행되면 안되거나 반드시 순차적으로 실행되어야 할 때 적합하다.
- `SingleThreadExecutor` 은** 한 번에 하나의 태스크만 접근을 허용하는 공유 자원을 다룰 때 자주 사용되어 **<span class="text-red">**경쟁 조건**</span>**이 발생하지 않도록 보장한다.**

#### ✅ ScheduledThreadPoolExecutor
**지연된 시간 후 실행되거나 일정한 간격으로 반복 실행되도록 태스크를 스케줄링한다.**
- 알림 전송, 시스템 상태 점검, 주기적인 유지보수 작업과 같이 정기적으로 반복 실행해야 하는 태스크에 적합하다.
- `ScheduledThreadPoolExecutor`는 수동 개입 없이도 태스크가 지정된 빈도로 실행되도록 보장한다.

#### ✅ WorkStealingPool
**`Work-Stealing`**** 알고리즘을 사용해 스레드 사이의 작업 부하를 동적으로 분산시켜 균형을 맞추는 스레드 풀,** **사용가능한 프로세서를 최대한 효율적으로 활용하도록 설계돼 있다.**
- 작고 독립적인 태스크가 많은 상황에서 CPU 코어의 사용률을 극도로 높여야 할 때 적합하다.

### Callable과 Future: 태스크 결과 처리
**Runnable**을 사용해 태스크를 제출하도록 했는데 **Runnable은 **<u>**반환값이 없을 때만**</u>** 사용이 가능하다.**

> 🤔 그렇다면 태스크 실행 완료 후 반환값을 받아 사용해야 할 때는 어떻게 할까?
> - `Callable`과 `Future`를 사용하면 된다.


#### 🧑‍💻 Callable 인터페이스
```java
public interface Callable {
    V call() throws Exception;
}
```

Runnable과 마찬가지로 함수형 인터페이스로 값을 반환하고 예외를 던질 수 있다는 점만 Runnable과 다르다.
```java
public class CallableExample {

    static final Map<Integer, Long> cache = new ConcurrentHashMap<>(
            Map.of(0, 0L, 1, 1L) // ①
    );

    public static void main(String[] args) throws Exception {
        try (ExecutorService threadPool = Executors.newCachedThreadPool()) {
            Future<Long> fibonacciNumber = threadPool.submit(new Callable<Long>() { // ②
                @Override
                public Long call() throws Exception {
                    return fibonacci(50); // ③
                }
            });
        }
    }

    private static Long fibonacci(int n) {
        if (cache.containsKey(n)) {
            return cache.get(n);
        } else {
            long result = fibonacci(n - 1) + fibonacci(n - 2);
            cache.put(n, result); // ④
            return result;
        }
    }
}
```

위 코드에서 `submit` 메서드는 `Callble<Long>` **타입의 태스크를** 받아 `Future<Long>`을 반환한다.
- 이 **Future** 객체는 비동기적으로 수행되는 계산 결과값을 나중에 받아올 수 있게 해준다.
Callable의 `call()` 메서드는 값을 반환한다.
- 예제에서는 피보나치 수열의 50번째 값을 반환
- Runnable의 `run()` 메서드는 값을 반환하지 않음.

#### 🧑‍💻 Future 인터페이스
```java
public interface Future<V> {
    boolean cancel(boolean mayInterruptIfRunning);
    boolean isCancelled();
    boolean isDone();
    V get() throws InterruptedException, ExecutionException;
    V get(long timeout, TimeUnit unit) 
        throws InterruptedException, ExecutionException, TimeoutException;
}
```

**Callable을 사용해 태스크를 제출하면 태스크가 아직 실행 완료되지 않았더라도 Future가 즉시 반환된다.**
- `isDone()` 메서드를 통해 태스크가 완료되었는지 확인할 수 있다.
- `get()` 메서드로 결과를 가져올 수 있다.
:::callout ⚠️ **get() 메서드는 블로킹 연산**
:::callout - `get()` 메서드를 호출한 스레드는 결과를 얻을 수 있을 때까지 블로킹

## ForkJoinPool
`자바 7`부터는 일반적인 **ThreadPoolExecutor**외에 <span class="text-red">**ForkJoinPool**</span>이라는 특별한 스레드 풀이 추가되었다.
**ForkJoinPool**은 **ExecutorService** 인터페이스를 구현하지만 **ThreadPoolExecutor**와는 핵심 설계와 작동원리에서 큰 차이가 있다.
### ForkJoinPool과 **ThreadPoolExecutor 차이**
1️⃣ **ForkJoinPool**은 <span class="text-red">**Work-Stealing**</span> 메커니즘을 사용한다.
각 스레드는 자신의 태스크를 가지고 있고 대기중인 스레드는 다른 스레드의 큐 뒤쪽에서 태스크를 훔쳐와 실행할 수 있다. 
**전통적인 스레드 풀**은 공유 큐에 태스크를 넣고 대기중인 유휴 스레드가 공유 큐에 있는 태스크를 가져가는데 이렇게 하면 스레드가 태스크를 가져가기 위해 경쟁하게 되고 오버헤드가 발생할 수 있다.

2️⃣ **ForkJoinPool은 작업량과 프로세서의 가용성에 따라 능동적으로 스레드 수를 조정할 수 있다.**
효율성을 유지하기 위해 스레드를 동적으로 추가하거나 일시 중단하거나 다시 활성화할 수 있다.
**전통적인 스레드 풀**은 보통 고정된 개수의 스레드를 유지하며 스레드 생성이나 종료가 상대적으로 덜 유동적이다.

3️⃣ **ForkJoinPool은 동기화가 거의 필요 없다.**
**ForkJoinPool**은 Work-Stealing 알고리즘 덕에 각 스레드가 더 자율적으로 동작할 수 있어 경쟁과 오버헤드가 줄고 성능이 크게 향상된다.
**전통적인 스레드 풀**은 락이나 조건 변수 같은 명시적 동기화 기법에 의존하여 태스크 접근이나 스레드 동기화를 관리한다.

4️⃣** ForkJoinPool은 큰 태스크를 더 작은 서브태스크로 재귀적으로 분할할 수 있도록 설계되어 있다.**
태스크가 트리 형태로 나눠지며 만약 한 태스크가 자신의 서브태스크 완료를 기다려야 한다면 실행을 잠시 보류하고 다른 태스크를 실행할 수 있다.
**전통적인 스레드 풀**은 태스크 사이의 의존성을 인식하지 못한다. 즉, 어떤 서브 태스크가 다른 선행 태스크의 결과를 기다려야 할 경우 그 서브태스크를 실행중인 스레드는 다른 스레드가 선행 태스크를 완료할 때까지 아무것도 하지 못하고 멈춰서 기다리게 되는 문제가 있다.

### 가상 스레드에서 ForkJoinPool을 사용하는 이유
일반 스레드 풀과 마찬가지로 <span class="text-red">**ForkJoinPool**</span>도 미리 **정의된 수의 워커 스레드**로 구성되어 있다.

각 **워커 스레드**는 `데크(deque)`로 만들어진 **자신의 작업 큐(work queue)**를 가지고 있다.
- 워커 스레드는 태스크를 `LIFO` 방식으로 넣고 꺼낸다.
- 다른 스레드는 반대 쪽 끝에서 `FIFO` 방식으로 태스크를 훔쳐올 수 있다.
워커 스레드가 **분할 후 정복 알고리즘의 일부인 태스크 분할**을 수행하면 분할 결과 새로 생성된 서브 태스크들을 자신의 작업 큐에 직접 추가한다.
워커 스레드에 속하지 않는 **외부 스레드**는 태스크를 제출하고 이 태스크들은 **제출 큐**를 통해 시스템에 들어온다.
**제출 큐**는 작업 큐와 비슷하지만 **ForkJoinPool**이 직접 관리하고 **동기화를 위해** 명시적으로 락같은 메커니즘을 사용하는 대신 `비교 후 교체(compare-and-swap, CAS)`같은 **원자적 연산**을 이용해 동시 접근을 효율적으로 관리한다.

#### 🔍 워커 스레드와 도둑 스레드가 같은 큐에 접근?!?
워커 스레드와 도둑 스레드가 동시에 같은 큐에 접근하려고 하면 경합이 발생하는 것처럼 보일 수 있다.
**ForkJoinPool**은 경합을 최소화하기 위해 두 스레드가 큐의 서로 다른 끝에서 태스크를 가져가도록 설계되어 있다.
- **작업 큐를 소유한 워커 스레드**: `pop` 메서드를 사용해 앞쪽에서 태스크를 가져온다.(`LIFO`)
- **도둑 스레드**: 뒤쪽에서 태스크를 가져간다. (`FIFO`)

#### 🔍 큐에 제출된지 오래된 태스크는 불공평하게 계속 뒤로 밀리나?
`push` 메서드는 새로운 태스크를 큐의 맨 앞에 추가한다.
워커 스레드는 태스크를 가장 나중에 추가된 태스크(큐의 맨 앞에 있는 태스크)를 가져간다. (`pop`)
CPU에는 최근 접근한 데이터를 저장하는 캐시가 있다. 최신 태스크를 우선 처리하면 그 태스크에 필요한 데이터가 이미 캐시에 들어 있을 확률이 높아진다. 
따라서 **LIFO 방식을 사용하면 캐시 미스가 줄어들어 성능이 향상된다.**
> 🤔 그러면 큐에 제출된 지 오래된 태스크는 불공평하게 계속 뒤로 밀리는거 아닌가?!?
> - 도둑 스레드의 역할이다.
> - 도둑 스레드는 큐 맨 뒤에 있는 가장 오래된 태스크를 가져와 실행한다.


#### ForkJoinPool 주의점
`ForkJoinPool`은 이용 가능한 모든 프로세서 코어를 활용해 <span class="text-red">**병렬 처리 속도 향상을 목표**</span>로 설계되어 <span class="text-red">**분할 후 정복 알고리즘을 이용해 그 목표를 달성**</span>할 수 있었다.
:::callout 💡 **기본 아이디어**
:::callout 큰 작업을 여러 개의 작은 작업으로 나누고 그 작은 작업들이 서로 독립적으로 계산될 수 있을 만큼 단순해질 때까지 계속 분할하는 것.

:::callout ⚠️ **주의 해야하는 점**
:::callout `ForkJoinPool`은 스레드 풀의 관리자일 뿐 작업을 작은 단위로 나누는 일을 직접 수행하지는 않는다.
:::callout - **작업을 어떻게 나눌지는 개발자의 책임이다.**
:::callout - `RecursiveTask`, `RecursiveAction` 같은 클래스들을 상속받아 자신의 태스크 클래스를 만들고 그 안에 분할 및 처리 로직을 구현한다.
하나의 태스크가 여러 개의 서브태스크로 분할되면 부모 태스크는 이 서브태스크들이 모두 실행 완료될 때까지 기다린다.
- 모든 서브태스크 실행이 완료되면 부모 태스크는 조인 과정을 통해 모든 서브태스크 결과를 하나로 합친다.
> 🚀 **이때 부모 태스크가 서브태스크들의 완료를 기다리는 동안 부모 태스크를 실행하던 워커 스레드는 놀지 않고 자신의 큐에 남은 다른 태스크를 처리하거나 다른 태스크의 워커 큐에서 태스크를 훔쳐와서 계속 일을 한다.**


### 비동기모드 ForkJoinPool
시간이 지나면서 `ForkJoinPool`의 사용범위는 점점 확장되었다.
특히 <u>**이벤트 스타일**</u>의 태스크처럼 서로 독립적으로 실행되며 결과를 합치기 위해 다른 태스크의 실행 완료를 기다릴 필요가 없는 태스크에도 사용되기 시작했다.
이런 경우 생성자 인자를 적절히 지정하여 **비동기 모드로 ForkJoinPool**을 생성할 수 있다.
- 비동기 모드에서는 **ForkJoinPool**이 `LIFO`대신 `FIFO` 정책으로 전환된다.
- 이 방식은 분할 정복처럼 재귀적인 모델이 아닌 이벤트나 외부 트리거처럼 제출된 순서대로 처리되는 것이 중요한 태스크들에 더 적합하다.
```java
ForkJoinPool forkJoinPool = new ForkJoinPool(4,
        ForkJoinPool.defaultForkJoinWorkerThreadFactory, null, true));
```

생성자의 마지막 파라미터인 `asyncMode` 값을 `true`로 설정하여 비동기 모드로 생성

:::callout ℹ️ **가상 스레드 스케줄링 - ForkJoinPool**
:::callout 가상 스레드를 사용할 때 `비동기 모드`가 사용되며 **ForkJoinPool 안에 있는 스레드가 모든 가상 스레드의 실행과 스케줄링을 담당한다.**
## 컨티뉴에이션(Continuation)
> 프로그래밍에서 **컨티뉴에이션**이란 프로그램의 현재 실행 상태를 저장하고 중단한 다음 **나중에 중단 지점에서 다시 이어서 실행할 수 있는 기능**을 의미한다.

컨티뉴에이션은 <span class="text-brown">**메서드**</span> 수준에서도 사용할 수 있다.
- 메서드가 특정 지점에서 **일지 정지 **됐다 다시 호출될 때 일시 정지된 지점부터 정확히 이어서 실행할 수 있으며 **실행 상태(변숫값 등)**도 일시 정지됐을 때 상태 그대로 유지된다.

☕** 자바 21부터 내부 패키지를 통해 컨티뉴에이션을 사용할 수 있다.**
### 컨티뉴에이션 동작 방식
```java
import jdk.internal.vm.Continuation;
import jdk.internal.vm.ContinuationScope;

// --add-exports java.base/jdk.internal.vm=ALL-UNNAMED
public class ContinuationExample {

    public static void main(String[] args) {
        ContinuationScope scope = new ContinuationScope("main");

        Continuation continuation = new Continuation(scope, () -> {
            System.out.println("Hello from continuation");
            Continuation.yield(scope);
            System.out.println("Hello again from continuation");
            Continuation.yield(scope);
            System.out.println("Done from continuation");
        });

        System.out.println("Before starting continuation");
        continuation.run();
        System.out.println("After starting continuation");
        continuation.run();
        System.out.println("After starting continuation again");
        continuation.run();
    }
}
```

`jdk.internal.vm` 패키지의 `Continuation` 과 `ContinuationScope` 클래스를 사용
이는 공개된 자바 API가 아니기 때문에 실행 시 다음과 같은 JVM 옵션을 지정해 줘야만 사용할 수 있다.
```java
--add-exports java.base/jdk.internal.vm=ALL-UNNAMED
```

`ContinuationScope` 은 컨티뉴에이션에 유일한 전용 스코프를 생성하고 `Continuation` 은 본질적으로 일시 정지되고 재개될 수 있는 코드 조각이다.

#### 👀 **예제 코드의 실행 결과**
```plain text
Before starting continuation
Hello from continuation
After starting continuation
Hello again from continuation
After starting continuation again
Done from continuation
```

1. `continuation.run()` 이 호출되면 **컨티뉴에이션 실행이 시작**되고 “Hello from continuation”을 출력한다.
2. 이후 `Continuation.yield(scope)`를 호출하면서 **일시 정지**되어 **제어권이 main 메서드**로 넘어간다.
3. main 메서드에서 **“After starting continuation”**을 출력하고 `continuation.run()`을 호출하면 다시 컨티뉴에이션 안으로 진입하여 일시 정지된 지점에서 실행이 이어져 **“Hello again from continuation”**을 출력하고 `yield`를 호출하면서 **제어권이 main 메서드로 넘어간다**.
4. main 메서드에서 **“After starting continuation again”**를 출력하고 `continuation.run()`을 호출하면 컨티뉴에이션 안의 `yield` 지점으로 진입하여 **“Done from continuation”**을 출력한 후 남아 있는 코드가 없으므로 컨티뉴에이션이 종료된다.

:::callout ℹ️ **가상 스레드에서 컨티뉴에이션 동작**
:::callout 메서드가 가상 스레드에서 실행되어** 블로킹 I/O 연산**을 만나면 컨티뉴에이션이 `Continuation.yield(scope)` 메서드를 호출해 가상 스레드를 일시 정지시켜 캐리어 스레드로부터 <span class="text-red">**언마운트**</span> 되어 일시 정지된 지점부터 이어서 실행된다.

> 🤔 **Continuation은 이런 일을 어떻게 해내는 걸까?**

프로그램에서 실행되는 모든 것은 스레드(특히 플랫폼 스레드)에 의해 실행된다.
각 스레드는 스택과 스택 프레임을 가지고 있으며 코드를 실행하면 메서드를 호출할 때마다 스택 아래 방향으로 **스택 프레임이 추가**된다.
1️⃣ 실행 중인 플랫폼 스레드는 `c.run()`을 호출하면서 <span class="text-blue">**컨티뉴에이션 c**</span>를 시작한다.
2️⃣ 이후 **메서드 a**를 호출하고 메서드 a는** 메서드 b**를 호출하고 메서드 b는 **메서드 c**를 호출한다.
3️⃣ **메서드 c**는 <span class="text-blue">**컨티뉴에이션 c**</span>의 `yield()` 메서드를 호출한다.
4️⃣ 컨티뉴에이션은 실행을 일시정지하고 반환한다.
5️⃣ <span class="text-blue">**컨티뉴에이션 c**</span>는 나중에 다른 플랫폼 스레드에 의해 실행되겠지만 지금은 반환해야 한다.
- 스레드는 작업을 완료하게 되지만 `c.run()`은 아직 작업이 끝난 것이 아닌 **일시 정지**된 것이다.
6️⃣ 일시 정지가 되면서 `c.run()`이 실제로 하는 일은 a에서 `c.yield()`까지의 모든 프레임을 컨티뉴에이션 객체 안에 저장하는 일이다.
7️⃣ 이후 <span class="text-blue">**컨티뉴에이션 c**</span>가 다시 <span class="text-red">**마운트**</span>된다면 반대 방향으로 이동된다.
- 모든 프레임이 다시 스레드의 스택으로 복사되고 c.yield()가 반환하면 일시 정지됐던 지점에서 다시 실행이 이어진다.
> ✔️ **정리**
> 가상 스레드(컨티뉴에이션)가 언마운트될 때 스레드의 스택 프레임이 다른 곳으로 복사되고 다시 마운트되면 복사된 스택 프레임이 다시 스레드의 스택으로 복사된다.

:::callout ⚠️ **이런 복사 작업은 비용이 많이 든다.**
:::callout 컨티뉴에이션이 일시 정지될 때마다 스택 전체가 복사되기 때문에 호출 스택이 아주 깊을 때는 비용이 더 많이 든다.

이 문제를 해결하기 위해서 **JVM**에는 지능적인** 지연 복사 메커니즘**이 구현되어 있다.
:::callout 💡**지연복사(lazy copy) 메커니즘**
:::callout - 불필요한 스택 프레임 복사를 피하기 위한 최적화 기법
:::callout - 처음 일시 중단될 때는 전체 스택 프레임을 컨티뉴에이션 객체로 복사
:::callout - 이 컨티뉴에이션 객체가 다시 재개될 때는 컨티뉴에이션에 있던 스택 프레임 전체를 스레드의 스택으로 다시 복사하지 않고 한 두개의 프레임만 복사한다.

`c.yield()`가 반환되면 c로 돌아가고 c가 반환되면 b로 돌아가야 한다.
- 하지만 스레드 스택에는 b가 없다.
- 대신 <u>**반환 배리어(return barrier)**</u>라는 매커니즘이 적용되어 있다.
> **반환 배리어**
> - 함수의 반환 지점에 주입된 작은 코드 조각으로 함수가 반환될 때 필요한 프레임을 컨티뉴에이션 스택으로부터 복사해 와야 하는지 확인한다.

b가 스레드 스택에 없기 때문에 b를 실행하려면 컨티뉴에이션으로부터 복사해 와야 하므로 c는 b로 돌아가지 않고 b를 컨티뉴에이션으로부터 복사해서 스레드 스택으로 가져오는 VM 코드가 있는 지점으로 돌아간다. 그러고 나서 b가 스레드 스택에 복사되면 b를 실행한다.

### 가상 스레드 직접 만들어보기
가상 스레드가 컨티뉴에이션을 어떻게 사용하는지 자세히 알아보기 위해 스레드와 비슷한 추상화 구조를 직접 만들어 확인해보자. (<u>NanoThread</u>)
#### 🤖 NanoThread 클래스
`Continuation API`를 바탕으로 만들어진 커스텀 가상 스레드
```java
public class NanoThread {

    public static final NanoThreadScheduler NANO_THREAD_SCHEDULER
                            = new NanoThreadScheduler();
    private static final AtomicInteger COUNTER
                            = new AtomicInteger(1);
    public static final ContinuationScope SCOPE
                            = new ContinuationScope("nanoThreadScope");

    private final Continuation continuation;
    private final int nid;

    private NanoThread(Runnable runnable) {
        this.nid = COUNTER.getAndIncrement();
        this.continuation = new Continuation(SCOPE, runnable);
    }

    public static void start(Runnable runnable) {
        var nanoThread = new NanoThread(runnable);
        NANO_THREAD_SCHEDULER.schedule(nanoThread);
    }

    public void run() {
        continuation.run();
    }

    public static NanoThread currentVThread() {
        return NanoThreadScheduler.CURRENT_NANO_THREAD.get();
    }

    @Override
    public String toString() {
        return "NanoThread-" + nid + "-" + Thread.currentThread().getName();
    }
}
```

1️⃣** run 메서드**
단순히 `컨티뉴에이션의 run()` 메서드를 호출하여 `Runnable` 로직을 실행하고 `Continuation.yield()`가 호출되면 가상스레드처럼 **NanoThread** 실행이 일시 정지되고 나중에 일시 정지된 지점부터 이어서 실행할 수 있다.

2️⃣ **start 메서드**
**NanoThread**를 생성한 후에 실행할 수 있도록 **스케줄러**에 전달한다.

#### ⏱️ NanoThread 스케줄러
**NanoThread** 인스턴스를 관리하고 실행하는데 핵심 역할을 담당한다.
- `WorkStealingPool`을 사용해 태스크를 스케줄링
- `ScheduledExecutorService`를 사용해 스레드 실행 일시 정지와 yield를 유발하는 I/O 집중 연산을 시뮬레이션
```java
public class NanoThreadScheduler {

    public static final ThreadLocal<NanoThread> CURRENT_NANO_THREAD = new ThreadLocal<>();
    public static final ScheduledExecutorService IO_EVENT_SCHEDULER = Executors.newSingleThreadScheduledExecutor();
    private final ExecutorService workStealingPool = Executors.newWorkStealingPool(2);

    public void schedule(NanoThread nanoThread) {
        workStealingPool.submit(() -> {
            CURRENT_NANO_THREAD.set(nanoThread);
            nanoThread.run();
            CURRENT_NANO_THREAD.remove();
        });
    }
}
```

1️⃣ `NanoThreadScheduler` 
**ThreadLocal** 변수인 `CURRENT_NANO_THREAD`를 사용해 워커 스레드에서 어떤 **NanoThread** 인스턴스가 실행중인지 추적한다.

2️⃣ `IO_EVENT_SCHEDULER` 
가상 스레드의 동작을 이해하는 데 꼭 필요한 I/O 연산을 시뮬레이션 한다.

3️⃣ `schedule()` 메서드 
**NanoThread** 인스턴스를 실제 태스크를 실행하는 `WorkStealingPool`에 제출한다.
- 위 예제에서 `WorkStealingPool` 는 2개의 워커 스레드를 가지고 있고 큐에 있는 태스크들을 계속 체크한다.

#### 📁 파일 전송 시뮬레이션
<u>**I/O 집중적 연산**</u>인 파일 전송을 시뮬레이션한다.
- `IO_EVENT_SCHEDULER` 가 지연을 발생시켜 파일 전송을 시뮬레이션
```java
public class FileOperation {

    private final Random random = new Random();

    public void transfer(String filePath) {
        System.out.println("Start transferring file: " + filePath);
        NanoThread nanoThread = NanoThread.currentVThread();
        IO_EVENT_SCHEDULER.schedule(() ->
                NANO_THREAD_SCHEDULER.schedule(nanoThread),
                random.nextInt(1000), TimeUnit.MILLISECONDS);
        CURRENT_NANO_THREAD.remove();
        Continuation.yield(SCOPE);
        System.out.println("Transfer completed for file: " + filePath);
    }
}
```

`IO_EVENT_SCHEDULER` 를 사용해 실제 I/O 연산 완료까지의 지연 시간 후에 **NanoThread** 실행을 재개하도록 스케줄링한다.
바로 `Continuation.yield(SCOPE)`가 호출되면 실행이 일시 정지되고 캐리어 스레드는 다른 **NanoThread**를 실행할 수 있게 된다.
지연시간이 지나면 파일 전송이 완료됐다 간주하여 `NANO_THREAD_SCHEDULER`에 **nanoThread**가 제출되어 `run()` 메서드가 호추로디고 일시 정지된 지점부터 이어서 실행한다.

#### 🧑‍💻 예제 실행
**NanoThread** 시스템을 통한 **파일 동시 전송 시나리오**
```java
public class NanoThreadDemo {

  public static void main(String[] args) throws Exception {
    FileOperation fileOperation = new FileOperation();
    for (int i = 0; i < 4; i++) {
      int finalI = i;
      NanoThread.start(() -> {
        System.out.println("Transfer: "
            + "File_" + finalI + " Running in VThread: "
            + NanoThread.currentVThread());

        fileOperation.transfer("File_" + finalI);

        System.out.println("Transfer: " + "File_" + finalI
          + " Completed in VThread: " + NanoThread.currentVThread());
      });
    }

    // 스케줄러가 모든 nanoThread 실행을 완료할 때까지 1분동안 대기
    Thread.sleep(Duration.ofMinutes(1));
  }
}
```

### 가상 스레드와 I/O 폴링
실제로 가상 스레드가 **I/O** 연산을 시작하면 **Continuation**이 일시정지되고 내부적으로는 스레드가 `LockSupport.park(`) 메서드를 실행한다.
```java
public static void park() {
    if (Thread.currentThread().isVirtual()) {
        VirtualThreads.park();
    } else {
        U.park(false, 0L);
    }
}
```

`VirtualThreads.park()` 메서드안에서는 몇 단계를 거친 후 `yieldContinuation()` 메서드를 호출하고 <u>**가상 스레드는 캐리어 스레드로부터 분리된다.**</u>
