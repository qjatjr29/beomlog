자바에서의 동시성 진화를 살펴보는 것으로 시작한다.
**플랫폼 스레드**에서 출발해 **Executor 프레임워크**, **Fork/Join**, **CompletableFuture**로 이어지는 흐름을 추적하고 리액티브 프로그래밍을 간략히 소개한다.
그동안의 트레이드오프를 살펴보고 `프로젝트 룸(Project Loom)`이 왜 근본적으로 다른 접근 방식을 선택했는지 이해한다.

## 자바 스레드의 간략한 역사
자바는 <span class="text-red">**동시성**</span>을 염두에 두고 설계됐으며 **멀티스레딩**을 언어 수준에서 지원하는 최초의 프로그래밍 언어 중 하나이다.
자바의 동시성은 기본적인 **동기화(synchronization)**와 **스레드 관리**로 시작했다.
`자바 5`에서 `java.util.concurrent `패키지가 도입되며 **Executor 프레임워크, 락, 동시성 지원 컬렉션**같은 중요한 새 기능이 추가됐다.
`자바 7`에서 **포크/조인(Fork/Join) 프레임워크**가 도입되면서 멀티코어 프로세서에서의 동시성 처리 능력을 제대로 활용할 수 있게 됐다.
최근에는 <span class="text-blue">**프로젝트 룸(Project Loom)**</span>에서 사용자 모드 경량 스레드와 구조적 동시성을 도입하면서 전통적인 스레드 기반 동시성의 한계와 복잡성을 해결하고 더 높은 효율로 쉽게 동시성을 활용하는 프로그램을 작성할 수 있게 됐다.
### 자바는 스레드로 만들어진다.
자바에서 스레드는 <u>**가장 작은 실행 단위**</u>다.
- 스레드: 하나의 프로그램 안에서 독립적인 실행 경로

#### 자바의 스레드 모델은 스레드 스케줄링 및 실행과 관련해 운영체제에 의존한다.
**운영체제**는 높은 실행 효율을 보장하기 위해 각 스레드에 CPU 시간을 할당하고 스레드 사이의 상태 전환도 관리를 한다.
스레드를 다수의 CPU에 분배하면 **병렬성**을 획득할 수 있고 동시성을 활용하는 애플리케이션의 성능을 높일 수 있다.

자바 프로그램에서 더 많은 스레드를 사용하면 실제로 더 많은 `실행환경(execution environment)`이 생성되며 이를 통해 여러 연산을 **동시에** 수행할 수 있다.
- 높은 수준의 병렬성과 동시성을 필요로 하는 애플리케이션에서 유용하다.
🚀 **애플리케이션은 다수의 스레드를 활용해 여러 가지 작업을 동시에 수행할 수 있고 처리량과 응답 속도를 높일 수 있다.**

#### 자바 생태계는 스레드 개념을 바탕으로 구현되고 작동한다.
자바의 메모리 관리를 담당하는 가비지 컬렉션, 단순한 “Hello World”를 출력하는 프로그램이든 모두 스레드를 바탕으로 한다.
👀 **따라서 스레드는 자바에서 필수 요소이다.**

### 스레드
자바 스레드는 자바 플래솜의 모든 레이어에 필수 불가결한 요소로 통합되어 있으며 단순히 코드를 실행하는 수준을 넘어 여러 가지 관점에서 핵심적인 역할을 한다.
- ex: 예외 처리, 디버깅, 프로파일링

#### 예외와 스레드
자바에서 모든 스레드는 각자의 고유한 호출 스택(call stack)을 가진다.
이 호출 스택에는 스레드가 생성되어 소멸되기 전까지의 모든 메서드 호출 내역이 기록되고 예외가 발생하면 개발자는 스레드 호출 스택에서 그 예외 발생 지점까지의 모든 메서드 호출 내역을 순서대로 확인하며 문제의 시작점을 파악할 수 있다.
#### 디버거와 스레드
자바 디버거는 실행 중인 애플리케이션에서 멈춰야 하는 지점을 어떻게 알아내는 걸까?
디버거는 자기 자신을 애플리케이션의 여러 스레드에 연결한다. 이를 통해 어떤 스레드를 조사할지 선택할 수 있고 스레드의 상태를 변경할 수도 있다.
#### 프로파일러와 스레드
프로파일링 도구는 애플리케이션이 어떻게 동작하고 있는지 자세하게 보여주기 위해 스레드에 의존한다.
스레드 정보를 사용해 개발자가 성능 저하 지점을 짚어낼 수 있게 해주고 멀티스레드 코드에서 발생하는 미묘한 타이밍 문제를 찾아 해결할 수 있게 해주며 애플리케이션 성능 개선 방법을 찾을 수 있도록 도와준다.
## 자바 1.0 스레드의 기원
`자바 1.0`은 스레드 지원 기능을 내장했다. `Thread` 클래스를 상속받거나 `Runnable` 인터페이스를 구현해 스레드를 생성할 수 있었다.
```java
public class ThreadCreationDemo {

    public static void main(String[] args) {
        // Extending Thread class
        Thread t1 = new ThreadByExtension("Worker-1"); // 1️⃣
        t1.start();

        // Implementing Runnable (classic)
        Thread t2 = new Thread(new RunnableImplementation(), "Worker-2"); // 2️⃣ 
        t2.start();

        // Anonymous inner class (Java 1.1)
        Thread t3 = new Thread(new Runnable() { // 3️⃣ 익명 내부 클래스를 사용하면 2번 방식을 더 간결하게 작성가능
            @Override
            public void run() {
                System.out.println("Anonymous: " + Thread.currentThread().getName());
            }
        }, "Worker-3");
        t3.start();

        // Lambda expression (Java 8)
        Thread t4 = new Thread(() -> // 4️⃣ 람다식을 사용하면 함수형 인터페이스 Runnable을 간단하게 구현 가능
        System.out.println("Lambda: " + Thread.currentThread().getName()), "Worker-4");
        t4.start();
    }
}

class ThreadByExtension extends Thread {
    public ThreadByExtension(String name) {
        super(name);
    }

    @Override
    public void run() {
        System.out.println("Extended Thread: " + getName());
    }
}

class RunnableImplementation implements Runnable {
    @Override
    public void run() {
        System.out.println("Runnable: " + Thread.currentThread().getName());
    }
}
```

1️⃣ **Thread 클래스를 상속받는 방식**은 스레드 메서드에 대한 직접 접근을 제공하지만 한 번 밖에 할 수 없는 **상속을 단지 스레드를 사용하는 데 쓴다는 단점**이 있다.
2️⃣ **Runnable 인터페이스 구현**은 태스크가 할 일을 정의하는 것과 스레드 관리를 분리하고 상속을 사용하지 않는다는 점에서 Thread 클래스 상속 방식보다 낫다.
### 스레드 시작하기
스레드를 생성하고 `Thread` 객체의 `start` 메서드가 호출될 때 **스레드 실행이 시작된다.**
:::callout ℹ️** start 메서드**
:::callout - 단순 `Runnable` 인터페이스의 `run` 메서드를 호출하는 것뿐 아니라 시스템 자원 할당 같은 준비 작업도 수행해 `start` 메서드 호출은 반드시 필요하다.
:::callout - 준비 작업 완료 후에 새로운 스레드에서 `run` 메서드를 호출한다.

:::callout ⚠️ `run` 메서드를 개발자가 직접 호출하지 않는 것이 중요하다.
:::callout - `run` 메서드를 직접 호출하면 새로운 스레드에서 `run` 메서드를 호출하는 것이 아닌 기존 스레드에서 `run` 메서드를 호출하게 된다.

#### Executor 프레임워크
> 최신 자바 애플리케이션에서는 `Thread` 클래스의 생성자를 통해 스레드를 생성하는 것보다 `Executor` 프레임워크를 사용하는 것이 훨씬 효율적이다.

`Executor`는 사용할 스레드를 미리 만들어 사용하는<span class="text-red">** 스레드 풀**</span>을 활용해 스레드 관리 작업을 추상화한다.
태스크가 `Executor`에 제출되면 큐에 저장되고 스레드 풀에 있는 스레드 하나가 큐에 저장된 태스크를 수행한다.
```java
try (ExecutorService executor = Executors.newFixedThreadPool(5)) {
    for (int i = 0; i < 10; i++) {
        executor.submit(() -> {
            //...
        });
    }
}
```

## 스레드에 숨어있는 비용
대부분의 웹 애플리케이션은 <span class="text-blue">**thread-per-request**</span> 모델을 사용한다
- **요청 하나에 하나의 스레드를 할당**
- 요청당 하나의 스레드가 서버의 처리를 거친 후 응답까지의 전체 라이프사이클을 관리
- `Apache Tomcat`, `Jetty`

요청이 서블릿 컨테이너로 들어오면 컨테이너는 요청을 스레드 풀에 있는 하나의 스레드에 할당한다.
이 방식은 대규모 동시 연결이 발생하는 상황에 <span class="text-blue">**처리량**</span>을 높일 수 있는 장점이 있다.

#### 많은 스레드를 생성하면 처리량을 더 높일 수 있을까?
최신 운영체제는 수백만 개의 동시 연결을 지원할 수 있다.
<u>운영체제에서 지원하는 동시 연결 수보다 더 많은 스레드를 생성하면 처리량을 더 높일 수 있지 않을까?</u>
- 그렇지 않다.

⚠️ **메모리 사용량**
스레드는 힙 외부에도 **다른 메모리 공간**을 사용한다. 
수천 개의 동시 요청이 발생하는 환경에서는 메모리 사용량이 급격히 늘어날 수 있고 스레드가 사용할 수 있는 전체 메모리 사용량에 따라 실제 사용 가능한 동시 연결 개수는 크게 달라질 수 있다.
애플리케이션이 물리 메모리를 다 사용한다고 가정했을 때 이는 메모리가 부족해 디스크 I/O를 유발하는 페이징이 발생하게 되고 이는 처리 속도를 급격히 떨어뜨린다.

⚠️ **스레드 개수 제한**
자바 스레드는 **운영체제가 제공하는 네이티브 스레드를 감싼** 스레드다.
애플리케이션에서 생성할 수 있는 최대 스레드 개수는 운영체제의 네이티브 스레드 최대 개수에 의해 제한된다.
애플리케이션의 확장성도 운영체제의 스레드 최대 한도에 따라 제한된다.

⚠️ **컨텍스트 전환 비용**
스레드 생성은 메모리 일부를 점유하는 것뿐 아니라 CPU에도 부하를 유발한다.
스레드를 전환할 때 컨텍스트 전환 작업을 해야하고 이는 CPU 사이클을 소모하고 이에 따른 성능 차이가 발생할 수 있다.
### 생성 가능한 스레드 수
간단한 테스트 프로그램 하나만 있다면 현재 시스템에서 생성할 수 있는 스레드의 최대 개수를 알아낼 수 있다.
```java
public class ThreadLimitTest {

    public static void main(String[] args) {
        var threadCount = new AtomicInteger(0);
        try {
            while (true) {
                var thread = new Thread(() -> {
                    threadCount.incrementAndGet();
                    LockSupport.park();
                });
                thread.start();
            }
        } catch (OutOfMemoryError error) {
            System.out.println("Reached thread limit: " + threadCount);
            error.printStackTrace();
        }
    }
}
```

이를 실행하면 메모리 고갈이나 시스템 한계에 도달하기 전에 몇 개의 스레드가 생성됐는지 확인할 수 있다.
## 고확장성 애플리케이션의 자원 효율성
스레드는 실제로 실행이 막힐 수 있어 효율적으로 이용되지 못할 때도 많다.
```java
public Credit calculateCredit(Long personId) {
    var person = getPerson(personId); // 데이터베이스 호출 후 응답이 올 때까지 스레드 실행이 막힘
    var assets = getAssets(person); // API호출 후 응답이 올 때까지 스레드 실행이 막힘
    var liabilities = getLiabilities(person); // 데이터베이스 호출 후 응답이 올 때까지 스레드 실행이 막힘
    importantWork(); // CPU가 많이 사용되는 작업
    return calculateCredits(assets, liabilities);
}
```

순서대로 5개의 메서드가 호출되는데 각 메서드는 이전 메서드 호출이 완료된 후 호출된다.
- 개별 메서드 호출이 느려질 수록 그만큼의 시간이 필요하다.
:::callout ⚠️ **블로킹(Blocking)**
:::callout - 이 비효율은 **입출력(I/O)** 작업을 수행하는 동안 실행이 막혀 스레드가 일을 하고 싶어도 할 수 없다는 것에서 발생한다.
:::callout - 스레드의 실행을 막아 낭비하면 시스템의 확장성은 제한된다.
### 병렬 실행 전략
위 코드에서의 메서드 호출은 서로 의존관계가 없다. 따라서 순서대로 실행할 필요 없이 병렬로 실행해도 전체 결과에 영향을 미치지 않는다.
- 블로킹을 완전히 막을 수는 없지만 메인 스레드가 블로킹되는 전체 시간을 줄일 수 있다.

<span class="text-blue">멀티 스레드</span>를 사용하도록 리팩토링을 해 성능을 개선할 수 있다.
**병렬 버전**은 스레드가 블로킹 되는 시간을 줄여 성능을 개선할 수 있지만 스레드 관리와 시스템 자원 사용에 대해서 문제가 있다.
- `필요할 때마다 스레드를 생성하는 방식`은 스레드 생애주기 관리나 자원 할당 관련 제어 메커니즘이 포함되어 있지 않다.
- 따라서 스레드 과다 사용으로 이어질 수 있고 이는 잠재적으로 시스템 자원을 고갈시킬 수 있다.
### Executor 프레임워크 
스레드를 무분별하게 생성하는 위험을 피하기 위해 <u>**자바 동시성 프레임워크**</u>에서 제공하는 `ExecutorService` 같은 기능을 사용하는 것이 좋다.
:::callout 💡 **ExecutorService**
:::callout 더 구조화된 방식으로 스레드 사용 가능
:::callout - 동시에 실행되는 스레드의 수를 제어하는 것뿐 아니라 스레드의 생애주기를 관리하여 효율적으로 재사용할 수 있게 해준다.

```java
public Credit calculateCreditWithExecutor(Long personId) throws ExecutionException, InterruptedException {
    try (ExecutorService executor = Executors.newFixedThreadPool(5)) { // 1️⃣
        var person = getPerson(personId);
        var assetsFuture = executor.submit(() -> getAssets(person)); // 2️⃣
        var liabilitiesFuture = executor.submit(() -> getLiabilities(person)); // 2️⃣
        executor.submit(() -> importantWork()); // 3️⃣
        return calculateCredits(assetsFuture.get(), liabilitiesFuture.get()); // 4️⃣
    }
}
```

1️⃣ **try 블록에서 빠져나갈 때 자동으로 제거되는 스레드 풀 생성**
2️⃣ 계산에 필수 **데이터를 가져오는 태스크 **제출
3️⃣ 계산 작업 **실행 흐름을 막지 않는 작업**을 제출
4️⃣ 필수 데이터를 가져오는 태스크 완료를 기다린 후 계산하여 반환
`Executor`는 스레드 생성 및 관리를 쉽게 해주고 적절한 작업 부하에 대해서 실행 속도를 높여주고 코드를 더 쉽게 조직화하고 깔끔하게 유지할 수 있게 해주면서 자바 애플리케이션의 동시 실행 작업 관리를 편리하게 만들어준다.
### 아직 남은 과제
`Executor` 프레임워크는 자바 애플리케이션에서 **자원 관리와 비동기 실행**에 있어 상당한 개선을 가져왔지만 활용성을 극대화하기 위해서는 <span class="text-red">**제약사항**</span>도 반드시 알아야 한다.
#### ⚠️ Future.get()에서 발생하는 블로킹
`Future`를 이용하면 작업을 비동기로 실행할 수 있지만 결과를 얻기 위해 `get()` 메서드를 호출되면 작업이 끝날 때까지 완료될 때까지 실행 흐름이 막힌다.
#### ⚠️ 캐시 일관성 유지 부하와 거짓 공유 위험성
스레드 풀의 작업은 서로 다른 CPU 코어에서 실행될 수 있다.
코어마다 캐시 상태가 달라질 수 있고 CPU는 **캐시 일관성**을 유지하기 위해 지속적으로 데이터를 동기화하고 무효화하는 작업을 수행하는데 오히려 성능이 저하될 수 있다.
또한 **거짓 공유(False Sharing)** 문제가 발생할 수 있다.
서로 다른 스레드가 실제로는 다른 변수를 수정하지만 우연히 같은 캐시 라인에 저장된 경우 CPU는 같은 데이터를 수정한다고 오해하여 불필요한 캐시 동기화를 수행하게 되고 성능이 저하된다.
> **캐시 라인 (cache line)**
> - 요즘 CPU는 데이터 접근 속도를 높이기 위해 L 캐시를 가지고 있는데 이는 적은 양의 데이터를 저장하고 CPU가 데이터를 필요로 할 때 메인 메모리가 아닌 CPU안 캐시에서 가져올 수 있도록 한다.
> - 이처럼 캐시에 저장되는 일정 크기의 데이터를 캐시 라인이라 부른다.

#### ⚠️ 합성력 부족
`Executor` 프레임워크는 명령형 스타일이라 합성력이 부족하다.
## 스레드 풀의 발전
`Executor` 프레임워크의 캐시 관련 성능 저하와 복잡한 태스크 의존 관계 문제를 <span class="text-red">**자바의 포크/조인 풀(fork/join pool)**</span>은 성능에 중심을 둔 설계와 특별한 알고리즘으로 해결했다.
**포크/조인 풀**이 구현된 `ExecutorService` 에는 더 복잡한 스레드 풀 메커니즘이 적용됐으며 태스크 스케줄링을 통해 성능과 자원 효율성을 높일 수 있었다.
### 캐시 선호도와 태스크 분배
**캐시 선호도**는 `CPU 캐시의 지역성(locality)`를 활용해 태스크를 완료하는 개념을 가리킨다.
- 태스크가 특정 CPU에서 실행되면 그 태스크와 <u>관련된 데이터도 해당 CPU 코어에 로딩</u>된다.
- 이 태스크와 관련있는 다른 태스크가 동일한 코어에서 실행되면 이 다른 태스크는 캐시에 저장된 데이터를 사용해 메모리 접근 시간을 절약
**포크/조인 풀**은 워커 스레드가 로컬 큐에 있는 태스크를 실행하기 때문에 캐시 선호도를 활용할 수 있다. 
#### 예시: 고해상도 이미지에 압축 수행 워커 스레드
1. 스레드는 이미지를 작은 영역의 타일들로 나누고 각 타일에 대해 `색상 분석`과 `데이터 인코딩` 같은 태스크를 로컬 큐에 저장한다.
2. 스레드가 특정 타일에 대한 `색상 분석` 태스크를 마치면 로컬 큐에서 동일한 타일을 `인코딩`하는 태스크를 꺼내 실행한다.
3. 해당 타일의 픽셀 데이터는 `색상 분석` 단계에서 코어의 캐시에 로드되어 있기 때문에 스레드는 메모리에서 데이터를 다시 가져올 필요 없이 **캐시에 있는 데이터**를 사용해 즉시 `인코딩`을 수행할 수 있다.
### Work-Stealing 알고리즘
**포크/조인 풀**은 스레드 사이 작업 부하를 효율적으로 분산하기 위해 동적인 <span class="text-red">**Work-Stealing 알고리즘**</span>을 사용한다.

:::callout 💡 <span class="text-red">**Work-Stealing 알고리즘**</span>
:::callout - **포크/조인 풀의 스레드들은 각자 전용 큐**를 가지고 있다.
:::callout - 한 스레드가 자신에게 할당된 작업을 모두 마치면 다른 스레드의 큐의 끝에 있는 태스크를 `훔쳐와` 수행한다.
:::callout - 태스크가 전부 완료되지 않은 상태에서 아무 일도 하지 않고 놀고 있는 스레드가 발생하지 않게 해 **CPU 사용률을 극대화**할 수 있다.
### 포크/조인 풀 사용 예제
> **포크/조인 풀은 태스크 전송 과정을 단순화하고 별도 추가조치 없이 자동으로 태스크를 효율적으로 분배한다.**

```java
ForkJoinPool forkJoinPool = new ForkJoinPool();
forkJoinPool.submit(() -> {
  // 병렬 태스크 처리 로직
}).join();
```


### CompletableFuture를 활용한 작업 흐름 조합
**Executor 프레임워크와 포크/조인 풀**을 사용해 성능 개선 효과는 있지만 <span class="text-red">복잡한 비동기 작업 흐름</span>을 조합하는 일은 여전히 어렵다.
이 문제를 해결하기 위해 **Java 8**에서 **`CompletableFuture`**를 도입했다.
:::callout ℹ️ **CompletableFuture**
:::callout - 조합성과 사용성에 초점을 맞춰 <span class="text-red">**비동기 프로그래밍**</span>을 쉽게 작성할 수 있게 해준다.

#### 조합력있는 평문형 API
`CompletableFuture`를 사용하면 비동기 코드 작성에서 발생하는 콜백 지옥을 피해 <span class="text-brown">**선언적인 함수형 스타일**</span>로 코드를 작성할 수 있다.
> `CompletableFuture` 에는 많은 API가 있어 잘 사용하면 다수의 비동기 연산을 쉽게 연결해 코드 **가독성과 유지보수성**을 개선할 수 있다.


#### CompletableFuture 장점
`CompletableFuture` 와 평문형 API를 사용해 비동기 코드를 일련의 **변환(transformation) **및 **중간 연산(intermediate computation)**을 사용해 작업 흐름을 정의하여 코드 가독성과 유지보수성을 높일 수 있고 버그 발생 가능성을 줄일 수 있다.

#### CompletableFuture 단점
1️⃣ **시간과 연습이 필요**
명령형 프로그래밍에 익숙한 개발자는 발상의 전환이 필요하다.
API를 의미에 맞게 사용하는 법을 배우기 위해 꽤 많은 시간을 투자해야 할 수 있다.
- 어떤 API를 어느 상황에 사용하는 것이 적합한지

2️⃣ **체이닝 스타일에서의 디버깅 흐름 제어**
`CompletableFuture` 의 체이닝 스타일이 적용된 코드를 디버깅을 할 때 행 단위로 순서대로 실행되지 않으므로 실행 흐름이 모호하고 때로는 순서가 뒤바뀔 수도 있어 힘들다.

## 자바 동시성의 혁명
### 가상 스레드의 약속
**프로젝트 룸**에서는 자바 동시성 처리에 <span class="text-red">**가상 스레드**</span>라 불리는 새로운 종류의 스레드를 도입했다.
- 기존 스레드 생성 비용과 비교하면 거의 비용이 들지 않을 정도로 가벼운 스레드라 필요할 때마다 생성하더라도 성능에 영향을 미치지 않는다.
- 높은 동시성 처리 능력과 확장성

### 기존 코드베이스와의 매끄러운 통합
> 기존 자바 코드를 크게 변경하지 않아도 통합해서 사용할 수 있다.


👀 기존 `Executor` 프레임워크를 사용하고 있었다면 `Executors.newVirtualThreadPerTaskExecutor()` 로 바로 가상 스레드를 사용할 수 있다.
```java
public Credit calculateCreditWithVirtualThread(Long personId) throws ExecutionException, InterruptedException {
    try (var executor = Executors.newVirtualThreadPerTaskExecutor()) { // ①
        var person = getPerson(personId);
        var assetsFuture = executor.submit(() -> getAssets(person));
        var liabilitiesFuture = executor.submit(() -> getLiabilities(person));
        executor.submit(this::importantWork);
        return calculateCredits(assetsFuture.get(), liabilitiesFuture.get());
    }
}
```

### 가상 스레드와 플랫폼 스레드
> **가상 스레드는 플랫폼 스레드 위에서 동작한다.**

- 플랫폼 스레드는 포크/조인 풀의 지원을 받아 가상 스레드도 포크/조인 풀의 장점을 모두 활용할 수 있다.
### 지능적인 블로킹 연산 처리
> 가상 스레드는 `sleep`이나 `네트워크 I/O` 같은 블로킹 연산을 만나면 실행을 멈추고 제어권을 자동으로 플랫폼 스레드에게 넘겨준다.
> - 제어권을 넘겨받은 플랫폼 스레드는 다른 가상 스레드를 실행하여 자원을 효율적으로 활용
> - 블로킹 연산이 완료되면 가상 스레드는 멈췄던 지점에서 다시 실행을 시작

### 가상 스레드의 장점

| **장점** | **설명** |
| ---- | ---- |
| **자원 효율성** | 가상 스레드는 가벼워 수백만 개를 생성해도 자원이 고갈되지 않다. |
| **코드 단순성** | 익숙한 블로킹 방식으로 코드를 작성하더라도 성능 손실이 발생하지 않아 추가적인 학습 곡선 없이 코드도 쉽게 유지 관리할 수 있다. |
| **최적 활용** | 가상 스레드는 블로킹 연산 수행중에는 제어권을 플랫폼 스레드에게 넘겨줘 플랫폼 스레드는 불필요하게 쉬지 않고 CPU를 사용해 성능이 높아진다. |
