:::callout 🎯 **챕터 목표**
:::callout 고도의 동시성을 가진 애플리케이션에서 컨텍스트 전파는 항상 어려운 문제였다.
:::callout 이 장에서는 <span class="text-red">**스코프드 밸류**</span>가 가상 스레드 환경에서 `ThreadLocal`보다 더 나은 대안이 되는 이유를 설명한다.
:::callout API를 살펴보고 구조적 동시성 맥락에서의 실용적인 예제를 통해 `ThreadLocal`에서 언제 어떻게 <span class="text-red">**스코프드 밸류**</span>로 전환해야 하는지를 이해하게 된다.
> <span class="text-red">**ScopedValue**</span>를 사용하면 값을 **특정 스코프에 구조적 방식으로 바인딩하면서도 컨텍스트와 일관성을 유지할 수 있다**.

기존 **ThreadLocal 변수는 다루기 번거롭고 메모리 누수가 발생할 위험이 높지만** **ScopedValue는 멀티스레드 애플리케이션을 다룰 때 더 간결하고 효율적으로 사용할 수 있다.**
## 컨텍스트 전달에 따르는 부담
#### ⚠️ 파라미터 전달 문제 - 일반적인 작업 스케줄링 프레임워크
코드의 여러 곳에서 데이터를 공유할 때 단순 메서드 인자로 데이터를 전달하기에는 어려운 상황이 발생할 수 있다. 특히 애플리케이션이 프레임워크에 의존하는 경우 더 그런 경우가 많다.

`ex) 사용자 코드가 프레임워크에 태스크를 등록하고 프레임워크는 해당 태스크를 실행하는 작업 스케줄링 프레임워크`
```java
public interface Job {
    void execute(JobContext context);
}

enum Priority { LOW, MEDIUM, HIGH }

public record JobContext(String jobName, Priority priority,
                         Map<String, Object> metadata) {
    public JobContext(String jobName, Priority priority) {
        this(jobName, priority, new HashMap<>());
        metadata.put("jobName", jobName);
        metadata.put("priority", priority);
        metadata.put("creationTime", Instant.now());
    }

    public Object getMetadataValue(String key) {
        return metadata.get(key);
    }
}
```


**이 프레임워크의 핵심 스케줄링 로직에서 JobContext를 생성하고 관리한다.**
```java
public class JobScheduler {
    public void schedule(Job job, String jobName, Priority priority) {
        JobContext context = new JobContext(jobName, priority);
        runJob(job, context);
    }

    private void runJob(Job job, JobContext context) {
        // 프레임워크가 사용자 코드를 호출하면서 context를 전달
        job.execute(context);
    }

    public Object getJobMetadata(String key, JobContext context) { 
        if (context == null) { return null; }
        return context.getMetadataValue(key);
    }
}
```


**이 프레임워크를 사용하려면 Job 인터페이스를 구현해야하고 프레임워크가 전달해준 context 객체를 사용해야 한다.**
```java
public class UserJob implements Job {

    private final JobScheduler jobScheduler;

    public UserJob(JobScheduler jobScheduler) {
        this.jobScheduler = jobScheduler;
    }
    
    @Override
    public void execute(JobContext context) {
        System.out.println("User job is running!");

        // 메타데이터 조회를 위해 사용자 코드가 프레임워크 코드를 호출
        Object creationTime
                  = jobScheduler.getJobMetadata("creationTime", context);⑤
        System.out.println("Job creation time: " + creationTime);

        // 다른 보조 메서드도 context 파라미터를 필요로 한다.
        processJobData(context);
    }

    private void processJobData(JobContext context) {
        // 이 메서드에서 context를 사용하지 않아도 
        // 프레임워크 메서드에 전달하기 위해 context를 받아야만 한다.
        Object priority = jobScheduler.getJobMetadata("priority", context);
        System.out.println("Processing job with priority: " + priority);
    }
}
```

프레임워크가 작업을 실행할 때마다 메타데이터를 담은 컨텍스트 객체를 생성한다.
이 컨텍스트는 프레임워크에 필수적이지만 사용자 코드에는 대부분 큰 의미가 없다.
사용자 코드에서 프레임워크의 내부 사정에 대해 알 필요가 없음에도 내부 사정에 해당하는 context를 전달받아야 한다.
### 파라미터 오염
작업 실행 흐름에 참여하는 모든 메서드는 그 메서드가 컨텍스트 정보를 전혀 사용하지 않더라도 이 파라미터를 선언해야 한다.
메서드 시그니처가 애플리케이션 로직과는 무관한 프레임워크 세부 사항으로 오염되는 파라미터 오염 현상을 만든다.
### 인터페이스 취약성
프레임워크가 더 발전하면 나중에 컨텍스트에 새로운 데이터가 추가될 수 있고 이렇게 되면 컨텍스트를 전달하는 사용자 코드의 모든 메서드 시그니처를 수정해야 할 수도 있다.
- 프레임워크가 필요로 하는 정보가 확장될 때 그 영향이 전체 코드베이스로 퍼져나가게 된다.
### 결합도와 테스트 용이성
사용자 코드는 프레임워크 구현 세부 사항에 강하게 결합되어 항상 유효한 컨텍스트를 제공해야 하여 테스트가 복잡해진다.
- 프레임워크와 무관한 비즈니스 로직에 대한 테스트에도 영향을 준다.
## ThreadLocal 소개
이런 문제를 해결하기 위해 프레임워크 코드를 <span class="text-blue">**ThreadLocal**</span>을 사용해 더 영리하게 설계할 수 있다.
```java
public class JobScheduler {
    private static final ThreadLocal<JobContext> jobContextHolder = new ThreadLocal<>(); // ①

    public void schedule(Job job, String jobName, Priority priority) {
        JobContext context = new JobContext(jobName, priority);
        try {
            jobContextHolder.set(context); // ②
            runJob(job);
        } finally {
            jobContextHolder.remove(); // ③
        }
    }

    private void runJob(Job job) { // ④
        job.execute();
    }

    public Object getJobMetadata(String key) {
        JobContext context = jobContextHolder.get(); // ⑤
        return (context != null) ? context.getMetadataValue(key) : null;
    }
}
```

1️⃣ **ThreadLocal** 변수를 만들어 **각 스레드마다 컨텍스트를 보관**
2️⃣ 작업 실행 전 **현재 스레드에 컨텍스트를 설정**
3️⃣ 메모리 누수를 막기 위해 **finally** 블록에서 **항상 컨텍스트를 제거**한다.
4️⃣ <u>**runJob 메서드는 더 이상 컨텍스트 파라미터가 필요하지 않다.**</u>
5️⃣ 프레임워크 메서드는 **필요할 때 ThreadLocal에서 컨텍스트를 가져올 수 있다.**

#### **🚀 사용자 코드는 프레임워크 컨텍스트를 다룰 필요가 없어 훨씬 깔끔해진다!**
```java
public class UserJob implements Job {

    private final JobScheduler jobScheduler;

    public UserJob(JobScheduler jobScheduler) {
        this.jobScheduler = jobScheduler;
    }

    @Override
    public void execute() { // ①
        System.out.println("User job is running!");

        // context 파라미터를 전달할 필요없이 프레임워크가 내부적으로 알아서 한다.
        Object creationTime = jobScheduler.getJobMetadata("creationTime"); // ②
        System.out.println("Job creation time: " + creationTime);

        // 보조 메서드도 깔끔해진다.
        processJobData();
    }

    private void processJobData() { // ③
        // Clean method signature - no framework parameters
        Object priority = jobScheduler.getJobMetadata("priority");
        System.out.println("Processing job with priority: " + priority);
    }
}
```

1️⃣ 사용자 코드에서 프레임워크 컨텍스트를 주고받는 불편함이 해결된다.
2️⃣ 프레임워크 서비스는 context 파라미터가 없어도 메타데이터를 조회할 수 있다.
3️⃣ 보조 메서드의 시그니처도 비즈니스 로직에 꼭 필요한 요소들만 포함하도록 깔끔해진다.

> 이를 통해 사용자 코드는 더 이상 프레임워크에서만 사용하는 파라미터에 신경 쓸 필요 없이 비즈니스 로직에만 집중하면 된다.

### ThreadLocal 변수의 한계
#### ⚠️ ThreadLocal은 변경에 제약이 없다.
`ThreadLocal` 변수의 `get()` 메서드를 호출할 수 있는 코드라면 언제든지 아무 제약없이 `set()`을 호출하여 데이터를 변경할 수 있다. 이는 데이터가 언제 어디에서 수정되는지 추적하기 어렵게 만든다.
#### ⚠️ ThreadLocal의 생애주기는 제한이 없다.
스레드 로컬 변수값이 지정되면 명시적으로 제거되지 않는 한 해당 스레드가 살아있는 동안 계속 유지된다.
- 스레드가 반복적으로 재사용되는 스레드 풀을 사용할 때 문제가 될 수 있다.
`remove()`를 호출하여 스레드 로컬 변수를 적절하게 제거해 주지 않으면 **스레드 로컬 변수의 값이 스레드에 계속 남아 스레드가 실행하는 서로 다른 태스크에 영향을 미치는 데이터 누수**가 발생할 수 있고 스레드 로컬 변수에 남은 민감한 데이터는 보안 취약점이 될 수도 있다.
#### ⚠️ InheritableThreadLocal (상속 문제)
**InheritableThreadLocal**을 사용할 경우 자식 스레드는 부모 스레드의 값을 자동으로 상속받는다.
- 많은 자식 스레드를 생성하면 숨은 비용이 커지게 된다.
**InheritableThreadLocal**은 <u>객체 자체가 아닌 참조를 자동으로 자식 스레드에 복사</u>한다.
- 커다란 객체를 모든 자식 스레드가 참조하면 커다란 객체가 더 오래 살아 가비지 컬렉션 되는 시점을 늦춘다.
- 자식 스레드가 데이터를 수정하지 않아도 모든 자식 스레드는 각자의 ThreadLocalMap을 통해 커다란 객체에 대한 참조를 유지한다.
적은 수의 플랫폼 스레드를 사용할 때는 이로인한 영향을 무시해도 될 만큼 작지만 가상 스레드를 사용할 때처럼 굉장히 많은 수의 스레드를 만들어 사용할 때는 이런 오버헤드도 문제가 될 수 있다.
### 가벼운 공유를 향하여
**각 가상 스레드가 자신만의 ThreadLocal 데이터를 유지하는 것이 기술적으로 가능은 하지만 메모리 오버헤드는 감당할 수 없다.**
가상 스레드는 수명이 짧아 **ThreadLocal**이 오랫동안 살아남아 발생하는 문제는 덜 심각하지만 많은 수의 중복된 데이터가 만들어지는 메모리 오버헤드는 문제다.
💡 **수많은 복제를 발생시키지 않고도 스레드별로 상속 가능한 데이터를 저장할 수 있는 새로운 메커니즘이 필요하다.**
- 저장되는 데이터가 불변이면 훨씬 좋다 (불변 객체 하나를 복제하지 않고도 여러 자식 스레드가 참조)
- 데이터에는 유한한 수명이 있어야 한다. (공유하는 메서드가 끝나면 연결된 스레드 로컬 정보는 의미가 없어야 한다)

## ScopedValue 핵심 구성요소
:::callout ℹ️ **ScopedValue **
:::callout **ScopedValue**는 암시적 메서드 파라미터처럼 동작하므로 `각 메서드 시그니처에 명시적으로 파라미터를 넣지 않고도 일련의 메서드 호출에 걸쳐 데이터를 전달`할 수 있게 한다.
이는 특히 깊이 중첩된 메서드 호출이나 콜백 구조를 다룰 때 코드가 훨씬 깔끔하고 유지보수하기 쉬워진다.
### 🤖 ScopedValue의 3가지 주요 특징

| 주요 특징 | 설명 | 장점 |
| ---- | ---- | ---- |
| 불변성 | 특정 스코프내에서 ScopedValue가 어떤 값에 바인딩되면 그 값은 ScopedValue가 살아있는 동안 변경할 수 없다. | 일관성 있고 예측 가능한 동작을 보장한다. |
| 스레드 제한적 바인딩 | 바인딩은 현재 스레드에 한정된다. | 스레드 사이 의도치 않은 데이터 공유를 방지하고 스레드 안정성을 높인다. |
| 유한한 수명 | ScopedValue의 바인딩은 특정 코드 블록의 실행 기간동안만 유지되어 블록이 끝나면 바인딩이 해제된다. | 자원 관리에 도움이 되고 메모리 누수 위험을 줄여준다. |

**ScopedValue**를 사용하려면 먼저 `정적 final 필드`로 선언한다.
```java
private static final ScopedValue<String> NAME = ScopedValue.newInstance();
```

:::callout 💡**ScopedValue 생성자**
:::callout **ScopedValue 생성자**는 `private`으로 지정되어 있어 반드시 팩토리 메서드를 사용해 생성해야 한다.
생성한 후에는 `where()` 메서드를 사용해 **ScopedValue에 값을 바인딩**하고 `run()` 메서드를 사용해 해당 **스코프 안에서 코드를 실행**한다.
```java
ScopedValue.where(NAME, "ko").run(() -> doSomething());
```

위 코드에서 `where()` 메서드는** NAME 스코프드 밸류**와 **현재 사용자를 바인딩**한다.
`run()` 메서드는 바인딩 된 값을 사용할 수 있는 스코프안에서 코드블록을 실행한다.

#### JobScheduler 클래스를 ScopedValue를 사용하도록 변경
```java
public class JobScheduler {
    private static final ScopedValue<JobContext> CONTEXT
                                            = ScopedValue.newInstance(); //①

    public void schedule(Job job, String jobName, Priority priority) {
        JobContext context = new JobContext(jobName, priority); //②

        ScopedValue.where(CONTEXT, context).run(() -> runJob(job)); //③
    }

    private void runJob(Job job) {
        job.execute(); //④
    }

    public static JobContext getContext() { //⑤
        return CONTEXT.get();
    }

    public static Object getJobMetadata(String key) {
        JobContext context = CONTEXT.get();
        if (context != null) {
            return context.getMetadataValue(key);
        }
        return null;
    }
}
```

1️⃣ 팩토리 메서드를 통해 ScopedValue 인스턴스 생성
2️⃣ 바인딩할 컨텍스트 객체 생성
3️⃣ 컨텍스트를 ScopedValue안에서 바인딩하고 ScopedValue에 접근할 수 있는 스코프안에서 작업 실행
4️⃣ 작업은 ScopedValue에 바인딩된 컨텍스트에 접근할 수 있는 상태로 실행된다.
5️⃣ 현재 컨텍스트에 대한 정적 접근 제공

:::callout 💡 `call()` 메서드
:::callout ScopedValue는 스코프 안에서 실행한 결과를 반환해야 하는 경우를 위해 call() 메서드를 제공한다.
:::callout - `run()`은 `Runnable`을 실행해 **void**를 반환
:::callout - `call()`은 `Callable`을 실행해 **값**을 반환
### ScopedValue 실행
**ScopedValue**의 수명은 유한하다. 먼저 값을 설정하고 난 후에 사용할 수 있다.
```java
public class ScopedValueExamples {

    public static void main(String[] args) throws InterruptedException {
        ScopedValue<String> NAME = ScopedValue.newInstance();

        Runnable task = () -> {
            if (NAME.isBound()) {
                System.out.println("Name is bound: " + NAME.get());
            } else {
                System.out.println("Name is not bound");
            }
        };

        // 스코프 안에서 실행
        ScopedValue.where(NAME, "Bazlur").run(task);
				// 스코프 밖에서 실행
        task.run();  // unbounded
    }
}

--------출력 결과---------
Name is bound: Bazlur
Name is not bound
```

ScopedValue는 `run()` 메서드 호출 내부의 동적 스코프 안에서만 바인딩 된 상태가 유지된다.
`run()` 메서드 실행이 완료되면 바인딩은 자동으로 해제되어 스코프의 경계가 분명하게 분리되는 것을 보장하고 관련 없는 코드 영역으로 값이 새어나가는 누수를 방지한다.

:::callout 💡 **ScopedValue**는 프로그램의 실행 흐름에 의해 결정되는 **동적 스코프**에서 작동한다.
:::callout 🤔 **동적 스코프란**
:::callout - 특정 메서드가 실행되는 동안 그리고 그 메서드가 직접/간접 적으로 호출하는 메서드들에서 값을 접근할 수 있음을 의미
:::callout - a → b, b → c 호출시 스코프는 c까지 이어지지만 c가 끝나면 종료된다.
:::callout **ScopedValue는 값을 해당 실행 흐름에 바인딩하고 run 메서드의 스코프안에서만 사용할 수 있게 한다.**
이렇게 일시적이고 정밀한 스코프 관리 덕분에 **ScopedValue**는 컨텍스트 데이터를 전달할 때 **ThreadLocal**보다 더 깔끔하고 안전하다.

#### 메인스레드 말고 다른 스레드에서 태스크를 실행하면 어떻게 될까?
```java
public class ScopedValueExamples {

    public static void main(String[] args) throws InterruptedException {
        ScopedValue<String> NAME = ScopedValue.newInstance();

        Runnable task = () -> {
            if (NAME.isBound()) {
                System.out.println("Name is bound: " + NAME.get());
            } else {
                System.out.println("Name is not bound");
            }
        };

        Thread thread = Thread.ofPlatform().unstarted(task); // 1
        ScopedValue.where(NAME, "Bazlur").run(thread::start); // 2
        thread.join();
    }
}

---------출력 결과---------
Name is not bound
```

1️⃣ 동일한 태스크를 실행하는 플랫폼 스레드를 생성
2️⃣ ScopedValue에 값을 바인딩하고 스코프 안에서 플랫폼 스레드를 시작

이렇게 출력되는 이유는 **ScopedValue**는 새로 생성되는 스레드에 자동으로 상속되지 않기 때문이다.
`thread::start`가 **NAME이 바인딩된 스코프 안에서 실행된다고 하더라도** **별개의 스레드에서 실행되는 실제 태스크는 바인딩된 ScopedValue를 상속받지 못한다.**
- 별개의 스레드가 `플랫폼 스레드`이든 `가상 스레드`이든 상관없이 모두 동일하게 동작한다.

#### 메인 스레드가 아닌 새로 생성되는 스레드 안에서 ScopedValue를 생성하면?

2️⃣ 새로 생성된 스레드의 컨텍스트 안에서 ScopedValue에 값을 바인딩하고 실행

#### ScopedValue는 평문형 API를 제공한다.
여러 개의 **ScopedValue**를 하나의 체인에서 함께 바인딩할 수도 있다.
```java
public class MultiScopedExample {
    private static final ScopedValue<String> USER_ID = ScopedValue.newInstance();
    private static final ScopedValue<String> SESSION_ID = ScopedValue.newInstance();

    public static void main(String[] args) {
        ScopedValue.where(USER_ID, "user123") //①
                .where(SESSION_ID, "session456") //②
                .run(() -> performTask()); //③
    }

    public static void performTask() {
        String userId = USER_ID.get();
        String sessionId = SESSION_ID.get();
        logAction();
    }

    public static void logAction() {
        String userId = USER_ID.get(); //④
        String sessionId = SESSION_ID.get();
    }
}
```

1️⃣ 첫 번째 ScopedValue 바인딩
2️⃣ 두 번째 ScopedValue 바인딩
3️⃣ 바인딩된 값을 갖는 2개의 ScopedValue를 사용할 수 있는 스코프에서 코드를 실행
4️⃣ 호출 스택 전체에 걸쳐 2개의 ScopedValue를 모두 사용할 수 있다.

#### orElse, orElseThrow 메서드
`orElse` 메서드는 <u>**ScopedValue에 바인딩된 값이 없을 때**</u> 기본값을 사용할 수 있게 해주고
`orElseThrow` 메서드는 예외를 던진다.
```java
public class ScopedValueDefaultsExample {
  private static final ScopedValue<String> USER_NAME = ScopedValue.newInstance();

  public static void main(String[] args) {
    // USER_NAME에 값이 없다면 기본값 사용
    String userNameUnbound = USER_NAME.orElse("Guest"); //①
    System.out.println("No binding -> user name defaults to: "
        + userNameUnbound);

    // USER_NAME에 값이 있는지 검증후 없다면 예외를 던진다.
    try {
      USER_NAME.orElseThrow(() ->
          new IllegalStateException("No user name bound yet!")); //②
    } catch (IllegalStateException e) {
      System.out.println("Caught exception: " + e.getMessage());
    }

    // USER_NAME에 값이 바인딩된 스코프안에서 USER_NAME 접근
    ScopedValue.where(USER_NAME, "Bazlur").run(() -> {
      String boundUserName = USER_NAME.orElse("Guest"); //③
      System.out.println("Within binding -> user name is: " + boundUserName);

      // 값이 바인딩되어 있으므로 예외를 던지지 않는다.
      String validatedName = USER_NAME.orElseThrow(()
          -> new IllegalStateException("No user name bound yet!")); //④
      System.out.println("Validated name: " + validatedName);
    });
  }
}
```

1️⃣ 값이 바인딩되어 있지 않을 때 기본값이 사용되게 해 ScopedValue를 안전하게 사용할 수 있다.
2️⃣ 바인딩되지 않았을 때 사용자 정의 예외를 던지게 할 수 있다.
3️⃣ 바인딩된 값이 있다면 기본값을 무시하고 바인딩된 값을 반환한다.
4️⃣ 바인딩된 값이 있다면 예외를 던지지 않는다.

#### 중첩 스코프에서 ScopedValue 리바인딩
**ScopedValue**의 강력한 기능 중 하나는 <span class="text-blue">**중첩된 스코프안에서 다시 바인딩할 수 있다는 점**</span>이다.
> **다시 바인딩한다?!**
> - **중첩된 스코프안에서 ****`동일한 이름의 ScopedValue`****에 **<span class="text-red">**새로운 값**</span>**을 지정하되 중첩된 스코프 안에서만 유효하도록 제한**하는 것을 의미

중첩된 스코프가 끝나면 **원래 값이 자동으로 복원**되어 컨텍스트를 깔끔하고 예측 가능하게 관리할 수 있다.
### ScopedValue와 구조적 동시성
**ScopedValue**는 <span class="text-brown">구조적 동시성</span>과 자연스럽게 통합되어 동작하도록 설계되었다.
ScopedValue는 일반적으로 자식 스레드에 자동으로 상속되지 않지만 `StructuredTaskScope` 안에서 사용될 때는 **해당 스코프 안에서 생성된 모든 자식 스레드가 ScopedValue 바인딩을 자동으로 상속**받는다.
이런 상속 메커니즘 덕에 부모 스레드와 자식 스레드 사이에 데이터를 명시적으로 인자로 넘기지 않아도 효율적으로 데이터를 공유할 수 있다.
`StructuredTaskScope` 를 빠져나오면 그 스코프안에서 생성된 모든 스레드는 인터럽트되고 가비지 컬렉션 대상이 되어 메모리 누수 문제가 발생하지 않는다.
### 성능 고려사항
<span class="text-red">ScopeValue</span>는 가상 스레드를 사용할 때 <span class="text-blue">ThreadLocal</span>보다 전반적으로 더 나은 성능을 보인다.

| **성능상 이점 요인** | **ThreadLocal** | **ScopedValue** |
| ---- | ---- | ---- |
| **오버헤드 감소** | 각 가상 스레드가 자신만의 복사복을 가져 메모리 사용량이 급증, 오버헤드 유발 | 특정 스코프안에서 스레드 사이에 데이터를 공유할 수 있어 메모리 사용을 최소화하여 성능 향상 |
| **가상 스레드와 구조적 동시성에 맞게 최적화** | ThreadLocal에서 병목문제가 발생 | 가상 스레드가 가볍기 때문에 병목 없이 효율적으로 데이터 공유 |

### 사용성 및 API 설계
성능 외에도 **ScopedValue**는 **ThreadLocal**보다 사용성이 더 좋다.

| **사용성 ** | **설명** | **이점** |
| ---- | ---- | ---- |
| **불변성 강제 ** | 한 스코프 안에서 ScopeValue에 바인딩된 값은 수정할 수 없다 | 코드의 동작을 이해하기 쉽게하고 의도치 않은 수정으로 인한 오류를 줄여준다. |
| **API 설계로 공유 데이터 생애주기 정의** | `run()` 메서드는 ScopedValue에 접근할 수 있는 스코프가 어디인지 분명히 보여준다 | 코드 가독성이 좋아지고 데이터가 어떻게 흐르는지 더 쉽게 이해할 수 있다. |
| **더 간결하고 직관적인 API** | `where()`, `run()`를 통해 스코프내 공유 데이터 읽기/쓰기를 구조적인 방식으로 수행 가능 | ThreadLocal API에 비해 ScopedValue API는 훨씬 자연스럽고 사용하기 쉽다. |
| **권한 객체** | ScopedValue는 누가 공유 데이터에 접근할 수 있는지 정밀하게 제어할 수 있다 | 적절한 접근 지정자를 지정해 선언하여 권한이 있는 컴포넌트만 접근하도록 제한할 수 있다. |
| **null 처리** | null 값을 ThreadLocal보다 훨씬 명확히 다룬다. | 값이 바인딩되지 않은 상태에서 `get()`을 호출하면 `NoSuchElementException`을 던져 실수를 초기에 잡아낼 수 있다. |

### ScopedValue로의 마이그레이션
#### ✅ ScopedValue로 전환하기전 고려해야 할 몇가지 요소

| **요소** | **설명** |
| ---- | ---- |
| **JDK 버전** | ScopedValue는 `JDK 25`부터 제공되므로 새 버전 JDK로의 업그레이드를 고려해야 한다 |
| **불변성** | ScopedValue는 불변성을 강제하므로 공유하려는 데이터가 반드시 불변이어야 한다는 점을 잊지 말아야 한다. |
| **공유 데이터 스코프 명확히 정의** | `where()`와 `run()` 메서드로 데이터가 의도한 컨텍스트 안에서만 접근되도록 보장하고 예기치 않은 부작용을 방지하도록 공유 데이터의 스코프를 명확하게 정의해야 한다. |
| **`null`**** 처리 방식** | ScopedValue는 바인딩되지 않은 값을 읽으려고 하면 `NoSuchElementException`을 던지지만 ThreadLocal은 `null`을 반환한다. |
| **테스트** | ScopedValue를 도입한 후에 동시성이나 데이터 일관성이 잘 유지되는지, 예상치 못한 문제를 발견하고 해결하도록 충분히 테스트해야 한다. |
