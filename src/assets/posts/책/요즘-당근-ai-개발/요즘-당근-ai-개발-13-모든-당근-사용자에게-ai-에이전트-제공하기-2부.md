## 🔍 KAMP에 당근 특화 기능 넣기
실제 사용자에게 제공하는** 당근 AI 에이전트**를 기획하고 만들다보니 기본 에이전트 구조에는 없는 당근의 사용자에게 필요로 하는 다음과 같은 기능을 개발했습니다.
1️⃣ **정해진 답변이 나가야 할 때 사람처럼 자연스럽게 답변하기**
2️⃣ **에이전트의 진행 상황 알려주기**
3️⃣** 최신성을 유지하며 당근 내부 정보 알려주기**
4️⃣ **AI로 맥락에 맞는 UI 제공하기**
### 정해진 답변이 나가야 할 때 사람처럼 자연스럽게 답변하기
에이전트를 개발하다 보면 정해진 답변을 내보내야 하는 경우가 있습니다.
- 내부 정책상 답변이 불가한 경우
- 사용자가 가벼운 인사말을 건네는 상황 등
이런 상황에서 항상 같은 문장으로 응답한다면 사용자는 기계적인 응답으로 느낄 것이고 AI 에이전트에 대한 부정적인 감정이 커질 수 있습니다.
✅ **우리는 에이전트가 정해진 답변이 나가야 하는 상황인지 인지하게만 하고 답변 도구를 호출하도록 했습니다.**
- 백엔드 시스템에서는 답변 도구에 사전 정의된 목록에서 무작위로 하나의 답변을 선택하여 응답하도록 개발
- 사용자에게는 자연스러운 경험을 제공하면서 특정 상황에 우리의 의도대로 답변을 관리
### 에이전트의 진행 상황 알려주기
⚠️ LLM 모델이 거대해지고 추론 기능이 생기며 성능이 올라갔지만 **응답이 나오기까지는 점점 오래 걸리고 있습니다.**
#### CoT(Chain of Thought, 생각의 사슬)
사용자가 느끼는 체감 시간을 줄이기 위해 다양한 시도가 이루어지고 있는데 특히 <span class="text-blue">**모델이 문제를 해결할 때 단계별로 추론한 과정을 사용자에게 명시적으로 보여주는 생각의 사슬(CoT)기법**</span>은 대부분의 서비스에 적용되어 있습니다.

**CoT를 보여주는 것의 단점**: 실시간으로 생성되는 추론 과정에서 잘못되거나 노출되면 안 되는 정보가 보여질 위험이 있다
- ex) 내부 API 엔드포인트, 데이터베이스 스키마 등 민감 정보

이 문제를 해결하면서 CoT를 보여주는 방법으로 선택한 방법은 <u>**실제 추론 과정을 실시간으로 서빙하는 대신 단계별 사전 정의된 메시지를 보여주는 것**</u>입니다.
멀티 AI 에이전트의 구조를 확정한 후 `사용자입력 ~ 응답 생성`까지의 단계를 검토하여 각 단계에 적합한 메시지를 설정합니다.

### **최신성을 유지하며 당근 내부 정보 알려주기**
<span class="text-red">**RAG**</span>는 에이전트가 답변을 생성할 때 실시간으로 외부 문서나 데이터베이스에서 관련 정보를 검색해 참조하는 기술로 모델이 부정확한 답변을 생성하는 `할루시네이션` 문제를 효과적으로 해결할 수 있습니다.
#### 벡터 서치
RAG에 사용할 수 있는 많은 기술중 의미론적인 검색을 할 수 있는 <span class="text-red">**벡터 서치**</span>를 활용하고자 했습니다.

사용자의 질문에 따라 <u>**실시간으로**</u> 벡터 서치를 진행하여 운영정책, FAQ, 게시글 등을 가져오려면 복잡한 준비가 필요합니다.
- KAMP에서는 사용자가 검색할 문서 혹은 데이터 소스와 데이터를 벡터로 만들 모델만 선택하면 자동으로 준비가 되도록 구현했습니다.

**벡터 서치를 위해서는 크게 인덱싱과 리트리벌 단계가 필요합니다.**

| **단계** | **설명** |
| ---- | ---- |
| **인덱싱(Indexing)** | 검색할 문서를 미리 벡터로 변환해 저장하는 과정 |
| **리트리벌(Retrieval)** | 사용자의 질문을 벡터로 변환해 인덱싱해둔 문서를 검색하는 과정 |

#### 인덱싱

✅ **먼저 검색에 사용할 당근 내부 문서들을 수집하고 데이터를 벡터로 변환하기 전 전처리 과정이 필요**
- 백엔드에서 사용자가 선택한 문서에 대해 자동으로 수행
- 다만 중고거래 게시글 등 대량의 데이터 검색이 필요한 경우 해당 서비스팀에서 구축한 벡터 서치 시스템을 사용
#### 리트리벌
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F3b890826-a19e-458f-8fed-5f29ecd283c0%2Fimage.png?table=block&id=366d67b9-9e80-80ed-97cc-ce38fd1afbe1&cache=v2" alt="image" width="1078" height="728" loading="lazy" />

사용자가 질문을 했을 때 RAG 도구를 이용해 인덱싱해둔 정보를 리트리벌
- 사용자가 구어체로 질문한 것을 정책 문서에 대해 검색이 잘되도록 정제하여 재작성
- 유사도 점수순으로 높은 것부터 리트리벌
## 🌟 AI로 맥락에 맞는 UI 제공하기
### 생성형 UI
전통적으로 웹사이트, 모바일 앱의 인터페이스는 디자이나와 개발자가 미리 설계한 **고정된 형태**였습니다.
> **이제 AI를 사용하면 사용자가 원하는 것을 파악해 필요한 UI 요소들을 골라 보여줄 수 있습니다.**


🎯 **당근 AI 에이전트에서도 생성형 UI를 활용하는 것이 필요했습니다.**
독립적이고 자율적으로 복잡한 작업을 처리하는 시스템이라는 에이전트의 정의를 살리려면 에이전트가 단순 답변만 제공하는 것이 아니라 사용자를 위해 액션까지 해야한다고 생각했기 때문입니다.

사용자가 대화하는 중 다양한 표현을 사용해도 AI가 그 의도를 파악해 미리 준비된 UI 컴포넌트중에 필요한 것들만 골라서 조립해 보여줄 수 있습니다.
