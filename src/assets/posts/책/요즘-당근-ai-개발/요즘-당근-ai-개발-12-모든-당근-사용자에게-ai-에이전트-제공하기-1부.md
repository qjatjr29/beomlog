고객 문의를 살펴보면 적절한 가이드만 제공하면 사용자가 스스로 해소할 수 있는 문의가 전체 문의의 약 40%입니다.
그렇다면 어떻게 사용자 스스로 해소하도록 유도할 수 있을까?
## 🔍 자주 묻는 질문을 자주 보지 않는 딜레마
보통 이런 문제 해결을 위한 적절한 가이드는 **자주 묻는 질문(FAQ)**에 있습니다.
🤔 사용자의 입장에서 당근의 **자주 묻는 질문**을 검색하고 읽어보는 일이 과연 쉬울까?
- 우리나라 기준 자주 묻는 질문은 `1,000`개 이상이며 각 서비스 기준 평균 `100`개의 문서가 존재
- 사용자는 이런 많은 문서를 두고 내 질문에 대한 답이 문서 모음에 있는지 판단하고 정확한 답을 얻기는 어려울 것입니다.
## 👋 문의하기로 바로 가는 사람이 하루에 2,000명이라면
고객센터를 찾는 사용자 대부분이 자주 묻는 질문을 검색하거나 읽어보지 않은 채 바로 문의를 합니다.
이런 행동으로 인해 문의 수가 많이 증가합니다. 25년 5월 기준 한국 중고거래 서비스로 하루에 들어오는 문의는 수천 개를 상회합니다.
운영에서는 문의답변 외에도 많은 다른 업무도 처리해야 해야 합니다.
이런 점들을 고려하여 사용자의 요청 사항을 빠르고 만족스럽게 해소하고자 다음과 같은 방식으로 해결해왔습니다.
1️⃣ 고객센터 근무자를 많이 고용
2️⃣ 근무자를 교육해 생산성을 올린다
3️⃣ 많은 부분을 자동화한다.
하지만 당근의 활성 사용자가 더 많아질 때도 고객센터 인원을 무한정 늘릴 수는 없습니다.
사람을 늘리는 방식이 아닌 <span class="text-red">**AI 시대를 맞아 많은 단계를 뛰어 넘어 사용자에게 최적화한 고객 서비스를 제공하는 맞춤형 AI 에이전트를 제공하면 어떨까?**</span>
## 🧑‍💻 AI 에이전트를 소개합니다.
### AI 에이전트란 무엇일까요?
`openAI`에 따르면 <span class="text-red">**AI 에이전트**</span>는 **사용자 대신 높은 수준의 독립성을 가지고 워크플로 작업을 수행하는 시스템**을 말합니다.
이 정의에 따르면 LLM을 사용하지만 워크플로를 제어하지 못한다면 AI 에이전트가 아니라고 할 수 있습니다.
`마이크로스프트`는 **사람을 위한 수동적인 반응에서 벗어나 자율성을 가지고 때로는 독립적으로, 때로는 협력하며 목표를 달성하는 시스템**을 <span class="text-red">**AI 에이전트**</span>라 정의합니다.
`구글`과 `IBM`도 <span class="text-red">**에이전트**</span>는 **사고, 계획, 기억을 가지며 소프트웨어와 시스템을 넘나들며 다단계 작업을 수행**한다 말합니다.

> 💡** 각 회사에서 AI 에이전트를 이야기할 때 조금씩 정의가 다르긴 하지만 공통된 부분이 있습니다.**
> <span class="text-red">**AI 에이전트**</span>는 **사람의 일을 일정 부분 도와주는 수준에서 벗어나 독립적이고 자율적으로 복잡한 작업을 처리하는 시스템**

### AI 에이전트의 구성 

| **구성** | **설명** |
| ---- | ---- |
| **모델(Model)** | 사고와 계획을 진행할 수 있는 높은 수준의 LLM 모델. |
| **도구(Tool)** | 액션을 진행하기 위해 호출할 수 있는 함수 혹은 API  |
| **지시사항(Instructions, Prompt)** | 에이전트의 목표를 설정하고 동작을 제어할 수 있는 가이드 |

#### 🚀 **핵심은 도구 사용 능력!!!**
LLM 모델 자체는 강력한 이해 능력을 갖추고 있지만 결국 훈련된 모델이라는 한계가 있어 실시간성을 가지기 어렵습니다.
또 일반적인 데이터로 훈련된 LLM을 내 도메인에서 잘 동작하게 만들기 위해 대량의 데이터를 LLM에 컨텍스트로 주입하기도 하는 데 그러다보면 모델의 기억 한계를 넘어가 제대로 동작하지 않는 경우도 많습니다.
이 때문에** 필요할 때에만 외부에서 필요한 내용을 검색해 적시에 LLM에 제공하는 작업도 필수**입니다.
#### 멀티 AI 에이전트 시스템
주요 테크 기업들은 모델과 도구 호출, 프롬프트의 통합을 더 쉽게 만드는데 주력합니다.
LLM 모델을 개발하는 것을 넘어 플랫폼으로의 전환을 앞두고 있는 것으로 **단일 AI 에이전트가 하나의 구성 요소**가 되며 **목표 달성에 필요한 여러 개의 에이전트를 만들어 에이전트 간 상호작용을 오케스트레이션**할 수 있도록 돕습니다.
즉, 단일 AI 에이전트를 구현하는 것을 넘어 멀티 AI 에이전트 시스템을 구축할 수 있도록 플랫폼을 제공하는 것입니다.

고객응대에서는 전문 역할을 나누는 것이 중요한데, 사용자의 요청을 처리하기 위해 각 서비스와 운영 정책에 대한 이해, 사용자의 활동 확인 등 방대한 정보를 인지하고 상황에 맞게 액션하는 것이 필요하기 때문입니다.
그래서 사람이 운영하는 고객센터에서도 서비스별 고객센터팀을 분리하거나 문의 응대와 전화 응대를 하는 팀을 별도로 설정하고 전문성을 살릴 수 있는 조직 구조를 고민합니다.

이 구조를 멀티 AI 에이전트 시스템에도 적용할 수 있습니다.

## KAMP에서 에이전트 오케스트레이션 하기
🤔 여러 **AI 에이전트**를 어떻게 협업하도록 만들 수 있을까요?
우리가 협업을 하는 것처럼 AI 에이전트간 협업도 같습니다.
하나의 에이저트가 다른 에이전트를 불러 협업하려면 서로 어떤 전문성을 가지고 있는지 무슨 문제를 해결하는지 인지하도록 만들어야 합니다.
### 멀티 에이전트 협업의 요구사항
#### 1️⃣ 에이전트간 상호 인식
하나의 에이전트가 다른 에이전트와 협업하려면 각 에이전트의 전문성과 해결 가능한 문제 영역을 파악해야 합니다.
> **에이전트 디스크립션**
> - 에이전트 발견을 위해서는 에이전트 디스크립션을 상세히 정의해야 합니다.
> - 해당 에이전트의 역할, 해결 가능한 문제 유형, 사용 방법, 출력 형태 등을 명시

#### 2️⃣ 명확한 소통 체계
에이전트간 협업할 때도 명확한 소통 체계를 만들어주는 것이 성능의 핵심을 차지합니다.
AI 에이전트는 자율적으로 판단하고 행동하므로 예측 불가능한 결과를 초래할 수 있습니다.
이때,** 구조화된 통신을 통해 커뮤니케이션의 기본 구조를 정의**하면 할루시네이션을 최소화하고 예측 가능성을 높여 전체 시스템의 신뢰성을 향상시킬 수 있습니다.
### 입력 스키마
> **각 에이전트의 입력 형식을 API 명세처럼 정의하는 역할**

이를 통해 <u>필수 파라미터 누락</u>이나 <u>잘못된 형식의 데이터 전달</u>을 방지할 수 있습니다.
### 출력 스키마
> **에이전트의 응답을 정해진 형식으로 제한하는 메커니즘**

#### 응답의 일관성
LLM은 본질적으로 확률적 모델으로 같은 질문에도 매번 다른 형태의 응답을 생성할 수 있습니다.
**출력 스키마**는 이런 응답의 변동성을 제어하는 중요한 도구가 됩니다.
- 스키마를 통해 같은 일관된 형식으로 강제할 수 있습니다.
#### 에러 처리
출력 스키마는 에러 처리 측면에서도 중요한 역할을 합니다.
스키마가 없다면 에이전트가 예상치 못한 형식으로 응답했을 때 전체 파이프라인이 중단될 수 있지만 미리 정의된 스키마가 있다면 검증 과정에서 오류를 조기에 발견하고 적절히 대응할 수 있습니다.
