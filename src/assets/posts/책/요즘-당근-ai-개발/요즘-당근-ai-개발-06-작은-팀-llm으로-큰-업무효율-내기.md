## 🔍 끝없는 온콜의 늪, 작은 팀의 절규
PM 1명과 백엔드 개발자 4명으로 구성된 작은 팀으로 사용자의 가입부터 탈퇴까지 과정에 필요한 모든 인증을 담당하고 있습니다.
당근 초창기에는 SMS 인증 방식의 단순한 가입 방식만 제공해 시스템이 단순했고 이슈로 인한 대응 리소스가 큰 문제가 되지 않았습니다.
하지만 국가의 문화와 규정, 사용자의 환경에 맞게 기능이 추가되었고 시스템 복잡도는 증가했습니다.

⚠️ 복잡도가 높아지면서 이슈에 대응하는<span class="text-red">** 온콜 업무**</span>가 팀 전체의 발목을 잡기 시작!!
매주 돌아가며 온콜을 담당하는 시스템은 팀에 큰 부담이 되었고 온콜 담당 주간에는 중요한 개발 작업을 진행하기 어려웠으며 계획했던 업무도 최소로 줄여야 했습니다.
가장 힘든 점은 `언제 어떤 문의가 들어올 지 예측할 수 없다는 것`으로 새벽에도 문의가 들어오고 개발을 하다가도 긴급 대응을 위해 작업을 중단하는 컨텍스트 스위칭은 집중력을 저하시키며 피로도가 쌓여갔습니다.
## 👋 외부 세계와 연결하는 MCP
LLM이 실제 업무에서 강력한 힘을 발휘하기 위해서는 사용하는 시스템과 직접 소통하고 상호작용할 수 있어야 합니다.
<span class="text-blue">**MCP**</span>는 이 LLM과 외부 세계를 잇는 다리 역할을 합니다!

## 🧑‍💻 직접 구축한 MCP 서버를 소개합니다.
LLM이 대신 온콜 업무를 처리하려면 실제 시스템의 데이터에 접근할 수 있어야 합니다.
그렇다고 데이터베이스를 열어줄 수는 없음!!
- 보안 
- LLM이 예상치 못한 동작을 할 위험도 존재
> 🚀 필요한 데이터만 안전하게 제공할 수 있는 MCP 서버들을 구축하기로 결정

### gRPC: 실시간 정보 조회
처음에는 메인 데이터베이스를 그대로 MCP로 연결하는 방안을 검토했지만 너무 위험하다고 판단했습니다.
LLM이 실수로라도 개인 정보에 접근하거나 민감 데이터를 조회할 가능성이 있기 때문.
따라서 `gRPC`를 통한 접근 방식을 선택했습니다.
`gRPC`는 <u>**미리 정의된 인터페이스**</u>를 통해서만 데이터를 주고받기 때문에 LLM이 접근할 수 있는 범위를 명확하게 제한할 수 있었습니다.
- 필요한 정보만 딱 필요한 형태로 제공

`gRPC`는 사용자의 실시간 정보를 조회하는 진입점 역할을 합니다.
- 기본적인 정보를 얻고 더 상세한 분석이 필요하면 빅쿼리나 서버로그 같은 다른 MCP 도구를 활용
- 데이터 조회뿐만 아니라 변경 작업도 gRPC를 통해서만 가능하도록 구현
LLM이 데이터베이스를 직접 수정하는 것보다 훨씬 안전하고 일관된 동작을 보장
### 빅쿼리: 데이터 기반 분석 지원
당근은 **데이터 기반**으로 의사결정을 내리는 문화가 강합니다.
그래서 보안상 문제가 없는 데이터는 빅쿼리에 모아 팀 간에 공유하고 있는데 이 데이터가 너무 방대한 것이 문제입니다.
- 수십 개의 테이블과 수백개의 컬럼
- 각각이 무엇을 의미하는지 파악하기도 어렵고 어떤 맥락에서 만든 데이터인지도 알기 힘듭니다.
- 데이터는 많은데 제대로 활용하지 못하는 상황이 발생

**빅쿼리 MCP 서버**는 이런 문제를 해결해줬습니다.
SQL문법과 DB구조를 모르더라도 자연어로 질문하면 LLM이 알아서 쿼리를 작성하고 실행하고 결과를 분석해 요약해줍니다.
### 센트리: 에러 상황 신속 파악
온콜 업무의 큰 부분을 차지하는 것은 `에러 대응` 부분입니다.
<span class="text-blue">**센트리**</span>는 예상치 못한 서버 에러나 클라이언트 에러가 발생하면 즉시 알림을 보내주는 도구입니다.
- 스택 트레이스를 제공해 어디서 에러가 발생했는지 빠르게 파악 가능
하지만 스택 트레이스만으로는 한계가 존재합니다.
- 왜 그 에러가 발생했는지?
- 어떤 상황에서 일어났는지?
이런 것들을 파악하려면 추가적인 조사가 필요합니다.

센트리 MCP는 이런 조사 과정을 자동화해 에러 알림이 오면 센트리 MCP가 해당 에러의 상세 정보를 가져오고 필요하면 깃허브 MCP나 서버 로그 MCP와 연계해 더 깊이있는 분석을 수행합니다.
### 깃허브: 코드 레벨의 원인 분석
깃허브 MCP는 에러가 발생한 파일의 코드를 가져올 뿐만 아니라 관련된 다른 파일들도 탐색해 문제의 근본 원인을 찾아냅니다.
### 데이터도그: 서비스간 연관성 파악
마이크로서비스 환경에서는 하나의 요청이 여러 서비스를 거쳐가는데 데이터도그는 이런 복잡한 상황을 한눈에 파악하는 도구입니다.
## 🤖 n8n으로 워크플로 자동화하기
MCP 서버들을 구축했지만 각 MCP가 제공하는 정보를 종합하고 복잡한 업무 흐름을 <span class="text-red">**자동화하기 위해서는 이들을 연결하는 도구**</span>가 필요했습니다.
n8n을 도입하여 단순 반복 업무를 자동화하고 복잡한 온콜 대응 프로세스도 체계화할 수 있었습니다.
### n8n의 강력한 통합 기능
**n8n**이 다른 자동화 도구들과 차별화되는 가장 큰 특징은 `통합의 용이성`입니다.
- 일상적으로 사용하는 슬랙, 구글 시트, 지메일은 물론이고 최신 AI 서비스들까지 폭넓게 지원합니다.

🌟 **특히 **<u>**AI 에이전트 기능**</u>**을 중요하게 생각했습니다.**
- chatGPT, 클로드, 제미나이 같은 LLM들을 워크플로에 쉽게 통합할 수 있었고 앞서 구축한 MCP 서버들과도 자연스럽게 연동할 수 있었습니다.
## 실전 사례1: 국가별 가입자 수 변화 분석 리포트
실제 구축한 시스템을 통해서 어떤 성과를 얻었는지 구체적인 사례를 통해 살펴보겠습니다.
첫 번째는 매일 아침 팀원들이 가장 먼저 확인하던** 지표 분석 업무를 자동화**한 이야기입니다.

팀에서 각 국가별 가입자 수와 로그인 지표를 매일 확인해야 했습니다.
매일 아침 출근 시 가장 먼저 빅쿼리를 열어 전날 지표를 확인하는 일을 했습니다.
- 국가별로 신규 가입자 수, 로그인 성공률, 인증 실패 원인 등을 SQL로 조회하여 엑셀에 옮겨 전일 대비 변화를 계산하고 이상 징후가 있는지 분석
이는 매일 30분에서 1시간은 걸렸습니다. 더 큰 문제는 사람이 하다보니 실수가 있다는 점이였습니다.

이 문제를 직접 해결하고자 했습니다 → `n8n을 이용한 신규 가입자 지표 요약 봇`
1️⃣ **매일 오전 9시 자동 실행**: 크론 스케줄러로 정해진 시간에 워크플로 시작
2️⃣ **빅쿼리에서 데이터 추출**: 각 국가별 가입자, 로그인, 인증 관련 지표 조회
3️⃣ **데이터 집계 및 분석**: 전일 대비 변화율 계산, 이상 징후 감지
4️⃣ **LLM으로 인사이트 도출**: 수치의 의미를 해석하고 주목할 만한 변화 포인트 정리
5️⃣ **슬랙으로 결과 전송**: 팀 채널에 보기 좋게 정리된 리포트 발송

이를 통해 팀원들은 반복되전 지표 확인 작업에서 해방되어 더 중요한 개발 업무에 집중할 수 있게 되었습니다.
## 실전 사례2: 에러 실시간 분석 ‘에러 박사’
두 번째로 온콜 담당자들의 큰 고민거리였던 **에러 대응을 자동화**한 사례입니다.

에러가 발생 시 즉시 슬랙으로 알림이 오도록 설정되어 있습니다.
슬랙에 `“rpc error: code = Internal desc = account not found”` 라는 메시지가 보입니다.
- 이 짧은 메시지로는 어떤 일이 일어났는지 알 수가 없습니다.
결국 센트리에 접속하여 에러 상세 정보를 확인하고 서버 로그를 찾아 전후 상황을 파악하고 코드를 열어서 어디서 문제가 발생했는지 추적합니다.
이 과정만 30분에서 1시간은 걸리게 되고 대부분의 경우 긴급한 대응이 필요 없는 일시적인 에러인 경우가 많았습니다.

이 문제를 해결하고자 `에러박사` 자동화 시스템을 만들었습니다. 
🎯 `에러박사` 목표
- **에러의 중요도 자동 판단**: 긴급 대응이 필요한지 아니면 업무 시간에 처리해도 되는지
- **초기 대응 시간 단축**: 에러의 원인과 영향 범위를 빠르게 파악해 대응 방향 제시
### 에러박사의 작동 원리
단순 에러 메시지를 보는 것이 아닌 서버 로그, 코드 등을 활용해 전체적인 맥락을 파악합니다.
**센트리 MCP**에서 센트리 이벤트의 상세 정보를 가져와 에러가 `언제`, `어디서`, `어떤` 상황에서 발생했는지 파악할 수 있습니다.
**깃허브 MCP**로 **에러가 발생한 코드를 직접 확인**합니다. 스택 트레이스에 나온 파일과 라인 번호의 실제 코드를 가져와 문제의 원인을 분석합니다.
**데이터도그 MCP**로는 **에러가 발생한 요청의 전체 흐름을 추적**합니다. 이를 통해 우리 서비스뿐만 아니라 연관된 다른 서비스의 상태도 함께 확인할 수 있습니다.
### 멀티턴 방식의 워크플로
> **멀티턴 (Multi-turn)**
> - 여러 번의 대화를 주고받으며 작업을 수행하는 방식
> - 각 단계에서 중간 결과를 확인하고 다음 단계로 넘어갑니다


처음에는 모든 정보를 한 번에 분석하고자 했으나 LLM에게 너무 많은 정보를 한꺼번에 주면 중요한 내용을 놓치거나 엉뚱한 분석을 하는 경우가 많았습니다.
그래서 단계별로 나누어 처리하는 **멀티턴 방식**을 채택했습니다.
1️⃣ **에러 식별**: **센트리** 에이전트가 슬랙 메시지에서 `issue_id`를 추출하고 기본 정보 수집
2️⃣ **흐름 파악**: **Trace** 에이전트가 **데이터도그**를 통해 요청의 전체 경로와 관련 서비스 확인
3️⃣ **로그 분석**: **서버 로그** MCP로 에러 전후의 상세 로그 확인
4️⃣ **코드 분석**: **깃허브** MCP로 에러 발생 지점의 코드와 관련 파일 분석
5️⃣ **종합 판단**: 솔루션이 모든 정보를 종합해 원인과 해결책 제시
6️⃣ **결과 전달**: 분석 결과를 읽기 쉬운 형태로 정리해 슬랙 댓글로 작성
