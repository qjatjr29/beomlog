`#LLM` `#운영자동화` `#프롬프트엔지니어링`

중고거래 플랫폼에서는 실시간으로 올라오는 수천 개의 게시글, 그 속에 생기는 개인, 단체간 이해관계, 사기 등 모든 것을 실시간으로 감지하고 대응해야 합니다.
이런 반복되는 일을 `LLM`을 활용해 벗어나고자 결심했습니다.
**신고가 접수되면 검토한 후 정책에 따라 판단을 내린 뒤 그에 맞는 사유를 작성하고 제재조치를 취하게 됩니다.**
이는 유사한 신고가 계속 들어오면서 반복되고 축적됩니다.
- 운영팀은 이런 끊임없는 흐름 속에서 일을 합니다.
## 🔍 반복에 매몰된 운영, 자동화를 고민하다.
중고거래 플랫폼의 운영 일은 하다보면 같은 업무가 반복됩니다.
- 신고 유형은 어느정도 정형화되어 있고 정책 판단 기준도 문서화되어 있습니다.
- 응대 문구도 거의 복붙입니다.
매일 비슷한 업무가 반복되면 리소스가 고갈되어 중요한 이슈를 놓칠 수도 급한 일만 처리하느라 시야가 점점 좁아집니다.
- 효율도 낮고 사람도 쉽게 번아웃이 됩니다.
그러다 한 가지 생각이 떠올랐습니다.
> **GPT로 자동화할 수 있지 않을까?**
> 반복적인 업무의 일부는 충분히 자동화할 수 있다는 가능성이 보였습니다.

## 👋 운영자가 만든 첫 AI 도구
자동화를 결심한 후 불법 게시글을 탐지하는 정규식 작업을 먼저 떠올렸습니다.
보통은 사용자가 신고한 게시글을 보고 수작업으로 정규식을 만들어 필터링 조건을 추가했습니다.
이 과정은 은근히 까다롭고 시간이 많이 들었습니다.
그래서 자연어로 `이런 조건의 게시글을 걸러줘` 라고 입력하면 그에 맞는 정규식을 자동으로 만들어주는 도구를 만들어보기로 했습니다.
- 사용자가 문장을 입력시 GPT를 호출해 정규식을 생성해 응답하는 구조
## 🧑‍💻 반복을 이해하고 자동화 설계하기
비개발자가 실제 운영 문제를 해결하기 위해 AI 도구를 설계하는 것은 단순 <span class="text-red">**자동화 구현**</span>을 넘어선 일입니다.
도구들이 어떤 식으로 동작하고 LLM이 어떤 식으로 입력을 받아들여 하는지를 이해하는 것부터 시작했습니다.
이 도구의 목표는 간단했습니다.
- 운영자가 `이런 유형의 게시글은 막고 싶어요`라고 말하면 그걸 정규식으로 변환하는 것
하지만 **GPT는 운영자 언어를 그대로 이해하지 않습니다.**

따라서 다음과 같은 조건으로 프롬프트를 다듬어야 했습니다.
```plain text
조건1. 주석 없이 단일 정규식만 반환
조건2. 운영팀이 사용할 수 있을 정도로 명확하고 짧게
조건3. 실제 운영 문맥을 간략히 요약해 GPT에게 전달
```

이후 프론트엔드에서 실제 게시글 샘플을 넣어 해당 정규식이 어떤 문장에 매칭되는지 테스트할 수 있도록 UI를 구성했습니다.

🤖 이 과정에서 `커서 AI`, `클로드`, `오픈AI` 를 활용했습니다.

| 사용 도구 | 사용 내용 |
| ---- | ---- |
| 커서 AI | GitHub 코드 기반으로 바로 수정 → 실행까지 가능, 예시 코드 수정 후 바로 동작 결과 확인 |
| 클로드 | 정책 요약 후 프롬프트로 변환, 출력결과가 json 형태 |
| 오픈AI GPT | 정규식 생성, 누락 탐지, 유사 문장 제안등 대부분의 처리 |

1. 클로드가 정책 문서 요약
2. 커서 AI가 프롬프트, 프론트, 백엔드 개발
3. 결과를 GPT로 실행
4. 프론트에 결과 적용
## 🤖 프롬프트 작성하기
도구를 만들면서 GPT가 만들어주는 결과가 너무 길거나 범용적이거나 엉뚱한 의미로 해석되는 경우가 많았습니다.
따라서 프롬프트를 다음과 같이 설계했습니다.
1️⃣ **운영자 시점에서 요청**
- 기프티콘 판매 사기 게시글을 정규식으로 막고 싶다.
2️⃣ **조건을 명시적으로 제시**
- 정규식을 짧고 오탐이 적고 운영팀이 실제 적용 가능한 형식으로
3️⃣ **정제된 요청 문장**
- 프롬프트에는 예시와 함께 `다음 문장에 맞는 정규식을 반환해줘. 단, 주석 없이 단일 정규식만` 이라는 문장을 넣었습니다.

> **LLM을 잘 쓰기 위해선 LLM이 뭘 모르는지를 아는게 먼저 **

이런 교훈을 얻고 나서 몇 가지 전략을 마련할 수 있었습니다.
1️⃣ **코드를 새로 짜기보단 조립하기**
- FastAPI, 리액트, 타입스크립트 템플릿을 그대로 복제해 사용
- 커서 AI로 수정과 실행을 하나의 창에서 실행
2️⃣ **LLM 프롬프트 설계에 시간을 더 쓸 것**
- **기능의 70%는 프롬프트 품질에서 결정**
- 응답이 일관되지 않으면 사용성은 바로 무너집니다.
3️⃣ **사용자 입장에서 설계하기**
- 결과는 읽기 쉬운 것이 중요했습니다.
4️⃣ **처음부터 완벽하려 들지 않기**
- **MVP로 시작하고 팀원들이 써보고 피드백을 준다**를 목표로

가장 큰 교훈은 기술보다 <span class="text-blue">**문제 정의**</span>가 더 중요하다는 점이였습니다.
- 어떤 모델, 프레임워크를 사용할지 보다 정말 필요한 기능이 뭔지 파악하는 것이 중요
## 운영팀의 반응과 실제 적용기
실제 운영 환경에 연결하고 팀원들이 직접 써보는 과정까지 가야 진짜 자동화가 시작됩니다.
공유를 하여 팀원 몇 명이 써보기 시작했고 다른 사람들에게도 소개했습니다.
- 팀원들의 피드백을 반영하며 조금씩 개선하는 과정이 진짜 자동화이다.

## 마치며: 완전 자동화를 향한 다음 발걸음
> 기술은 도구일 뿐입니다.

<span class="text-red">**중요한 건 그 도구를 어디에 어떻게 쓸지를 아는 사람입니다.**</span>
**비개발자도 운영자도 문제 정의만 잘하면 AI 시대에 가장 강력한 문제 해결자가 될 수 있다.**

## 🤔 이 챕터를 읽고나서,,, 
**1️⃣ 빠르게 MVP를 만들고 피드백을 통해 개선한다**
이 내용이 이전 챕터에서부터 계속해서 나오고 있다. 처음부터 완벽한 것을 만드려고 하는 것이 아닌 초기 버전을 만들어 팀원들에게 공유하고 이에 대한 반응을 보며 수정해 나가는 과정이 역시나 모든 과정에서 중요하다…!

**2️⃣ 도메인 지식 과 명확한 문제 정의**
비개발자였지만 Cursor와 Claude를 이용해 뚝딱 결과물을 만들어낼 수 있었던 진짜 이유는 운영팀의 업무 흐름(도메인)을 완벽히 이해하고 해결하고자 하는 문제를 정확히 정의했기 때문이라고 생각한다.
결국 AI 시대에서 코딩(기술) 스킬 그 자체보다 훨씬 중요한 것은 **내가 개발하고 있는 제품의 도메인을 깊게 이해하고 사용자가 진짜 불편해하는 문제를 명확히 정의하는 능력**이라는 것을 깨달았다. 
개발자인 나에게 이 역량이 더더욱 필요하겠다!!
