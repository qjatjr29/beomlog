`#LLM` `#중고거래` `#프롬프트엔지니어링` `#벡터DB` `#데이터검색`
LLM을 활용하여 중고거래 게시글에서 스마트폰 정보를 추출하고 이를 통해 시세를 산출한 방법을 소개하고자 합니다.
- LLM으로 게시글을 **분류/정제**하는 과정
- 빅쿼리를 이용해 정보를 **후처리**하고 시세를 **집계**하는 과정
- 벡터 DB 기반으로 유사 게시글을 **추천**하는 과정
## 스마트폰 시세 조회는 왜 필요할까요?
많은 중고거래 판매자들이 물품의 적절한 가격을 결정하는 것을 어려워합니다.
- 다양한 상품이 혼재 - 종류도 다양, 상태도 가지각색
- 따라서 물품의 정확한 시세를 한눈에 파악하기 어려움
중고거래팀은 사용자가 물품의 시세를 한눈에 확인하고 더 쉽게 가격을 결정할 수 있도록 스마트폰 시세 조회 서비스를 테스트하기로 했습니다.
- 구체적인 물품 상태에 따라 시세가 어느 정도인지 파악할 수 있는 서비스
- 머신러닝을 활용해 당근 중고거래 데이터를 기반으로 정확한 시세를 제공한 시도
## 상품 정보 추출하기
가장 큰 문제는 게시글에서 상품 정보를 추출하는 것입니다.
당근은 판매자의 글쓰기 허들을 낮추기 위해서 게시글에 구체적인 기종이나 물품 상태를 입력하도록 요구하지 않습니다.
하지만 구체적인 물품 상태 별 스마트폰 시세를 제공하기 위해서는 다양한 조건을 알아내야 했습니다.
기존에는 이런 데이터를 추출하기 위해서는 복잡한 정규식을 만들거나 ML 모델을 만들었습니다.
하지만 LLM을 도입해 데이터를 추출할 수 있게 되었고 프롬프트 수정만으로도 추출 정확도를 높일 수 있었습니다.
### ↗️ **파이프라인**
- 게시글이 생성될 때마다 LLM을 적용해 빅쿼리에 적재
- 빅쿼리에 적재한 이후 데이터 후처리
## 데이터 기반 시세 집계
위 과정을 통해 원시 데이터를 얻을 수 있었고 이제 이 데이터를 집계하여 시세 정보를 만들어낼 수 있습니다.
- `빅쿼리`와 `MySQL`두 개의 저장소를 사용
- 빅쿼리에서 주간 시세 조회 처리 같은 대용량 작업을 마친 후 집계 결과만을 MySQL로 옮겨 저장
- 사용자가 화면에 진입할 시 MySQL을 활용해 시세 조회 결과를 도출
### 빅쿼리 → MySQL
집계 결과를 빅쿼리에서 MySQL로 옮겨오는 작업을 멱등하게 설계해 운영의 편의성을 높였습니다.
데이터가 중복으로 저장되지 않도록 막는 규칙으로 <u>**고유 인덱스**</u>를 설정하여 `Upsert`방식 작업을 설계해 동일한 작업을 여러 번 반복해도 결과가 달라지지 않는 <span class="text-red">**멱등성**</span>을 확보할 수 있었습니다.
## 유사 게시글 제공
이를 통해 사용자가 원하는 조건의 상품 시세를 구체적인 가격 범위로 제공하게 됐습니다.
그런데 일반적으로 당근에서 물건을 팔기 전 비슷한 물건을 하나하나 확인해보게 됩니다.
불편함을 덜기 위한 목적으로 가격 범위를 제공하지만 더 정확한 가격 책정을 위해 다른 게시글을 직접 확인할 수도 있겠다고 생각해 이 과정도 편리하게 만들기 위해 시세 조회 화면에서 시세 통계 데이터뿐만 아니라 유사 게시글도 제공하기로 했습니다.
### 임베딩
이 유사 게시글 제공 기능은 게시글의 임베딩을 활용해 구현했습니다.
> 📖 **임베딩**
> - 텍스트, 이미지 등의 개체를 수학적인 형태로 바꾸어 표현하는 방법
> - 텍스트의 의미를 수학적으로 잘 변환하기 때문에 의미상으로 유사한 게시글을 빠르게 찾아낼 수 있습니다.

`예시: iPhone 과 아이폰`
이 두 개가 같은 의미라는 것은 단순히 문자열의 유사도로는 알아낼 수 없지만 임베딩 모델을 사용한다면 두 단어는 비슷한 벡터로 변환이 되고 따라서 사용자가 어떤 것으로 검색하든 같은 결과를 제공할 수 있게 됩니다.
➡️ 벡터 저장과 검색에 최적화된 데이터베이스인 벡터 DB중 <span class="text-blue">파인콘</span>을 도입하여 <span class="text-blue">벡터 서빙</span>을 최적화
## 마치며: 더 좋은 경험을 제공하는 것에 진심입니다.
LLM과 임베딩 모델등 새로운 기술을 활용해 당근의 자체 데이터 기반으로 시세 조회 기능을 만들어간 과정을 소개했습니다.
이 과정에서 얻은 교훈은 아래와 같습니다.
1️⃣ LLM이 똑똑하고 좋은 도구지만 모든 과정을 프롬프트 엔지니어링으로 해결하려들기보다는 후처리 과정을 따로 작성하는 게 더 효율적일 때도 있다.
2️⃣ 내가 필요한 장점을 가진 저장소를 선택하면 효율적으로 일할 수 있다.
3️⃣ 내 작업 유형에 잘 맞는 임베딩 모델을 사용하면 문제를 쉽게 풀어낼 수 있다.
## 🤔 이 챕터를 읽고나서,,,
### **1️⃣ 프롬프트와 후처리**
LLM이 모든 예외 상황을 프롬프트 엔지니어링 하나로만 해결하려 들면 모델이 무거워지고 오류가 발생하기 쉽다. 따라서 이번 챕터에서 프롬프트로 1차 추출을 하고 빅쿼리를 통해 후처리를 별도로 진행했다는 점이 인상 깊었다. AI에게 100%를 기대하기보다 AI가 잘하는 80%를 하도록 하고 나머지 20%는 견고한 백엔드나 데이터 로직으로 처리하는 것이 더 좋은 결과가 나올 수 있겠다는 것을 느꼈다.
### **2️⃣ 목적에 맞는 저장소 분리**
대용량 데이터를 집계하는 작업(빅쿼리)과 사용자가 화면에서 빠르게 시세를 조회하는 작업(MySQL)을 분리해 처리한 것이 매우 실무에서 중요하겠구나라고 생각하게 되었습니다. 복잡한 연산은 분석용 DB에서 끝내고, 서비스용 DB에는 그 '결과'만 서빙하는 방식이다. 데이터를 읽고 쓰는 책임을 분리하는 아키텍처가 실제 대규모 서비스에서 어떻게 트래픽과 리소스를 효율적으로 관리하는지 확인할 수 있는 좋은 사례였다.
### **3️⃣ 멱등성 설계의 중요성**
빅쿼리에서 MySQL로 데이터를 옮길 때 동일한 작업을 여러 번 반복해도 결과가 중복되거나 틀어지지 않도록 고유 인덱스 기반의 'Upsert' 방식을 적용했다는 내용이 있었는데 이전 프로젝트에서 멱등성을 중요하게 생각한 지점이 있었는데 정말 실무에서도 중요하게 생각한다는 것을 다시 느꼈습니다. 분산 시스템이나 데이터 파이프라인에서 네트워크 지연이나 재시도로 인해 데이터가 두 번 적재되는 일은 흔하게 발생하기 때문에 시스템의 안정성을 위해서는 멱등성을 보장하는 설계가 선택이 아닌 필수임을 다시 한번 깨달았다.
