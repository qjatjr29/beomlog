# 락
락(Lock)은 **여러 트랜잭션이 동시에 같은 데이터에 접근할 때 일관성을 보장하기 위한 메커니즘**입니다.
## 공유락과 배타락
### 공유락 (Shared Lock, S-Lock)
> **데이터를 읽는 중이니 다른 트랜잭션에서 읽을 수 있지만 수정은 안됌.**

- 여러 트랜잭션이 동시에 공유락 획득 가능 (읽기 + 읽기 = 허용)
- 공유락이 걸린 데이터에 배타락 획득 불가 (읽기 + 쓰기 = 충돌)
- `SELECT ... FOR SHARE`
```sql
SELECT * FROM accounts WHERE id = 1 FOR SHARE;  -- MySQL 8.0 문법
```

### 배타락 (Exclusive Lock, X-Lock)
> **데이터를 수정하는 중이니 다른 트랜잭션은 읽기도 쓰기도 안됌.**

- 하나의 트랜잭션만 배타락 획득 가능
- 배타락이 걸린 데이터에 공유락도 배타락도 획득 불가
- 획득 방법: `SELECT ... FOR UPDATE`, `UPDATE`, `DELETE`, `INSERT`
```sql
-- 배타락 획득
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
UPDATE accounts SET balance = 500 WHERE id = 1;  -- 자동으로 배타락
```

### 의도락 (Intention Lock)
테이블 레벨에서 `테이블의 특정 행에 락을 걸 것이다`라고 미리 선언하는 락.
- 테이블 락과 행 락이 충돌하는지 빠르게 확인하기 위해 사용
- 테이블 전체에 락을 걸기 전 다른 트랜잭션이 해당 테이블의 특정 행을 수정중인지 확인하여 성능을 향상시키고 데드락을 방지

| 의도락 | 의미 |
| ---- | ---- |
| IS (Intention Shared) | 테이블의 특정 행에 공유락을 걸 예정 |
| IX (Intention Exclusive) | 테이블의 특정 행에 배타락을 걸 예정 |

## 락의 범위
**컬럼 락 (Column Lock)**
- 특정 행의 특정 컬럼 하나에만 락을 거는 방식
- 컬럼 단위로 락을 관리할 시 오버헤드가 더 커서 성능이 나빠집니다.
**행 락 (Row Lock)**
- 하나의 행에만 락을 거는 방식
- 동시성이 가장 높고 RDBMS의 기본 락 단위
**페이지 락 (Page Lock)**
- 디스크의 물리적 저장 단위인 하나의 페이지 전체에 락을 거는 방식
**테이블 락 (Table Lock)**
- **테이블 전체에 락을 거는 방식.** 
- 동시성이 가장 낮지만 락 관리 오버헤드가 없다.
- DDL 문(ALTER, DROP)이나 테이블 전체 업데이트 시 사용
## 낙관적 락, 비관적 락
낙관적 락과 비관적 락은 단순히 락을 구현하는 방법이 아니라, **트랜잭션 간의 데이터 충돌을 처리하는 접근 전략**입니다.
### 비관적 락 (Pessimistic Lock)
> **충돌이 발생할 것이라고 가정하고 데이터에 접근하기 전에 미리 락을 설정 **
> - DB 레벨의 락 사용


충돌이 빈번할 것으로 예상해 데이터를 읽는 시점에 미리 <u>**DB 레벨의 배타락**</u>을 획득하는 방법입니다.

🐳 **동작 과정**
1. **데이터 조회 및 락 획득: **트랜잭션이 시작되고 데이터를 읽을 때 즉시 배타락을 겁니다. (ex: `SELECT … FOR UPDATE`)
2. **트랜잭션 대기: **다른 트랜잭션이 동일한 데이터에 접근하고자 할 때 이전 트랜잭션이 끝날 때까지 **대기 상태(Blocking)**됩니다.
3. **작업 수행: **락을 확보한 상태에서 비즈니스 로직을 처리한 후 데이터를 수정(UPDATE)합니다.
4. **커밋 및 락 해제: **작업이 완료되어 커밋되거나 롤백이 되어 트랜잭션이 종료됨과 동시에 락이 해제되고 대기 중이던 다음 트랜잭션이 락을 획득하고 작업을 시작합니다.

✅ **장단점**

| **장점** | **단점** |
| ---- | ---- |
| 데이터 정합성을 강력하게 보장 | 트랜잭션이 길어지면 락 보유 시간이 늘어나 전체적인 시스템 **동시성**과 **처리량**이 크게 저하될 수 있습니다 |
| 충돌 시 애플리케이션 레벨에서 복잡한 재시도 로직을 짤 필요 없음 | 서로 다른 데이터를 잠그고 대기하는 **데드락**발생 위험 존재 |
| 요청이 순차적으로 처리 |  |

🎯 **적합한 상황**
- **충돌 가능성이 높을 때: **여러 사용자가 동시에 같은 데이터를 수정하려는 빈도가 잦은 환경
- **충돌 시 재시도 비용이 클 때 : **로직이 복잡하여 한 번 실패했을 때 다시 처음부터 연산하는 비용이 DB 대기 비용보다 큰 경우.
- **예시**
  - **금융 시스템**: 은행 계좌 이체, 결제 처리 등 정합성이 절대적으로 중요한 도메인.
  - **재고 관리:** 한정 수량 상품의 재고 차감, 선착순 티켓 예매 시스템 등.
### 낙관적 락 (Optimistic Lock)
> 데이터 충돌이 드물 것이라 예상하여 **실제 DB 락을 걸지 않고 진행**하다가 최종적으로 데이터를 수정(Commit)하는 직전에 **충돌 여부를 검사**하는 전략입니다.
> - 애플리케이션 레벨의 버전 관리 사용

DB 레벨의 락이 아닌 **버전(version) 컬럼**이나 **타임스탬프**를 이용해 애플리케이션 레벨에서 구현.
🐳 **동작 과정**
1. **데이터 조회 및 버전 확인:** 데이터를 읽어오면서 레코드의 현재 `version` (또는 timestamp) 값을 함께 가져옵니다. (예: `version = 5`)
2. **락 없이 작업 수행:** DB에 락을 걸지 않은 상태에서 애플리케이션 메모리 상에서 비즈니스 로직을 수행합니다.
3. **데이터 수정 및 충돌 검사:** 수정을 시도할 때 내가 처음에 읽었던 `version`이 DB의 현재 `version`과 일치하는지 확인합니다.
  - `UPDATE ... WHERE id = 1 AND version = 5;`
4. **결과 처리**
  - 데이터가 수정되며 `version`을 1 증가시킵니다.
  - 다른 트랜잭션이 먼저 데이터를 수정해 DB의 버전이 이미 올라간 상태(**충돌 발생**): 트랜잭션을 롤백하고 애플리케이션 레벨에서 예외 처리 및 **재시도**로직을 수행합니다.
✅ **장단점**

| **장점** | **단점** |
| ---- | ---- |
| DB에 락을 걸지 않으므로 트랜잭션 간 대기가 발생하지 않아 동시 처리 성능이 매우 뛰어납니다. | 충돌이 발생했을 때 롤백 처리를 하고 재시도하는 로직을 직접 구현해야 합니다.  |
| 데드락 위험도 거의 없습니다. | 충돌이 빈번한 환경에서는 잦은 재시도로 인해 오히려 성능이 급격히 저하될 수 있습니다. |

🎯 **적합한 상황**
- **충돌 가능성이 낮은 경우:** 데이터 업데이트보다 조회가 압도적으로 많은 환경.
- **락 보유 시간을 최소화해야 하는 경우:** 시스템 전체의 처리량을 극대화해야 하는 서비스
- **예시**
  - 게시글 편집, 위키 문서 수정, 사용자 프로필 업데이트 등 동시에 같은 글을 두 명이 수정할 확률이 낮은 기능
  - 조회수가 매우 높은 일반 상품의 정보 수정 등

# MySQL InnoDB 스토리지 엔진 레벨에서의 Lock
### MySQL InnoDB 락의 계층
```plain text
┌────────────────────────────────────┐
│           Table Lock               │  ← 테이블 전체
├────────────────────────────────────┤
│    Intention Lock (의도 락)         │  ← 테이블 레벨, 하위 락의 의도 표시
├────────────────────────────────────┤
│    Next-Key Lock                   │  ← Record Lock + Gap Lock
│    ├── Record Lock (Row Lock)      │  ← 특정 행(인덱스 레코드)을 잠금
│    └── Gap Lock                    │  ← 행과 행 사이의 빈 공간을 잠금
└────────────────────────────────────┘
```


## Record Lock
> **실제 데이터 행이 아닌 **<u>**인덱스 레코드 하나**</u>**에 걸리는 락**

InnoDB는 데이터를 잠글 때 행(Row) 자체가 아니라 `인덱스(Index)`를 잠급니다.

### 💡 **인덱스 레코드**
- 데이터를 빠르게 찾기 위해 **[정렬된 키 값 + 실제 데이터의 위치(또는 기본키 값)]** 형태로 별도의 공간에 저장된 데이터 조각
- InnoDB에서는 모든 테이블이 기본키(PK)를 기준으로 정렬된 클러스터링 인덱스(B-Tree)구조로 저장됩니다.
- 즉, 인덱스 레코드란 데이터를 탐색하기 위해 B-Tree 구조에 저장되어 있는 각각의 데이터 항목을 말합니다.

### 🤔** 왜 레코드 자체가 아닌 인덱스 레코드에 락을 거는 것일까?!**
- InnoDB의 설계 철학과 동시성 처리 방식 때문
1️⃣** B-Tree 탐색 구조**
InnoDB에서 데이터를 읽거나 수정하기 위해서는 B-Tree 구조의 인덱스를 타고 내려가면서 데이터를 찾아야 합니다.
원하는 데이터를 찾아가는 길목에 있는 인덱스 레코드(노드)들을 하나씩 거치게 되는데 이 탐색 과정에서 발견된 인덱스 레코드에 락을 걸어야만 `지금 이 데이터를 찾아서 작업 중이니, 다른 트랜잭션은 이 경로의 데이터나 그 주변(Gap)을 건드리지 마라`라고 확실하게 통제할 수 있습니다.

2️⃣ **갭 락(Gap Lock) 구현**
InnoDB는 여러 트랜잭션이 동시에 실행될 때 발생하는 팬텀 리드 현상을 막기 위해 데이터가 있는 곳뿐만 아니라 **데이터가 없는 빈 공간(Gap)**에도 다른 트랜잭션이 데이터를 끼워 넣지 못하게 락을 걸어야 합니다(넥스트 키 락).
만약 인덱스가 아닌 실제 데이터(Row) 자체에만 락을 건다면 <u>**데이터가 없는 공간**</u>에는 락을 걸 물리적인 실체가 없습니다. 
하지만 인덱스라는 <u>**정렬된 자료구조**</u>를 기준으로 락을 걸게 되면 **"10번 인덱스 레코드와 20번 인덱스 레코드 사이의 공간"**이라는 기준이 생겨 갭 락을 구현할 수 있게 됩니다.

3️⃣** 락 관리의 효율성**
물리적인 디스크에 흩어져 있는 데이터 자체를 찾아가서 락을 관리하는 것보다 메모리(버퍼 풀)에 올라와 있는 정렬된 인덱스 구조에서 인덱스 레코드의 헤더에 있는 락 비트를 조작하는 것이 메모리 효율적이고 탐색 및 락 관리가 훨씬 빠릅니다.

### 🤔 **인덱스가 없는 컬럼으로 WHERE 조건을 걸면?**
InnoDB는 어떤 행을 잠가야 할지 인덱스로 특정할 수 없어 테이블을 처음부터 끝까지 뒤지는 **Full Table Scan**을 수행합니다.
이때 스캔하는 **모든 행의 인덱스에 락**을 겁니다.
이는 테이블 락과 같은 상황으로 실무에서 주의해야할 부분입니다.
```sql
-- 나쁜 예: name에 인덱스 없음 → 전체 행 락
SELECT * FROM accounts WHERE name = 'Alice' FOR UPDATE;

-- 좋은 예: id는 PK → id=1 행만 락
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
```


## Gap Lock
> 새로운 데이터가 끼어드는 것을 막기 위해 **빈 공간**을 잠그는 락

실제로 존재하는 데이터 레코드가 아니라 **행과 행 사이의 빈 공간(Gap)**을 잠그는 락입니다. 
- 동일한 트랜잭션 내에서 같은 쿼리를 두 번 실행했을 때 다른 트랜잭션이 그 사이에 데이터를 INSERT하여 결과가 달라지는 현상(**Phantom Read**)을 방지하기 위함

### 특징
- **Repeatable Read 격리 수준 이상에서만 동작.**
- Gap Lock끼리는 서로 충돌하지 않고 두 트랜잭션이 같은 Gap에 Gap Lock을 동시에 보유 가능
- Gap Lock이 걸린 구간에 **INSERT Intention Lock**이 충돌하여 INSERT가 블로킹

## Next-Key Lock
> Record Lock과 그 레코드 앞의 Gap Lock을 합친 형태

InnoDB가 행을 잠글 때 사용하는 기본 락 단위입니다.
- `FOR UPDATE`, `UPDATE`, `DELETE` 실행 시 기본으로 적용된다.
- 레코드 자체를 잠가서 수정, 삭제를 막고 동시에 그 앞 공간을 잠가서 새로운 데이터가 삽입되는 것도 막습니다. (Phantom Read)

✅ `id: 1, 5, 10` 이 있을 때 각 레코드의 **Next-Key Lock 범위**
```plain text
(-∞,  1]  ← id=1의 Next-Key Lock (Gap(-∞~1) + Record(1))
( 1,  5]  ← id=5의 Next-Key Lock (Gap(1~5)  + Record(5))
( 5, 10]  ← id=10의 Next-Key Lock (Gap(5~10) + Record(10))
(10, +∞)  ← 마지막 이후의 Gap Lock (Supremum pseudo-record)
```

### **예외 규칙**
`PK`나 `Unique 인덱스`로 <u>**단일 행**</u>을 정확히 조회할 때는 Gap Lock이 생략되고 **Record Lock**
만 걸립니다.
- “어차피 값은 중복될 수 없으므로 이 조건에 부합하는 레코드는 1개뿐이다” 라고 판단하여 주변 간격(Gap)을 잠글 필요가 없음.
## Insert Intention Lock
> INSERT 직전에 획득하는 특수한 형태의 갭 락

나는 이 빈 공간의 특정 위치에 데이터를 넣을 것이라 라고 알리는 락입니다.
일반 Gap Lock과 달리 Insert Intention Lock끼리는 삽입하려는 정확한 위치(PK 값 등)가 다르다면 서로 충돌하지 않습니다.
덕분에 여러 INSERT가 같은 Gap에서 동시에 진행해 성능을 높여줍니다.
# 👨‍💻 MySQL 락 실습 
### ⚙️ 환경 세팅
```sql
-- 테이블 생성 (id: PK, name: 인덱스 없음)
CREATE TABLE accounts (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    balance INT
);

-- 초기 데이터 삽입 (id가 1, 5, 10 인 간격이 있는 데이터)
INSERT INTO accounts (id, name, balance) VALUES 
(1, 'Beomsic', 1000), 
(5, 'kbs', 1000), 
(10, 'beomseok', 1000);
```

### FOR UPDATE 사용 이유
InnoDB의 특징 중 하나는 데이터를 읽을 때 기본적으로 락을 걸지 않는다(Non-Locking Consistent Read)는 점입니다. 다른 트랜잭션이 데이터를 수정 중이더라도 `MVCC` 기술을 통해 수정되기 전의 스냅샷을 읽어옵니다.
따라서 단순히 `SELECT`만 해서는 락이 발생하지 않아 테스트를 할 수 없습니다. **직접 눈으로 락을 관찰하고 다른 트랜잭션이 대기(Blocking)하는 충돌 상황을 의도적으로 살펴보기 위해** 데이터를 읽으면서 동시에 배타락(Exclusive Lock)을 쥐는 `FOR UPDATE`를 사용합니다.
## 🧪 PK 단일 행 조회(Record Lock 확인)
### 🎯** 목적**
기본키(PK)나 유니크 인덱스로 **정확히 한 건의 데이터**만 조회할 때 InnoDB가 불필요한 빈 공간(Gap)은 잠그지 않고 **정확히 해당 행(인덱스 레코드)만 잠근다는 것**을 확인
- PK를 이용한 접근이 동시성 환경에서 얼마나 효율적이고 다른 작업에 방해를 주지 않는지 확인
### 👨‍💻 실습
1️⃣ **세션1: PK를 이용해 특정 행 선점**
```sql
START TRANSACTION;
-- id가 5인 데이터를 수정할 목적(FOR UPDATE)으로 락을 획득합니다.
SELECT * FROM accounts WHERE id = 5 FOR UPDATE;
-- (트랜잭션 유지)
```


2️⃣ **세션2: 어떤 락이 걸렸는지 확인해보기**
```sql
SELECT INDEX_NAME, LOCK_TYPE, LOCK_MODE, LOCK_STATUS, LOCK_DATA 
FROM performance_schema.data_locks 
WHERE OBJECT_NAME = 'accounts';
```

## 🧪 범위 조회 (Next-Key Lock 과 Gap Lock)
### 🎯 **목적**
범위를 조건으로 검색했을 때 존재하는 데이터뿐만 아니라 데이터와 데이터 사이의 빈 공간(Gap)까지 잠가버리는 현상(Next-Key Lock)을 확인
- 범위 검색 중에는 다른 트랜잭션이 그 범위 안에 새로운 데이터를 `INSERT` 할 수 없습니다. 
- 이를 통해 `Phantom Read`을 DB가 어떻게 막아내는지 눈으로 확인
### 👨‍💻 실습
1️⃣ **세션1: 범위에 락걸기 **
```sql
COMMIT; -- 이전 실습 종료
START TRANSACTION;
-- 4~7 사이의 공간 전체에 락을 획득합니다. (현재 존재하는 데이터는 5뿐)
SELECT * FROM accounts WHERE id BETWEEN 4 AND 7 FOR UPDATE;
```

2️⃣ **세션2: 락 확인 및 빈 공간에 INSERT 시도**
```sql
-- 락 확인: LOCK_MODE에 'X'와 'X, GAP'이 나타납니다. 
-- 즉, 레코드 락과 갭 락이 동시에 걸렸음을 증명합니다.
SELECT INDEX_NAME, LOCK_TYPE, LOCK_MODE, LOCK_STATUS, LOCK_DATA 
FROM performance_schema.data_locks 
WHERE OBJECT_NAME = 'accounts';

-- 빈 공간(id=4)에 새로운 데이터 삽입 시도
INSERT INTO accounts (id, name, balance) VALUES (4, 'Dave', 1000);
```

## 🧪 인덱스 없는 컬럼 조건으로 조회 (Full Table Lock)
### 🎯 목적
인덱스가 없는 컬럼을 `WHERE` 조건으로 사용하여 `FOR UPDATE`를 걸었을 때 InnoDB가 **테이블 전체를 스캔하며 모든 행과 빈 공간에 락을 걸어버리는 상황** 확인
### 👨‍💻 실습
1️⃣ **세션1: 인덱스 없는 name 컬럼으로 조회**
```sql
COMMIT; -- 이전 실습 종료
START TRANSACTION;
-- name 컬럼은 인덱스가 없으므로, InnoDB는 'Bob'을 찾기 위해 테이블 전체를 뒤집니다.
SELECT * FROM accounts WHERE name = 'Bob' FOR UPDATE;
```


2️⃣ **세션2: 상관없는 데이터 업데이트 해보기**
```sql
-- 아무 관계 없는 id=10 의 데이터를 수정 시도해봅니다.
UPDATE accounts SET balance = 500 WHERE id = 10;
```
