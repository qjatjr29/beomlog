## 💡핵심 개념 요약
<span class="text-red">**추상 클래스(Abstract Class)**</span>는 클래스간 `IS-A`관계를 바탕으로 **공통된 상태(필드)와 행위(메소드)**을 추상화하여 재사용성과 확장성을 제공해줍니다.
<span class="text-blue">**인터페이스(Interface)**</span>는 클래스가 `CAN-DO`를 명시하는 것으로 서로 관련 없는 클래스들이라도 **동일한 기능(행위)을 수행할 수 있도록 보장하는 명세서(계약) **역할을 하며 시스템의 결합도를 낮추고 다형성을 제공해줍니다.
## 🔍 인터페이스와 추상클래스
### 추상 클래스(Abstract Class)
`abstract` 키워드로 선언하며 클래스 간의 `IS-A (~은 ~이다)` 관계를 구축하는 것에 초점을 둡니다.
- **상태 보존**: 일반 변수(인스턴스 필드)를 가질 수 있으므로 객체의 상태를 저장하고 관리할 수 있습니다.
- **초기화**: 생성자를 가질 수 있습니다. 하위 클래스는 `super()`를 통해 상위 클래스의 생성자를 호출하여 공통 상태를 안전하게 초기화합니다.
- 구현된 일반 메소드와 하위 클래스에 구현을 강제하는 추상 메소드를 동시에 가질 수 있습니다.
- **단일 상속**: Java의 클래스 상속 규칙에 따라 **다중 상속이 불가능**하며 오직 하나의 추상 클래스만 상속받을 수 있습니다.
- **인스턴스화 불가**: `new` 연산자를 통한 직접 생성이 불가능합니다.
### 인터페이스(Interface)
`interface` 키워드로 선언하며 구현체가 어떤 기능을 제공해야 하는지 `CAN-DO (~을 할 수 있다)`를 명시합니다.
- **인스턴스화 불가**: 역시 `new` 연산자로 직접 생성할 수 없습니다.
- **다중 구현**: 클래스는 여러 개의 인터페이스를 동시에 `implements` 할 수 있으며, 인터페이스끼리도 다중 상속(`extends`)이 가능합니다.
- 인스턴스 필드를 가질 수 없습니다. 오직 상수(`public static final`)만 허용됩니다.
### 핵심 비교

| **구분** | **추상 클래스 (Abstract Class)** | **인터페이스 (Interface)** |
| ---- | ---- | ---- |
| **관계** | **IS-A** (개는 동물이다) | **CAN-DO** (오리는 수영할 수 있다) |
| **키워드** | `abstract class` / `extends` | `interface` / `implements` |
| **다중 상속** | ❌ **불가능** (단일 상속만 허용) | ✅ **가능** (다중 구현 및 다중 상속 지원) |
| **상태(필드)** | 인스턴스 변수, 상수 등 모든 필드 선언 가능 | 오직 상수(`public static final`)만 가능 |
| **생성자** | ✅ **있음** (`super()`로 호출) | ❌ **없음** (상태가 없으므로 초기화할 필요 없음) |
| **기본 메소드** | 일반 메소드, 추상 메소드 모두 가능 | 추상 메소드 (Java 8+부터 default/static 허용) |
| **설계 목적** | 관련된 하위 클래스들의 **공통 상태와 로직 공유** | 서로 다른 클래스들의 **표준화된 통신 규약(API) 정의** |

## 🤔 언제 무엇을 쓸까?
### 추상 클래스
- 공통된 상태를 공유해야 할 때
- 생성자를 통해 상태 초기화가 필요할 때
- 추상 클래스는 상속할 객체들의 공통점을 찾아 추상화시켜 놓은 것으로 같은 부모 클래스를 상속하며 부모 클래스가 가진 기능들을 구현해야할 경우 사용한다.(`IS-A`)
### 인터페이스
- 서로 관련 없는 클래스들에 추가적인 동일한 기능을 부여할 때(`CAN-DO`)
- 다중 구현이 필요할 때
## 🧑‍💻 실제 예시
### 추상 클래스
```java
abstract class Animal {
    protected String name; // 상태(필드)를 가질 수 있음
    
    Animal(String name) { this.name = name; } // 생성자로 상태 초기화
    
    public String getName() { return name; } // 구현된 공통 로직
    
    abstract void sound(); // 하위 클래스에서 반드시 구현해야 할 추상 로직
    abstract void move();
}

public class Dog extends Animal {
    public Dog(String name) {
        super(name);
    }

    @Override
    public void sound() { System.out.println("멍멍!"); }

    @Override
    public void move() { System.out.println("네 발로 달린다") }
}
```

### 인터페이스
```java
interface Flyable  { void fly(); }
interface Swimmable { void swim(); }

class Duck extends Animal implements Flyable, Swimmable {
    Duck(String name) { super(name); } // 부모 추상 클래스의 생성자 호출
    
    @Override public void sound() { System.out.println("꽥꽥!"); }
    @Override public void move()  { System.out.println("두 발로 달린다") }
    @Override public void fly()   { System.out.println("하늘을 난다"); }
    @Override public void swim()  { System.out.println("물갈퀴로 수영한다"); }
}
```

## Java 인터페이스 변화 (Java 8+)
과거 인터페이스는 오직 추상 메소드만 허용이 되었었는데 현재는 `default`, `static`, `private` 메소드가 도입이 되었습니다.
### Java 8: `default` 메소드 도입 - API 하위 호환성 보장
전 세계 수백만 개의 프로젝트가 자바의 `List`인터페이스를 구현하여 사용하고 있는 상황에서 `List`에 새로운 기능인 `sort()`를 추가하고 싶어졌습니다.
```java
public interface List<E> extends SequencedCollection<E> {
    void sort(); // 🚨 추상 메소드를 추가하는 순간 문제 발생!!!
}
```

추상 메소드가 추가되면 기존에 `List`를 구현해 두었던 전 세계의 모든 클래스(`ArrayList`, `LinkedList`, 사용자가 직접 만든 커스텀 리스트 등)에서 **구현체가 없어 컴파일 에러**가 발생합니다.
인터페이스 내부에 `default` 키워드를 붙여 기본 구현을 제공하도록 했습니다.
이를 통해 기존 클래스들을 단 한 줄도 수정하지 않고(하위 호환성을 유지하며) 새로운 기능을 API에 추가할 수 있게 되었습니다.
```java
public interface List<E> extends SequencedCollection<E> {
    default void sort(Comparator<? super E> c) {
        //...
    }
}
```

### Java 8: `static` 메소드 도입 - 유틸리티 클래스의 파편화 해결
과거에는 인터페이스와 관련된 "정적 유틸리티 기능"을 제공하기 위해 별도의 **유틸리티 클래스**를 만들어야 했습니다.
예를 들어 `Collection` 인터페이스를 다루기 위해 `Collections` 라는 유틸리티 클래스가 별도로 존재했고 이로 인해 API가 파편화되었습니다.
```java
public interface Collection<E> extends Iterable<E> {
    int size();
    boolean isEmpty();
    // 구현부가 있는 static 메서드는 절대 들어올 수 없었음!
}

// Collections (유틸리티 클래스)
public class Collections {
    // ⚠️ 객체 생성을 막기 위해 생성자를 private으로 숨김 (유틸리티 클래스의 특징)
    private Collections() {
    }

    // Collection을 다루기 위한 정적 메서드들이 여기에 다 모여있음
    public static <T extends Comparable<? super T>> void sort(List<T> list) {
        list.sort(null);
    }
    
    public static <T> List<T> unmodifiableList(List<? extends T> list) {
        return (list instanceof RandomAccess ? ... );
    }
}
```

`Java 8`부터는 인터페이스가 `static` 메서드를 가질 수 있게 되면서 새로운 API들은 더 이상 **유틸리티** 클래스를 만들지 않게 되었습니다.
대표적인 예가 `Java 8`에 도입된 <span class="text-red">**Stream API**</span>입니다. 
- `Streams`라는 클래스는 존재하지 않습니다.
```java
public interface Stream<T> extends BaseStream<T, Stream<T>> {
    
    // 추상 메서드들
    Stream<T> filter(Predicate<? super T> predicate);
    
    // 💡 인터페이스 내부에 직접 static으로 구현!
    // Streams.of() 가 아니라 Stream.of() 로 바로 호출
    public static<T> Stream<T> of(T... values) {
        return Arrays.stream(values);
    }
    
    public static<T> Stream<T> empty() {
        return StreamSupport.stream(Spliterators.<T>emptySpliterator(), false);
    }
}
```

`Collections` 클래스 같은 경우는 하위 호환성을 위해 그대로 존재
- 기존 것은 놔두고 앞으로 새로 만드는 것에 대해서는 해당 방식을 사용하도록 가이드라인을 바꾼 것.
### Java 9: `private` 메소드의 도입 - 캡슐화와 중복 코드 제거
`Java 8`에서 `default` 메소드를 쓰다 보니 여러 `default` 메소드 간에 **중복된 로직**이 발생했습니다.
```java
public interface Payment {
    default void payByCreditCard() {
        System.out.println("결제 로그 저장"); // 중복 1
        System.out.println("신용카드 결제");
    }
    
    default void payByPoint() {
        System.out.println("결제 로그 저장"); // 중복 2
        System.out.println("포인트 결제");
    }
}
```

코드를 작성하다보면 중복 코드가 발생하게 되고 중복을 제거하기 위해 새로운 메소드를 만들면 문제가 발생합니다.
```java
public interface Payment {
    default void payByCreditCard() {
        commonLog(); // 중복 제거를 위해 추출한 메서드 호출
        System.out.println("신용카드 결제");
    }
    
    // 🚨 문제 발생: 자바 8에서는 모든 메서드가 무조건 public!!
    // 내부에서만 쓰려고 만든 헬퍼 메서드가 외부로 노출되어 버림.
    default void commonLog() { 
        System.out.println("결제 로그 저장");
    }
}
```

위 예제에서 중복을 제거하기 위해 새로운 `commonLog()`와 같은 메소드를 만들면 이 메소드가 `public`으로 외부에 노출이 됩니다.
따라서 인터페이스를 사용하는 클라이언트(외부 클래스)가 마음대로 호출해버릴 수 있게 되고 이는 내부 구현 로직이 외부로 노출되어 캡슐화 붕괴(보안 이슈)가 발생합니다.
오직 인터페이스 내부의 `default` 메소드에서만 호출할 수 있는 `private` 메소드를 도입하여 내부 구현을 완벽히 숨긴 채 중복 코드를 제거했습니다.
```java
public interface Payment {
    default void payByCreditCard() {
        commonLog(); // private 헬퍼 메소드 호출
        System.out.println("신용카드 결제");
    }
    
    default void payByPoint() {
        commonLog(); // private 헬퍼 메소드 호출
        System.out.println("포인트 결제");
    }
    
    // 외부로 노출되지 않는 인터페이스 내부 전용 로직!
    private void commonLog() { 
        System.out.println("결제 로그 저장");
    }
}
```
