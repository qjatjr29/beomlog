## 블로킹(Blocking) vs 논블로킹(Non-blocking)
**`제어권`**의 관점입니다. 
- 호출된 함수가 바로 리턴을 하는가?

### 블로킹
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F2dcdbf3a-316b-44e9-9e03-b5d1456d97da%2Fimage.png?table=block&id=341d67b9-9e80-8078-8626-f67a14324186&cache=v2" alt="image" width="1560" height="1000" loading="lazy" />

호출된 함수가 자신의 일을 끝낼 때까지 제어권을 계속 가지고 있습니다.
호출자는 리턴이 올 때까지 아무것도 하지 못하고 기다려야 합니다.
### 논블로킹
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2Fffb41372-eeb3-4231-a306-4b996fe69eb7%2Fimage.png?table=block&id=341d67b9-9e80-80b2-9f4a-d73d84e064f0&cache=v2" alt="image" width="1246" height="858" loading="lazy" />

호출된 함수가 작업을 다 못 끝냈더라도 일단 바로 리턴합니다.
호출자는 제어권을 즉시 돌려받아 다른 작업을 할 수 있습니다.
## 동기(Synchronous) vs 비동기(Asynchronous)
**`작업 완료를 누가 관리하는가`**에 대한 관점입니다.
### 동기
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F7f80fdf8-926c-4427-87e7-318d789dc677%2Fimage.png?table=block&id=341d67b9-9e80-8055-b810-cba91b5f63e9&cache=v2" alt="image" width="1364" height="816" loading="lazy" />

작업을 요청한 쪽이 완료 여부를 **직접** 챙깁니다.
즉, 호출자가 결과를 확인하고 그 순서에 맞춰 다음 로직을 진행합니다.
### 비동기
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F725adefd-56ec-4bab-bcaf-ed46e7f4895e%2Fimage.png?table=block&id=341d67b9-9e80-8072-acac-e50e79e1acb5&cache=v2" alt="image" width="1312" height="962" loading="lazy" />

작업을 요청 한 뒤 호출자는 결과에 신경쓰지 않고 다른 일을 합니다.
작업이 완료되면 운영체제나 호출된 함수가 콜백이나 이벤트를 통해 결과를 알려줍니다.
## 🚀 4가지 조합
이 두 가지가 결합하여 실무에서 마주하는 4가지 모델이 생길 수 있습니다.
### 1️⃣ 동기 + 블로킹
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F4737590f-47da-4660-8cab-44a7599c901b%2Fimage.png?table=block&id=341d67b9-9e80-80ff-85bb-e93658ad8853&cache=v2" alt="image" width="2272" height="692" loading="lazy" />

가장 전통적이고 직관적인 방식입니다.
요청을 보낸 뒤 끝날 때까지 호출자가 기다리며 완료 여부도 호출자가 직접 챙깁니다.

📌 **특징**
- 이해하기 쉽습니다.
- 코드 흐름이 순차적이라 디버깅이 쉽습니다.
- 하지만 I/O 동안 스레드가 놀게 되어 자원 효율이 떨어집니다.
### 2️⃣ 동기 + 논블로킹
함수는 바로 리턴하는데 완료 여부는 호출자가 계속 직접 확인합니다.
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2Fb58dc3ec-7ad9-4b47-ae9f-ad75eee1360e%2Fimage.png?table=block&id=341d67b9-9e80-8020-9678-c7453e7d1132&cache=v2" alt="image" width="2266" height="1116" loading="lazy" />

즉, 기다리면서 멈추지는 않지만 끝났는지 계속해서 체크를 합니다(주기적 체크, `폴링`)
📌 **특징**
- 제어권은 빨리 돌아옵니다.
- 하지만 완료 확인을 계속해야 하므로 호출자 로직이 복잡해집니다.
- 잘못 구현시 `폴링` 때문에 CPU 낭비가 발생합니다.
### 3️⃣ 비동기 + 논블로킹
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2Fd8c65190-5815-4d69-b59f-3ea7eb27f9d4%2Fimage.png?table=block&id=341d67b9-9e80-809b-b95a-c0aa2631892c&cache=v2" alt="image" width="2266" height="1278" loading="lazy" />

호출 직후 제어권을 돌려받고 결과는 나중에 이벤트나 콜백으로 전달받습니다.
📌 **특징**
- 적은 수의 스레드로 많은 동시 요청을 처리하기 좋습니다.
- I/O 대기 시간 동안 스레드를 묶어두지 않습니다.
- 이벤트 루프, 콜백, Promise, Future, Reactor 모델과 잘 연결됩니다.

🤖 **예시**
- Node.js 이벤트 루프 기반 I/O
- Nginx의 이벤트 기반 처리
- Spring WebFlux
- epoll, kqueue 같은 이벤트 통지 기반 모델 위의 서버 구조
### 4️⃣ 비동기 + 블로킹
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F01ea3daf-b235-4b56-b4df-ad4bad2926ef%2Fimage.png?table=block&id=341d67b9-9e80-80bd-b02f-dec0d837d0e6&cache=v2" alt="image" width="2254" height="886" loading="lazy" />

개념적으로 가능. 완료는 비동기적으로 통지받도록 설계했는데 정작 호출자가 중간에 멈춰 기다리면 결과적으로 blocking이 됩니다.

👀 **예시**
- 비동기 API를 호출했지만 곧바로 결과나 나올 때까지 기다리도록 하는 경우
- 이벤트 기반 시스템 안에서 내부적으로 blocking DB 드라이버를 호출하는 경우

📌 **특징**
- 비동기 구조의 장점을 거의 살리지 못합니다.
- 코드만 복잡해지고 성능 이점은 사라질 수 있습니다.
- 실무에서는 안티패턴으로 자주 언급됩니다.

## 🔍 궁금증
### 🙋 비동기 + 논블로킹이 좋아보이는데 이것만 사용하면 되는거 아닌가?
`비동기 + 논블로킹`은 높은 동시성과 I/O 효율이 중요한 환경에서 매우 효율적이지만 모든 시스템에 무조건 적합한 방법은 아닙니다.
`비동기+논블로킹`은 주로 `I/O-bound` 환경에서 큰 효과를 냅니다. 
- 네트워크 응답 대기, 디스크 대기, 외부 API 대기 처럼 CPU가 계산하는 시간보다 기다리는 시간이 더 긴 경우에는 스레드를 묶어두지 않는 것이 큰 이점이 됩니다.
반대로 `CPU-bound` 작업에서는 이야기가 다릅니다. 결국 CPU가 직접 계산을 끝내야 하므로 비동기로 바꾼다고 해서 처리 시간이 줄어드는 것은 아닙니다. 
오히려 작업 분산, 콜백 연결, 컨텍스트 전환, 상태 관리 비용만 늘어나 코드가 더 복잡해질 수 있습니다.
또한 `동기 + 블로킹` 코드는 읽기 쉽고 디버깅이 단순합니다. 호출 순서가 눈에 잘 보이고 스택 트레이스도 자연스러워서 작은 서비스나 단순 CRUD 서버에서는 오히려 생산성이 더 좋습니다.

### 🙋 비동기 + 논블로킹** 시스템에서 동기 블로킹 호출을 하면?!?**
`비동기 + 논블로킹` 모델은 I/O 대기 시간 동안 스레드를 쉬게 두지 않고 다른 요청을 계속 처리할 수 있도록 설계된 구조입니다. 
즉, **적은 수의 스레드로 높은 동시성을 확보하는 것이 핵심!!**
하지만 이 구조 안에 동기식 외부 API 호출이나 블로킹 DB 드라이버 같은 요소가 끼어들면 해당 스레드는 응답이 올 때까지 반환되지 못하고 묶이게 됩니다. 이 순간부터 그 스레드는 비동기 시스템의 장점을 전혀 활용하지 못하는 상태가 됩니다.
이렇게 블로킹되는 요청이 늘어나면 가용 스레드 수가 부족해지고 처리되지 못한 요청은 큐에 쌓입니다. **이는 평균 응답 시간이 증가하고 타임아웃과 재시도가 발생하며 결국 시스템 전체 처리량이 급격히 떨어질 수 있는 문제가 됩니다.**
