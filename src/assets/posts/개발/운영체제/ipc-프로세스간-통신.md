프로세스 운영체제로부터 자신만의 **독립적인 메모리공간(주소 공간)**을 가지고 있습니다.
따라서 서로 데이터를 주고 받기 위해서는 `IPC(Inter Process Communication)`방식을 이용해야 합니다.
## 📖 IPC (Inter Process Communication)
### 🤔 IPC는 왜 필요할까요?
우리가 사용하는 프로그램들은 여러 프로세스가 협력해야 하는 경우가 많습니다.
- **데이터 공유**
  - ex) 웹 브라우저의 다운로드 프로세스와 화면 렌더링 프로세스가 파일 상태를 공유하는 경우
- **작업 분산**
  - ex) 많은 양의 계산을 여러 프로세스로 쪼개서 병렬로 처리하고 결과를 모으는 경우

따라서 이렇게 독립적인 메모리를 가진 프로세스들이 서로 데이터를 주고받기 위해서는 IPC 가 필요합니다.

### **IPC의 2가지 방법**

| **방법** | **설명** |
| ---- | ---- |
| **Shared Memory** | 프로세스들이 공통 메모리 영역을 직접 읽고 쓰는 방법 |
| **Message Passing** | 커널을 통해 메시지를 전달하는 방법 |

## **🤝 **Shared Memory (공유 메모리)
프로세스들이 통신할 수 있도록 공통으로 사용할 수 있는 메모리 영역을 할당하는 방식입니다.
- 프로세스가 커널에 시스템 콜을 통해 공유 메모리 할당을 요청하면 커널이 특정 메모리 영역을 두 프로세스 주소 공간에 공통으로 매핑을 해줍니다.
- 그 이후로는 커널의 개입 없이 자신의 메모리에 값을 읽고 쓰듯 직접 접근하여 사용합니다.

### ✅ 장 단점

| **장점** | **단점** |
| ---- | ---- |
| **속도가 빠르다** - 일반 메모리 접근과 동일한 속도 | ⚠️** 동기화 문제 **- 여러 프로세스가 동시 접근시 데이터 훼손 위험 |
| **큰 데이터** 처리시 복사할 필요가 없고 참조만으로 통신  | **복잡성** - 직접 접근 제어를 구현해야 함 |

만약 프로세스 A가 공유 메모리에 데이터를 쓰는 도중에 프로세스 B가 데이터를 읽어가거나 데이터를 덮어 쓰는 상황이 생긴다면 데이터가 섞이는 `Race Condition` 문제가 발생할 수 있습니다.
따라서 공유 메모리를 사용하기 위해서는 <span class="text-red">**동기화 기법**</span>이 반드시 필요합니다!
- `뮤텍스`
- `세마포어`

### 🧵 잠깐! 스레드는?
스레드는 **하나의 프로세스 안에서 스택(Stack) 영역만 독립적으로 가지고 코드(Code), 데이터(Data), 힙(Heap) 영역은 서로 공유**합니다.
> **💡 즉, 스레드는 별도의 IPC를 구축할 필요 없이 너무나 쉽게 데이터를 공유할 수 있습니다!**


하지만 메모리를 공유한다는 것은 동일하기 때문에 공유 메모리 방식이 겪는 **`동기화 문제(Race Condition)`****를 스레드도 똑같이 겪게 됩니다.** 
따라서 멀티 스레드 프로그래밍에서도 <span class="text-red">**동기화**</span> 처리가 필요합니다.

## ✉️ Message Passing (메시지 패싱)
프로세스들이 메모리를 직접 공유하지 않고 OS(커널)을 중재자로 삼아 메시지를 송수신하는 방식입니다.
- `Send` 와 `Receive` 시스템 콜을 사용합니다.

- `Direct Communication`: 메시지를 받을 프로세스의 이름을 명시적으로 표시
- `Indirect Communication`: 메시지를 받을 프로세스의 이름을 명시적으로 표시하지 않고 mailbox를 통해 간접적으로 전달
- 두 가지 방식 모두 커널을 통해서 전달하는 것은 같습니다.

### ✅ 장 단점

| **장점** | **단점** |
| ---- | ---- |
| **안전** - 커널이 통신을 통제해 동기화 문제를 신경 쓸 필요가 없음 | **속도 저하** - 매번 시스템 콜(커널 모드 전환)과 메모리 복사가 발생 |
| **구현 편리**: 충돌 위험이 없고 코드가 직관적 | 대용량 데이터를 보내기에는 오버헤드가 큼 |


### ▶️ 메시지 패싱의 방식 - 파이프(Pipe)
`단방향 통신`
- 한 프로세스는 쓰기만 다른 프로세스는 읽기만 가능
- 양방향 통신을 위해서는 파이프 2개가 필요

**예시**
```bash
ls | grep txt
```

- `|` 가 파이프를 의미
- `ls`** 결과를 **`grep`**에 입력으로 전달 = 프로세스간 데이터 전달**

### 💬 메시지 큐 
운영체제 메모리 공간에 만들어진 `Queue)`에 메시지를 던져 넣는 방식입니다.
**큐의 이름(key)**만 알면 전혀 모르는 프로세스들끼리도 통신을 할 수 있습니다.
받는 프로세스가 당장 없더라도 큐에 메시지를 쌓아둘 수 있으며 메시지에 번호를 붙여 수신자가 큐에 쌓은 메시지 중 원하는 특정 메시지만 가져갈 수 있습니다.

## 💡 공유 메모리 - 메시지 패싱 요약

| 비교 항목 | 🤝 공유 메모리 (Shared Memory) | ✉️ 메시지 패싱 (Message Passing) |
| ---- | ---- | ---- |
| **통신 매개체** | 프로세스 간 매핑된 공통 메모리 | 운영체제 (커널) |
| **데이터 전송 속도** | **매우 빠름** | 느림 (시스템 콜, 오버헤드 발생) |
| **동기화 문제** | **발생함 (개발자가 직접 처리해야 함)** | 발생 안 함 (OS가 알아서 처리) |
| **적합한 상황** | 대용량 데이터 공유, 높은 성능 필요 시 | 적은 양의 데이터, 안전성/편의성 우선 시 |
