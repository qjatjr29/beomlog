## 📒 파일과 파일시스템
### File
> **A named collection of related information**
> - 관련 정보를 이름을 통해서 저장하는 것.

일반적으로 하드디스크 같은 비휘발성 보조기억장치에 저장되며 운영체제는 다양한 저장 장치를 `파일`이라는 동일한 단위로 볼 수 있게 해줍니다.

### **File Attribute (Metadata)**
파일 이름, 유형, 저장 위치, 크기, 접근 권한(읽기/쓰기/실행), 시간 정보(생성/변경 등), 소유자 등 파일을 관리하기 위한 모든 정보입니다.

### File System
운영체제 내에서 파일, 메타데이터, 디렉토리 정보를 관리하고 파일의 저장 방법 및 보호 기법을 결정하는 부분.

### Directory 
해당 디렉토리에 속한 파일들의 이름 및 메타데이터 일부를 보관하는 일종의 특별한 <span class="text-red">**파일**</span>.

### Partition (= Logical Disk)
물리적 디스크를 논리적으로 나눈 단위(예: C드라이브, D드라이브)
- 하나의 (물리적) 디스크안에 여러 파티션을 두는 것은 일반적
- 여러개의 물리적인 디스크를 하나의 파티션으로 구성하기도 합니다.
- 각각의 파티션에 파일 시스템을 설치하거나 스와핑(Swapping) 용도로 사용 가능

## 파일 접근 연산과 버퍼 캐싱
`open()` 시스템 콜은 파일의 데이터를 직접 읽는 것이 아니라 **디스크에 있는 파일의 메타데이터를 메모리로 가져오는 작업 **
```bash
open("/a/b/c")
→ 디스크로부터 파일 c의 메타데이터를 메모리로 가지고 오는 작업
→ 이를 위해 directory path를 search
```

1. `open("/a/b/c")` 호출 시 루트 디렉토리(`/`)부터 탐색하여 `c` 파일의 메타데이터 위치를 찾습니다.
2. 운영체제는 각 프로세스의 'Open File Table' 배열에 해당 메타데이터의 포인터를 저장하고 이 배열의 **인덱스(File Descriptor)**를 사용자 프로세스에게 반환합니다.
3. 이후 `read`/`write` 시에는 디스크를 다시 탐색할 필요 없이 파일 디스크립터만으로 요청
  - 디스크를 다시 루트부터 뒤질 필요 없이 `fd` 배열 인덱스를 타고 곧바로 메모리에 올라와 있는 메타데이터 위치 정보로 찾아가 실제 데이터 블록을 읽어오는 것이 가능

### **버퍼 캐싱**
디스크에서 읽어온 파일 내용을 운영체제의 메모리 영역(버퍼 캐시)에 먼저 저장한 뒤 사용자 프로그램에 전달합니다.
- 동일한 데이터 요청 시 디스크까지 가지 않고 캐시에서 즉시 반환

**모든 I/O 요청은 시스템 콜을 통해 운영체제에게 CPU 제어권**이 넘어가므로 페이징 시스템(클럭 알고리즘 사용)과 달리 버퍼 캐싱에서는 **LRU, LFU 알고리즘**을 사용 가능.

## 🔒 파일 보호(File Protection)
각 파일에 대해 누구에게 어떤 접근 권한(Read/Write/Execution)을 허락할지 제어하는 방법


| 제어 기법 | 설명 | 특징 및 단점 |
| ---- | ---- | ---- |
| **Access Control Matrix** | 행렬을 이용해 사용자와 파일 간의 권한을 매핑 | 희소 행렬이 되어 공간 낭비 발생. (보완: Access Control List, Capability) |
| **Grouping** | 사용자를 소유자, 동일 그룹, 외부 사용자의 3그룹으로 분류 | 일반적인 운영체제(UNIX 등)에서 주로 사용. 단 9개의 bit만으로 권한 표현 가능 |
| **Password** | 파일/권한별로 비밀번호를 부여 | 파일이 많아질 경우 관리 및 암기 문제 발생 |

### Access Control Matrix
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F68303538-b2bd-4af8-bc3a-7a28b851581e%2Fimage.png?table=block&id=33bd67b9-9e80-8007-a3ff-eac054127534&cache=v2" alt="image" width="1694" height="554" loading="lazy" />


`희소 행렬`
- 특정 사용자가 자신만 사용하기 위해서 만들어놓은 파일은 다른 사용자에게 접근권한이 없지만 행렬의 칸을 만들어 낭비

**링크드리스트 형태로 만드는 형식**

| <span class="text-red">**Access Control List**</span> | 파일별로 누구에게 어떤 접근 권한이 있는지 표시 | 파일을 주체로 파일에 접근권한이 있는 사용자들을 링크드리스트로 연결 |
| ---- | ---- | ---- |
| <span class="text-blue">**Capability**</span> | 사용자별로 자신이 접근 권한을 가진 파일 및 해당 권한 표시 | 사용자를 주체로 접근권한이 있는 파일들을 링크드리스트로 연결 |


### Grouping
일반적인 운영체제에서는 그룹핑 방법을 통해서 접근권한을 제어

<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F6a063903-a41f-4320-ae0f-d7246d606744%2Fimage.png?table=block&id=33bd67b9-9e80-8051-bb59-f7e98fd00b20&cache=v2" alt="image" width="470" height="548" loading="lazy" />

**그룹핑은 모든 사용자에 대해서 접근권한을 다루는 것이 아닌 사용자 그룹을 3가지로 나눕니다.**
- 그 파일의 `소유주` 에 대해서 권한(read, write, execution)이 있는지
- 해당 소유자와 동일 `그룹`에 속한 사용자에 대해서 권한이 있는지 표시
- 나머지 전체 외부 사용자에 대해서 권한이 있는지 표시

총 <span class="text-red">**9개의 bit**</span>만 사용하여 파일 하나에 대한 접근 권한을 나타낼 수 있음.

### Password
파일마다 Password를 두는 방법으로 모든 접근 권한에 대해 하나의 password를 사용합니다.
- password가 많아져 관리가 어려움

## ➡️ 파일 할당 방법(File Allocation)
디스크는 동일한 크기의 논리적 블록(Sector 단위)으로 나뉘어 관리합니다.
이를 파일에 할당하는 3가지 방법
- Contiguous Allocation
- Linked Allocation
- Indexed Allocation

### Contiguous Allocation (연속 할당)
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F3922f3b3-93e5-4ccd-abaa-1e363e038db5%2Fimage.png?table=block&id=33bd67b9-9e80-80b1-8506-cb8f90d66497&cache=v2" alt="image" width="1190" height="1102" loading="lazy" />

하나의 파일을 디스크 상에 연속해서 저장합니다.

**장점**
HDD와 같은 저장장치는 데이터 전송 시간보다 헤드 이동 시간(seek)과 회전 지연 시간(rotation)이 큰 비중을 차지하기 때문에 블록이 연속되어 있으면 한 번의 seek 이후 많은 데이터를 연속적으로 읽을 수 있어 순차 접근 성능이 매우 좋음
또한 파일의 시작 블록 번호와 길이만 알고 있으면 원하는 블록의 위치를 쉽게 계산할 수 있으므로 직접 접근도 비교적 단순합니다.
- 구조 자체는 단순하면서도 읽기 성능이 우수

**단점**
연속 할당은 파일들이 생성되고 삭제되는 과정에서 디스크 중간중간에 사용하지 못하는 빈 공간들이 흩어져 생기는 **외부 단편화** 문제가 발생합니다.
또한 파일 크기가 증가했을 때 바로 **뒤쪽 블록이 비어 있지 않으면 현재 위치에서 파일을 확장할 수 없어** 더 큰 연속 공간으로 파일 전체를 옮겨야 하는 문제가 발생합니다.

이 문제를 줄이기 위해 처음부터 파일이 더 커질 것을 예상하고 여유 공간을 미리 확보해놓는 방법을 사용할 수 있지만 실제로 사용하지 않는 공간까지 미리 잡아두게 되므로 공간 활용 측면에서는 비효율이 발생할 수 있습니다.

이러한 특성 때문에 연속 할당은 일반적인 범용 파일 시스템 전체보다는, 빠른 순차 I/O가 특히 중요한 영역에 더 적합합니다.
- 대표적으로 swap area처럼 대용량 데이터를 빠르게 내리고 다시 올려야 하는 경우 **공간 효율성보다 속도가 더 중요**하므로 연속 할당의 장점이 부각

### Linked Allocation (연결 할당)
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F6fa70c8b-b08a-40fa-9bf6-b0e986b55700%2Fimage.png?table=block&id=33bd67b9-9e80-80eb-a36d-e1376e276f83&cache=v2" alt="image" width="1294" height="1296" loading="lazy" />

연결 할당은 파일의 블록들을 디스크의 빈 공간 어디에나 저장할 수 있도록 하고 각 블록이 다음 블록의 위치를 가리키는 포인터를 가지는 방식입니다. 즉, 파일 전체가 한 위치에 연속해서 저장될 필요는 없고 첫 번째 블록만 알면 나머지는 포인터를 따라가며 찾을 수 있습니다.

**장점: 연속된 빈 공간이 필요하지 않다**
파일을 저장할 때 디스크 곳곳의 빈 블록을 활용할 수 있으므로 연속 할당에서 문제였던 **외부 단편화**가 해결됩니다.
또한 파일이 커졌을 때도 새로운 빈 블록 하나를 찾아 뒤에 연결하면 되므로 파일 크기 증가에 대응 가능합니다.
- 공간 활용 측면에서는 연속 할당보다 유리

**단점 - 직접 접근이 어려움, 신뢰성**
예를 들어 파일의 4번째 블록을 읽고 싶더라도 디렉토리는 보통 첫 번째 블록의 위치만 알고 있으므로 1번째, 2번째, 3번째 블록을 순서대로 따라간 뒤에야 4번째 블록에 도달할 수 있습니다. 
즉, 디스크 장치 자체는 랜덤 액세스를 지원하더라도 파일 관리 방식이 연결 구조이면 파일 내부에서는 사실상 **순차 접근**을 해야 합니다.
다른 단점은 **신뢰성** 문제로 중간 블록의 포인터 정보가 손상되면 그 뒤의 나머지 블록들을 찾을 수 없게 되어 파일의 상당 부분을 잃어버릴 수 있습니다. 
또한 각 블록의 일부 공간을 다음 블록 위치를 저장하는 데 사용해야 하므로 순수 데이터 저장 공간이 줄어드는 비효율도 있습니다.

이러한 단점을 보완하기 위한 대표적인 방식이 **`FAT(File Allocation Table)`**입니다. 
`FAT`는 연결 할당의 아이디어를 유지하면서도 각 블록 내부에 포인터를 저장하는 대신 별도의 테이블에 다음 블록 정보를 모아 관리함으로써 공간 효율과 관리 편의성을 개선한 방식입니다.

### Indexed Allocation (인덱스 할당)
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2Fbc447646-05c3-4e85-a8b2-bb358e0a9f29%2Fimage.png?table=block&id=33bd67b9-9e80-8088-8921-eebc9c1117dc&cache=v2" alt="image" width="1326" height="1210" loading="lazy" />

인덱스 할당은 파일마다 별도의 **인덱스 블록**을 두고 그 안에 해당 파일이 사용하는 데이터 블록들의 위치 정보를 배열처럼 모아 저장하는 방식입니다. 
- 디렉토리는 데이터 블록들의 위치를 직접 들고 있는 대신 이 인덱스 블록의 위치를 가리킴

**장점**
예를 들어 파일의 4번째 블록을 읽고 싶다면 앞의 블록들을 차례대로 따라갈 필요 없이 인덱스 블록에서 네 번째 항목만 확인하면 바로 해당 데이터 블록에 접근 가능. (**직접 접근**)
- linked allocation과 달리 랜덤 액세스에 유리
또한 데이터 블록 자체는 디스크 상에서 흩어져 있어도 되므로 연속 할당처럼 큰 연속 공간이 필요하지 않아 **외부 단편화 문제**도 없습니다. 

**단점**
작은 파일에는 비효율이 생길 수 있습니다. 실제 데이터는 몇 블록 쓰지 않더라도 인덱스 블록 하나를 별도로 할당해야 해 파일 크기가 작을수록 상대적으로 공간 낭비가 커질 수 있습니다.
또한 파일이 매우 커지면 하나의 인덱스 블록만으로는 모든 데이터 블록의 주소를 담기 어려운 문제가 발생합니다. 
이를 해결하기 위해 인덱스 블록을 다시 연결하는 방식이나, single indirect, double indirect, triple indirect 같은 다단계 인덱스 구조를 사용할 수 있습니다.

## 💡 **실제 파일 시스템의 구현**
### 1️⃣ UNIX 파일 시스템 구조
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F9efd76ae-60d9-4596-81e3-80ff218c7fc0%2Fimage.png?table=block&id=33bd67b9-9e80-8014-9501-c4d6f1f24971&cache=v2" alt="image" width="1040" height="598" loading="lazy" />


유닉스의 파일시스템의 저장되는 구조는 크게 **4가지 영역**으로 나뉩니다.
1. **Boot block**
  - 부팅 시 운영체제 커널의 위치를 찾아 메모리에 적재하는 역할을 하며 어떤 파일 시스템이든 항상 파티션의 맨 앞에 위치합니다.
2. **Super block**
  - 파일 시스템의 전체적인 정보를 관리합니다. 
  - 전체 데이터 블록의 개수, 비어있는 블록의 위치, inode list와 실제 Data block의 경계 등
3. **Inode list**
  - 각 파일마다 고유하게 하나씩 부여되는 `inode`라는 자료구조가 모여있는 영역
  - 파일의 메타데이터(소유자, 권한, 생성 시간 등)가 여기에 저장
  - **파일의 이름은 ****`inode`****에 없고 디렉토리가 (파일 이름 - inode 번호) 형태로 관리**
4. **Data block**
  - 파일의 실제 데이터 내용이 저장되는 영역

### 2️⃣ FAT 파일 시스템
> **Windows 계열에서 FAT 파일시스템을 사용하는 경우가 존재**

**FAT 시스템**은 **연결 할당(Linked Allocation)의 문제**를 개선한 구조
파일의 메타데이터 중 핵심인 이름, 권한, 사이즈, 그리고 **첫 번째 데이터 블록의 위치**는 디렉토리가 보관
**다음 블록의 위치 정보(포인터)**를 각 데이터 블록 내부가 아니라 <span class="text-blue">**FAT(File Allocation Table)**</span>**이라는 별도의 배열 테이블에 따로 모아 저장**
FAT 테이블의 크기는 전체 데이터 블록의 개수와 동일하며 테이블 내의 인덱스가 현재 블록 번호를, 배열의 값이 다음 블록 번호를 의미합니다. 
FAT 테이블 전체를 메모리에 올려두면 디스크 헤드를 이동할 필요 없이 메모리 상에서 포인터를 따라가며 위치를 즉시 계산할 수 있어 **순차 접근의 한계를 극복하고 직접 접근(Random Access)이 가능**해집니다
- 포인터 유실로 인한 신뢰성 문제도 FAT를 이중 백업하여 해결

## **빈 공간 관리 (Free-Space Management)**
디스크 상의 비어있는 블록을 효율적으로 찾고 관리하기 위해 다음과 같은 기법들이 사용됩니다.

**Bit map / Bit vector**
- 블록 1개당 1비트를 할당해 0이면 빈 블록, 1이면 사용 중으로 표시
- 연속적인 빈 공간(연속 할당 시)을 찾는데 효과적
- 부가적인 비트맵 저장 공간(배열)이 필요(단점)

**Linked list**
- 비어있는 블록들을 포인터로 연결해 두는 방식
- 공간 낭비는 없지만 새로운 빈 블록을 찾기 위해 찾는 과정이 필요해 속도가 느려 현실적으로 거의 쓰이지 않습니다.

**Grouping**
- Linked list의 변형으로 첫 번째 빈 블록이 다른 빈 블록 `n-1`개의 위치(포인터)를 배열처럼 들고 있고 마지막 포인터가 다시 다음 빈 블록들의 첫 번째를 가리키게 하는 방식

**Counting**
- 연속된 빈 블록을 하나씩 기록하지 않고 빈 공간의 시작 블록 위치와 거기서부터 연속해서 몇 개가 비어있는지를 `쌍`으로 묶어 관리

## 📂 Directory 구현
디렉토리란 그 안에 들어있는 파일과 하위 디렉토리들의 메타데이터(또는 메타데이터의 위치 포인터)를 관리하는 특별한 <span class="text-red">**파일**</span>입니다.
1️⃣ **Linear List**
`파일 이름 - 메타데이터` 쌍을 고정된 크기로 순차적으로 나열하는 방식입니다. 
구현이 단순하지만 특정 파일을 찾으려면 최악의 경우 리스트 전체를 순차 탐색해야 하므로 속도가 느립니다.

2️⃣ **Hash Table**
파일 이름에 **해시 함수**를 적용하여 메타데이터 위치를 바로 찾아가는 방식입니다. 
순차 탐색이 필요 없어 속도가 매우 빠르지만 해시 충돌(Hash Collision) 문제가 발생할 수 있습니다.

**긴 파일 이름 지원**
- 디렉토리 엔트리는 보통 고정 크기를 가지는데 파일 이름이 길어지면 엔트리 내부에 이름의 일부만 적고 뒷부분은 별도의 공간에 저장한 뒤 그 포인터를 기록하는 방식을 사용

## VFS 와 NFS
### VFS (Virtual File System)
> 다양한 **file system**에 대해 동일한 시스템 콜 인터페이스(API)를 통해 접근할 수 있게 해주는 OS의 layer


파일시스템은 많은 종류가 있습니다.
- UNIX 에서도 여러 종류의 파일시스템이 존재
사용자가 파일시스템을 이용하기 위해서는 <span class="text-red">시스템 콜</span>을 이용해야 하는데  이때, 파일시스템의 종류별로 서로 다른 시스템 콜 인터페이스를 사용해야 한다면 사용자가 혼란스러울 수 있습니다.
**VFS**는 사용자가 파일 시스템의 종류에 관계없이 동일한 시스템 콜 인터페이스, API(예: `open`, `read`, `write`)를 사용해 접근할 수 있도록 묶어주는 **운영체제 내부의 가상 추상화 계층**입니다.

### NFS (Network File System)
> 분산 시스템에서는 네트워크를 통해 파일이 공유될 수 있음.
> - NFS는 분산 환경에서의 대표적인 파일 공유 방법

분산 환경에서 네트워크를 통해 다른 컴퓨터의 파일을 마치 내 로컬 디스크에 있는 파일인 것처럼 사용할 수 있게 해주는 대표적인 파일 공유 프로토콜입니다.

## Page Cache 와 Buffer Cache
운영체제는 성능 향상을 위해 메모리의 일부를 캐시로 사용하는데 접근하는 목적과 영역에 따라 두 가지로 나뉘었습니다
### Page Cache
가상 메모리 관점에서 프로세스의 주소 공간을 구성하는** 페이지 프레임**을 캐싱합니다 (디스크의 Swap area와 연동).

### Buffer Cache
파일 시스템 관점에서 디스크에서 읽어온 논리적 파일 데이터 블록을 운영체제가 캐싱

### **Unified Buffer Cache**
최근의 운영체제(Linux 등)는 이 두 가지를 분리하지 않고 통합하여 관리합니다. 즉, 버퍼 캐시 역시 4KB 페이지 단위로 관리하여 공간을 유동적으로 분배
- 과거에는 파일 데이터를 읽을 때 OS가 버퍼 캐시로 데이터를 읽어오고 이를 다시 사용자 프로세스의 페이지 캐시(메모리 영역)로 복사하는 오버헤드가 있었습니다.
- Unified Buffer Cache 환경에서는 버퍼 캐시에 올라온 데이터를 다시 복사할 필요 없이 해당 페이지 자체를 사용자 프로세스의 논리적 가상 주소 영역에 그대로 매핑하여 성능 향상
