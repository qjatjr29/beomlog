제가 개발한 프로그램이 어떻게 물리적인 RAM에 올라가고 8GB밖에 안 되는 RAM으로 어떻게 10GB짜리 게임을 돌릴 수 있는지등을 확인하기 위해서 OS의 메모리 관리에 대해서 학습해보고자 합니다. 백엔드 개발자로서 메모리 누수(Memory Leak)나 성능 저하를 추적할 때 꼭 필요할 것이라 생각합니다.
## 📌 **주소 바인딩과 단편화**
### **논리적 주소와 물리적 주소**

| **논리적 주소 (Logical address, virtual address)** | 프로세스마다 독립적으로 가지는 주소 공간. | 각 프로세스마다 0번지부터 시작. **CPU가 보는 주소는 논리적 주소** |
| ---- | ---- | ---- |
| **물리적 주소 (Physical address)** | 메모리에 실제 올라가는 위치 |  |

프로세스는 자신이 0번지부터 메모리를 갖는다고 느낍니다. 하지만 실제로는 여러 프로세스가 하나의 RAM을 나눠 쓰므로, CPU가 낸 논리적 주소를 실제 물리적 주소로 바꿔야 합니다.

> 📖 **Symbolic Address**
> 프로그래머가 프로그램을 만들때는 숫자 주소를 통해서 프로그래밍하지 않습니다.
> - 이름 기반 주소 (변수명, 함수명)
> - 컴파일과 링크 과정을 통해서 가상 주소로 구체화

### **주소 바인딩**
```bash
Symbolic Address → Logical Address → Physical address
```

주소 바인딩은 프로그램의 주소를 실제 메모리 위치와 연결하는 과정입니다. 

<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2Fce6516f3-505d-446e-ad31-28bd990e333c%2Fimage.png?table=block&id=323d67b9-9e80-80c3-aa73-d6c97d4e0d7d&cache=v2" alt="image" width="1378" height="870" loading="lazy" />

이 주소 변환이 언제 결정되느냐에 따라 보통 3가지로 나눕니다.

| 방식 | 설명 |
| ---- | ---- |
| **Compile time binding** | 메모리 위치가 고정되어 있으면 절대 주소를 생성 |
| **Load time binding** | 프로그램이 메모리에 올라갈 때 주소를 결정 |
| **Execution time(=Run time) binding** | 실행 중에도 물리 위치가 바뀔 수 있고 하드웨어(MMU)가 주소 변환을 담당 |

1️⃣ **Compile time binding**
컴파일 시점에 프로그램이 메모리의 어느 위치에 올라갈지가 **이미 정해져 있는 경우** 컴파일러는 그 위치를 기준으로 **절대 코드(absolute code)**를 생성
- 프로그램은 정해진 주소에만 올라갈 수 있습니다.
- 다른 메모리 공간이 비어 있어도 지정된 위치가 아니면 그대로 실행할 수 없습니다.
- 메모리 시작 위치가 바뀌면 다시 컴파일
예전에 한 번에 프로그램 하나만 실행되는 환경에서 이런 방식이 사용

2️⃣ **Load time binding**
**프로그램이 메모리에 올라가는 순간** 실제 위치를 결정하는 방식
컴파일러는 보통 재배치 가능한 코드(relocatable code)를 만들어 둡니다. 그리고 로더가 메모리의 빈 공간을 보고 그 위치에 맞게 결정

3️⃣ **Run time binding**
**실행 중**에도 프로세스의 메모리 위치가 바뀔 수 있는 방식
- 실행 중에도 프로세스를 다른 물리 메모리 위치로 옮길 수 있습니다.
- 가상 메모리 사용이 가능합니다.
- 페이징, 페이지 교체, 스와핑 같은 현대 메모리 관리 기법이 가능
- **CPU가 주소를 참조할 때마다 MMU가 논리 주소를 물리 주소로 동적으로 변환**

✅ 현대 OS는 **Run time binding**을 사용
> 현대 OS는 메모리가 꽉 차면 안 쓰는 프로세스의 메모리를 잠시 하드디스크(Swap)로 빼고, 나중에 다시 RAM으로 부르는 작업(스와핑/가상 메모리)을 쉴 새 없이 합니다. 이렇게 메모리 위치가 실행 중에도 휙휙 바뀌어야 하므로, 접근할 때마다 주소를 계산해 주는 **Run Time Binding**과 이를 아주 빠르게 처리해 주는 하드웨어인 **MMU**가 반드시 필요

프로세스를 유연하게 옮길 수 있고 가상 메모리를 쓸 수 있기 때문
## 🤖 **MMU (Memory Management Unit)**
**CPU가 만든 논리적 주소를 물리적 주소로 변환(매핑)해주는 하드웨어입니다.**
- 주소 변환뿐만 아니라 프로세스가 허가되지 않은 메모리 영역을 침범하지 못하도록 막는 **메모리 보호(Memory Protection)** 역할도 수행

### **MMU의 두 가지 레지스터**
MMU는 **두 개의 레지스터**를 사용하여 주소 변환과 메모리 보호를 처리합니다.
1️⃣ **Base Register (Relocation Register)**
- 현재 실행 중인 프로세스가 실제 물리적 메모리(RAM)에 적재된 **시작 주소**를 담고 있습니다.
- CPU가 <u>**0번지**</u>부터 시작하는 논리적 주소를 요청하면 MMU는 이 논리적 주소에 **Base Register**의 값을 더하여 **실제 물리적 주소를 계산**
*(ex: Base Register가 300,000이고 논리 주소 50번지를 요청하면 실제 물리 주소 300,050번지에 접근)*

2️⃣ **Limit Register**
- 해당 프로세스에 할당된 논리적 주소의 **최대 크기(범위)**를 담고 있습니다.
- CPU가 주소를 요청할 때 가장 먼저 확인하여 요청된 논리적 주소가 **Limit Register**의 값보다 크거나 같다면 <u>**다른 프로세스의 영역을 침범하려는 불법적인 접근(Illegal Address)**</u>으로 간주합니다. 
- 이 경우 `예외(Trap)`를 발생시켜 운영체제로 제어권을 넘기고 해당 프로세스를 강제 종료

### **MMU의 동작 과정**
CPU가 데이터를 요청했을 때 MMU 내부에서는 다음과 같은 검증과 변환이 일어납니다.
1. CPU가 특정 논리적 주소의 데이터 접근을 시도
2. MMU가 요청된 주소와 `Limit Register`의 값을 비교하여 범위를 초과하면 Trap(오류)을 발생
3. 안전한 접근임이 확인되면 논리적 주소에 `Base Register`값을 더함
4. 계산된 최종 물리적 주소를 통해 RAM에서 데이터를 읽거나 씁니다

## Swapping
다중 프로그래밍 환경에서 모든 프로세스를 메인 메모리(RAM)에 올려둘 수는 없습니다. 
시스템의 메모리가 부족해지면 **당장 실행되지 않는 프로세스나 중요도가 떨어지는 프로세스를 일시적으로 메모리에서 쫓아내어 디스크로 보내는 기법**을 `Swapping`이라고 합니다.

📖 **용어 정리**
- **Swap Area (Backing Store):** 쫓겨난 프로세스들이 머무는 하드디스크 내의 공간
- **Swap Out:** 메모리에 있던 프로세스를 디스크의 <u>Swap Area</u>로 쫓아내는 과정
- **Swap In:** 디스크의 <u>Swap Area</u>에 있던 프로세스를 다시 메모리로 불러오는 과정

👀 **Swapping의 주체 (중기 스케줄러)**
누구를 쫓아낼지 결정하는 것은 **중기 스케줄러(Medium-term Scheduler)**가 처리합니다. 
주로 오랫동안 잠들어 있거나(Sleep) I/O 작업을 기다리고 있는 프로세스들이 최우선 `Swap Out` 대상이 됩니다.

🔗 **Swapping과 Run time Binding**
프로세스가 `Swap Out` 되었다가 나중에 다시 `Swap In` 될 때 **반드시 예전과 똑같은 메모리 위치(물리 주소)로 돌아온다는 보장이 없습니다**. 
- 빈 공간이 다른 곳에 생겼다면 그곳으로 들어가야 합니다.
따라서, 실행 중에도 메모리의 적재 위치가 유연하게 바뀔 수 있도록 주소를 매번 새롭게 계산해 주는 **Run time Binding** 이 **Swapping** 환경에 가장 적합합니다.
## 📌 연속 할당과 불연속 할당

| **할당 방법** | **설명** | **방법** |
| ---- | ---- | ---- |
| **연속 할당** | 각각의 프로세스가 메모리의 연속적인 공간에 적재되도록 하는 것 | 고정 분할 방식, 가변 분할 방식 |
| **불연속 할당** | 하나의 프로세스가 메모리의 여러 영역에 분산되어 올라가는 방식 | Paging, Segmentation, Paged Segmentation |

## ⏩ 연속 할당 (Contiguous Allocation)
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F184ee058-2b80-44a3-8e6c-0771ae18274a%2Fimage.png?table=block&id=323d67b9-9e80-8062-a1eb-c39c7f2b5720&cache=v2" alt="image" width="1910" height="996" loading="lazy" />


📌 **고정 분할 방식**
- 물리적 메모리를 몇 개의 분할(partition)으로 나누어 놓는 방식
- 분할의 크기가 모두 동일하거나 서로 다른 방식이 존재
- 분할당 하나의 프로그램 적재
- 동시에 메모리에 올라갈 수 있는 프로그램의 수가 파티션 개수만큼으로 제한되며 파티션보다 큰 프로그램은 아예 실행할 수 없음.
- **내부 단편화** 발생(파티션이 비어도 프로그램이 커 들어가지 못하는 **외부 단편화**도 발생 가능)

📌 **가변 분할 방식**
- 프로그램이 실행될 때 프로그램의 크기만큼 할당
- 분할의 크기, 개수가 동적으로 변함
- 내부 단편화는 발생하지 않지만 프로그램들이 실행되고 종료되기를 반복하다보면 중간중간 빈 공간들이 생겨 **외부 단편화**가 발생
### **내부 단편화와 외부 단편화**
단편화는 **메모리 공간이 비효율적으로 쪼개져 낭비**되는 현상입니다.

| **내부 단편화(Internal Fragmentation)** | 프로세스가 요청한 양보다 더 큰 메모리가 할당되어 **할당된 공간 내부에 남게 된** 메모리 공간 |
| ---- | ---- |
| **외부 단편화(External Fragmentation** | 메모리 전체를 합치면 남은 공간이 충분하지만 빈 공간들이 작게 흩어져 있어 새로운 프로세스를 올릴 수 없는 상태 |

### 🆚 고정 분할과 가변 분할 비교

| 비교 항목 | 고정 분할 방식 (Fixed Partitioning) | 가변 분할 방식 (Variable Partitioning) |
| ---- | ---- | ---- |
| **분할 시점** | 시스템 부팅 시 미리 분할 | 프로그램 실행(적재) 시 동적 분할 |
| **분할 크기** | 고정됨 (모두 같거나 다를 수 있음)  | 프로그램 크기에 딱 맞게 가변적임  |
| **내부 단편화** | 발생함 (파티션보다 프로그램이 작을 때) | 발생하지 않음 (크기만큼만 주므로)  |
| **외부 단편화** | 발생할 수 있음 | **심하게 발생** (중간중간 구멍 뚫림) |

### ⚠️ 가변 분할 방식의 문제
가변 분할 방식은 프로세스 크기에 딱 맞춰 메모리를 할당하여 내부 단편화는 막을 수 있지만 프로그램들이 수없이 실행되고 종료되는 과정을 반복하면서 메모리 공간 곳곳에 구멍(Hole)들이 무수히 생겨나는 문제가 발생합니다.

⚠️ **가용 공간(Hole)의 발생**
`Hole`: 프로세스가 사용하다가 종료되어 반납한 빈 메모리 공간(가용 공간)
- 가변 분할 환경에서는 크기가 제각각인 Hole들이 물리적 메모리 여러 곳에 흩어져 있음
- 운영체제는 현재 사용 중인 '할당 공간'과 비어있는 '가용 공간(Hole)'를 지속적으로 추적하고 관리

⚠️ **동적 메모리 할당 문제(Dynamic Storage-Allocation Problem)**
새로운 프로세스가 실행을 위해 `n` 크기의 메모리를 요청했을 때 ***"수많은 Hole들 중에서 도대체 어느 빈 공간에 이 프로세스를 집어넣어야 가장 효율적일까?"*** 를 결정하는 문제입니다.
- `First-fit`: 메모리의 처음부터 탐색하다가 `n` 크기 이상인 Hole을 발견하면 그 즉시 할당
- `Best-fit`: 메모리에 있는 모든 Hole을 모두 확인하여 `n` 크기를 만족하면서도 가장 크기가 작은 **Hole**에 할당
- `Worst-fit`: 모든 Hole을 다 확인하여 가장 크기가 큰 Hole에 프로세스를 할당

💡 **압축 (Compaction)**
이런 외부 단편화 문제를 해결하는 방법으로 압축방법이 등장했습니다.
- 메모리 상에서 실행 중인 프로세스들을 한쪽으로, 흩어진 Hole들을 반대쪽으로 모아 하나의 거대한 빈 공간으로 만드는 기법
이는 실행 중인 프로세스들을 통째로 다른 메모리 위치로 옮겨야하는 과정에서 프로세스의 실행을 멈춰야 하고 모든 프로세스의 주소를 새롭게 다시 매핑해야 하므로 시스템에 엄청난 성능 저하를 일으켜 비용이 너무 많이 들어서 실시간으로 자주 수행하는 것은 매우 어렵습니다.
## 🚀 불연속 할당 (Noncontiguous allocation)
불연속 할당은 하나의 프로세스가 물리적 메모리의 연속된 공간에 덩어리째 올라가는 것이 아니라, 여러 조각으로 나뉘어 메모리의 다양한 위치에 분산되는 방식입니다.
대표적인 방법으로 `Paging` `Segmentation` `Paged Segmentation` 이 있습니다.
## 📂 페이징
<u>**Process의 가상 메모리**</u>를 <span class="text-blue">**동일한 사이즈의 page**</span> 단위로 나누어 메모리에 할당
- page 단위로 불연속적으로 저장

👾 **동작 방식**
- OS는 프로세스를 페이지 단위로 잘라 물리 메모리의 빈 프레임 아무 곳에나 불연속적으로 넣습니다
- 빈 공간의 크기가 모두 프레임 단위로 균일하므로 어떤 페이지든 들어갈 수 있습니다
> 
> **Page (페이지):** 프로세스의 논리적 메모리를 동일한 크기로 자른 조각.
> **Frame (프레임):** 물리적 메모리(RAM)를 페이지와 똑같은 크기로 자른 조각

💡 **페이징의 단편화 해결 결과**
- ✅ **외부 단편화 해결:** 프로세스가 통째로 들어갈 큰 빈 공간(Hole)을 찾을 필요가 없으므로 외부 단편화가 완벽히 사라집니다. 압축(Compaction) 작업도 필요 없습니다.
- ⚠️ **내부 단편화 발생 가능:** 프로그램의 크기가 항상 페이지 크기의 배수로 딱 떨어지지 않기 때문에 마지막 페이지에서 남는 공간(내부 단편화)이 발생할 수 있습니다.
### **Page Table**
프로세스가 메모리에 불연속적으로 흩어져 있기 때문에 기존 연속 할당 방식처럼 단일 `Base Register` 값에 논리 주소를 더하는 방식으로는 주소 변환을 할 수 없습니다.
따라서 논리적 주소를 구성하는 각각의 페이지가 실제 어떤 프레임에 적재되어 있는지 1:1로 매핑해 둔 `페이지 테이블 (Page Table)`이 필요합니다.

프로세스마다 큰 크기의 페이지 테이블이 필요합니다 (32bit 환경에서 약 100만개의 엔트리)
- 이렇게 큰 테이블을 CPU 내부 레지스터에 모두 담을 수 없고 하드디스크에 둘 수도 없어 페이지 테이블은 메인 메모리에 저장됩니다.

**페이징 환경에서의 MMU 레지스터 변화**
연속 할당 시절에 쓰던 2개의 레지스터(Base / Limit Register)는 페이징 환경에서 메인 메모리에 있는 **페이지 테이블을 찾고 보호하기 위한 용도**로 그 역할이 바뀝니다.

| **페이징 레지스터** | **역할** |
| ---- | ---- |
| **PTBR (Page-Table Base Register)** | 메인 메모리에 저장된 **해당 프로세스의 페이지 테이블 시작 위치**를 가리킵니다. |
| **PTLR (Page-Table Length Register)** | 프로세스가 사용하는 페이지의 개수. **페이지 테이블의 크기**를 보관하여 잘못된 범위의 접근을 막습니다. |

- CPU는 PTBR(Page Table Base Register)을 통해 해당 프로세스의 페이지 테이블 시작 주소를 참조

⚠️ **문제 - 2번의 메모리 접근**
페이지 테이블이 메인 메모리에 위치하게 되면서 CPU가 어떤 데이터에 접근하려고 할 때마다** 2번의 메모리 접근**이 발생하게 됩니다.
1. `PTBR`을 통해 메인 메모리에 있는 `Page Table`에 먼저 접근하여 물리적 프레임 번호를 알아냅니다.
2. 변환된 물리적 주소를 가지고 `실제 데이터`가 있는 메인 메모리에 다시 접근
이로 인해 메모리 접근 시간이 2배로 늘어나는 막대한 오버헤드가 발생하게 되며 이를 해결하기 위해 등장한 것이 바로 주소 변환 전용 하드웨어 캐시인 **TLB**입니다
### TLB (Translation Lookaside Buffer)
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2Fa3c85027-ed76-4617-8abe-70b84e15a8ec%2Fimage.png?table=block&id=323d67b9-9e80-806a-aaa1-ced1f34bd3f4&cache=v2" alt="image" width="1534" height="964" loading="lazy" />

메모리를 접근하기 위해서 매번 2번의 메모리에 접근하는 건 비용이 큰 작업이기 때문에 속도 향상을 위해 별로의 하드웨어를 사용합니다. (`TLB`)
- 주소 변환을 위한 캐시 메모리

TLB는 page table에서 빈번히 참조되는 <span class="text-blue">**일부 엔트리**</span>를 캐시하고 있습니다.
따라서 CPU가 주소를 요청하면 메인 메모리의 페이지 테이블로 가기 전에 먼저 TLB를 뒤져봅니다. (TLB Hit 시 곧바로 물리 주소 획득)

🧩 **특징**
`TLB`는 페이지 테이블의 일부만 가지고 있기 때문에 위에서부터 인덱스로 바로 접근할 수 없으며 `[논리 페이지 번호 p : 물리 프레임 번호 f]` 쌍으로 데이터를 저장해야 합니다.

**병렬 탐색 (Parallel Search)**
- 원하는 페이지 번호를 찾기 위해 TLB 전체를 순차적으로 검색하면 너무 느립니다. 
- 이를 해결하기 위해 모든 엔트리를 한 번에 동시에 검색할 수 있는 **연관 레지스터(Associative Register)** 하드웨어로 구현되어 있습니다.

> 
> ❤️‍🩹 **TLB와 문맥 교환(Context Switch)**
> 페이지 테이블은 프로세스마다 독립적으로 존재합니다. 따라서 CPU를 차지하는 프로세스가 바뀌면 이전 프로세스의 주소 변환 정보가 담긴 T**LB 안의 내용도 전부 비워(Flush)줘**야 합니다.
> 이런 과정은 context switch의 오버헤드를 가중시키는 요인 중 하나가 됩니다.


### MultiLevel page table
🤔 **왜 페이지 테이블을 여러 단계로 나누어서 처리??**
현대 컴퓨터는 address space가 큰 프로그램을 지원해줍니다.
- `32bit` 운영체제 기준으로 하나의 프로세스는 `2^32`, `4GB`의 가상 주소 공간을 가지게 됩니다.
- 페이지 크기가 `4KB`이라면 `4GB`공간을 관리하기 위해서는 프로세스당 약 100만개의 페이지 테이블 엔트리가 필요
- 각 엔트리의 크기가 `4Byte`라고 하면 **프로세스 1개당 4MB의 페이지 테이블 공간**이 필요

💣 **문제점**
이렇게 4GB 중 실제로 사용하는 메모리 공간은 일부분인데도 논리적 주소의 연속성을 유지하기 위해서 사용하지 않는 나머지 페이지 테이블 엔트리들도 모두 메모리에 미리 만들어두어야 합니다.
여러 프로세스가 동시에 실행하게 된다면 메인 메모리는 사용하지도 않는 페이지 테이블로 가득 차버리는 낭비가 발생하게 됩니다.

💡 **Two-Level Page Table**
안쪽 페이지 테이블과 바깥쪽 페이지 테이블로 나누어 관리
- 페이지 테이블을 한 번 더 페이징
CPU가 논리적인 주소를 주게 되면 페이지 테이블을 2단계를 거쳐서 주소 변환을 한 후 메모리 접근

<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2Fdcec5c17-4e73-4b87-88b8-aff5d2f15e33%2Fimage.png?table=block&id=323d67b9-9e80-8024-9a10-e2c973e3743c&cache=v2" alt="image" width="822" height="434" loading="lazy" />

1️⃣ **바깥쪽 페이지 테이블**
- 전체 논리적 메모리 크기에 맞춰 만들어집니다. 
- 여기에는 물리적 프레임 주소가 아니라 안쪽 페이지 테이블이 어디 있는지가 저장
2️⃣ **안쪽 페이지 테이블**
- 실제 물리적 프레임 번호가 저장

이전에는 안 쓰는 주소까지 모두 다 만들어 두었다면 프로세스가 실제로 사용하는 메모리 영역에 대해서만 안쪽 페이지 테이블을 생성합니다.
- 프로세스가 사용하지 않는 영역에 대해서는 바깥 페이지 테이블에서 연결을 Null로 끊고 안쪽 페이지 테이블을 만들지 않습니다. 

### Valid / InValid Bit 
페이지 테이블의 각 엔트리에는 주소 변환 정보 외에도 이 페이지의 상태를 알려주는 비트가 있습니다.

| **비트** | **설명** |  |
| ---- | ---- | ---- |
| **Valid** | 해당 페이지가 실제 물리적 메모리(Frame)에 정상적으로 올라와 있음을 의미 |  |
| **Invalid** | 프로세스가 해당 주소 영역을 아예 사용하지 않는 경우 | 당장 필요하지 않아서 **메모리에서 쫓겨나 디스크의 Swap 영역에 있는 경우** |

## 🗂️ 세그먼테이션 (Segmentation)
프로그램의 주소 공간을 같은 크기로 자르는 것이 아닌 **의미있는 단위**인 `세그멘트` 단위로 자르는 것.
- code 세그먼트, data 세그먼트, stack 세그먼트
- 각 세그먼트는 그 역할에 따라 크기가 모두 다릅니다.
- 이렇게 나누어진 각각의 세그먼트들은 물리적 메모리의 서로 다른 위치에 불연속적으로 적재

**⚠️ 세그먼테이션의 문제 (외부 단편화)**
세그먼트들은 그 크기가 균일하지 않기 때문에 메모리에 넣고 빼기를 반복하다 보면 또다시 크기가 제각각인 빈 공간(Hole)들이 생겨납니다. 
즉, 가변 분할 방식에서의 **Dynamic Storage-Allocation 문제**와 **외부 단편화(External Fragmentation)** 문제가 발생합니다.

### 세그먼트 테이블
페이지 테이블처럼 세그먼테이션 환경에서는 주소 변환을 위해 `세그먼트 테이블(Segment Table)`을 사용
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2Fb1d51daa-92a4-4dca-9179-34ef7c8a5926%2Fimage.png?table=block&id=323d67b9-9e80-8078-bd87-e07aec8415ed&cache=v2" alt="image" width="1214" height="662" loading="lazy" />

CPU가 생성하는 논리 주소는 두 부분으로 나뉩니다.
- `[Segment 번호(s), Offset(d)]`

🤖 **세그먼트 테이블의 두 가지 정보**
페이지 테이블의 엔트리가 `프레임 번호`만 가지고 있는 것과 달리 **세그먼트 테이블의 각 엔트리는 세그먼트마다 **<span class="text-blue">**크기**</span>**가 다르기 때문에 2가지 정보**를 반드시 함께 저장해야 합니다.
1. **Base:** 해당 세그먼트가 물리적 메모리에서 시작하는 주소
2. **Limit:** 해당 세그먼트의 크기(길이)

👊 **변환 과정**
1️⃣ CPU가 논리 주소`[s, d]`를 주면 MMU는 먼저 `Segment-Table Length Register (STLR)`를 확인합니다. 
- 요청한 **세그먼트 번호(s)**가 프로그램이 사용하는 **전체 세그먼트 개수보다 크다면 **즉시 `Trap`(오류)을 발생

2️⃣ 안전한 세그먼트라면 `Segment-Table Base Register (STBR)`를 통해 메모리에 있는 세그먼트 테이블로 찾아갑니다. 
- 요청한 **Offset(d)** 값이 테이블에 적혀있는 해당 세그먼트의 **Limit**보다 큰지 확인
- 크다면 자신의 세그먼트 영역을 벗어난 접근이므로 `Segmentation Fault(Trap)`를 발생

3️⃣ 테이블에 적힌 **Base(시작 주소)** 값에 **Offset(d)** 값을 더하여 최종 물리적 메모리 주소를 계산
## 🚀 정리
CPU가 논리적인 주소를 가지고 있으면 물리적인 메모리 주소로 변환하는 주소 변환과정에 대해서 학습을 해보았습니다.
하지만 이 주소변환 과정에서 운영체제가 하는 일은 없습니다. (모두 `MMU` - 하드웨어가 처리)
> 어떤 프로세스가 CPU를 가지고 있으면서 메모리 접근을 할 때 운영체제의 도움을 받을까요??

- 전혀 받지 않습니다.
만약 주소 변환을 할 때마다 운영체제가 개입을 해야 한다고 하면 CPU가 프로세스에서 운영체제로 넘어간 후 주소변환 이후에 다시 프로세스에게 돌아가야 합니다. 따라서 주소변환은 하드웨어적으로 처리가 됩니다.
이제 다음으로는 가상 메모리(Virtual Memory)를 학습해보면서 운영체제가 하는 역할에 대해서 학습해보겠습니다.
