## ⭐ CPU Scheduler
> **`Ready`** 상태의 프로세스 중**(ready queue)**에서 실행할 프로세스를 선택해 CPU를 할당


CPU는 하나인데 실행할 프로세스는 많을 때 `다음엔 누구한테 CPU를 주지?` 라는 것을 결정을 해주어야 합니다. <span class="text-red">**CPU Scheduler**</span>는 Ready 상태의 프로세스들이 모여있는 Ready Queue에서 다음에 실행할 프로세스를 선택해 CPU를 할당합니다. 
- `단기 스케줄러(Short-Term Scheduler)`
### Dispatcher
**CPU 제어권을 CPU scheduler에 의해 선택된 프로세스에게 넘기는 역할**
- `context switch`

현재 실행 중이던 프로세스의 상태를 `PCB`에 저장하고 새롭게 선택된 프로세스의 상태를 복구하는 문맥 교환(Context Switch)이 일어납니다. 
디스패처가 하나의 프로세스를 정지시키고 다른 프로세스를 실행하기까지 걸리는 시간을 디스패치 지연(Dispatch Latency)이라고 부르며 이 시간을 최소화하는 것이 전체 시스템의 응답 속도에 매우 중요합니다.

### CPU 스케줄링이 발생하는 상황
운영체제는 프로세스의 상태 변화가 일어날 때 스케줄링을 트리거합니다.
- 실행 상태 → 대기 상태: I/O 요청이 발생한 경우 
- 실행 상태 → 준비 상태: 타이머 인터럽트가 발생하여 할당된 시간을 다 쓴 경우입니다.
- 대기 상태에서 준비 상태로 전환될 때 I/O 작업이 완료되어 다시 CPU를 받을 준비가 된 경우입니다.
- 프로세스가 정상적으로 혹은 비정상적으로 종료될 때 발생합니다.

### 🎯 CPU 스케줄링의 목적
> 💡 모든 프로세스가 공평하고 효율적으로 작업하도록 도와주는 것


| **목적** | **설명** |
| ---- | ---- |
| **공평성** | 모든 프로세스가 공평하게 CPU를 할당받아야 합니다. |
| **효율성** | CPU가 유휴 시간(Idle time) 없이 계속 일하도록 하여 활용률을 극대화해야 합니다. |
| **안정성** | 중요 프로세스나 시스템 프로세스가 먼저 처리되도록 우선순위를 부여해 시스템을 안정적으로 유지해야 합니다. |
| **확장성** | 프로세스 수가 급격히 증가하더라도 안정적으로 동작해야 합니다. |
| **반응 시간 보장** | 사용자가 입력을 했을 때 적절한 시간 내에 화면에 응답을 보여주어야 합니다. |
| **무한 연기 방지** | 특정 프로세스가 영원히 CPU를 할당받지 못하는 기아 현상(Starvation)을 막아야 합니다. |

### 👀 선점형(Preemptive) / 비선점형(Non-preemptive)
<u>**비선점형**</u>은 CPU를 한 번 할당받았다면 다 쓰고 자진 반납하거나 I/O 작업을 만나 대기 상태로 가기 전까지 CPU 할당을 절대적으로 보장해주는 방법입니다. 
- context switch 오버헤드가 적지만 처리 시간이 긴 프로세스가 CPU를 독점하면 다른 프로세스들이 무작정 기다려야 하는 단점이 있습니다.

<u>**선점형**</u>은 하나의 프로세스가 CPU를 할당받아 실행 중이더라도 운영체제가 강제로 제어권을 빼앗아 다른 우선순위가 높은 프로세스에게 할당할 수 있는 방법입니다.
- 응답성이 좋고 시분할 시스템에 적합하지만 잦은 문맥 교환으로 인한 오버헤드가 발생할 수 있습니다.

### 🔍 스케줄링 성능 척도(Criteria)

| **Criteria** | **설명** |
| ---- | ---- |
| **CPU utilization (이용률)** | 전체 시간중 CPU가 일한 시간의 비율 |
| **Thoughput (처리량)** | 주어진 시간동안 몇 개의 일을 만료했는가 |
| **Turnaround time (소요 시간, 반환 시간)** | 프로세스가 생성되어 종료될 때까지 걸린 총 시간 |
| **Watiting time (대기 시간)** | 프로세스가 ready queue에서 CPU를 기다린 시간의 총 합 |
| **Response Time (응답 시간)** | 프로세스가 처음으로 CPU를 할당받기까지 걸린 시간 |

## 🤖 스케줄링 알고리즘
### **FCFS (First Come First Served) **
`비선점형`
Ready Queue에 먼저 들어온 프로세스가 먼저 CPU를 받습니다(FIFO)
- CPU burst가 긴 작업이 앞에 오면 뒤의 짧은 작업들이 길게 대기해 평균 대기 시간이 증가할 수 있습니다

> ⚠️ <span class="text-red">**Convoy**</span>** 현상**
> 몇 개의 시간이 오래 걸리는 프로세스로 인해 전체 OS가 느려지는 현상


### SJF (Shortest Job First)
`비선점형` `선점형`
- CPU burst가 가장 짧은 프로세스에 CPU를 먼저 할당
비선점형은 **CPU burst time(또는 예측값)**이 가장 짧은 프로세스부터 실행합니다.
선점형은 보통** SRTF(Shortest Remaining Time First)**로 **남은 시간이 더 짧은 작업이 오면 선점**하는 방식입니다.
긴 작업이 계속 뒤로 밀려 실행을 못할 수 있는 **Starvation 현상이 발생할 수 있습니다.**

### Priority Scheduling
`비선점형` `선점형`
- 우선순위가 제일 높은 프로세스에게 CPU를 할당
낮은 우선순위의 프로세스는 영원히 실행되지 않을 수 있는 **Starvation 현상**이 발생할 수 있습니다.

💡 **Starvation 현상을 막기 위한 방법**
- **Aging**: 오래 기다린 프로세스의 우선순위를 점진적으로 올려서 결국 실행되도록 합니다.

### Round Robin
`선점형`
각 프로세스에 **할당시간(타임 퀀텀, time quantum)**만큼 CPU를 주고 시간이 끝나면 강제로 준비 큐 뒤로 보냅니다.
💡 **장점**: 모든 프로세스가 기아 현상(Starvation) 없이 공평하게 자원을 배분받으며, 응답성이 뛰어납니다.
⚠️ 문제: 퀀텀이 너무 작으면 `content switch` 과정이 잦아져 오버헤드가 커지고 너무 크면 `FCFS`처럼 되어 응답성이 떨어지게 됩니다.

### Multilevel Queue(다단계 큐) 와 **MLFQ**
> 두 가지 모두 **Ready queue**를 여러 개로 나누지만 `프로세스가 큐 사이를 이동할 수 있는가`가 핵심 차이


1️⃣ **Multilevel Queue**
Ready queue를 여러 개로 **분할**해 각 큐에 서로 다른 스케줄링 알고리즘을 적용합니다.
- 프로세스가 큐 간에 이동할 수 없으며 고정된 우선순위를 가집니다.
- 큐 간 우선순위 기반 스케줄링으로 시스템 효율성이 높습니다.

⚠️ 낮은 우선순위 큐의 프로세스가 실행되지 않는 **Starvation** **현상**이 일어날 수 있습니다.

2️⃣ **Multilevel Feedback Queue (MLFQ)**
MLQ의 단점을 보완하여 프로세스의 CPU 사용 행동에 따라 큐 사이를 이동할 수 있게 만든 스케줄링방식입니다.
- 새 프로세스는 최상위 우선순위 큐에서 시작
- CPU를 너무 오래 사용하면 낮은 우선순위 큐로 이동
- `우선순위(A) > 우선순위(B)` 일 경우 A가 실행, B는 대기
- `우선순위(A) = 우선순위(B)` 일 경우 **Round Robin **방식으로 실행
- I/O 위주 작업은 높은 우선순위를 유지
- 특정 시간마다 모든 프로세스를 최상위 큐로 옮기는 **우선순위 상향 조정(Priority Boost)**을 수행(**Starvation** 방지)
