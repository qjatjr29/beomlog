<span class="text-brown">**프로세스 관리**</span>는 운영체제의 핵심 역할입니다.
프로그램이 실행되어 프로세스가 생성된 이후 작업을 마칠때 까지 어떤 상태를 거치고 각 상태에서 고려할 사항이 무엇인지, OS가 프로레스를 어떻게 관리하는지(PCB)를 살펴보고자 합니다.
또한 스레드와 프로세스의 역할을 비교하여 멀티프로세스와 멀티스레드의 장단점과 트레이드오프를 이해하고자 합니다.

## 💻 프로그램, 프로세스, 스레드
<span class="text-blue">**프로그램**</span>은 하드 디스크같은 저장장치에 저장되어 있는 '정적인(수동적인) 파일'입니다. 
- 예: `.exe`, `.app`

<span class="text-red">**프로세스**</span>는 프로그램을 실행되어 메모리(RAM)에 올라와 OS로 부터 CPU를 할당받을 수 있는 <u>**동적인 작업 단위**</u>입니다. (실행중인 프로그램)
> 🧑‍🍳 **프로그램과 프로세스에 대한 비유 (레시피-요리)**
> - **프로그램**은 요리사가 어떤 재료를 가지고 어떻게 조리할 지 절차를 적어 놓은 것.
> - 레시피가 실제 요리로 탄생하기 위해서는 주방에서 레시피에 따라 요리를 하는 과정 - **프로세스**


<span class="text-red">**스레드**</span>는 하나의 프로세스 안에서 실제로 작업을 수행하는 <u>**실행 흐름의 최소 단위**</u>입니다. 
> 🧑‍🍳 **프로세스와 스레드 비유**
> - **프로세스**는 요리 작업 전체와 같고 **스레드**는 요리를 완성하기 위해 수행하는 각각의 조리과정


### 🤖 프로세스 Context(문맥)
**프로세스를 실행하기 위해 필요한 모든 상태 정보.**
즉, CPU가 어떤 프로세스를 실행하다가 멈췄다가 다시 실행하기 위해 필요한 정보 전체입니다.

## 📦 프로세스의 메모리 구조
운영체제는 프로세스가 작업할 수 있도록 <u>**독립된 가상 메모리 공간**</u>을 할당합니다. 
- 이 공간은 크게 4가지 영역으로 나뉩니다.

<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F72b7e6e3-5dfe-4427-94cd-d982cd66c73a%2Fimage.png?table=block&id=31bd67b9-9e80-80c7-8c38-edab950fe6ed&cache=v2" alt="image" width="720" height="319" loading="lazy" />



| 영역 | 저장되는 데이터 | 설명 |
| ---- | ---- | ---- |
| **Code (Text)** | 사용자가 작성한 **프로그램의 소스 코드**(기계어) | 프로그래머가 작성한 소스 코드가 기계어 명령어로 번역되어 저장되는 공간 |
| **Data ** | **전역 변수(Global)**, **정적 변수(Static)** | 프로그램이 시작될 때 할당되고 종료될 때 해제되는 전역 변수와 정적 변수가 저장되는 공간 |
| **Heap** | 런타임에 **동적으로 할당**되는 데이터 | 런타임에 사용자가 **동적으로 메모리를 할당**할 때 사용하는 공간 |
| **Stack** | **지역 변수(Local)**, **매개 변수**, **함수 호출(Return) 주소** | 함수 호출과 관련된 지역 변수, 매개 변수, 리턴 주소 등이 임시로 저장되는 공간 |

### Code (Text) 영역
프로그래머가 작성한 소스 코드가 기계어 명령어로 번역되어 저장되는 공간
CPU가 이 영역의 명령어를 하나씩 가져와 실행(Program Counter 활용)하고 프로그램이 실행되는 동안 내용이 변경되면 안 되므로 `읽기 전용(Read-Only)`으로 할당

### Data 영역
Data영역은 변수의 초기화 여부에 따라 두 가지 영역으로 다시 나뉘게 됩니다.

1️⃣ **GVAR (초기값이 있는 변수)**
- **초기화된** 전역변수, 정적 변수, 상수 등이 저장되는 영역

2️⃣ **BSS (초기값이 없는 변수)**
- **초기화되지 않았거나 0으로 초기화**된 전역 변수, 정적 변수가 저장되는 영역

> 💡 **이렇게 초기화되지 않은 데이터(BSS)까지 실제 파일에 저장하면 **
> **프로그램의 크기가 불필요하게 커져 값이 0인 변수들은 BSS 영역으로 묶어 디스크 공간을 절약하고 프로그램 실행 시에만 메모리에 할당하여 효율을 높이고자 하는 것!!**

### **Heap 영역**
**런타임**에 사용자가 **동적으로 메모리를 할당**할 때 사용하는 공간
- 사용 후에는 반드시 해제(메모리 누수 방지)해야 합니다.

### Stack 영역
함수 호출과 관련된 지역 변수, 매개 변수, 리턴 주소 등이 임시로 저장되는 공간
- 함수가 호출될 때 할당되고 끝나면 반환됩니다. 
- **컴파일 타임**에 크기가 결정되며 메모리의 높은 주소에서 낮은 주소 방향으로 할당

## 🔄** 프로세스의 생명 주기 (상태 변화)**
프로세스는 상태가 변경되며 수행이 된다.
<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F140e6a7a-b8c5-4541-86d5-c81d6d7dc495%2Fimage.png?table=block&id=31bd67b9-9e80-80bb-875a-f72c2dd8df6d&cache=v2" alt="image" width="1104" height="564" loading="lazy" />


| **상태** | **설명** |
| ---- | ---- |
| **Running** | CPU를 잡고 instruction을 수행중인 상태 |
| **Ready** | CPU를 기다리는 상태 (준비가 된 상태) |
| **Blocked (wait, sleep)** | CPU를 주어도 당장 instruction을 수행할 수 없는 상태 (예시: 요청한 I/O가 완료되지 않아 기다리는 상태) |

> 📖 **Dispatch(디스패치)**
> - Ready 상태의 프로세스를 선택하여 CPU를 할당하고 실행 상태로 전환하는 작업
> - 스케줄러가 `누가 실행할지` 결정하면 → 디스패처가 실제로 CPU를 넘겨 실행하게 하는 단계

## 🧵 스레드
하나의 프로세스 안에서 CPU가 작업을 수행하는 <u>**실행 흐름의 최소 단위**</u>입니다.

각 스레드는 보통 다음 요소들을 가집니다.
- **Program Counter (PC)**: 이 스레드가 다음에 실행할 명령어 위치
- **레지스터 집합**: 연산 중간값, 스택 포인터 등
- **스택(Stack) 영역**: 함수 호출 프레임, 지역 변수, 매개 변수가 쌓이는 독립적인 스택

동일한 프로세스에 속한 스레드들은 다음을 공유합니다.
- Code 영역
- Data 영역
- Heap 영역
- 기타 OS 자원

즉, **각 스레드는 스택만 따로 가지고 나머지 대부분은 공유**합니다.
이 때문에 스레드 간 통신은 빠르지만 <span class="text-red">동기화 문제(레이스 컨디션, 데드락 등)</span>에 주의해야 합니다.

### 스레드와 성능
- **동시성 관점:** 하나의 코어에서도 여러 스레드를 빠르게 전환(컨텍스트 스위칭)하여 `동시에 실행되는 것처럼` 보이게 할 수 있어 응답성을 높일 수 있습니다.
- **병렬성 관점:** 멀티코어 환경에서는 서로 다른 스레드를 서로 다른 코어에서 실제로 동시에 실행시켜 CPU 바운드 작업 처리량을 높일 수 있습니다.

### Thread의 장점 👍
1️⃣ **Responsiveness (응답성)**
- 멀티스레드 환경에서 **하나의 스레드가 Blocked(대기) 상태가 되어도 다른 스레드들은 계속 실행**

2️⃣ **Resource Sharing (자원 공유)**
- 같은 프로세스에 속한 스레드들은 **코드, 데이터, 힙, 열린 파일, 소켓 등 프로세스의 자원을 공유**하여 효율적으로 사용

3️⃣ **Economy (경제성)**
- **프로세스 생성 및 스위칭보다 스레드 생성 및 스위칭의 오버헤드가 훨씬 작습니다.**

## 📂 O**S의 프로세스·스레드 관리**
### 📦 PCB (Process Control Block)
> 운영체제가 **각 프로세스**를 관리하기 위해 커널 내부에 유지하는 **프로세스 정보 모음 (자료구조)**


### 🤔 **왜 필요한가?!?**
멀티태스킹 과정에서 여러 프로세스가 번갈아 실행되면 어떤 프로세스가 어디까지 실행했는지(다음에 다시 시작할 위치)와 어떤 자원을 사용하고 있는지를 OS가 기억을 해야합니다.
따라서, 이런 정보가 없다면 <u>타이머 인터럽트나, I/O 작업</u> 이후에 이어서 `이이서 실행` 하는 것이 불가능하고 스케줄러도 올바른 선택을 할 수 없게 됩니다.

<img src="https://www.notion.so/image/https%3A%2F%2Fprod-files-secure.s3.us-west-2.amazonaws.com%2Fce63075d-cee3-44a7-8d31-efcf5784b0e7%2F3917684d-20c1-4dbc-bea3-bda51fa33d57%2Fimage.png?table=block&id=31bd67b9-9e80-8071-a691-e71655df3188&cache=v2" alt="image" width="688" height="672" loading="lazy" />



| 분류 | 항목 | 설명 |
| ---- | ---- | ---- |
| **1️⃣ OS 관리/스케줄링 정보** | **Process State** | 현재 프로세스가 Ready, Running, Waiting 중 **어떤 상태**인지 |
|  | **Process ID (PID)** | 프로세스를 식별하기 위한 **고유 번호** |
|  | **Scheduling info / Priority** | 프로세스의 **우선순위**.  |
| **2️⃣ CPU 수행 관련 하드웨어 값 **⭐ | **Program Counter (PC)** | 코드의 몇 번째 줄까지 실행했는지(**다음 실행할 명령어 주소**)를 기억 |
|  | **Registers** | 실행 중이던 계산 결과값(**임시 데이터**)들이 담긴 레지스터 값들 |
| **3️⃣ 메모리 관련 값** | **Memory Management info** | 프로세스의 Code, Data, Stack 영역이 물리적 RAM의 어느 주소에 위치하는지 |
| **4️⃣ 파일 / 자원 관련 값** | **Open File Descriptors** | 어떤 파일들을 열어서 쓰고 있는지 어떤 I/O 장치를 점유하고 있는지 추적하여 자원을 관리하고 종료 시 회수 |

### 📦 TCB (Thread Control Block)
> 운영체제가 각 스레드를 관리하기 위해 유지하는 **스레드 정보 모음(자료구조)**입니다

**멀티스레드 프로세스**에서는 아래처럼 이해하면 됩니다.
- **하나의** `PCB` 아래에 **여러 개의 **`TCB`가 매달려 있는 구조


| 항목 | 설명 |
| ---- | ---- |
| **TID (Thread ID)** | 스레드를 구분하는 식별자 |
| **Thread State** | Running/Ready/Blocked 등 스레드 상태 |
| **레지스터 저장 영역** | 레지스터 값들 |
| **스택 포인터** | 유저 스택, 커널 스택 위치 |
| **스케줄링 정보** | 스레드 우선순위, 타임 슬라이스 등 |
| **PCB 포인터** | 어느 프로세스 PCB에 속한 스레드인지 가리키는 포인터 |

## 🚀 Context Switch
> **CPU를 한 프로세스에서 다른 프로세스로 넘겨주는 과정**


CPU가 다른 프로세스에게 넘어갈 때 다음과 같이 동작합니다.
1. 현재 실행중이던 프로세스/스레드의 상태(문맥)를 해당 프로세스의 `PCB/TCB`에 저장
2. 다음에 실행할 프로세스/스레드 선택 (스케줄러)
3. 새로 할당된 프로세스/스레드의 상태를 `PCB/TCB`에서 가져와 다시 로드

🆚 **프로세스 전환 vs 스레드 전환**
**프로세스 간 전환**

**같은 프로세스 내 스레드 간 전환**

## ⭐ **멀티 프로세스 / 멀티 스레드**

| 관점 | 멀티 프로세스 | 멀티 스레드 |
| ---- | ---- | ---- |
| **주소공간** | 프로세스마다 독립 | 하나의 프로세스 안에서 공유 |
| **메모리 사용량** | 큼 | 상대적으로 적음 |
| **통신 비용** | IPC 필요, 무거움 | 공유 메모리 활용, 매우 빠름 |
| **안정성** | 한 프로세스 장애 시 다른 프로세스 영향 적음 | 한 스레드 장애가 프로세스 전체에 영향 가능 |
| **컨텍스트 스위칭 비용** | 큼 (페이지 테이블, 캐시 영향) | 작음 (레지스터/스택 위주) |

### **멀티 프로세스 (Multi Process)**
독립된 여러 프로세스를 만들어 동시에(또는 동시성으로) 실행하는 방식
- 각 프로세스는 **서로 완전히 분리된 주소공간**

**장점**
- 프로세스 간 메모리 격리가 강해, **한 프로세스가 죽어도 다른 프로세스에 영향이 적음** → 안정성 높음

**단점**
- 프로세스마다 주소공간과 자원을 따로 가져야 해서 **메모리 사용량이 큼**
- 프로세스 간 통신(IPC: 파이프, 소켓, 공유 메모리 등)이 상대적으로 **무겁고 복잡**
- 컨텍스트 스위칭 시 주소공간 전환으로 인한 캐시/TLB 손실 등 **오버헤드가 큼**

### **멀티 스레드 (Multi Thread)**
하나의 프로세스 안에서 여러 스레드가 **스택과 레지스터만 분리**해서 실행하는 방식

**장점**
- 프로세스를 여러 개 띄우는 것보다 **메모리 사용량이 훨씬 적음**
- 같은 주소공간을 공유하므로 **스레드 간 통신이 매우 빠름** (전역변수/힙 공유)
- 같은 프로세스 내에서의 스레드 컨텍스트 스위칭은 **프로세스 스위칭보다 가벼움**

**단점**
- 메모리를 공유하기 때문에 <span class="text-red">동기화 문제(레이스 컨디션, 데드락 등)</span>에 매우 취약
- 한 스레드가 메모리를 망가뜨리면(예: 잘못된 포인터) **프로세스 전체가 죽을 수 있음**
- 디버깅이 어렵고 잘못 설계하면 오히려 병목과 성능 저하를 초래할 수 있음

## 🤔 궁금한 내용
### 🙋 스레드는 왜 스레드라는 이름을 갖게 됐을까?
`Thread`의 사전적 의미는 우리가 옷을 꿰맬 때 쓰는 <span class="text-red">**실**</span>입니다.
컴퓨터 공학에서 해당 단어를 선택한 이유는 <u> </u><u>**프로그램이 실행되는 연속적인 흐름**</u>을 이 <span class="text-red">**실**</span>로 빗대어 표현한 것이기 때문입니다.
- 프로세스 내에서 독립적인 실행 단위를 꿰매어지는 <span class="text-red">**실**</span>에 비유하여 **실행의 실 (Thread of execution)**이라 부름.

### 🙋 왜 스레드는 독립적인 stack 영역은 가지면서 프로세스 메모리의 다른 영역들은 공유할까?
스레드마다 **독립적인 함수 호출 흐름**을 가지려면 서로의 호출 기록/지역 변수와 섞이지 않도록 **스택은 반드시 분리**되어야 합니다.

> 🤔 **스택을 공유하게 된다면??**
> - **A 스레드가 함수 리턴 주소를 덮어써서 B 스레드가 엉뚱한 곳으로 리턴**
> **➡ 독립된 실행 흐름 **❌


또, 같은 프로세스 안의 스레드들은 **한 작업을 협력해서 빠르게 처리**하는 게 목적!!
따라서 같은 코드를 실행하고 같은 전역/정적 변수를 사용해 같은 힙에 할당된 객체를 다뤄 스레드간 데이터 교환과정 없이 빠르고 메모리 절약!!
